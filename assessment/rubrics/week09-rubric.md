# Week 9: 애자일/스크럼 - 평가 루브릭

> **평가 대상**: 사용자 스토리 작성 및 스프린트 계획 보고서  
> **제출 형식**: A4 2-3페이지 + 제품 백로그 (별도)  
> **제출 기한**: Week 9 종료 시점

---

## 📋 실습 과제 설명

**과제명**: 애자일 프로젝트 백로그 작성 및 첫 스프린트 계획

**과제 내용**:
선정한 프로젝트를 애자일 방식으로 접근하여 사용자 스토리를 작성하고, 스토리 포인트를 부여한 후, 2주 스프린트 계획을 수립하며, 스프린트 회고 항목을 제시하는 보고서를 작성하세요.

**요구사항**:
1. Epic 2개 이상 정의
2. 사용자 스토리 10개 이상 작성 (INVEST 기준 충족)
3. 각 스토리에 스토리 포인트 부여 (피보나치 수열)
4. 2주 스프린트 계획 수립 (벨로시티 가정)
5. 완료 정의(DoD) 작성
6. 스프린트 회고 템플릿 제시 (잘한 점, 개선할 점, 액션 아이템)

---

## 📊 평가 항목 (총 100점)

### 1. 사용자 스토리 품질 (30점)
- 사용자 스토리 형식의 정확성
- INVEST 기준 충족 여부
- 스토리의 현실성 및 가치 명확성

### 2. 스프린트 계획 타당성 (25점)
- 스토리 포인트 부여의 적절성
- 스프린트 목표의 명확성
- 벨로시티 고려한 스프린트 백로그 구성

### 3. 애자일 원칙 이해도 (25점)
- Epic과 Story의 관계 이해
- 완료 정의(DoD)의 구체성
- 스프린트 회고 항목의 적절성

### 4. 보고서 완성도 (20점)
- 제품 백로그 시각화
- 논리적 구성 및 가독성
- 애자일/스크럼 용어의 정확한 사용

## 💡 피드백 예시

### 탁월한 답변 예시 (90-100점)
**강점**:
- Epic "사용자 인증 기능"에 Story 3개(로그인, 회원가입, 비밀번호 재설정)를 논리적으로 배치했습니다.
- 사용자 스토리 "As a 쇼핑몰 고객, I want 장바구니에 상품을 담고 싶다, So that 여러 상품을 한 번에 결제할 수 있다"가 INVEST 기준을 완벽히 충족합니다.
- 스토리 포인트를 피보나치로 일관되게 부여하고, "로그인(3점)보다 OAuth 통합(8점)이 더 복잡"이라는 상대적 크기 비교가 명확합니다.
- 스프린트 목표 "사용자가 이메일과 소셜 로그인으로 로그인할 수 있다"가 구체적이고 측정 가능합니다.
- DoD에 "단위 테스트 커버리지 80% 이상, 코드 리뷰 2명 승인, API 문서 업데이트, Staging 배포 완료"를 포함하여 실무적입니다.

**추가 학습 제안**:
- 벨로시티 트렌드 차트를 추가하여 3-5 스프린트 평균 벨로시티를 시각화해보세요.
- 칸반 보드를 설계하고 WIP 제한을 설정해보세요.
- Sprint Retrospective 기법(Start-Stop-Continue, 4Ls)을 학습해보세요.

---

### 우수한 답변 예시 (75-89점)
**강점**:
- 사용자 스토리를 올바른 형식으로 작성했습니다.
- 스토리 포인트를 피보나치로 부여했습니다.
- 스프린트 계획이 타당합니다.

**개선 필요 사항**:
- 일부 사용자 스토리가 너무 큽니다. "사용자 인증 시스템 구축(13점)"은 Epic에 가깝습니다. "로그인 기능(5점)", "회원가입 기능(3점)"처럼 작은 스토리로 분해하세요.
- INVEST의 "Testable" 기준 충족을 위해 각 스토리에 수용 기준을 추가하세요. 예: "Given 사용자가 이메일과 비밀번호를 입력했을 때, When 로그인 버튼을 클릭하면, Then 홈 화면으로 이동한다."
- DoD에 "완료"라고만 하지 말고, "무엇이 완료인가?"를 구체적으로 정의하세요. "코드 작성, 테스트 통과, 리뷰 완료, 문서화, 배포"처럼 체크리스트 형태로 작성하세요.

### 보통 수준 답변 예시 (60-74점)
**강점**:
- 사용자 스토리와 스프린트 계획의 기본 구조를 갖췄습니다.
- 애자일 개념을 이해하고 있습니다.

**개선 필요 사항**:
- 사용자 스토리 형식이 부정확합니다. Week 9 강의 자료의 "사용자 스토리 형식" 섹션을 다시 학습하세요:
  ```
  As a [사용자 역할],
  I want [원하는 기능],
  So that [달성하고자 하는 가치/이유].
  ```
  "사용자가 로그인한다"가 아니라 "As a 쇼핑몰 고객, I want 이메일로 로그인하고 싶다, So that 내 주문 내역을 확인할 수 있다"로 작성하세요.
- INVEST 기준을 명시적으로 검증하세요. 각 스토리에 대해 I-N-V-E-S-T 체크리스트를 작성해보세요.
- 스토리 포인트가 1, 2, 4, 8로 부여되었습니다. 애자일에서는 피보나치 수열(1, 2, 3, 5, 8, 13, 21)을 사용합니다. 4 대신 3이나 5를 사용하세요.

**도움이 될 자료**:
- Week 9 강의 자료: 사용자 스토리 예시, INVEST 기준
- Scrum Guide: https://scrumguides.org
- "User Stories Applied" by Mike Cohn

### 부족한 답변 예시 (0-59점)
**주요 문제점**:
- 사용자 스토리 형식이 잘못되었습니다.
- INVEST 기준을 충족하지 않습니다.
- 스프린트 계획이 비현실적입니다.
- 애자일 개념 이해가 부족합니다.

**즉시 해야 할 일**:
1. Week 9 강의 자료를 처음부터 끝까지 다시 학습하세요.
2. 애자일 선언문의 4가지 핵심 가치를 이해하세요:
   - 프로세스보다 **개인과 상호작용**
   - 문서보다 **작동하는 소프트웨어**
   - 계약보다 **고객과의 협력**
   - 계획보다 **변화에 대응**
3. Epic과 Story의 차이를 명확히 하세요:
   - Epic: 큰 단위의 기능 (예: "사용자 인증 시스템")
   - Story: 작은 단위의 기능 (예: "이메일 로그인")
4. 사용자 스토리 예제를 5개 이상 직접 작성하고, INVEST 기준으로 검증해보세요.
5. Scrum Guide를 정독하세요 (30페이지, 1시간 소요).
6. 강사와 1:1 미팅을 예약하여 개념을 다시 확인하세요.

**재제출 권장**:
애자일/스크럼은 현대 소프트웨어 개발의 표준입니다. Week 10에서 Jira를 실습할 때도 이 개념이 필수입니다. 지금 확실히 이해하지 않으면 이후 학습이 어렵습니다. 위 자료를 학습한 후 보고서를 수정하여 재제출하시기 바랍니다.

## ✅ 자가 점검 체크리스트

제출 전에 다음 항목을 확인하세요:

### Epic 및 사용자 스토리
- [ ] Epic을 2개 이상 정의했는가?
- [ ] 사용자 스토리를 10개 이상 작성했는가?
- [ ] 모든 스토리가 "As a [역할], I want [기능], So that [가치]" 형식인가?
- [ ] 각 스토리가 INVEST 기준을 충족하는가?
  - [ ] Independent (독립적)
  - [ ] Negotiable (협상 가능)
  - [ ] Valuable (가치 있음)
  - [ ] Estimable (추정 가능)
  - [ ] Small (작은 크기)
  - [ ] Testable (테스트 가능)
- [ ] 수용 기준(Acceptance Criteria)을 포함했는가?

### 스프린트 계획
- [ ] 각 스토리에 스토리 포인트를 부여했는가?
- [ ] 스토리 포인트가 피보나치 수열(1, 2, 3, 5, 8, 13, 21)인가?
- [ ] 스프린트 목표를 명확히 정의했는가?
- [ ] 벨로시티를 가정하고 2주 스프린트에 적절한 양을 선정했는가?
- [ ] 스프린트 백로그가 제품 백로그에서 명확히 구분되는가?

### 완료 정의(DoD)
- [ ] DoD를 작성했는가?
- [ ] DoD에 코드 작성, 테스트, 리뷰, 문서화, 배포 등을 포함했는가?
- [ ] DoD가 구체적이고 측정 가능한가?

### 스프린트 회고
- [ ] 스프린트 회고 항목을 제시했는가?
- [ ] 잘한 점, 개선할 점, 액션 아이템이 각각 포함되었는가?
- [ ] 회고 항목이 실무적이고 구체적인가?

### 형식

### 자주 묻는 질문 (FAQ)

**Q1: Epic과 Story의 차이는 무엇인가요?**
A:
- Epic: 큰 단위의 기능, 여러 스프린트에 걸쳐 완료. 예: "사용자 인증 시스템"
- Story: 작은 단위의 기능, 1-2 스프린트 내 완료. 예: "이메일 로그인"
- 1개 Epic = 여러 개 Story

**Q2: 스토리 포인트는 시간인가요?**
A: 아니요. 스토리 포인트는 상대적 크기입니다. "로그인(3점)"이 "OAuth 통합(8점)"보다 약 2.5배 쉽다는 의미입니다. 시간은 벨로시티를 통해 간접적으로 추정합니다.

**Q3: 스토리 포인트를 어떻게 결정하나요?**
A: 플래닝 포커를 사용하세요:
1. 팀원들이 동시에 카드를 공개 (1, 2, 3, 5, 8, 13)
2. 가장 높은/낮은 추정치를 낸 사람이 이유 설명
3. 재투표하여 합의

**Q4: 벨로시티는 어떻게 설정하나요?**
A: 첫 스프린트는 예상으로 시작합니다 (예: 20포인트). 3-5 스프린트 후 평균을 사용하세요. 예: Sprint 1=18, Sprint 2=22, Sprint 3=20 → 평균 20포인트.

**Q5: DoD(Definition of Done)와 수용 기준의 차이는?**
A:
- DoD: 모든 스토리에 공통 적용되는 완료 기준 (예: "테스트 통과")
- 수용 기준: 특정 스토리에만 적용되는 기준 (예: "로그인 후 홈 화면 이동")

**Q6: INVEST의 'Small'은 얼마나 작아야 하나요?**
A: 1-2 스프린트(2-4주) 내에 완료 가능해야 합니다. 8-13포인트 이상이면 더 작은 스토리로 분해하세요.

**Q7: 스프린트 회고는 언제 하나요?**
A: 스프린트 종료 직전, 스프린트 리뷰 후에 합니다. 리뷰(제품 점검)와 회고(프로세스 점검)는 별도입니다.
