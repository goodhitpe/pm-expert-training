# Week 15: 클라우드 컴퓨팅 - 평가 루브릭

> **평가 대상**: 클라우드 아키텍처 설계 + 클라우드 서비스 체험 + 마이그레이션 계획  
> **제출 형식**: 아키텍처 다이어그램 + 체험 스크린샷 + 마이그레이션 계획서 (A4 2페이지)  
> **제출 기한**: Week 15 종료 시점

---

## 📋 실습 과제 설명

**과제명**: PM을 위한 클라우드 컴퓨팅 기초

**과제 내용**:
PM으로서 클라우드 프로젝트를 관리하거나 온프레미스에서 클라우드로의 마이그레이션 프로젝트를 수행하기 위한 기본 지식을 습득합니다. 클라우드 아키텍트 수준이 아닌, PM이 클라우드 서비스를 이해하고 프로젝트 계획을 수립하며 비용과 리스크를 관리하기 위한 실용적 지식에 집중합니다.

**요구사항**:
1. **과제 1: 클라우드 아키텍처 설계**
   - 웹 애플리케이션을 위한 클라우드 아키텍처 설계
   - 3-tier 아키텍처 (Web, App, DB)
   - 고가용성(HA) 고려
   - Auto Scaling 적용
   - 다이어그램 작성 (draw.io, Lucidchart 등 사용)

2. **과제 2: 클라우드 서비스 체험**
   - AWS, Azure, GCP 중 하나의 무료 계정 생성
   - 가상 머신(VM) 생성
   - 웹 서버 설치 및 간단한 페이지 배포
   - 객체 스토리지에 파일 업로드
   - 주요 단계별 스크린샷 제출

3. **과제 3: 클라우드 마이그레이션 계획**
   - 온프레미스 시스템을 클라우드로 마이그레이션하는 계획 수립
   - 현재 시스템 분석
   - 마이그레이션 전략 선택 (6R 중 선택)
   - 단계별 계획
   - 리스크 및 대응 방안 (A4 2페이지)

## 💡 피드백 예시

### 탁월한 답변 예시 (90-100점)
**강점**:
- 클라우드 아키텍처가 매우 체계적입니다. 3개의 가용 영역에 걸쳐 웹/앱/DB 서버를 배치하고, Application Load Balancer로 트래픽을 분산했습니다.
- Auto Scaling 정책을 CPU 70% 이상일 때 스케일 아웃, 30% 이하일 때 스케일 인으로 명확히 정의했습니다.
- AWS 실습에서 EC2, S3, RDS를 모두 체험하고, 각 서비스의 용도를 명확히 이해했습니다.
- 마이그레이션 계획에서 "Rehost(Lift and Shift)" 전략을 선택한 근거가 명확했습니다: "레거시 애플리케이션의 코드 수정 리스크를 피하고, 6개월 내 마이그레이션 완료를 목표로 함."
- 리스크 대응 방안이 구체적입니다. 특히 "다운타임 최소화를 위해 DNS 전환 방식 사용, 롤백을 위해 1주일간 병렬 운영"이라는 계획이 현실적입니다.

**추가 학습 제안**:
- AWS Well-Architected Framework를 학습하여 클라우드 아키텍처 모범 사례를 더 깊이 이해해보세요.
- Terraform이나 CloudFormation 같은 IaC(Infrastructure as Code) 도구를 체험해보시면 좋습니다.

### 우수한 답변 예시 (75-89점)
**강점**:
- 클라우드 아키텍처의 주요 구성 요소를 포함했습니다.
- 클라우드 서비스 실습을 성공적으로 완료했습니다.
- 마이그레이션 계획의 기본 틀이 잘 갖춰져 있습니다.

**개선 필요 사항**:
- 클라우드 아키텍처에서 데이터베이스가 단일 인스턴스입니다. 고가용성을 위해 Primary-Replica 구성이나 Multi-AZ 배포를 추가하세요.
- Auto Scaling 정책이 "부하 증가 시"라고만 표현되어 모호합니다. 구체적인 메트릭(CPU 사용률, 요청 수 등)과 임계값을 정의하세요.
- 클라우드 실습에서 보안 그룹 설정 스크린샷이 누락되었습니다. 어떤 포트를 열었는지 확인할 수 있도록 추가하세요.
- 마이그레이션 계획에서 "데이터 마이그레이션" 방법이 구체적이지 않습니다. 데이터량, 마이그레이션 도구(AWS DMS, rsync 등), 소요 시간을 명시하세요.

### 보통 수준 답변 예시 (60-74점)
**강점**:
- 클라우드 아키텍처를 작성했습니다.
- 클라우드 계정을 생성하고 기본 실습을 수행했습니다.
- 마이그레이션 계획의 개요를 제시했습니다.

**개선 필요 사항**:
- 클라우드 아키텍처가 단일 가용 영역입니다. Week 15 강의의 "고가용성" 섹션을 다시 읽고, 다중 가용 영역 배치를 설계하세요.
- 로드 밸런서가 없어서 단일 실패 지점(SPOF)이 됩니다. ALB/ELB를 추가하세요.
- VPC와 서브넷 구성이 표시되지 않았습니다. 퍼블릭 서브넷(웹)과 프라이빗 서브넷(앱, DB)을 분리하는 것이 보안상 중요합니다.
- 클라우드 실습에서 객체 스토리지를 사용하지 않았습니다. S3(AWS) 또는 Blob Storage(Azure)를 체험하고 스크린샷을 추가하세요.
- 마이그레이션 전략이 "클라우드로 이동"이라고만 표현되어 불명확합니다. 6R 전략 중 하나를 선택하고 이유를 설명하세요.

- AWS Architecture Center: https://aws.amazon.com/architecture/
- Azure Architecture Center: https://docs.microsoft.com/azure/architecture/
