## 📋 실습 과제 설명

**과제명**: PM을 위한 데이터베이스 기초

**과제 내용**:
PM으로서 데이터베이스 프로젝트를 관리하거나 데이터베이스를 포함한 시스템 개발 프로젝트를 수행하기 위한 기본 지식을 습득합니다. 데이터베이스 전문가 수준이 아닌, PM이 요구사항을 이해하고 개발팀과 소통하기 위한 실용적 지식에 집중합니다.

**요구사항**:
1. **과제 1: 데이터베이스 설계**
   - 온라인 쇼핑몰 데이터베이스 설계
   - ER 다이어그램 작성
   - 테이블 정의 (최소 5개)
   - 관계 정의 (1:N, N:M)
   - 3NF까지 정규화

2. **과제 2: SQL 실습**
   - MySQL 또는 PostgreSQL 설치
   - 데이터베이스 및 테이블 생성
   - 샘플 데이터 삽입 (각 테이블 10건 이상)
   - 다양한 쿼리 작성 (SELECT, INSERT, UPDATE, DELETE, JOIN, 집계 함수)
   - 실행 결과 스크린샷 제출

3. **과제 3: RDBMS vs NoSQL 비교**
   - 특정 프로젝트 시나리오 선정
   - RDBMS와 NoSQL 중 어느 것이 적합한지 분석
   - 선택 근거 제시 (A4 1페이지)

### 3. RDBMS vs NoSQL 분석 (30점)

| 등급 | 점수 | 기준 |
|------|------|------|
| **탁월** | 27-30 | • 구체적이고 현실적인 프로젝트 시나리오 제시<br>  (예: "실시간 주식 거래 시스템", "소셜 미디어 플랫폼" 등)<br>• RDBMS와 NoSQL의 특성을 정확히 이해<br>  - 스키마 유연성<br>  - 트랜잭션 지원 (ACID vs BASE)<br>  - 확장성 (Scale-up vs Scale-out)<br>  - 쿼리 복잡도<br>• 프로젝트 요구사항을 기준으로 체계적으로 비교<br>  - 데이터 구조 (정형 vs 비정형)<br>  - 트랜잭션 필요성<br>  - 데이터량 및 확장 계획<br>  - 읽기/쓰기 비율<br>  - 개발 일정 및 예산<br>• PM 관점의 선택 기준 명확<br>  (성능, 비용, 개발 복잡도, 팀 역량 등)<br>• 선택한 데이터베이스의 구체적 제품 제시<br>  (MySQL/PostgreSQL vs MongoDB/Redis 등)<br>• 마이그레이션 계획이나 하이브리드 접근 언급 |
| **우수** | 23-26 | • 프로젝트 시나리오 제시<br>• RDBMS와 NoSQL 특성 비교<br>• 요구사항 기준 비교<br>• 선택 근거 제시<br>• PM 관점 일부 반영<br>• 구체적 제품 언급 |
| **보통** | 18-22 | • 시나리오 제시하나 추상적<br>• RDBMS와 NoSQL 기본 차이 언급<br>• 비교 기준 모호<br>• 선택 근거 약함<br>• PM 관점 미흡<br>• 구체적 제품 미언급 |
| **부족** | 0-17 | • 시나리오 불명확<br>• RDBMS와 NoSQL 이해 부족<br>• 비교 분석 없음<br>• 선택 근거 없음<br>• PM 관점 결여 |

## 💡 피드백 예시

### 탁월한 답변 예시 (90-100점)
**강점**:
- ER 다이어그램이 매우 체계적입니다. 회원, 상품, 주문, 결제 등 온라인 쇼핑몰의 핵심 기능을 모두 포함했고, 향후 확장 가능성(리뷰, 쿠폰)도 고려했습니다.
- N:M 관계(주문-상품)를 주문상세 테이블로 올바르게 해소했고, 3NF까지 정규화하여 데이터 중복을 제거했습니다.
- SQL 실습에서 LEFT JOIN을 활용하여 "주문하지 않은 상품"을 조회하는 등 실무 중심의 쿼리를 작성했습니다.
- RDBMS vs NoSQL 비교에서 "은행 송금 시스템"이라는 구체적 시나리오를 제시하고, ACID 트랜잭션이 필수적이므로 RDBMS(PostgreSQL)를 선택한다는 논리가 명확했습니다.

**추가 학습 제안**:
- 인덱스 전략을 더 깊이 학습해보세요. 어떤 컬럼에 인덱스를 생성하면 좋을지 고민해보세요.
- 실제 MongoDB를 설치하여 NoSQL의 동작 방식을 체험해보시면 더욱 좋습니다.

### 우수한 답변 예시 (75-89점)
**강점**:
- ER 다이어그램이 잘 작성되었고, 주요 엔티티와 관계를 포함했습니다.
- SQL 실습을 성공적으로 완료했고, 다양한 쿼리를 작성했습니다.
- RDBMS와 NoSQL의 차이를 이해하고 있습니다.

**개선 필요 사항**:
- ER 다이어그램에서 "카테고리" 엔티티가 누락되었습니다. 상품-카테고리의 관계를 추가하세요.
- 정규화 과정을 보여주세요. 정규화 전후의 테이블 구조를 비교하면 이해도를 더 잘 보여줄 수 있습니다.
- SQL 실습에서 서브쿼리를 사용한 예제가 없습니다. 예를 들어, "평균 주문 금액보다 높은 주문"을 조회하는 쿼리를 추가해보세요.
```sql
SELECT * FROM Orders 
WHERE total_amount > (SELECT AVG(total_amount) FROM Orders);
```
- RDBMS vs NoSQL 비교에서 "대용량 데이터"라는 표현이 모호합니다. 구체적인 수치(예: "일일 100만 건 이상의 로그 데이터")를 제시하세요.

### 보통 수준 답변 예시 (60-74점)
**강점**:
- 기본적인 ER 다이어그램을 작성했습니다.
- SQL 실습을 수행했습니다.

**개선 필요 사항**:
- ER 다이어그램에서 엔티티가 4개만 있습니다. 최소 5개 이상의 엔티티를 포함하세요.
- 외래 키가 명확히 표시되지 않았습니다. ER 다이어그램에서 FK를 명시하고, 참조 무결성을 설명하세요.
- 정규화가 2NF 수준에 머물렀습니다. Week 14 강의의 "정규화" 섹션을 다시 읽고, 3NF까지 수행하세요.
  - 예: 주문 테이블에 고객명이 있다면, 이는 고객 테이블을 참조하도록 변경해야 합니다.
- SQL 실습에서 JOIN을 사용하지 않았습니다. 최소한 INNER JOIN으로 두 테이블을 연결하는 쿼리를 작성하세요.
```sql
SELECT o.order_id, u.name, o.total_amount
FROM Orders o
INNER JOIN Users u ON o.user_id = u.user_id;
```
- RDBMS vs NoSQL 비교가 표면적입니다. "NoSQL이 빠르다"는 언급만 있고, 왜 빠른지, 어떤 상황에서 빠른지 설명이 없습니다.

**도움이 될 자료**:
- Week 14 강의 자료: 정규화, SQL JOIN
- 온라인 SQL 실습 사이트: https://www.db-fiddle.com/

- SQL 실습에서 테이블 생성에 오류가 있습니다.
- RDBMS vs NoSQL 비교가 없거나 매우 간략합니다.

6. 강사와 1:1 미팅을 예약하여 데이터베이스 기초 개념을 재학습하세요.
