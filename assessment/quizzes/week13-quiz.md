### 문제 1
**질문**: SDLC(소프트웨어 개발 생명주기)의 올바른 순서는?

A) 요구사항 분석 → 구현 → 설계 → 테스트 → 배포 → 유지보수  
B) 설계 → 요구사항 분석 → 구현 → 테스트 → 배포 → 유지보수  
C) 요구사항 분석 → 설계 → 구현 → 테스트 → 배포 → 유지보수  
D) 요구사항 분석 → 설계 → 테스트 → 구현 → 배포 → 유지보수

**정답**: C

**해설**: SDLC의 표준 순서는 요구사항 분석 → 설계 → 구현 → 테스트 → 배포 → 유지보수입니다. 이 순서는 논리적인 개발 흐름을 나타내며, 각 단계는 이전 단계의 결과물을 기반으로 진행됩니다.

**관련 학습 내용**: Week 13 > 소프트웨어 개발 생명주기 (SDLC) > SDLC 단계

### 문제 2
**질문**: 폭포수 모델(Waterfall Model)의 주요 단점은?

A) 문서화가 부족함  
B) 변경 사항에 대응하기 어려움  
C) 각 단계가 명확하지 않음  
D) 너무 짧은 개발 기간

**정답**: B

**해설**: 폭포수 모델은 순차적으로 진행되기 때문에 한 단계가 완료되면 이전 단계로 돌아가기 어렵습니다. 따라서 요구사항 변경이나 뒤늦게 발견된 문제에 대응하기 어렵다는 것이 주요 단점입니다. 반면 각 단계가 명확하고 문서화가 충실하다는 것은 장점입니다.

**관련 학습 내용**: Week 13 > SDLC 모델 > 폭포수 모델

### 문제 3
**질문**: PM 관점에서 기능적 요구사항(Functional Requirements)의 예시로 **가장 적절한** 것은?

A) 시스템 응답 시간은 3초 이내여야 한다  
B) 사용자는 비밀번호를 재설정할 수 있어야 한다  
C) 시스템은 동시에 1000명의 사용자를 지원해야 한다  
D) 데이터는 암호화되어 저장되어야 한다

**정답**: B

**해설**: 기능적 요구사항은 시스템이 수행해야 할 구체적인 기능을 정의합니다. "사용자는 비밀번호를 재설정할 수 있어야 한다"는 명확한 기능 요구사항입니다. A, C, D는 성능, 확장성, 보안에 관한 비기능적 요구사항입니다.

**관련 학습 내용**: Week 13 > SDLC 단계 > 요구사항 분석

### 문제 4
**질문**: Git에서 브랜치(Branch)를 사용하는 주요 목적은?

A) 코드의 크기를 줄이기 위해  
B) 여러 기능을 독립적으로 개발하기 위해  
C) 컴파일 속도를 높이기 위해  
D) 데이터베이스를 백업하기 위해

**정답**: B

**해설**: Git의 브랜치는 메인 코드에 영향을 주지 않고 새로운 기능을 독립적으로 개발할 수 있게 합니다. 여러 개발자가 동시에 다른 기능을 작업할 수 있으며, 완성된 기능은 병합(merge)을 통해 메인 브랜치에 통합됩니다.

**관련 학습 내용**: Week 13 > 버전 관리 (Version Control) > Git 기초

### 문제 5
**질문**: 단위 테스트(Unit Test)에 대한 설명으로 **틀린** 것은?

A) 개별 함수나 메서드를 테스트한다  
B) 개발자가 직접 작성한다  
C) 전체 시스템의 통합을 테스트한다  
D) 빠르게 실행할 수 있다

**정답**: C

**해설**: 단위 테스트는 가장 작은 단위(함수, 메서드)를 테스트하는 것으로, 개발자가 작성하며 빠르게 실행됩니다. 전체 시스템의 통합을 테스트하는 것은 시스템 테스트 또는 통합 테스트의 역할입니다.

**관련 학습 내용**: Week 13 > 소프트웨어 테스팅 > 테스트 레벨

### 문제 6
**질문**: CI/CD에서 "CI"가 의미하는 것은?

A) Code Integration (코드 통합)  
B) Continuous Integration (지속적 통합)  
C) Complete Implementation (완전한 구현)  
D) Central Infrastructure (중앙 인프라)

**정답**: B

**해설**: CI는 Continuous Integration(지속적 통합)을 의미하며, 코드 변경 시 자동으로 빌드하고 테스트를 실행하여 통합 문제를 조기에 발견하는 것입니다. CD는 Continuous Deployment/Delivery(지속적 배포/전달)를 의미합니다.

**관련 학습 내용**: Week 13 > CI/CD > CI (지속적 통합)

### 문제 7
**질문**: 마이크로서비스 아키텍처의 주요 특징이 **아닌** 것은?

A) 각 서비스는 독립적으로 배포 가능  
B) 모든 기능이 하나의 서버에서 실행  
C) 서비스 간 API로 통신  
D) 확장성과 유연성이 높음

**정답**: B

**해설**: 마이크로서비스 아키텍처는 애플리케이션을 작고 독립적인 서비스로 분리합니다. 각 서비스는 독립적으로 개발, 배포, 확장할 수 있으며 API를 통해 통신합니다. "모든 기능이 하나의 서버에서 실행"되는 것은 모놀리식(Monolithic) 아키텍처의 특징입니다.

**관련 학습 내용**: Week 13 > 소프트웨어 아키텍처 > 마이크로서비스 아키텍처

### 문제 8
**질문**: PM이 개발자와 효과적으로 협업하기 위해 이해해야 하는 것으로 **가장 거리가 먼** 것은?

A) 기술 스택의 기본 개념  
B) 모든 함수의 상세 코드 구현  
C) 개발 프로세스와 일정 추정  
D) 기술적 리스크와 기술 부채

**정답**: B

**해설**: PM은 기술에 대한 기본적인 이해는 필요하지만, 모든 함수의 상세 코드를 이해할 필요는 없습니다. PM은 전체적인 시스템 구조, 기술 스택의 개념, 일정 추정의 현실성, 기술적 리스크를 파악하는 것이 중요합니다. 상세한 코드 구현은 개발자의 영역입니다.

**관련 학습 내용**: Week 13 > PM이 알아야 할 소프트웨어 개발 지식 > 개발자와의 커뮤니케이션

### 문제 9
**질문**: 코드 리뷰(Code Review)의 주요 목적으로 **가장 적절하지 않은** 것은?

A) 코드 품질 향상  
B) 버그 조기 발견  
C) 팀원 간 지식 공유  
D) 개발자 성과 평가

**정답**: D

**해설**: 코드 리뷰의 주요 목적은 코드 품질 향상, 버그 조기 발견, 지식 공유입니다. 코드 리뷰는 협업과 학습의 도구이지, 개발자를 평가하기 위한 수단이 아닙니다. 평가 목적으로 사용하면 방어적 태도를 유발하여 코드 리뷰의 효과가 떨어질 수 있습니다.

**관련 학습 내용**: Week 13 > 버전 관리 > 코드 리뷰

### 문제 10
**질문**: SOLID 원칙 중 "S"가 의미하는 것은?

A) Software (소프트웨어)  
B) System (시스템)  
C) Single Responsibility (단일 책임)  
D) Standard (표준)

**정답**: C

**해설**: SOLID는 객체 지향 설계의 5가지 원칙으로, S는 Single Responsibility(단일 책임 원칙)을 의미합니다. 하나의 클래스나 모듈은 하나의 책임만 가져야 한다는 원칙입니다. 나머지는 O(Open/Closed), L(Liskov Substitution), I(Interface Segregation), D(Dependency Inversion)입니다.

**관련 학습 내용**: Week 13 > 소프트웨어 아키텍처 > 디자인 원칙
