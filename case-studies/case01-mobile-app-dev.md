# Case Study 01: Mobile Fitness App Development

## Project Overview and Background

**Company:** FitLife Technologies Inc.  
**Industry:** Health & Wellness Technology  
**Project Name:** FitLife Pro Mobile App  
**Project Duration:** 9 months  
**Budget:** $750,000  
**Project Start Date:** January 15, 2024  
**Target Launch Date:** October 1, 2024

### Background
FitLife Technologies is a mid-sized health and wellness company with 200 employees that has been providing web-based fitness tracking services for the past 5 years. The company currently has 150,000 active users on their web platform. Market research indicates that 78% of their users access fitness apps primarily on mobile devices, and competitor analysis shows FitLife is losing market share to mobile-first competitors.

The executive team has approved a strategic initiative to develop a native mobile application (iOS and Android) to increase user engagement, reduce churn rate (currently at 35% annually), and capture the mobile-first demographic (ages 18-35). The app must integrate with existing backend systems while providing enhanced features like real-time workout tracking, AI-powered personal training recommendations, and social fitness challenges.

## Business Objectives

1. **Revenue Growth:** Increase annual recurring revenue by 40% within 12 months post-launch through new subscriptions
2. **User Acquisition:** Acquire 75,000 new mobile app users within 6 months of launch
3. **User Engagement:** Increase daily active users (DAU) from current 12% to 35% within 8 months post-launch
4. **Churn Reduction:** Reduce annual churn rate from 35% to 20% within 12 months post-launch
5. **Market Position:** Achieve top 10 ranking in health & fitness category on both app stores within 6 months
6. **User Satisfaction:** Achieve minimum 4.5-star rating on app stores with at least 5,000 reviews

## Project Scope

### In Scope
- Native iOS app (iOS 15.0 and above)
- Native Android app (Android 10.0 and above)
- User authentication and profile management
- Workout tracking (cardio, strength training, yoga, sports)
- Integration with Apple Health and Google Fit
- AI-powered workout recommendations engine
- Social features (friend connections, challenges, leaderboards)
- Nutrition logging and calorie tracking
- Progress analytics and visualization dashboards
- Push notifications for reminders and achievements
- In-app subscription payment processing
- Offline mode for workout tracking
- Integration with existing FitLife backend APIs
- Migration tools for existing web users
- App store optimization and submission
- Beta testing program (100 internal users, 500 external beta testers)

### Out of Scope
- Smartwatch/wearable device integration (planned for Phase 2)
- Video streaming of workout classes (planned for Phase 2)
- Direct messaging between users
- Marketplace for fitness equipment
- Integration with third-party fitness devices (Peloton, Mirror, etc.)
- Web application updates or redesign
- Telemedicine or virtual personal training sessions
- Multi-language support (English only for MVP)

## Key Challenges and Issues

### Challenge 1: Cross-Platform Feature Parity
**Description:** Ensuring identical functionality and user experience across iOS and Android platforms with different native capabilities and design guidelines.

**Impact:** High - User satisfaction and brand consistency  
**Current Status:** Ongoing concern  
**Mitigation Strategy:** 
- Shared component library for common features
- Weekly cross-platform sync meetings
- Automated UI testing to verify parity
- Platform-specific acceptance criteria in user stories

### Challenge 2: Legacy Backend Integration
**Description:** The existing backend API was designed for web applications and lacks mobile-optimized endpoints. Real-time workout tracking requires WebSocket connections not currently supported.

**Impact:** High - Technical architecture and performance  
**Current Status:** Backend team working on API enhancements  
**Mitigation Strategy:**
- Parallel backend API development stream
- Temporary workarounds using polling for MVP
- GraphQL layer to optimize data fetching
- Close collaboration between mobile and backend teams

### Challenge 3: AI Recommendation Engine Performance
**Description:** The AI-powered workout recommendation engine requires significant processing power and may not perform well on older devices. Training data quality is also a concern.

**Impact:** Medium - User experience and feature differentiation  
**Current Status:** Prototype in development  
**Mitigation Strategy:**
- Cloud-based processing for AI recommendations
- Caching of recommendations with background updates
- Graceful degradation for older devices
- Minimum viable recommendation algorithm for MVP

### Challenge 4: User Data Migration Complexity
**Description:** Migrating 150,000 existing web users' historical data (workouts, nutrition logs, progress) to the mobile platform without data loss or corruption.

**Impact:** High - User retention and trust  
**Current Status:** Migration tool in design phase  
**Mitigation Strategy:**
- Phased migration approach
- Extensive testing with production data copies
- Rollback procedures
- User communication and support plan

### Challenge 5: Resource Constraints
**Description:** Development team has competing priorities with maintenance of web platform. Two senior developers are only available part-time due to other commitments.

**Impact:** Medium - Timeline and quality  
**Current Status:** Resource allocation approved by management  
**Mitigation Strategy:**
- Clear priority definition with executive sponsor
- Contractor engagement for additional capacity
- Simplified MVP scope where possible
- Buffer time built into schedule

### Challenge 6: App Store Approval Uncertainties
**Description:** First-time mobile app submission with unknown app store review requirements and potential for rejection, especially around health data and in-app purchases.

**Impact:** Medium - Launch timeline  
**Current Status:** Research and preparation ongoing  
**Mitigation Strategy:**
- Early review of app store guidelines
- Legal review of privacy policy and terms
- 2-week buffer before launch date
- Pre-submission consultation with app store representatives

### Challenge 7: Performance on Low-End Devices
**Description:** App must perform well on devices as old as 4 years, but feature-rich functionality may cause performance issues.

**Impact:** Medium - User base and market reach  
**Current Status:** Performance testing framework being established  
**Mitigation Strategy:**
- Performance budgets defined for key screens
- Regular performance testing on target devices
- Optimization sprints scheduled
- Feature toggles for resource-intensive features

## Quality Requirements

### Functional Quality
- **Feature Completeness:** 100% of MVP user stories must meet acceptance criteria
- **Cross-Platform Parity:** 95% feature parity between iOS and Android
- **Data Accuracy:** 99.9% accuracy in workout tracking and calorie calculations
- **Integration Success:** 100% successful sync with Apple Health and Google Fit APIs

### Performance Quality
- **App Launch Time:** Under 2 seconds on target devices
- **Screen Load Time:** Under 1 second for 90% of screen transitions
- **API Response Time:** P95 under 500ms for all API calls
- **Offline Functionality:** 100% of workout tracking available offline
- **Battery Usage:** Under 5% battery drain per hour during active workout tracking
- **Memory Footprint:** Under 150MB RAM usage during normal operation

### Reliability Quality
- **Crash Rate:** Less than 0.5% crash rate per user session
- **Uptime:** 99.5% backend availability (4 hours downtime per month max)
- **Data Loss:** Zero tolerance for user data loss
- **Sync Reliability:** 99% success rate for data synchronization

### Usability Quality
- **App Store Rating:** Minimum 4.5 stars average
- **User Onboarding:** 90% of users complete onboarding flow
- **Task Completion:** 95% success rate for primary user tasks
- **Accessibility:** WCAG 2.1 Level AA compliance for accessibility features

### Security Quality
- **Data Encryption:** 100% of data encrypted in transit (TLS 1.3) and at rest (AES-256)
- **Authentication:** OAuth 2.0 with optional biometric authentication
- **Security Vulnerabilities:** Zero critical or high-severity vulnerabilities at launch
- **Privacy Compliance:** 100% GDPR, CCPA, and HIPAA compliance

### Compatibility Quality
- **iOS Compatibility:** iOS 15.0 and above (covering 95% of active iOS devices)
- **Android Compatibility:** Android 10.0 and above (covering 92% of active Android devices)
- **Screen Sizes:** Support for phone screens 4.7" to 6.9"
- **Network Conditions:** Functional on 3G networks with degraded performance warnings

## Questions and Exercises for Students

### Exercise 1: Stakeholder Management
**Scenario:** The Marketing Director wants to add a feature for users to share workout achievements directly to Instagram and TikTok. This wasn't in the original scope. How do you handle this request?

**Tasks:**
1. Analyze the impact of this change request on timeline, budget, and resources
2. Identify which stakeholders need to be involved in this decision
3. Draft a change request document with your recommendation
4. Create a communication plan for delivering your decision to the Marketing Director

### Exercise 2: Risk Management
**Scenario:** It's June 15, and the backend API integration is 3 weeks behind schedule (Risk R03 has materialized). The iOS team is blocked and cannot proceed with Sprint 5 work.

**Tasks:**
1. Assess the current situation and its impact on project milestones
2. Identify 3 potential solutions with pros/cons for each
3. Create a recovery plan including timeline adjustments
4. Draft a status report for the executive sponsor
5. Identify any new risks introduced by your solution

### Exercise 3: Budget Management
**Scenario:** In Sprint 7, you discover that cloud infrastructure costs are tracking 25% higher than budgeted due to increased AI processing needs.

**Tasks:**
1. Perform a variance analysis and forecast total budget impact
2. Identify cost optimization opportunities
3. Prepare options for presentation to the CFO
4. Create a revised budget forecast for remaining project phases
5. Recommend preventive measures for future projects

### Exercise 4: Quality Assurance
**Scenario:** During external beta testing, you receive feedback that the Android app crashes frequently on Samsung devices running Android 11, affecting approximately 15% of beta testers.

**Tasks:**
1. Prioritize this issue relative to other open bugs
2. Create an action plan to resolve the issue
3. Assess impact on the launch timeline (currently 10 days to submission)
4. Decide whether to proceed with launch or delay
5. Draft communication to beta testers about the issue

### Exercise 5: Scope Management
**Scenario:** The development team reports that the AI recommendation engine will only achieve 70% accuracy by launch date, below the 85% target. Achieving 85% would require an additional 4 weeks.

**Tasks:**
1. Evaluate the options: launch with 70% accuracy, delay launch, or remove the feature
2. Assess the impact of each option on business objectives
3. Make a recommendation with justification
4. Create a plan for achieving 85% accuracy post-launch if you decide to launch with 70%
5. Identify lessons learned for future AI/ML projects

### Exercise 6: Sprint Planning
**Scenario:** You are planning Sprint 5 (June 1-14). You have 4 developers (2 iOS, 2 Android) and need to complete health platform integration and begin analytics features.

**Tasks:**
1. Create user stories for Sprint 5 with acceptance criteria
2. Estimate story points for each user story
3. Allocate stories to team members considering dependencies
4. Identify any risks or blockers for the sprint
5. Define the sprint goal and success criteria

### Exercise 7: Change Management
**Scenario:** A competitor just launched a similar app with a viral "buddy workout" feature where friends can work out together in real-time. The CEO wants this feature added immediately.

**Tasks:**
1. Perform impact analysis (scope, schedule, budget, resources)
2. Evaluate alternatives (include in current project, Phase 2, never)
3. Assess the competitive threat realistically
4. Make a recommendation with supporting analysis
5. Create a decision document for the steering committee

### Exercise 8: Communication Planning
**Scenario:** You need to prepare for the Sprint 6 review on July 14, where you'll demo progress to all stakeholders including the CEO.

**Tasks:**
1. Identify what should be demonstrated and to whom
2. Create an agenda for the 90-minute review meeting
3. Prepare talking points for addressing the 2-week delay from Sprint 4
4. Design a dashboard showing project health (scope, schedule, budget, quality)
5. Anticipate difficult questions and prepare responses

### Exercise 9: Resource Management
**Scenario:** Your lead iOS developer has accepted another job and will leave in 3 weeks (during Sprint 6). Finding a replacement typically takes 6-8 weeks.

**Tasks:**
1. Assess the impact on project timeline and deliverables
2. Create a knowledge transfer plan
3. Identify options for replacing the resource (internal transfer, contractor, etc.)
4. Adjust project plan to mitigate impact
5. Update risk register with lessons learned

### Exercise 10: Post-Launch Analysis
**Scenario:** It's 30 days post-launch (November 1). Results: 8,000 downloads, 3.8-star rating, 22% DAU, 65% 7-day retention. Some targets are missed, some exceeded.

**Tasks:**
1. Analyze results against success criteria
2. Identify what went well and what didn't
3. Investigate root causes for the lower-than-expected download numbers
4. Create a 90-day improvement plan
5. Prepare a lessons learned document for future projects

### Exercise 11: Critical Path Analysis
**Tasks:**
1. Identify the critical path in the project schedule
2. Calculate float/slack for non-critical activities
3. Determine which activities can be compressed to recover 2 weeks if needed
4. Analyze the impact of compressing different activities
5. Create a fast-track scenario that reduces timeline by 3 weeks

### Exercise 12: Earned Value Management
**Scenario:** As of July 1 (midway through project):
- Planned Value (PV): $400,000
- Actual Cost (AC): $435,000
- Earned Value (EV): $380,000

**Tasks:**
1. Calculate CPI (Cost Performance Index) and SPI (Schedule Performance Index)
2. Calculate variance (CV and SV)
3. Forecast EAC (Estimate at Completion)
4. Interpret the results and identify problems
5. Create an action plan to get back on track
