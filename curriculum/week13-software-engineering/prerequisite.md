## 1️⃣ 프로그래밍이란?

### 쉬운 설명
**프로그래밍**은 컴퓨터에게 일을 시키기 위한 명령어를 작성하는 것입니다.

### 실생활 비유
요리 레시피를 생각해보세요:
- **레시피** = 프로그래밍 코드
- **요리사** = 컴퓨터
- **완성된 요리** = 프로그램 실행 결과

레시피에 "양파를 다지세요", "물을 끓이세요"라고 적으면 요리사가 따라하듯이, 
프로그래밍 코드에 명령을 작성하면 컴퓨터가 실행합니다.

### 예시: 간단한 Python 코드
```python
# 사용자 이름을 입력받아 인사하는 프로그램
def greet(name):
    return f"안녕하세요, {name}님!"

print(greet("PM"))
# 출력: 안녕하세요, PM님!
```

### PM이 알아야 하는 이유
- 개발자와 소통할 때 기본 개념 필요
- 개발 일정 산정 시 작업량 이해
- 코드 리뷰나 Pull Request 검토 시 맥락 파악

---

## 2️⃣ 소스 코드 (Source Code)

### 정의
프로그래머가 작성한 프로그램의 원본 텍스트 파일입니다.

### 특징
- 사람이 읽고 쓸 수 있는 형태
- 다양한 프로그래밍 언어로 작성 (Python, Java, JavaScript 등)
- 확장자로 언어 구분 (`.py`, `.java`, `.js` 등)

### 예시
```javascript
// JavaScript 소스 코드 예시
function calculateProjectCost(hours, hourlyRate) {
    return hours * hourlyRate;
}

const totalCost = calculateProjectCost(100, 150000);
console.log(totalCost); // 출력: 15000000
```

### PM 관점
- 소스 코드는 **프로젝트의 핵심 자산**
- 버전 관리 필수 (Git 사용)
- 코드 품질 = 프로젝트 품질

---

## 3️⃣ 컴파일 vs 실행

### 컴파일 (Compile)
사람이 작성한 코드를 컴퓨터가 이해할 수 있는 형태로 변환하는 과정

**비유**: 한국어 문서를 기계어로 번역하는 것

### 실행 (Execute/Run)
변환된 프로그램을 실제로 동작시키는 것

### 프로그래밍 언어별 차이

| 언어 유형 | 예시 | 특징 | PM이 알아야 할 점 |
|----------|------|------|------------------|
| **컴파일 언어** | C, C++, Java | 실행 전 컴파일 필요 | 빌드 시간 고려 필요 |
| **인터프리터 언어** | Python, JavaScript | 즉시 실행 가능 | 빠른 프로토타이핑 가능 |

---

## 4️⃣ 버전 관리 (Version Control)

### 왜 필요한가?
여러 사람이 함께 코드를 작성할 때:
- 누가, 언제, 무엇을 변경했는지 추적
- 이전 버전으로 되돌리기
- 동시 작업 시 충돌 방지

### 일상 비유
Google 문서의 "버전 기록" 기능과 비슷합니다.

### Git이란?
가장 많이 사용되는 버전 관리 시스템

**핵심 개념**:
- **Repository (저장소)**: 프로젝트 코드가 저장된 공간
- **Commit**: 변경 사항을 저장하는 것 (저장 시점 기록)
- **Branch**: 독립적인 작업 공간 (실험용 복사본)
- **Merge**: 다른 Branch의 변경사항을 합치는 것

### PM이 알아야 하는 이유
- 개발 진행 상황 파악
- 코드 충돌(Conflict) 이슈 이해
- Branch 전략 (개발/테스트/배포) 이해

## 5️⃣ 소프트웨어 개발 프로세스 기초

### 개발의 기본 단계
```
1. 요구사항 분석
   ↓
2. 설계
   ↓
3. 개발 (코딩)
   ↓
4. 테스트
   ↓
5. 배포
   ↓
6. 유지보수
```

### Waterfall vs Agile (간단 비교)

**Waterfall (폭포수)**:
- 순차적 진행 (1→2→3→4→5→6)
- 각 단계 완료 후 다음 단계
- 변경 어려움

**Agile (애자일)**:
- 반복적 진행 (작은 사이클 반복)
- 빠른 피드백과 변경 반영
- 유연성 높음

---

## 6️⃣ 테스트 (Testing)

### 정의
프로그램이 의도대로 동작하는지 확인하는 과정

### 테스트 유형 (PM 관점)

| 테스트 유형 | 설명 | PM의 역할 |
|------------|------|----------|
| **단위 테스트** | 개별 함수/모듈 테스트 | 개발자가 작성, PM은 커버리지 확인 |
| **통합 테스트** | 여러 모듈이 함께 작동하는지 | 테스트 계획에 포함 |
| **시스템 테스트** | 전체 시스템 테스트 | PM이 주도적으로 관리 |
| **사용자 승인 테스트** | 실제 사용자가 테스트 | PM이 조율 및 관리 |

### 왜 중요한가?
- 버그를 일찍 발견할수록 수정 비용 적음
- 품질 보증
- 고객 만족도 향상

---

## 7️⃣ 디버깅 (Debugging)

### 정의
프로그램의 오류(버그)를 찾아서 수정하는 과정

### 비유
자동차 정비사가 이상한 소리가 나는 원인을 찾는 것

### 버그 (Bug)란?
프로그램의 오류나 예상치 못한 동작

**역사적 배경**: 1947년 실제 나방(bug)이 컴퓨터에 들어가 오류를 일으킨 사건에서 유래

### PM 관점
- 버그 수정은 시간이 걸림 (일정에 반영)
- 버그 우선순위 관리 필요 (Critical, High, Medium, Low)
- 버그 추적 시스템 활용 (Jira 등)

---

## 8️⃣ 빌드 (Build)

### 정의
소스 코드를 실행 가능한 프로그램으로 만드는 과정

### 단계
1. 컴파일 (코드 → 기계어)
2. 의존성 해결 (필요한 라이브러리 포함)
3. 패키징 (실행 파일 생성)

### PM이 알아야 하는 이유
- 빌드 시간이 길면 개발 속도 저하
- 빌드 실패 = 배포 불가
- CI/CD에서 자동 빌드 사용

---

## 9️⃣ 배포 (Deployment)

### 정의
완성된 소프트웨어를 실제 사용 환경에 설치하는 과정

### 배포 환경
- **개발(Development)**: 개발자가 작업하는 환경
- **테스트(Test/Staging)**: 테스트를 위한 환경
- **운영(Production)**: 실제 사용자가 사용하는 환경

### 배포 전략
- **Blue-Green 배포**: 새 버전과 구 버전을 번갈아 전환
- **Canary 배포**: 일부 사용자에게만 먼저 배포
- **롤링 배포**: 점진적으로 교체

## 🔟 주요 IT 용어 정리

### 개발 관련
| 용어 | 의미 | 예시 |
|------|------|------|
| **IDE** | 통합 개발 환경 (코드 작성 도구) | VS Code, IntelliJ |
| **Framework** | 개발을 쉽게 해주는 틀 | React, Django |
| **Library** | 재사용 가능한 코드 모음 | jQuery, NumPy |
| **API** | 프로그램 간 소통 인터페이스 | REST API, GraphQL |

### 버전 관리
| 용어 | 의미 |
|------|------|
| **Commit** | 변경사항 저장 |
| **Push** | 로컬 → 원격 저장소로 업로드 |
| **Pull** | 원격 → 로컬 저장소로 다운로드 |
| **Branch** | 독립적인 작업 공간 |
| **Merge** | 브랜치 합치기 |
| **Conflict** | 코드 충돌 (수동 해결 필요) |
