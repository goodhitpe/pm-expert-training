## 📚 이 주차를 위한 준비

Week 9에서는 애자일 방법론과 스크럼 프레임워크를 학습합니다. 
애자일은 단순한 프로세스가 아닌 **사고방식의 전환**입니다. 
기존의 전통적 방법론과는 다른 철학과 가치관을 먼저 이해하는 것이 중요합니다.

## 1️⃣ 애자일이 생긴 이유

### 전통적 방법론의 한계

**폭포수 모델의 문제점**:
```
요구사항 분석 (3개월)
     ↓
설계 (2개월)
     ↓
개발 (6개월)
     ↓
테스트 (2개월)
     ↓
배포 및 운영
```

**실제로 발생하는 문제들**:
1. **고객이 원하는 것을 정확히 모름** 
   - 처음에 "A가 필요해요"라고 했지만
   - 6개월 후 실제 제품 보고 "아, 내가 원한 건 B였네요"

2. **시장 변화에 대응 불가**
   - 1년 프로젝트 중간에 경쟁사가 신제품 출시
   - 하지만 우리는 계획대로 진행할 수밖에 없음

3. **문제를 너무 늦게 발견**
   - 테스트 단계에서야 큰 문제 발견
   - 되돌리기엔 너무 늦음

### 애자일의 탄생
2001년, 17명의 소프트웨어 개발 전문가들이 모여 **애자일 선언문**을 작성했습니다.
핵심 메시지: **"변화에 빠르게 대응하자!"**

---

## 2️⃣ 애자일 마인드셋 핵심

### 계획 vs 대응

**전통적 사고방식**:
```
"완벽한 계획을 세우면 성공한다"
"계획대로 실행하는 것이 중요하다"
"변경은 나쁜 것이다"
```

**애자일 사고방식**:
```
"계획은 중요하지만, 변화에 대응하는 것이 더 중요하다"
"빠르게 시도하고 배우는 것이 중요하다"
"변경은 자연스러운 것이다"
```

### 실생활 비유

**여행 계획으로 비유**:

**전통적 방식 (폭포수)**:
- 6개월 전부터 모든 일정 확정
- 호텔, 레스토랑, 관광지 모두 예약
- 날씨가 안 좋아도 계획대로 진행
- 새로 발견한 멋진 장소가 있어도 일정 변경 불가

**애자일 방식**:
- 큰 틀만 정하고 출발 (어느 도시, 며칠 머물지)
- 매일 아침 날씨와 상황 보고 당일 일정 결정
- 현지인 추천으로 더 좋은 곳 발견하면 바로 변경
- 유연하게 대응하며 최고의 경험 추구

## 3️⃣ 애자일 4대 가치

### 가치 1: 개인과 상호작용 > 프로세스와 도구

**의미**:
도구나 절차보다 **사람 간의 소통**이 더 중요하다

**예시**:
- ❌ 나쁜 예: "문서로만 소통, 회의 금지"
- ✅ 좋은 예: "5분 대화로 1시간 회의 대체"

**실생활 비유**:
```
자동차 수리:
- 프로세스 중심: 매뉴얼만 보고 기계적으로 수리
- 사람 중심: 정비사와 이야기하며 진짜 문제 파악
```

---

### 가치 2: 작동하는 소프트웨어 > 포괄적인 문서

**의미**:
완벽한 문서를 만들기보다 **실제로 작동하는 것**을 보여주자

**예시**:
- ❌ 나쁜 예: 200페이지 기획서 작성, 개발은 시작도 안 함
- ✅ 좋은 예: 간단한 프로토타입 만들어서 피드백 받기

**실생활 비유**:
```
요리:
- 문서 중심: 레시피를 완벽하게 적기만 하고 요리 안 함
- 작동하는 것 중심: 일단 만들어서 맛보고 개선
```

---

### 가치 3: 고객과의 협력 > 계약 협상

**의미**:
계약서의 조항을 따지기보다 **고객과 함께 문제 해결**

**예시**:
- ❌ 나쁜 예: "계약서에 이렇게 적혀있으니 이것만 합니다"
- ✅ 좋은 예: "계약과 조금 다르지만 고객에게 더 필요한 걸 만들어요"

**실생활 비유**:
```
집 인테리어:
- 계약 중심: "계약서에 벽지만 있으니 벽지만 바꿉니다"
- 협력 중심: "벽지보다 조명 바꾸는 게 더 나을 것 같은데요?"
```

---

### 가치 4: 변화에 대응 > 계획을 따르기

**의미**:
계획을 고집하기보다 **상황에 맞게 조정**

**예시**:
- ❌ 나쁜 예: "계획이니까 무조건 이대로 해야 해"
- ✅ 좋은 예: "상황이 바뀌었으니 계획을 수정하자"

**실생활 비유**:
```
식당 창업:
- 계획 고집: 한식 계획했으니 손님 없어도 계속 한식
- 변화 대응: 손님 반응 보고 메뉴 조정, 주력 상품 변경
```

## 4️⃣ 반복과 점진

### 반복 (Iteration)

**의미**:
한 번에 다 만들지 않고, **작은 사이클을 여러 번** 반복

**예시**:
```
전통적 방식 (한 번에 완성):
[12개월] → 완성된 제품 출시

애자일 방식 (2주마다 반복):
[2주] → 버전 0.1
[2주] → 버전 0.2
[2주] → 버전 0.3
...
[2주] → 버전 1.0
```

### 점진 (Incremental)

**의미**:
작은 기능부터 **조금씩 추가**하며 완성도 높임

### 비유: 그림 그리기

**전통적 방식**:
```
1. 왼쪽 위부터 정교하게 완성
2. 천천히 오른쪽으로 이동
3. 마지막에 전체 그림 완성
→ 중간에 문제 발견 시 수정 어려움
```

**애자일 방식**:
```
1. 전체 밑그림 먼저 (스케치)
2. 핵심 부분 색칠
3. 디테일 추가
4. 계속 개선
→ 매 단계마다 전체 그림 확인 가능
```

## 5️⃣ 실패를 빨리, 자주

### Fast Fail (빠른 실패)

**전통적 사고**:
"실패는 나쁜 것이다. 절대 실패하면 안 된다"

**애자일 사고**:
"실패는 학습이다. 빨리 실패할수록 빨리 배운다"

### 왜 빨리 실패하는 것이 좋은가?

**케이스 1: 늦게 실패**
```
12개월 개발 → 출시 → 사용자 반응 나쁨 → 실패
결과: 12개월 + 많은 비용 낭비
```

**케이스 2: 빨리 실패**
```
2주 개발 → 테스트 → 방향 수정
2주 개발 → 테스트 → 방향 수정
2주 개발 → 테스트 → 성공!
결과: 6주 만에 올바른 방향 찾음
```

### 실생활 비유
```
새로운 레스토랑 메뉴:
- 전통적: 대량 재료 구매 → 메뉴 출시 → 반응 나쁨 → 큰 손실
- 애자일: 소량 테스트 → 고객 반응 확인 → 조정 → 정식 출시
```

## 6️⃣ 자기 조직화 팀

### 의미
위에서 지시하기보다 **팀 스스로 결정**하고 책임진다

### 전통적 vs 애자일

**전통적 팀**:
```
관리자: "이렇게 하세요" (지시)
팀원: "네, 알겠습니다" (수동적)
```

**애자일 팀**:
```
관리자: "무엇을 달성해야 하는지" (목표 제시)
팀: "어떻게 할지는 우리가 결정" (능동적)
```

### 왜 효과적인가?

1. **동기부여**: 자율성이 높을수록 몰입도 증가
2. **창의성**: 스스로 방법을 찾으면 더 나은 해결책 도출
3. **책임감**: 자신이 결정한 것에 더 책임감 느낌

### 실생활 비유
```
프로젝트 발표 준비:

전통적:
- 교수: "PPT 30장, 글자 크기 18pt, 발표 20분"
- 학생: 지시대로만 준비

애자일:
- 교수: "청중을 설득해야 합니다"
- 학생: 영상, 데모, 스토리텔링 등 자유롭게 준비
```

## 7️⃣ 고객 중심 사고

### 의미
내가 만들고 싶은 것이 아닌 **고객이 필요로 하는 것** 만들기

### 자주 하는 실수

**기능 중심**:
"이 기능 멋져! 넣자!"

**고객 가치 중심**:
"고객이 이 문제를 해결해야 해. 어떤 기능이 필요할까?"

### MVP (Minimum Viable Product)

**의미**: 최소한의 기능으로 고객 가치를 전달하는 제품

**예시: 교통 수단 개발**

❌ **나쁜 MVP**:
```
1차: 바퀴만 (움직이지도 않음)
2차: 바퀴 + 프레임 (여전히 못 씀)
3차: 바퀴 + 프레임 + 엔진 (드디어 완성!)
→ 고객은 3차까지 아무 가치도 못 받음
```

✅ **좋은 MVP**:
```
1차: 스케이트보드 (느리지만 이동 가능!)
2차: 스쿠터 (더 빠름!)
3차: 자전거 (훨씬 편함!)
4차: 오토바이 (완벽!)
→ 매 단계마다 고객이 사용 가능
```

## 8️⃣ 투명성과 검사

### 투명성 (Transparency)
모든 것을 **공개**한다

**예시**:
- 진행 상황을 벽에 붙인 보드로 공유
- 문제가 있으면 숨기지 않고 공유
- 코드도 팀원 모두가 볼 수 있음

### 검사 (Inspection)
자주 **점검**한다

**예시**:
- 매일 15분 회의 (Daily Scrum)
- 2주마다 결과 리뷰
- 2주마다 프로세스 회고

### 적응 (Adaptation)
문제 발견 시 즉시 **조정**한다

**예시**:
- 일정이 늦어지면 범위 조정
- 방법이 안 맞으면 다른 방법 시도
- 고객 피드백 받으면 즉시 반영

### 3개의 관계
```
투명성 → 검사 → 적응 → 투명성 → ...
(계속 반복)
```

## 9️⃣ 애자일이 맞는 프로젝트 vs 안 맞는 프로젝트

### 애자일이 잘 맞는 경우 ✅

1. **요구사항이 불명확**
   - 만들면서 배워야 함
   - 고객도 정확히 뭘 원하는지 모름

2. **변화가 많은 환경**
   - 시장 변화 빠름
   - 경쟁사 동향에 민감

3. **혁신이 필요**
   - 새로운 시도
   - 실험과 학습 필요

**예시**: 스타트업 신제품, AI 서비스, 모바일 앱

### 폭포수가 더 맞는 경우 ⚠️

1. **요구사항이 명확하고 변경 없음**
   - 법규 준수 필수
   - 이미 정해진 표준

2. **변경 비용이 매우 높음**
   - 하드웨어 제조
   - 건설 프로젝트

3. **예측 가능성이 중요**
   - 고정 예산/일정 필수
   - 계약 조건 엄격

**예시**: 공공 인프라, 의료 기기, 항공 시스템

### 자주 묻는 질문 (FAQ)

**Q1: 애자일은 IT 업계에서만 쓰나요?**

**Q2: 애자일이 폭포수보다 무조건 좋은가요?**  
A: 아니요. 프로젝트 특성에 따라 다릅니다. 요구사항이 명확하고 변경이 적으면 폭포수가 더 효율적일 수 있습니다.

**Q3: 애자일에서는 계획을 안 세우나요?**  
A: 계획을 세웁니다! 다만 상세한 장기 계획보다는, 큰 방향 + 유연한 조정을 선호합니다.

**Q4: 모든 팀원이 애자일에 동의해야 하나요?**  
A: 이상적으로는 그렇습니다. 하지만 한 명씩 설득하고 작은 성공 경험을 쌓으면서 확산시킬 수 있습니다.
