## 📖 Part 1: 애자일 방법론 (Agile Methodology)

### 1.1 애자일의 탄생 배경

#### 전통적 방법론의 한계
- **변화 대응 어려움**: 초기 요구사항 확정 후 변경 어려움
- **긴 개발 주기**: 6개월~2년 후에야 결과물 확인
- **높은 실패율**: Standish Group 통계 - 프로젝트 성공률 29%
- **과도한 문서화**: 실제 코딩보다 문서 작성에 더 많은 시간
- **고객 불만족**: 최종 제품이 요구사항과 달라짐

#### 애자일의 등장 (2001년)
- **배경**: 17명의 소프트웨어 개발자가 스노버드(미국 유타주)에서 모임
- **목적**: 경량 개발 방법론의 공통점 도출
- **결과**: 애자일 선언문(Agile Manifesto) 발표

### 1.2 애자일 선언문 (Agile Manifesto)

#### 4가지 핵심 가치

```
우리는 소프트웨어를 개발하고, 또 다른 사람의 개발을 도와주면서
소프트웨어 개발의 더 나은 방법들을 찾아가고 있다.

이 작업을 통해 우리는 다음을 가치 있게 여기게 되었다:

1. 프로세스와 도구보다 개인과 상호작용을
2. 포괄적인 문서보다 작동하는 소프트웨어를
3. 계약 협상보다 고객과의 협력을
4. 계획을 따르기보다 변화에 대응하기를

가치 있게 여긴다. 이 말은, 왼쪽에 있는 것들도 가치가 있지만,
우리는 오른쪽에 있는 것들에 더 높은 가치를 둔다는 것이다.
```

#### 상세 설명

**1. 개인과 상호작용 > 프로세스와 도구**
- ❌ 나쁜 예: "프로세스대로 했는데 실패했으니 프로세스 탓"
- ✅ 좋은 예: "팀원 간 대화로 문제 빠르게 해결"
- 💡 의미: 도구보다 사람 간 커뮤니케이션이 중요

**2. 작동하는 소프트웨어 > 포괄적인 문서**
- ❌ 나쁜 예: 100페이지 문서 작성, 코드는 없음
- ✅ 좋은 예: 간단한 문서 + 실제 동작하는 프로토타입
- 💡 의미: 문서는 필요한 만큼만, 실제 동작이 우선

**3. 고객과의 협력 > 계약 협상**
- ❌ 나쁜 예: "계약서에 없으니 못 합니다"
- ✅ 좋은 예: "고객 피드백 반영해서 더 나은 제품 만들기"
- 💡 의미: 계약 준수보다 고객 만족이 목표

**4. 변화에 대응 > 계획 따르기**
- ❌ 나쁜 예: "계획대로 진행 중이라 변경 불가"
- ✅ 좋은 예: "시장 변화에 맞춰 제품 방향 조정"
- 💡 의미: 변화는 자연스러운 것, 유연하게 대응

### 1.3 애자일 12가지 원칙

1. **조기 및 지속적 인도**: 가치 있는 소프트웨어를 일찍, 자주 인도
2. **변경 환영**: 요구사항 변경을 경쟁 우위로 활용
3. **짧은 인도 주기**: 몇 주에서 몇 달 단위로 작동하는 소프트웨어 인도
4. **비즈니스-개발 협업**: 매일 함께 일하기
5. **동기부여된 개인**: 환경과 지원 제공, 신뢰
6. **대면 대화**: 가장 효과적인 정보 전달 방법
7. **작동하는 소프트웨어**: 진척의 주요 척도
8. **지속 가능한 개발**: 일정한 속도 유지
9. **기술적 우수성**: 좋은 설계로 민첩성 향상
10. **단순성**: 하지 않아도 되는 일을 최대화
11. **자기 조직화 팀**: 최고의 아키텍처, 요구사항, 설계는 자기 조직화 팀에서
12. **정기적 성찰**: 더 효과적이 되는 방법 모색 및 조정

### 1.4 애자일 vs 워터폴 (Waterfall)

| 구분 | 워터폴 (전통적) | 애자일 |
|-----|---------------|--------|
| **철학** | 예측적 (Predictive) | 적응적 (Adaptive) |
| **접근** | 순차적 (Sequential) | 반복적 (Iterative) |
| **요구사항** | 초기 확정, 변경 어려움 | 점진적 발견, 변경 환영 |
| **계획** | 상세한 사전 계획 | 충분한 계획 + 지속적 조정 |
| **인도** | 프로젝트 종료 시 일괄 | 주기적 증분 인도 |
| **고객 참여** | 초기/종료 시 | 지속적 협업 |
| **팀 구조** | 역할 명확 분리 | 교차 기능팀 (Cross-functional) |
| **문서화** | 상세 문서 필수 | 필요한 만큼 |
| **리스크** | 후반부에 집중 | 조기 발견 및 대응 |
| **적합 프로젝트** | 요구사항 명확, 안정적 | 요구사항 불명확, 변화 많음 |

#### 실제 사례

**워터폴 실패 사례**: 영국 NHS IT 시스템
- 10년 계획, 120억 파운드 투자
- 요구사항 변경 미대응
- 결과: 프로젝트 폐기

**애자일 성공 사례**: Spotify
- 2주 스프린트로 빠른 출시
- 사용자 피드백 즉시 반영
- 결과: 시장 선도 음악 스트리밍 서비스

## 🏃 Part 2: 스크럼 프레임워크 (Scrum Framework)

### 2.1 스크럼 개요

#### 스크럼이란?
- **정의**: 복잡한 제품 개발을 위한 경량 프레임워크
- **창시자**: Jeff Sutherland & Ken Schwaber (1990년대 초)
- **특징**: 
  - 규칙이 명확하고 간단
  - 경험적 프로세스 제어 (Empirical Process Control)
  - 3-3-3-5 구조 (3 역할, 3 산출물, 3+2 이벤트)

#### 스크럼의 3가지 기둥

```
        투명성
       (Transparency)
          /  \
         /    \
    검사      적응
  (Inspection) (Adaptation)
```

**1. 투명성 (Transparency)**
- 모든 정보가 공개됨
- 공통 언어 사용
- 완료 정의(DoD) 명확

**2. 검사 (Inspection)**
- 정기적 진행 상황 점검
- 산출물 검토
- 목표 대비 편차 확인

**3. 적응 (Adaptation)**
- 문제 발견 시 즉시 조정
- 프로세스 개선
- 방향 전환 (Pivot)

### 2.2 스크럼 역할 (3 Roles)

#### 2.2.1 제품 책임자 (Product Owner, PO)

**주요 책임**:
- 제품 비전 제시 및 공유
- 제품 백로그 관리 및 우선순위 결정
- 제품 가치 최대화
- 개발 팀과 이해관계자 간 의사소통

**구체적 활동**:
- 사용자 스토리 작성 및 정제 (Refinement)
- 스프린트 리뷰에서 완성된 기능 검수
- ROI를 고려한 우선순위 설정
- 시장 트렌드 및 경쟁사 분석

**성공하는 PO의 특징**:
- ✅ 결단력 있음 (우선순위 명확)
- ✅ 이해관계자와 좋은 관계
- ✅ 비즈니스 지식 풍부
- ✅ 개발 팀 신뢰

**실습 시나리오**:
```
상황: 온라인 쇼핑몰 PO
백로그 항목 10개, 스프린트 용량 30 스토리 포인트

우선순위 결정 기준:
1. 비즈니스 가치 (매출 기여도)
2. 리스크 감소
3. 고객 만족도
4. 기술적 의존성
```

#### 2.2.2 스크럼 마스터 (Scrum Master, SM)

**주요 책임**:
- 스크럼 프로세스 코치 및 퍼실리테이터
- 장애물 제거 (Impediment Removal)
- 팀 보호 (외부 방해 차단)
- 조직 변화 지원

**구체적 활동**:
- 스크럼 교육 및 코칭
- 회의 퍼실리테이션 (중립적 진행)
- 팀 간 협업 촉진
- 프로세스 개선 주도

**SM이 하지 않는 것**:
- ❌ 팀 관리자 아님 (Manager가 아님)
- ❌ 작업 배정 안 함
- ❌ 기술적 결정 안 함
- ❌ 프로젝트 매니저 아님

**SM vs PM 비교**:
| 구분 | Scrum Master | Project Manager |
|-----|-------------|-----------------|
| 역할 | 서번트 리더 | 관리자 |
| 권한 | 없음 (코치) | 있음 (지시) |
| 초점 | 프로세스 | 결과 |
| 팀 | 자기 조직화 지원 | 작업 배정 |

#### 2.2.3 개발 팀 (Development Team)

**특징**:
- **교차 기능 (Cross-functional)**: 모든 필요한 기술 보유
- **자기 조직화 (Self-organizing)**: 스스로 작업 방법 결정
- **책임 공유**: 팀 전체가 책임
- **적정 규모**: 3-9명 (이상적: 7±2명)

**역할 없음**:
- 팀 내 타이틀 없음
- "시니어 개발자", "주니어 디자이너" 같은 구분 없음
- 모두가 "개발 팀원"

**구체적 책임**:
- 스프린트 목표 달성
- 제품 증분 개발
- 품질 보장
- DoD 준수
- 일일 스크럼 참여

### 2.3 스크럼 이벤트 (5 Events)

#### 2.3.1 스프린트 (Sprint)

**정의**: 1개월 이하의 고정된 시간(Timebox) 동안 "완료(Done)" 되고 사용 가능하며 잠재적으로 출시 가능한 제품 증분을 만드는 기간

**특징**:
- **고정 기간**: 1-4주 (일반적으로 2주)
- **타임박스**: 연장 불가
- **일관성**: 동일한 기간 유지
- **연속성**: 스프린트 종료 즉시 다음 스프린트 시작

**스프린트 구조**:
```
Day 1: Sprint Planning (오전)
Day 2-9: Daily Scrum (매일 15분)
         개발 작업
Day 10: Sprint Review (오후)
        Sprint Retrospective (오후)
```

**스프린트 중 금지사항**:
- ❌ 스프린트 목표 변경
- ❌ 품질 목표 하향 조정
- ❌ 개발 팀 구성 변경

**스프린트 취소**:
- PO만 취소 가능
- 스프린트 목표가 더 이상 의미 없을 때
- 매우 드문 경우 (거의 안 함)

#### 2.3.2 스프린트 계획 (Sprint Planning)

**목적**: 스프린트에서 무엇을 어떻게 할지 결정

**시간**: 2주 스프린트 기준 최대 4시간 (1주당 2시간)

**구조**:
```
Part 1: 무엇을 (What) - 2시간
- 스프린트 목표 설정
- 제품 백로그 항목 선정

Part 2: 어떻게 (How) - 2시간
- 선정된 항목을 작업으로 분해
- 스프린트 백로그 작성
```

**참석자**: PO, SM, 개발 팀 (전원 필수)

**산출물**: 
- 스프린트 목표 (Sprint Goal)
- 스프린트 백로그 (Sprint Backlog)

**실습 예제: 온라인 쇼핑몰**

```
스프린트 목표: "사용자가 상품을 장바구니에 담고 결제할 수 있다"

선정된 백로그 항목:
1. 장바구니 기능 (8 SP)
2. 결제 페이지 (13 SP)
3. 주문 확인 이메일 (5 SP)
총 26 SP (팀 벨로시티: 30 SP)

작업 분해 (Task Breakdown):
장바구니 기능 (8 SP)
├─ DB 테이블 설계 (2시간)
├─ API 개발 (4시간)
├─ UI 개발 (4시간)
├─ 단위 테스트 (2시간)
└─ 통합 테스트 (2시간)
```

#### 2.3.3 일일 스크럼 (Daily Scrum)

**목적**: 스프린트 목표 달성을 위한 진행 상황 점검 및 계획 조정

**시간**: 매일 같은 시간, 15분 이내

**장소**: 고정된 장소 (Stand-up Meeting)

**참석자**: 개발 팀 (필수), SM (퍼실리테이션), PO (선택)

**3가지 질문**:
1. **어제 무엇을 했는가?** (What did I do yesterday?)
2. **오늘 무엇을 할 것인가?** (What will I do today?)
3. **장애물은 무엇인가?** (Any impediments?)

**좋은 예시**:
```
개발자 A:
"어제 장바구니 API 개발 완료했습니다.
오늘은 UI와 API 연동하겠습니다.
장애물은 없습니다."

개발자 B:
"어제 결제 페이지 UI 80% 완료했습니다.
오늘 나머지 20% 완료하고 테스트하겠습니다.
PG사 API 문서가 불명확해서 PO님께 확인 필요합니다."
```

**나쁜 예시**:
```
개발자 C:
"어제 회의 3개 참석하고, 이메일 20개 답장하고,
점심 먹고, 커피 마시고, 코드 10줄 작성했습니다.
오늘은... 음... 아직 잘 모르겠고...
장애물은... 생각해봐야겠어요."
```

**Daily Scrum 안티패턴**:
- ❌ 보고 회의로 전락 (SM이나 PO에게 보고)
- ❌ 문제 해결 회의 (15분 초과)
- ❌ 일부만 참여
- ❌ 형식적으로 진행

#### 2.3.4 스프린트 리뷰 (Sprint Review)

**목적**: 완성된 증분 시연 및 제품 백로그 조정

**시간**: 2주 스프린트 기준 최대 2시간

**참석자**: 스크럼 팀 + 이해관계자 (고객, 경영진 등)

**진행 순서**:
```
1. 스프린트 목표 검토 (5분)
2. 완성된 기능 시연 (60분)
   - 실제 동작하는 제품 시연
   - 데모, not 프레젠테이션
3. 이해관계자 피드백 (30분)
4. 제품 백로그 업데이트 (25분)
   - 새로운 요구사항 추가
   - 우선순위 조정
```

**시연 체크리스트**:
- ✅ 실제 환경에서 동작
- ✅ 완료 정의(DoD) 충족
- ✅ 사용자 관점에서 시연
- ✅ 실패한 항목은 시연 안 함

**피드백 수집**:
- 무엇이 마음에 드는가?
- 무엇을 개선하면 좋을까?
- 다음 스프린트 우선순위는?

#### 2.3.5 스프린트 회고 (Sprint Retrospective)

**목적**: 팀, 프로세스, 도구 개선 방안 도출

**시간**: 2주 스프린트 기준 1.5시간

**참석자**: 스크럼 팀 (PO, SM, 개발 팀)

**원칙**:
- 안전한 환경 (비난 금지)
- 솔직한 대화
- 구체적 개선 액션

**5가지 회고 기법**:

##### 1. Start-Stop-Continue

```
┌──────────┬──────────┬──────────┐
│  Start   │   Stop   │ Continue │
│(시작하기) │(중단하기) │(계속하기) │
├──────────┼──────────┼──────────┤
│ 페어    │ 야근     │ 코드    │
│ 프로그래밍│          │ 리뷰    │
├──────────┼──────────┼──────────┤
│ TDD     │ 회의 중  │ Daily   │
│         │ 멀티태스킹│ Scrum   │
└──────────┴──────────┴──────────┘
```

##### 2. Mad-Sad-Glad

```
😠 Mad (화나는 것):
- 빌드 서버가 자주 다운됨
- 요구사항이 계속 변경됨

😢 Sad (슬픈 것):
- 테스트 커버리지가 낮음
- 기술 부채 증가

😊 Glad (기쁜 것):
- 고객 만족도 높음
- 팀워크 좋아짐
```

##### 3. Sailboat (돛단배)

```
       ⛵ (목표)
      /|\
     / | \
    /  |  \
   🌬️ (바람-촉진) 💎 (보물-목표)
        |
    ⚓ (닻-장애물)  🪨 (암초-리스크)
```

- **바람**: 팀을 앞으로 나가게 하는 것
- **닻**: 팀을 붙잡는 것
- **암초**: 잠재적 위험
- **보물**: 달성하고자 하는 목표

##### 4. 4Ls

- **Liked** (좋았던 것): 새로운 도구 도입
- **Learned** (배운 것): 성능 최적화 기법
- **Lacked** (부족했던 것): 문서화
- **Longed for** (바라는 것): 더 나은 개발 환경

##### 5. Timeline (타임라인)

```
Week 1         Week 2
  |              |
  😊 킥오프       😢 버그 발견
       😠 요구사항 변경
            😊 문제 해결
                  😊 배포 성공
```

**회고 액션 아이템 예시**:
```
1. 빌드 서버 안정화
   - 담당: 개발자 A
   - 기한: 다음 스프린트 중반
   - 측정: 다운타임 50% 감소

2. 테스트 커버리지 향상
   - 담당: 전체 팀
   - 기한: 2주 후
   - 측정: 70% → 80%
```

### 2.4 스크럼 산출물 (3 Artifacts)

#### 2.4.1 제품 백로그 (Product Backlog)

**정의**: 제품에 필요한 모든 것의 우선순위화된 목록

**특징**:
- 살아있는 문서 (Living Document)
- 지속적으로 변화
- PO가 관리
- 공개적 (Transparent)

**항목 유형**:
- 기능 (Features)
- 버그 수정 (Bug Fixes)
- 기술 작업 (Technical Work)
- 지식 습득 (Learning)

**우선순위 기준**:
1. 비즈니스 가치
2. 리스크 감소
3. 의존성
4. 시간 민감도

**예시: 온라인 쇼핑몰 제품 백로그**

| ID | 사용자 스토리 | 우선순위 | SP | 상태 |
|----|-------------|---------|----|----|
| 1 | 사용자 등록/로그인 | 높음 | 8 | 완료 |
| 2 | 상품 검색 | 높음 | 13 | 완료 |
| 3 | 장바구니 | 높음 | 8 | 진행중 |
| 4 | 결제 | 높음 | 13 | 대기 |
| 5 | 주문 내역 조회 | 중간 | 5 | 대기 |
| 6 | 위시리스트 | 낮음 | 5 | 대기 |

#### 2.4.2 스프린트 백로그 (Sprint Backlog)

**정의**: 스프린트 동안 완료할 제품 백로그 항목 + 작업 계획

**구성요소**:
- 스프린트 목표 (Sprint Goal)
- 선정된 제품 백로그 항목
- 작업 분해 (Tasks)
- 번다운 차트 (Burndown Chart)

**스프린트 백로그 예시**:

```
스프린트 목표: 사용자가 상품을 구매할 수 있다

선정 항목:
1. 장바구니 (8 SP)
   └─ 작업:
      - DB 테이블 설계 (2h) [완료]
      - 장바구니 API (4h) [진행중]
      - 장바구니 UI (4h) [대기]
      - 테스트 (4h) [대기]

2. 결제 (13 SP)
   └─ 작업:
      - PG사 연동 조사 (4h) [완료]
      - 결제 API (8h) [대기]
      - 결제 UI (6h) [대기]
      - 테스트 (4h) [대기]
```

**번다운 차트**:
```
남은 작업(시간)
 40│\
   │ \
 30│  \
   │   \___
 20│       \___
   │           \___
 10│               \
   │                 \
  0└─────────────────────
   1  3  5  7  9  11  일
   
   —— 이상선 (계획)
   ··· 실제선
```

#### 2.4.3 증분 (Increment)

**정의**: 스프린트 종료 시점의 완료된 제품 백로그 항목의 합

**특징**:
- 완료 정의(DoD) 충족
- 잠재적으로 출시 가능 (Potentially Shippable)
- 누적적 (이전 증분 + 현재 증분)
- 통합 및 테스트 완료

**완료 정의 (Definition of Done) 예시**:
```
✅ 코드 작성 완료
✅ 단위 테스트 작성 및 통과 (커버리지 80% 이상)
✅ 코드 리뷰 완료 (2명 승인)
✅ 통합 테스트 통과
✅ 사용자 인수 테스트 완료
✅ 문서 업데이트
✅ 스테이징 환경 배포
✅ PO 승인
```

## 📝 Part 3: 사용자 스토리 (User Story)

### 3.1 사용자 스토리 형식

**템플릿**:
```
As a [사용자 역할],
I want [기능],
So that [가치/이유].

[역할]로서,
[기능]을 원한다,
[가치]를 얻기 위해.
```

**좋은 예시**:
```
As a 온라인 쇼핑객,
I want 장바구니에 상품을 담고 싶다,
So that 나중에 한 번에 결제할 수 있다.
```

**나쁜 예시**:
```
As a 시스템,
I want 데이터베이스를 업데이트한다,
So that 데이터를 저장한다.

문제: 
- 사용자 관점 아님
- 기술적 용어 사용
- 가치 불명확
```

### 3.2 INVEST 기준

**I - Independent (독립적)**
- 다른 스토리와 독립적
- 순서 바뀌어도 OK
- 의존성 최소화

**N - Negotiable (협상 가능)**
- 대화의 시작점
- 세부사항은 협의
- 계약서가 아님

**V - Valuable (가치 있음)**
- 사용자/비즈니스 가치
- 기술적 가치만으로는 불충분
- PO가 가치 판단

**E - Estimable (추정 가능)**
- 크기 추정 가능
- 너무 크면 분해
- 너무 모호하면 명확화

**S - Small (작은 크기)**
- 1 스프린트 내 완료
- 너무 크면 Epic → Story로 분해
- 8-13 SP 권장

**T - Testable (테스트 가능)**
- 인수 기준 명확
- 통과/실패 판단 가능
- 자동화 가능 선호

### 3.3 스토리 포인트 (Story Point)

**정의**: 사용자 스토리의 상대적 크기 추정 단위

**고려 요소**:
- 복잡도 (Complexity)
- 양 (Volume)
- 불확실성 (Uncertainty)

**피보나치 수열 사용**:
```
1, 2, 3, 5, 8, 13, 21, 34, ...

1 SP: 매우 간단 (1-2시간)
3 SP: 간단 (반나절)
5 SP: 보통 (1일)
8 SP: 복잡 (2-3일)
13 SP: 매우 복잡 (1주) - 분해 고려
21+ SP: Epic - 반드시 분해
```

**스토리 포인트 예시**:
```
기준 스토리 (5 SP):
"사용자가 비밀번호를 재설정할 수 있다"

비교 스토리:
- 로그인 (3 SP) - 기준보다 간단
- 장바구니 (8 SP) - 기준보다 복잡
- 결제 (13 SP) - 기준보다 매우 복잡
```

### 3.4 Planning Poker (플래닝 포커)

**목적**: 팀 전체가 합의하는 추정치 도출

**준비물**: 포커 카드 (0, 1/2, 1, 2, 3, 5, 8, 13, 20, 40, 100, ?, ☕)

**진행 순서**:
```
1단계: PO가 스토리 설명
2단계: 팀원들이 질문 (명확화)
3단계: 각자 카드 선택 (숨김)
4단계: 동시에 카드 공개
5단계: 가장 높은/낮은 추정치 제시자 설명
6단계: 토론 및 재추정
7단계: 합의 시까지 반복
```

**실습 예제**:

```
스토리: "사용자가 상품 리뷰를 작성할 수 있다"

1라운드 추정:
- 개발자 A: 5
- 개발자 B: 8
- 개발자 C: 3
- 개발자 D: 8

토론:
개발자 C (3): "간단한 텍스트 입력만 추가하면 돼요"
개발자 B (8): "이미지 업로드, 별점, 욕설 필터링도 필요해요"

PO 명확화: "이번 스프린트는 텍스트 + 별점만"

2라운드 추정:
- 모두: 5

최종: 5 SP
```

**특수 카드**:
- **?**: 추정 불가 (정보 부족)
- **☕**: 휴식 필요 (집중력 저하)
- **100**: Epic (분해 필요)

### 3.5 Epic과 Theme

**Epic**: 여러 스토리로 분해되는 큰 기능
```
Epic: "사용자 관리 시스템"
├─ Story 1: 회원 가입 (8 SP)
├─ Story 2: 로그인 (3 SP)
├─ Story 3: 비밀번호 재설정 (5 SP)
├─ Story 4: 프로필 수정 (5 SP)
└─ Story 5: 회원 탈퇴 (3 SP)
```

**Theme**: 관련된 스토리들의 그룹
```
Theme: "결제 관련"
- 장바구니
- 결제
- 주문 확인
- 배송 추적
```

## 📊 Part 4: 애자일 메트릭 (Agile Metrics)

### 4.1 벨로시티 (Velocity)

**정의**: 스프린트당 완료한 스토리 포인트의 합

**계산 방법**:
```
Sprint 1: 25 SP
Sprint 2: 28 SP
Sprint 3: 23 SP
Sprint 4: 27 SP
Sprint 5: 26 SP

평균 벨로시티 = (25 + 28 + 23 + 27 + 26) / 5 = 25.8 SP
```

**활용**:
- 릴리즈 계획 수립
- 팀 생산성 추이 파악
- 용량 계획 (Capacity Planning)

**주의사항**:
- ❌ 팀 간 비교 금지
- ❌ 개인 평가 지표로 사용 금지
- ✅ 팀 내부 계획 도구로만 사용

### 4.2 번다운 차트 (Burndown Chart)

**종류**:
1. **Sprint Burndown**: 스프린트 내 남은 작업
2. **Release Burndown**: 릴리즈까지 남은 스토리

**해석**:
```
이상적:  \
           \
            \
             
실제:     \___
              \___
                  \
```

- 이상선보다 위: 진행 느림 (위험)
- 이상선보다 아래: 진행 빠름 (좋음)
- 수평선: 진행 없음 (문제)

### 4.3 누적 흐름 다이어그램 (CFD)

```
스토리 수
  ↑
  │          Done
  │      Testing
  │  Development
  │To Do
  └─────────────→ 시간
```

**활용**:
- 진행 상황 시각화
- 병목 지점 파악
- WIP 추적

## 🔄 Part 5: 칸반 (Kanban)

### 5.1 칸반 개요

**정의**: 작업 흐름을 시각화하고 진행 중인 작업(WIP)을 제한하는 방법

**유래**: 일본 도요타 생산 시스템

**핵심 원칙**:
1. 작업 흐름 시각화
2. WIP 제한
3. 흐름 관리
4. 프로세스 정책 명확화
5. 피드백 루프
6. 협력적 개선

### 5.2 칸반 보드

**기본 구조**:
```
┌────────┬──────────┬──────────┬────────┐
│ Backlog│   To Do  │In Progress│  Done  │
│        │          │ (WIP:3)  │        │
├────────┼──────────┼──────────┼────────┤
│ Task A │ Task B   │ Task C   │Task D  │
│ Task E │ Task F   │ Task G   │Task H  │
│ Task I │          │ Task J   │Task K  │
│ Task L │          │          │        │
└────────┴──────────┴──────────┴────────┘
```

**확장 구조**:
```
Backlog → Selected → Design → Dev → Test → Review → Deploy → Done
```

### 5.3 WIP (Work In Progress) 제한

**이유**:
- 멀티태스킹 감소
- 사이클 타임 단축
- 병목 현상 조기 발견
- 품질 향상

**WIP 제한 예시**:
```
컬럼별 제한:
- Design: 2개
- Development: 3개
- Testing: 2개

전체 WIP: 7개
```

### 5.4 칸반 메트릭

**리드 타임 (Lead Time)**:
- 정의: 요청부터 완료까지 시간
- 측정: Backlog → Done

**사이클 타임 (Cycle Time)**:
- 정의: 작업 시작부터 완료까지 시간
- 측정: In Progress → Done

**처리량 (Throughput)**:
- 정의: 단위 시간당 완료 작업 수
- 측정: Done 컬럼 항목 수 / 시간

### 5.5 칸반 vs 스크럼

| 구분 | 스크럼 | 칸반 |
|-----|--------|------|
| **역할** | 정의됨 (PO, SM, 팀) | 기존 역할 유지 |
| **이터레이션** | 스프린트 (1-4주) | 없음 (연속적) |
| **변경** | 스프린트 중 제한 | 언제든 가능 |
| **백로그** | 스프린트 백로그 | 없음 (칸반 보드) |
| **추정** | 스토리 포인트 | 선택적 |
| **메트릭** | 벨로시티 | 리드/사이클 타임 |
| **회의** | 4가지 (Planning, Daily, Review, Retro) | 선택적 |
| **적합** | 제품 개발 | 운영, 유지보수 |

**언제 무엇을 사용할까?**
- **스크럼**: 신제품 개발, 명확한 릴리즈 목표
- **칸반**: 지속적 서비스, 우선순위 자주 변경
- **스크럼반**: 스크럼 + 칸반 장점 결합

## 💼 6. 실습 과제

### 과제 1: 사용자 스토리 작성 및 INVEST 검증

**시나리오**: 온라인 강의 플랫폼 개발

**과제**:
1. 주요 기능 10개를 사용자 스토리로 작성
2. 각 스토리에 INVEST 기준 적용 및 검증
3. Epic 1개와 하위 Story 3-5개 작성

**제출 양식**:
```
Story 1:
As a 수강생,
I want 강의를 검색하고 싶다,
So that 원하는 강의를 찾을 수 있다.

INVEST 검증:
- Independent: ✅ 다른 기능과 독립적
- Negotiable: ✅ 검색 방식은 협의 가능
- Valuable: ✅ 사용자 가치 명확
- Estimable: ✅ 크기 추정 가능 (5 SP)
- Small: ✅ 1 스프린트 내 완료 가능
- Testable: ✅ 검색 결과 확인 가능
```

### 과제 2: 스프린트 계획 수립

**주어진 조건**:
- 팀 벨로시티: 25 SP
- 스프린트 기간: 2주
- 제품 백로그: 과제 1에서 작성한 스토리

**과제**:
1. Planning Poker로 각 스토리에 스토리 포인트 부여
2. 우선순위 결정 (비즈니스 가치 기준)
3. 스프린트 목표 설정
4. 스프린트 백로그 작성 (작업 분해 포함)
5. 번다운 차트 작성

**제출물**:
- 스프린트 계획 문서 (2-3페이지)
- 번다운 차트 (Excel 또는 도구)

### 과제 3: 회고 실습

**시나리오**: 방금 끝난 스프린트 회고

**과제**:
1. 5가지 회고 기법 중 2가지 선택
2. 각 기법으로 회고 진행 (가상 시나리오)
3. 액션 아이템 3개 도출 (SMART 기준)

**SMART 액션 아이템 예시**:
```
Specific: 빌드 시간 단축
Measurable: 10분 → 5분
Achievable: 캐싱 및 병렬화
Relevant: 개발 생산성 향상
Time-bound: 2주 내
```

### 서술형 문제

**6. 애자일과 워터폴의 차이점을 3가지 이상 설명하시오.**

**모범 답안**:
1. **접근 방식**: 워터폴은 순차적, 애자일은 반복적
2. **요구사항**: 워터폴은 초기 확정, 애자일은 점진적 발견
3. **고객 참여**: 워터폴은 초기/종료 시, 애자일은 지속적
4. **인도**: 워터폴은 프로젝트 종료 시, 애자일은 주기적 증분

---

**7. 스크럼의 3가지 기둥을 설명하시오.**

**모범 답안**:
1. **투명성(Transparency)**: 모든 정보가 공개되고 공통 언어 사용
2. **검사(Inspection)**: 정기적으로 진행 상황과 산출물 점검
3. **적응(Adaptation)**: 문제 발견 시 즉시 조정 및 개선

---

**8. Planning Poker의 진행 순서를 단계별로 설명하시오.**

**모범 답안**:
1. PO가 스토리 설명
2. 팀원들이 질문하여 명확화
3. 각자 포커 카드 선택 (숨김)
4. 동시에 카드 공개
5. 가장 높은/낮은 추정치 제시자가 이유 설명
6. 토론 후 재추정
7. 합의 도출 시까지 반복

---

**9. 스프린트 회고에서 다루는 3가지 주요 질문은 무엇인가?**

**모범 답안**:
1. 무엇이 잘 되었는가? (What went well?)
2. 무엇을 개선할 수 있는가? (What could be improved?)
3. 다음 스프린트에서 무엇을 실행할 것인가? (What will we commit to improve?)

---

**10. 칸반과 스크럼의 차이점을 4가지 이상 설명하시오.**

**모범 답안**:
| 구분 | 스크럼 | 칸반 |
|-----|--------|------|
| 역할 | 정의됨 (PO, SM, 팀) | 기존 역할 유지 |
| 이터레이션 | 스프린트 (고정) | 없음 (연속적) |
| 변경 | 스프린트 중 제한 | 언제든 가능 |
| 메트릭 | 벨로시티 | 리드/사이클 타임 |
| 적합 | 제품 개발 | 운영, 유지보수 |
