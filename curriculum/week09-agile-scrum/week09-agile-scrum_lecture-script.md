### [00:15-01:30] 섹션 1: 애자일 기초 (75분)

#### [00:15-00:45] 1.1 애자일 탄생 배경 및 선언문 (30분)

**강의 내용**:
```
📖 애자일의 탄생

🕰️ 역사적 배경 (2001년)
"유타주 스키 리조트, 17명의 소프트웨어 개발자"
"폭포수 방법론의 한계를 느낀 전문가들"

❌ 전통적 방법론의 문제:
1. 긴 개발 기간 (1-2년)
2. 출시 시 요구사항 이미 변경됨
3. 고객은 프로젝트 종료 시에야 제품 확인
4. 변경 비용 막대
5. 실패율 높음 (50% 이상)

예시:
```
2000년 요구사항: "DVD 대여 웹사이트"
2년 후 출시: 시장은 이미 스트리밍으로 이동
결과: 쓸모없는 제품
```

💡 애자일의 핵심 아이디어:
"빠르게 만들고, 피드백 받고, 개선하자!"
"완벽한 계획보다 빠른 적응이 더 중요하다"

📜 애자일 선언문 (Agile Manifesto)

4가지 핵심 가치:

1️⃣ 프로세스와 도구보다 **개인과 상호작용**을
```
❌ 전통적: "프로세스 문서 준수가 최우선"
✅ 애자일: "팀원 간 대화와 협력이 최우선"

예: 문제 발생 시
전통적: 문서 확인 → 절차 따라 보고 → 승인 대기
애자일: 즉시 팀에게 말하고 함께 해결
```

2️⃣ 포괄적인 문서보다 **작동하는 소프트웨어**를
```
❌ 전통적: 100페이지 요구사항 명세서
✅ 애자일: 동작하는 프로토타입

💡 "문서는 필요하지만, 코드가 진실"
```

3️⃣ 계약 협상보다 **고객과의 협력**을
```
❌ 전통적: 계약서에 명시된 것만
✅ 애자일: 고객과 지속적 협업

예: 새로운 아이디어 제안
전통적: "계약 범위 아님, 변경 계약 필요"
애자일: "좋네요! 다음 스프린트에 반영합시다"
```

4️⃣ 계획을 따르기보다 **변화에 대응**을
```
❌ 전통적: 초기 계획 엄격히 준수
✅ 애자일: 계획은 가이드, 변경 환영

💡 "계획은 중요하지만, 계획대로 안 되는 게 현실"
```

🎯 12가지 원칙 (주요 5가지 강조)

1. **가치 있는 소프트웨어의 조기 및 지속적 인도**
   "2주마다 사용 가능한 기능 제공"

2. **요구사항 변경을 환영**
   "변경은 경쟁 우위 기회!"

3. **짧은 주기로 인도 (2주-2개월)**
   "빠른 피드백 루프"

4. **비즈니스와 개발자의 매일 협업**
   "고객이 팀의 일부"

5. **자기 조직화 팀**
   "팀이 스스로 결정"

💬 질문: "왜 변경을 환영할까요? 변경은 나쁜 거 아닌가요?"
→ 답: "변경 = 시장이 바뀌었다 = 배우고 있다는 증거
      고정된 요구사항 = 시장 무시 = 실패"

🎯 애자일의 핵심 개념 정리:
```
속도 > 완벽
피드백 > 예측
협업 > 계약
적응 > 계획
가치 > 문서
```
```

**교수법**:
- 역사적 맥락으로 흥미 유발
- 전통적 방법론과 대비하여 설명
- 실제 사례로 이해 증진
- 4가지 가치를 화이트보드에 크게

#### [00:45-01:15] 1.2 애자일 vs 폭포수 (30분)

**강의 내용**:
```
⚖️ 전통적(폭포수) vs 애자일 비교

화이트보드에 비교표 그리기:

구분           │ 폭포수              │ 애자일
──────────────┼─────────────────────┼──────────────────
접근 방식      │ 예측적              │ 적응적
요구사항       │ 초기 확정           │ 점진적 발견
변경           │ 통제 (비용↑)       │ 환영
인도           │ 프로젝트 종료 시    │ 2주마다
고객 참여      │ 초기/종료           │ 매일
팀 구조        │ 역할별 분업         │ 교차 기능팀
문서화         │ 상세               │ 필요한 만큼
리스크         │ 후기 집중          │ 조기 발견
적합 프로젝트  │ 요구사항 명확       │ 요구사항 불명확

💡 시각화: 개발 흐름 비교

폭포수:
```
요구사항 → 설계 → 개발 → 테스트 → 배포
(6개월)  (2개월) (8개월) (2개월) (1개월)
총 19개월, 출시 1회

문제: 19개월 후 시장이 바뀌었다면? 💥
```

애자일:
```
반복 1: 요구→설계→개발→테스트→배포 (2주)
반복 2: 요구→설계→개발→테스트→배포 (2주)
반복 3: 요구→설계→개발→테스트→배포 (2주)
...
총 19개월, 출시 40회

장점: 매번 피드백, 계속 개선 ✅
```

📊 실제 사례 비교

사례 1: 전자 상거래 웹사이트 개발

폭포수 접근:
```
1. 요구사항: 3개월 (모든 기능 정의)
2. 설계: 2개월
3. 개발: 6개월
4. 테스트: 2개월
5. 배포: 1개월
───────────────────
총: 14개월

14개월 후:
고객: "생각했던 것과 달라요. 이 기능은 필요 없고..."
PM: "이미 개발 완료했는데... 변경 비용 5천만원"
고객: "😱"
```

애자일 접근:
```
스프린트 1 (2주): 로그인 기능 → 배포
  고객: "로그인 화면 색상 변경해주세요" ✅

스프린트 2 (2주): 상품 검색 → 배포
  고객: "검색이 느려요" → 즉시 개선 ✅

스프린트 3 (2주): 장바구니 → 배포
  고객: "대신 위시리스트 먼저 만들어주세요" ✅

...
6개월 후: 핵심 기능 완성, 지속적 개선 중
고객: "우리가 원하는 게 바로 이거예요!" 🎉
```

💡 핵심 차이점 5가지

1️⃣ 리스크 시점
```
폭포수: 후반에 리스크 폭발 💣
  (테스트 시 "아, 이거 안 되네요")

애자일: 초반에 리스크 발견 🔍
  (첫 스프린트에 "이 기술 어려워요" → 빠른 대응)
```

2️⃣ 고객 만족
```
폭포수: 마지막에 만족도 확인
  → 불만족 시 수정 비용 막대

애자일: 2주마다 만족도 확인
  → 불만족 시 즉시 조정
```

3️⃣ 팀 구조
```
폭포수:
  분석가 → 설계자 → 개발자 → 테스터
  (역할 분리, 인수인계)

애자일:
  교차 기능팀 (모든 역할 함께)
  (분석도 하고, 개발도 하고, 테스트도 하고)
```

4️⃣ 성공 측정
```
폭포수: 계획 준수율
  "일정대로 완료했나?"

애자일: 고객 가치
  "고객에게 가치를 전달했나?"
```

5️⃣ 변경 대응
```
폭포수:
  변경 요청 → 변경 위원회 → 영향 분석 → 승인 → 변경
  (수주~수개월)

애자일:
  변경 요청 → 백로그에 추가 → 다음 스프린트에 반영
  (2주)
```

🎯 언제 어떤 방법론을 사용할까?

폭포수가 적합한 경우:
✅ 요구사항이 명확하고 변하지 않음
✅ 규제가 엄격 (의료, 항공)
✅ 고정가 계약
✅ 예측 가능성이 중요

예: 건물 건축, 하드웨어 제조

애자일이 적합한 경우:
✅ 요구사항이 불명확하거나 변함
✅ 혁신과 속도가 중요
✅ 고객 피드백 필수
✅ 기술 불확실성

예: 웹/앱 개발, 스타트업 제품

💡 현실:
"대부분의 IT 프로젝트는 애자일이 적합합니다!"
"최근 추세: 하이브리드 (애자일 + 전통적)"

💬 그룹 토론 (10분):
"여러분의 프로젝트는 어느 방법론이 적합할까요?"
- 소그룹 토론
- 이유 공유
```

### [01:45-03:15] 섹션 2: 스크럼 프레임워크 (90분)

#### [01:45-02:30] 2.1 스크럼 3-5-3 구조 (45분)

**강의 내용**:
```
🏉 스크럼이란?

어원: 럭비의 스크럼 (팀 전체가 함께 전진)
특징: 가볍고, 이해하기 쉽고, 마스터하기 어려움

💡 스크럼 3-5-3 구조
- 3가지 역할
- 5가지 이벤트
- 3가지 산출물

화이트보드에 큰 그림 그리기

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎭 3가지 역할

1️⃣ Product Owner (제품 책임자)

역할:
✅ 제품 비전 제시
✅ 제품 백로그 관리
✅ 우선순위 결정
✅ ROI 최대화
✅ 이해관계자 대표

특징:
- 1명만! (위원회 아님)
- 최종 결정권
- 비즈니스 이해

잘못된 PO:
❌ "모든 기능이 중요해요" (우선순위 없음)
❌ "개발팀한테 물어보세요" (결정 회피)
❌ 스프린트 중 우선순위 변경

올바른 PO:
✅ "이번 스프린트는 결제 기능에 집중"
✅ "이 기능은 ROI가 높아요"
✅ "다음 분기 비전은 이렇습니다"

2️⃣ Scrum Master (스크럼 마스터)

역할:
✅ 스크럼 프로세스 코치
✅ 장애물 제거
✅ 팀 보호 (외부 방해 차단)
✅ 퍼실리테이터
✅ 서번트 리더

특징:
- PM이 아님!
- 명령하지 않음
- 팀을 섬김

잘못된 SM:
❌ "이렇게 해라" (명령)
❌ "왜 늦어?" (추궁)
❌ 회의만 주관

올바른 SM:
✅ "무엇이 필요한가요?" (지원)
✅ "이 장애물을 제거했습니다"
✅ "팀이 스스로 결정하도록 도왔습니다"

SM의 하루:
- 장애물 제거 (서버 권한, 정보 수집)
- 회의 퍼실리테이션
- 팀 코칭 ("이렇게 하면 더 좋아요")
- 조직 변화 촉진

3️⃣ 개발 팀 (Development Team)

특징:
✅ 교차 기능팀 (개발, 디자인, 테스트 모두)
✅ 자기 조직화 (스스로 결정)
✅ 3-9명 (이상적: 7±2)
✅ 역할 구분 없음 (모두 "개발자")

권한과 책임:
- 스프린트 백로그 작성
- 작업 방법 결정
- 품질 책임
- 스프린트 목표 달성

잘못된 팀:
❌ "제 일이 아니에요" (역할 분리)
❌ "PM이 시키면 해요" (수동적)
❌ 15명 (너무 큼)

올바른 팀:
✅ "제가 도울까요?" (협업)
✅ "우리가 결정했어요" (자율)
✅ 7명 (최적)

📅 5가지 이벤트 (모두 Timebox!)

0️⃣ 스프린트 (Sprint) - 컨테이너
- 1-4주 (보통 2주)
- 고정 기간
- 완료 정의 충족한 증분 생산

1️⃣ 스프린트 계획 (Sprint Planning)

시간: 2주 스프린트 기준 4시간

Part 1: 무엇을? (What)
- PO: 우선순위 높은 백로그 제시
- 팀: 스프린트 목표 합의
- 팀: 백로그 항목 선정

Part 2: 어떻게? (How)
- 팀: 작업 분해
- 팀: 스프린트 백로그 작성
- 팀: 용량 확인

예시:
```
PO: "이번 스프린트는 결제 기능입니다"
팀: "3개 사용자 스토리 가능합니다"
  - 신용카드 결제
  - 결제 확인 이메일
  - 환불 처리

스프린트 목표: "사용자가 신용카드로 결제할 수 있다"

작업 분해:
- 결제 API 연동 (8시간)
- UI 개발 (5시간)
- 보안 검증 (3시간)
- 테스트 (4시간)
총 20시간
```

2️⃣ 일일 스크럼 (Daily Scrum)

시간: 15분 정각
장소: 같은 시간, 같은 장소
방식: 서서 (Stand-up)

3가지 질문:
1. 어제 무엇을 했나?
2. 오늘 무엇을 할 것인가?
3. 장애물은 무엇인가?

예시:
```
개발자 A: 
  어제: 로그인 API 완료
  오늘: 로그인 UI 작업
  장애물: 디자인 가이드 필요

개발자 B:
  어제: 데이터베이스 스키마 설계
  오늘: 데이터베이스 구현
  장애물: DB 서버 접근 권한 없음

SM: "디자인 가이드와 DB 권한, 제가 오늘 중 해결하겠습니다"
```

❌ 하지 말아야 할 것:
- 30분 회의로 변질
- 상세 기술 논의 (따로 하기)
- PM에게 보고하는 자리로 변질

3️⃣ 스프린트 리뷰 (Sprint Review)

시간: 2주 스프린트 기준 2시간
참석: 팀 + 이해관계자 + 고객

목적:
- 완성된 증분 시연
- 피드백 수집
- 백로그 업데이트

형식:
- 데모 (동작하는 소프트웨어)
- 질의응답
- 다음 계획 논의

예시:
```
Dev: "결제 기능을 시연하겠습니다"
      (라이브 데모)
고객: "좋네요! 그런데 할부 옵션도 추가할 수 있나요?"
PO: "백로그에 추가하겠습니다. 우선순위는 중간입니다"
```

💡 파워포인트 금지! 동작하는 제품만!

4️⃣ 스프린트 회고 (Sprint Retrospective)

시간: 2주 스프린트 기준 1.5시간
참석: 팀만 (내부 회의)

목적: 프로세스 개선

3가지 질문:
1. 잘한 점 (Keep)
2. 개선할 점 (Improve)
3. 액션 아이템 (Action)

기법:
- Start-Stop-Continue
- Mad-Sad-Glad
- 4Ls (Liked, Learned, Lacked, Longed for)

예시:
```
잘한 점:
✅ 페어 프로그래밍으로 버그 감소
✅ 매일 코드 리뷰

개선할 점:
❌ 회의가 너무 길었음
❌ 테스트 환경 불안정

액션 아이템:
🎯 회의 시간 30분으로 제한
🎯 테스트 환경 재구축 (SM이 담당)
```

💡 회고가 스크럼의 핵심!
   (지속적 개선 = 애자일의 본질)

📋 3가지 산출물

1️⃣ 제품 백로그 (Product Backlog)

내용:

우선순위 기준:

────────┼─────────────────────────────────┼────────

   4    | 사용자가 결제할 수 있다           |  13

2️⃣ 스프린트 백로그 (Sprint Backlog)

내용:

- 작업으로 분해
- 번다운 차트

예시:
```

작업:

□ 비밀번호 암호화 (3h)

□ 통합 테스트 (2h)

#### [02:30-03:15] 2.2 사용자 스토리 작성 (45분)

**강의 내용**:
```
📝 사용자 스토리 (User Story)

정의: 사용자 관점에서 기술한 기능

📌 형식:
```
As a [사용자 유형],
I want [기능],
So that [가치/이유].
```

한글:
[사용자 유형]로서,
[기능]을 원한다,
[가치/이유] 때문에.
```

예시:

좋은 사용자 스토리:
```
As a 쇼핑몰 고객,
I want 장바구니에 상품을 담고 싶다,
So that 한 번에 결제할 수 있다.
```

왜 좋은가?
✅ 사용자 명확 (쇼핑몰 고객)
✅ 기능 명확 (장바구니)
✅ 가치 명확 (한번에 결제)

나쁜 예시:
```
❌ "로그인 기능 구현"
  → 사용자 관점 아님, 기술 중심

❌ "사용자가 로그인하고, 프로필을 수정하고, 로그아웃할 수 있다"
  → 너무 큼, 여러 스토리로 분리

❌ "데이터베이스에 세션 테이블 추가"
  → 기술 구현, 사용자 가치 없음
```

🎯 INVEST 기준

좋은 사용자 스토리는:

I - Independent (독립적)
```
✅ 다른 스토리와 독립적으로 개발 가능
❌ "A 기능 완료 후에만 B 가능"
```

N - Negotiable (협상 가능)
```
✅ 세부 사항은 논의 가능
❌ 상세한 기술 스펙 (계약서 아님)
```

V - Valuable (가치 있음)
```
✅ 사용자/비즈니스에 가치 제공
❌ "코드 리팩토링" (간접적 가치)
```

E - Estimable (추정 가능)
```
✅ 팀이 노력을 추정 가능
❌ "AI로 뭔가 하기" (너무 모호)
```

S - Small (작음)
```
✅ 한 스프린트에 완료 가능
❌ "전체 관리자 기능" (너무 큼)
```

T - Testable (테스트 가능)
```
✅ 명확한 인수 기준
❌ "사용하기 쉬운 UI" (주관적)
```

📊 인수 기준 (Acceptance Criteria)

정의: 스토리가 "완료"되었다고 판단하는 기준

형식: Given-When-Then

예시:
```
사용자 스토리: 사용자가 로그인할 수 있다

인수 기준:
Given 사용자가 등록된 이메일과 비밀번호를 가지고 있을 때,
When 로그인 페이지에서 이메일과 비밀번호를 입력하고 로그인 버튼을 클릭하면,
Then 홈 페이지로 이동하고 "환영합니다, [이름]"이 표시된다.

Given 사용자가 잘못된 비밀번호를 입력했을 때,
When 로그인 버튼을 클릭하면,
Then "이메일 또는 비밀번호가 잘못되었습니다" 오류 메시지가 표시된다.

Given 사용자가 5회 연속 로그인 실패했을 때,
When 6번째 시도하면,
Then 계정이 10분간 잠기고 "너무 많은 시도. 10분 후 다시 시도하세요" 메시지가 표시된다.
```

💡 인수 기준 = 테스트 케이스

🎨 스토리 포인트

정의: 스토리의 상대적 크기/복잡도

특징:
- 시간이 아님!
- 팀 내 상대적
- 피보나치 수열 (1, 2, 3, 5, 8, 13, 21...)

예시:
```
스토리 A (포인트 3): 로그인 UI
스토리 B (포인트 5): 로그인 API + 인증
스토리 C (포인트 8): 결제 통합

해석:
- B는 A의 약 1.7배 복잡
- C는 A의 약 2.7배 복잡
```

추정 방법: Planning Poker
1. 각자 카드 선택 (숫자)
2. 동시에 공개
3. 차이 크면 논의
4. 합의할 때까지 반복

💬 질문: "왜 시간이 아니라 포인트인가요?"
→ 답: "시간은 사람마다 다름. 상대적 크기는 객관적.
      속도(Velocity)가 쌓이면 예측 가능."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📝 실습: 사용자 스토리 작성 (20분)

시나리오: "피트니스 앱 개발"

요구사항:
- 운동 기록
- 칼로리 계산
- 친구와 공유
- 목표 설정
- 통계 확인

과제:
1. 사용자 스토리 10개 작성
2. INVEST 기준 확인
3. 각 스토리에 인수 기준 2개
4. 스토리 포인트 추정

⏰ 15분 작성, 5분 발표

예시 답안:
```
1. As a 운동하는 사용자,
   I want 오늘 한 운동을 기록하고 싶다,
   So that 내 진척을 추적할 수 있다.
   
   인수 기준:
   - 운동 종류, 시간, 칼로리를 입력 가능
   - 저장 후 목록에 표시
   
   포인트: 3

2. As a 다이어트 중인 사용자,

   인수 기준:
   - 운동별 칼로리 자동 계산
   - 일일 총 칼로리 표시
   
   포인트: 5
```
```

**교수법**:
- 사용자 스토리 형식 반복 강조
- INVEST 기준 하나씩 명확히
- 실습으로 체득
- 우수 사례 공유

## 📌 강의 Tips

### 강사 준비사항
- [ ] 레고 세트 (팀당 1세트)
- [ ] 타이머 (여러 개)
- [ ] 스크럼 보드 (화이트보드 또는 벽)
- [ ] 포스트잇 (다양한 색상)
- [ ] 사용자 스토리 카드 템플릿
- [ ] Planning Poker 카드 세트

### 교수법 포인트
- 🎮 **체험 중심**: 레고 시뮬레이션으로 실전 경험
- 🔁 **반복 강조**: 스프린트 3회로 개선 체감
- 💬 **토론 유도**: 회고를 실제로 경험
- 📊 **비교 학습**: 폭포수 vs 애자일 명확히
- 🎯 **실용성**: 바로 적용 가능한 팁

### 자주 하는 질문
1. **Q: "스크럼 마스터와 PM의 차이는?"**
   A: PM은 프로젝트 책임, SM은 프로세스 코치. 역할이 다름.

2. **Q: "2주 스프린트가 너무 짧은데요?"**
   A: 처음엔 그럴 수 있음. 익숙해지면 적절. 4주까지 가능.

3. **Q: "문서를 정말 안 만드나요?"**
   A: 필요한 문서는 만듦. 과도한 문서를 피할 뿐.

4. **Q: "우리 조직은 애자일이 안 맞아요"**
   A: 처음부터 맞는 조직은 없음. 변화가 필요.

### 주의사항
- ⚠️ 레고 시뮬레이션 시간 엄수 (타이머 필수)
- ⚠️ 역할 순환하여 모두 경험하도록
- ⚠️ 피드백 시 너무 가혹하지 않게
- ⚠️ "애자일이 만능"이라는 오해 방지
