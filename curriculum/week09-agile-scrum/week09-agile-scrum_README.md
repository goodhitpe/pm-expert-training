## 📚 주요 내용

### 1. 애자일 개요

#### 1.1 애자일 선언문 (Agile Manifesto)

**4가지 핵심 가치:**
1. **프로세스와 도구보다 개인과 상호작용**을
2. **포괄적인 문서보다 작동하는 소프트웨어**를
3. **계약 협상보다 고객과의 협력**을
4. **계획을 따르기보다 변화에 대응**을

**12가지 원칙:**
- 가치 있는 소프트웨어의 조기 및 지속적 인도
- 요구사항 변경 환영
- 짧은 주기로 소프트웨어 인도
- 비즈니스와 개발자의 협업
- 동기부여된 개인 중심
- 대면 대화 선호
- 작동하는 소프트웨어가 진척의 척도
- 지속 가능한 개발 속도
- 기술적 우수성과 좋은 설계
- 단순성
- 자기 조직화 팀
- 정기적 성찰과 조정

#### 1.2 애자일 vs 전통적(폭포수) 방법론

| 구분 | 전통적 (Waterfall) | 애자일 (Agile) |
|------|-------------------|----------------|
| 접근 방식 | 예측적 (Predictive) | 적응적 (Adaptive) |
| 요구사항 | 초기에 확정 | 점진적 발견 |
| 변경 | 통제 대상 | 환영 대상 |
| 인도 | 프로젝트 종료 시 | 반복적 인도 |
| 고객 참여 | 초기/종료 시 | 지속적 |
| 팀 구조 | 역할 명확 | 교차 기능팀 |
| 문서화 | 상세 | 필요한 만큼 |

#### 1.3 애자일 방법론 종류
- **스크럼 (Scrum)**: 반복적 개발 프레임워크
- **칸반 (Kanban)**: 흐름 기반 관리
- **XP (Extreme Programming)**: 기술 실천법 중심
- **린 (Lean)**: 낭비 제거 중심
- **크리스탈 (Crystal)**: 팀 규모별 맞춤형

### 2. 스크럼 프레임워크

#### 2.1 스크럼 역할

**Product Owner (PO):**
- 제품 비전 제시
- 제품 백로그 관리
- 우선순위 결정
- ROI 최대화

**Scrum Master (SM):**
- 스크럼 프로세스 코치
- 장애물 제거
- 팀 보호
- 퍼실리테이터

**개발 팀 (Development Team):**
- 교차 기능팀 (Cross-functional)
- 자기 조직화 (Self-organizing)
- 3-9명 권장
- 스프린트 목표 달성

#### 2.2 스크럼 이벤트

**스프린트 (Sprint):**
- 1-4주 (통상 2주)
- 시간 박스 (Timebox)
- 잠재적으로 출시 가능한 증분 생산

**스프린트 계획 (Sprint Planning):**
- 스프린트 목표 설정
- 백로그 항목 선정
- 작업 분해
- 시간: 2주 스프린트 기준 4시간

**일일 스크럼 (Daily Scrum):**
- 15분 이내
- 서서 진행 (Stand-up)
- 3가지 질문:
  1. 어제 무엇을 했는가?
  2. 오늘 무엇을 할 것인가?
  3. 장애물은 무엇인가?

**스프린트 리뷰 (Sprint Review):**
- 완성된 증분 시연
- 이해관계자 피드백
- 제품 백로그 업데이트
- 시간: 2주 스프린트 기준 2시간

**스프린트 회고 (Sprint Retrospective):**
- 프로세스 개선
- 잘한 점, 개선할 점, 액션 아이템
- 시간: 2주 스프린트 기준 1.5시간

#### 2.3 스크럼 산출물

**제품 백로그 (Product Backlog):**
- 우선순위화된 요구사항 목록
- 사용자 스토리 형식
- 지속적으로 업데이트

**스프린트 백로그 (Sprint Backlog):**
- 선정된 제품 백로그 항목
- 작업 분해
- 번다운 차트로 추적

**증분 (Increment):**
- 완료 정의(DoD) 충족
- 잠재적 출시 가능
- 누적적

### 3. 사용자 스토리

#### 3.1 사용자 스토리 형식
```
As a [사용자 유형],
I want [기능],
So that [가치/이유].
```

**예시:**
- As a 쇼핑몰 고객, I want 장바구니에 상품을 담고 싶다, So that 한 번에 결제할 수 있다.

#### 3.2 INVEST 기준
- **I**ndependent: 독립적
- **N**egotiable: 협상 가능
- **V**aluable: 가치 있음
- **E**stimable: 추정 가능
- **S**mall: 작은 크기
- **T**estable: 테스트 가능

#### 3.3 스토리 포인트
- 상대적 크기 추정
- 피보나치 수열 (1, 2, 3, 5, 8, 13, 21...)
- 플래닝 포커로 추정

#### 3.4 완료 정의 (Definition of Done)
- 코드 작성 완료
- 단위 테스트 통과
- 코드 리뷰 완료
- 통합 테스트 통과
- 문서화 완료
- PO 승인

### 4. 애자일 추정 및 계획

#### 4.1 플래닝 포커 (Planning Poker)
1. PO가 스토리 설명
2. 팀원들이 카드 선택 (동시에 공개)
3. 가장 높은/낮은 추정치 제시자 설명
4. 재추정 (합의 시까지)

#### 4.2 벨로시티 (Velocity)
- 스프린트당 완료한 스토리 포인트 합계
- 3-5 스프린트 평균값 사용
- 향후 계획 수립에 활용

#### 4.3 번다운 차트 (Burndown Chart)
```
스토리 포인트
 ↑
 │\
 │ \  이상선
 │  \
 │   \_____ 실제선
 │         \
 └──────────────> 일
```

#### 4.4 릴리즈 계획
- 제품 백로그 총 스토리 포인트
- 팀 벨로시티
- 필요 스프린트 수 = 총 포인트 / 벨로시티

### 5. 칸반 (Kanban)

#### 5.1 칸반 보드
```
┌──────┬───────┬──────┬──────┐
│ TO DO│ IN    │ TEST │ DONE │
│      │PROGRESS│     │      │
├──────┼───────┼──────┼──────┤
│ 스토리A│ 스토리B│스토리C│스토리D│
│ 스토리E│       │      │스토리F│
└──────┴───────┴──────┴──────┘
```

#### 5.2 WIP (Work In Progress) 제한
- 동시 진행 작업 수 제한
- 병목 현상 방지
- 흐름 최적화

#### 5.3 칸반 vs 스크럼
| 구분 | 스크럼 | 칸반 |
|------|--------|------|
| 역할 | 정의됨 (PO, SM, 팀) | 기존 역할 유지 |
| 이터레이션 | 스프린트 (고정) | 없음 (연속적) |
| 변경 | 스프린트 중 제한 | 언제든 가능 |
| 메트릭 | 벨로시티 | 리드 타임, 사이클 타임 |

### 6. 애자일 확장

#### 6.1 SAFe (Scaled Agile Framework)
- 대규모 조직용
- 팀 → 프로그램 → 포트폴리오

#### 6.2 LeSS (Large-Scale Scrum)
- 스크럼 원칙 유지
- 여러 팀 조율

## 💡 실습 과제

### 과제 1: 사용자 스토리 작성
선정한 프로젝트의 주요 기능 10개를 사용자 스토리 형식으로 작성하고, INVEST 기준으로 검증하시오.

### 과제 2: 스프린트 계획
사용자 스토리에 스토리 포인트를 부여하고, 2주 스프린트 계획을 수립하시오 (벨로시티 20 가정).
