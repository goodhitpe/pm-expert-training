## 1️⃣ 데이터베이스(Database)란?

### 쉬운 설명
**데이터베이스**는 체계적으로 정리된 데이터의 저장 공간입니다. 
엑셀처럼 표 형태로 데이터를 저장하지만, 훨씬 더 강력하고 안전하게 관리할 수 있습니다.

### 실생활 비유
도서관을 생각해보세요:
- **도서관** = 데이터베이스
- **책장(카테고리별)** = 테이블
- **각각의 책** = 데이터 레코드
- **사서** = 데이터베이스 관리 시스템(DBMS)

도서관에서 사서가 책을 체계적으로 분류하고 찾아주듯이, 
데이터베이스 시스템이 데이터를 정리하고 빠르게 찾아줍니다.

### PM이 알아야 하는 이유
- 모든 서비스의 핵심 데이터를 저장하는 곳
- 데이터 손실은 곧 사업 손실 (백업 전략 필수)
- 데이터베이스 선택이 서비스 성능과 비용에 큰 영향
- 개인정보 보호법 준수 시 데이터베이스 보안 중요

---

## 2️⃣ 테이블(Table)과 레코드(Record)

### 정의
- **테이블**: 엑셀의 시트처럼 데이터를 행과 열로 정리한 구조
- **레코드(행)**: 하나의 완전한 데이터 항목
- **컬럼(열)**: 데이터의 속성

### 예시: 회원 테이블
```
┌────┬──────────┬─────────────────┬──────┬────────────┐
│ ID │ 이름     │ 이메일           │ 나이  │ 가입일      │
├────┼──────────┼─────────────────┼──────┼────────────┤
│ 1  │ 홍길동   │ hong@email.com   │ 30   │ 2024-01-15 │
│ 2  │ 김철수   │ kim@email.com    │ 25   │ 2024-02-20 │
│ 3  │ 이영희   │ lee@email.com    │ 28   │ 2024-03-10 │
└────┴──────────┴─────────────────┴──────┴────────────┘
```

### 실생활 비유
학생 명단을 생각해보세요:
- **전체 명단** = 테이블
- **한 학생의 정보** = 레코드(행)
- **이름, 학번, 전화번호** = 컬럼(열)

### PM 관점
- 서비스에 필요한 데이터 구조 설계 시 테이블 구조 이해 필요
- 테이블이 많을수록 관계가 복잡해져 개발 시간 증가
- 사용자 데이터, 주문 데이터, 상품 데이터 등 각각의 테이블로 관리

## 3️⃣ SQL (Structured Query Language)

### 정의
데이터베이스에게 명령을 내리는 언어입니다.
"이 조건에 맞는 데이터를 찾아줘", "이 데이터를 저장해줘"라고 요청하는 것입니다.

### 기본 명령어 (PM이 알아두면 좋은 것들)

**데이터 조회 (SELECT):**
```sql
-- 모든 회원 조회
SELECT * FROM users;

-- 30세 이상 회원만 조회
SELECT * FROM users WHERE age >= 30;
```

**데이터 추가 (INSERT):**
```sql
-- 새 회원 추가
INSERT INTO users (name, email, age) 
VALUES ('박민수', 'park@email.com', 32);
```

**데이터 수정 (UPDATE):**
```sql
-- 회원 정보 수정
UPDATE users SET age = 31 WHERE name = '홍길동';
```

**데이터 삭제 (DELETE):**
```sql
-- 회원 탈퇴
DELETE FROM users WHERE id = 1;
```

### 실생활 비유
식당 주문과 비슷합니다:
- **"김치찌개 하나요!"** = `INSERT` (주문 추가)
- **"주문 내역 보여주세요"** = `SELECT` (조회)
- **"찌개를 된장찌개로 변경해주세요"** = `UPDATE` (수정)
- **"주문 취소요"** = `DELETE` (삭제)

### PM이 알아야 하는 이유
- 개발자가 "간단한 SELECT 쿼리"라고 할 때 의미 파악
- 데이터 조회 요청 시 SQL로 표현하면 정확한 소통
- 복잡한 쿼리는 성능에 영향 (개발 일정에 반영)

## 4️⃣ 기본 키(Primary Key)와 외래 키(Foreign Key)

### 기본 키 (Primary Key)
각 레코드를 유일하게 식별하는 ID입니다.

**예시:**
- 주민등록번호 = 사람을 구분하는 기본 키
- 회원 ID = 회원 테이블의 기본 키
- 주문번호 = 주문 테이블의 기본 키

### 외래 키 (Foreign Key)
다른 테이블을 참조하는 키입니다. 테이블 간의 관계를 만듭니다.

### 실생활 비유
학교 시스템을 생각해보세요:
```
학생 테이블:
┌─────────┬──────────┬─────────┐
│ 학번(PK)│ 이름     │ 학과코드 │
├─────────┼──────────┼─────────┤
│ 2024001 │ 홍길동   │ CS      │
│ 2024002 │ 김철수   │ BA      │
└─────────┴──────────┴─────────┘

학과 테이블:
┌──────────┬──────────┐
│ 학과코드(PK)│ 학과명  │
├──────────┼──────────┤
│ CS       │ 컴퓨터   │
│ BA       │ 경영     │
└──────────┴──────────┘
```
학생 테이블의 "학과코드"는 학과 테이블을 가리키는 외래 키입니다.

### PM 관점
- 테이블 간 관계가 복잡할수록 데이터 정합성 관리 중요
- 외래 키 설정 시 데이터 삭제/수정 시 영향 범위 고려 필요
- ER 다이어그램에서 테이블 관계를 시각적으로 확인

---

## 5️⃣ 관계형 데이터베이스 (RDBMS)

### 정의
테이블 간의 관계를 기반으로 데이터를 저장하고 관리하는 데이터베이스입니다.

### 대표적인 RDBMS
- **MySQL**: 가장 널리 사용, 오픈소스, 웹 서비스에 적합
- **PostgreSQL**: 오픈소스, 복잡한 쿼리와 대용량 처리에 강함
- **Oracle**: 엔터프라이즈급, 고가, 대기업에서 주로 사용
- **SQL Server**: Microsoft 제품, Windows 환경에 적합

### 실생활 비유
백화점의 통합 관리 시스템:
- **회원 정보** = 회원 테이블
- **구매 내역** = 주문 테이블
- **상품 정보** = 상품 테이블
- 이 모든 것이 "회원 ID"로 연결되어 있습니다.

### PM이 알아야 하는 이유
- 프로젝트 초기 데이터베이스 선택 시 의사결정 참여
- 라이선스 비용 차이 큰 편 (Oracle은 고가, MySQL/PostgreSQL은 무료)
- RDBMS는 정형화된 데이터, 트랜잭션이 중요한 서비스에 적합

---

## 6️⃣ 트랜잭션(Transaction)

### 정의
여러 작업을 하나의 논리적 단위로 묶은 것입니다.
**"모두 성공"하거나 **"모두 실패"**해야 합니다 (중간 상태 없음).

### 실생활 비유
은행 송금을 생각해보세요:
```
1. A의 계좌에서 10만원 출금
2. B의 계좌에 10만원 입금
```

만약 1번은 성공했는데 2번이 실패하면? → 돈이 사라집니다! 😱

트랜잭션은 이를 방지합니다:
- 1번과 2번 모두 성공 → 완료(Commit)
- 하나라도 실패 → 모두 취소(Rollback)

### ACID 특성 (간단히)
- **Atomicity (원자성)**: 전부 성공 or 전부 실패
- **Consistency (일관성)**: 데이터가 항상 일관된 상태 유지
- **Isolation (격리성)**: 동시 실행되는 작업들이 서로 영향 없음
- **Durability (지속성)**: 성공하면 영구적으로 저장

### PM이 알아야 하는 이유
- 결제, 예약, 송금 등 중요한 기능은 트랜잭션 필수
- 트랜잭션이 많으면 성능 저하 가능 (대규모 트래픽 시 고려)
- "트랜잭션 롤백"은 오류 복구 메커니즘

## 7️⃣ 인덱스(Index)

### 정의
데이터를 빠르게 찾기 위한 "색인"입니다.
책 뒤의 찾아보기와 같은 역할입니다.

### 실생활 비유
백과사전에서 "데이터베이스"라는 단어를 찾는다면:

**인덱스 없을 때:**
- 첫 페이지부터 하나씩 읽으며 찾기 → 느림 😰

**인덱스 있을 때:**
- 뒤의 "찾아보기"에서 "데이터베이스 → 123페이지" 확인 → 빠름 😊

### 장단점

| 장점 | 단점 |
|------|------|
| ✅ 조회 속도 엄청 빠름 | ❌ 저장 공간 더 필요 |
| ✅ 검색 성능 향상 | ❌ 데이터 추가/수정 시 느려짐 |

### PM 관점
- 자주 검색하는 컬럼에 인덱스 설정 (예: 회원 이메일, 상품명)
- 인덱스가 많으면 쓰기 성능 저하 (트레이드오프)
- "인덱스 최적화"는 성능 개선의 중요한 방법

## 8️⃣ NoSQL 데이터베이스

### 정의
"Not Only SQL" - 관계형이 아닌 다양한 형태의 데이터베이스입니다.
테이블 구조가 고정되어 있지 않아 유연합니다.

### RDBMS vs NoSQL

| 구분 | RDBMS | NoSQL |
|------|-------|-------|
| **데이터 구조** | 고정된 테이블 | 유연 (JSON 등) |
| **확장** | 서버 성능 업그레이드 | 서버 대수 증가 |
| **복잡한 관계** | 잘 지원 | 제한적 |
| **속도** | 보통 | 빠름 (단순 조회) |
| **적합한 경우** | 은행, 회계, 쇼핑몰 | SNS, 로그, 캐시 |

### 대표적인 NoSQL
- **MongoDB**: 문서(JSON) 형태 저장, 유연한 스키마
- **Redis**: 메모리에 저장, 초고속, 캐싱에 적합
- **Cassandra**: 대용량 분산 처리

### 실생활 비유

**RDBMS = 정리된 서류 캐비닛**
- 모든 서류가 정해진 양식에 맞춰 정리됨
- 찾기 쉽지만 새 양식 추가는 번거로움

**NoSQL = 유연한 노트**
- 각 페이지마다 다른 형식으로 자유롭게 기록
- 빠르게 추가하고 변경 가능
- 복잡한 연결 관계는 직접 관리 필요

### PM이 알아야 하는 이유
- 프로젝트 특성에 따라 RDBMS vs NoSQL 선택
- SNS 피드, 실시간 채팅 → NoSQL이 적합
- 전자상거래, 재무 → RDBMS가 적합
- 두 가지를 함께 사용하는 경우도 많음 (하이브리드)

## 9️⃣ 데이터 정규화(Normalization)

### 정의
데이터 중복을 최소화하고 데이터 일관성을 높이는 설계 기법입니다.

### 왜 필요한가?
**비정규화된 테이블 (나쁜 예):**
```
주문 테이블:
┌────┬──────┬──────────┬─────────────────┬──────┐
│ ID │ 고객 │ 상품명   │ 고객 이메일     │ 가격 │
├────┼──────┼──────────┼─────────────────┼──────┤
│ 1  │ 홍길동│ 노트북  │ hong@email.com  │ 100만 │
│ 2  │ 홍길동│ 마우스  │ hong@email.com  │ 3만   │
└────┴──────┴──────────┴─────────────────┴──────┘
```
문제: 홍길동의 이메일이 중복 저장됨. 이메일 변경 시 모든 행을 수정해야 함.

**정규화된 테이블 (좋은 예):**
```
고객 테이블:
┌────┬──────┬─────────────────┐
│ ID │ 이름 │ 이메일           │
├────┼──────┼─────────────────┤
│ 1  │ 홍길동│ hong@email.com │
└────┴──────┴─────────────────┘

주문 테이블:
┌────┬────────┬──────────┬──────┐
│ ID │ 고객ID │ 상품명   │ 가격 │
├────┼────────┼──────────┼──────┤
│ 1  │ 1      │ 노트북   │ 100만│
│ 2  │ 1      │ 마우스   │ 3만  │
└────┴────────┴──────────┴──────┘
```

### 실생활 비유
수첩에 친구 전화번호를 적는다면:
- **나쁜 방법**: 매번 만난 날짜와 함께 전화번호 적기 → 번호 바뀌면 전부 수정
- **좋은 방법**: 전화번호부에 한 번만 적고, 날짜에는 이름만 적기

### PM 관점
- 정규화는 데이터 일관성 향상, 저장 공간 절약
- 과도한 정규화는 조회 시 테이블 여러 개 조인 → 성능 저하
- 서비스 특성에 따라 적절한 수준의 정규화 필요

## 🔟 데이터 백업과 복구

### 정의
데이터 손실을 방지하기 위해 정기적으로 복사본을 만드는 것입니다.

### 백업 전략
- **전체 백업**: 모든 데이터를 복사 (시간 오래 걸림, 안전)
- **증분 백업**: 변경된 부분만 복사 (빠름, 복구 시 여러 파일 필요)

### 실생활 비유
- **휴대폰 데이터 백업**: 사진, 연락처를 클라우드에 저장
- **문서 버전 관리**: "보고서_최종.docx", "보고서_최종_진짜최종.docx"

### 중요한 개념
- **RPO (Recovery Point Objective)**: 얼마나 최근 데이터까지 복구할 것인가
  - 예: 일일 백업 → 최대 24시간 데이터 손실 가능
- **RTO (Recovery Time Objective)**: 복구에 걸리는 시간
  - 예: 서비스 중단 1시간 이내 복구

### PM이 알아야 하는 이유
- 백업 전략은 사업 연속성 계획(BCP)의 핵심
- 백업 비용 vs 데이터 손실 리스크 트레이드오프
- 정기적인 복구 테스트 필요 (백업만 하고 복구 안 되면 무용지물)
- 개인정보 백업 시 보안 규정 준수 필요

### 자주 묻는 질문 (FAQ)

**Q1: SQL을 직접 작성할 줄 알아야 하나요?**  
A: 아니요! PM은 SQL을 읽고 이해하는 수준이면 충분합니다. 복잡한 쿼리는 개발자의 영역입니다.

**Q2: RDBMS와 NoSQL 중 어떤 게 더 좋나요?**  
A: 정답은 없습니다! 서비스 특성에 따라 다릅니다. 은행 시스템은 RDBMS, 실시간 채팅은 NoSQL이 적합합니다.

**Q3: 데이터베이스 관리는 누가 하나요?**  
A: DBA(Database Administrator)가 전문적으로 관리하지만, PM도 기본 개념을 알아야 프로젝트 관리가 수월합니다.

**Q4: 클라우드 데이터베이스는 뭔가요?**  
A: AWS RDS, Azure SQL 같은 서비스로, 서버 관리 없이 데이터베이스 사용 가능합니다. Week 15 클라우드 수업에서 배웁니다.
