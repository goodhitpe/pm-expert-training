### [00:15-01:30] 섹션 1: 자원 관리 기초 (75분)

#### [00:15-00:45] 1.1 자원 관리 프로세스 (30분)

**강의 내용**:
```
👥 자원이란?

4가지 자원 유형:
1️⃣ 인적 자원: 팀원, 전문가
2️⃣ 물적 자원: 장비, 서버, 사무실
3️⃣ 금전적 자원: 예산
4️⃣ 정보 자원: 지식, 데이터, 문서

💡 인적 자원이 가장 중요하고 관리가 어렵습니다!

🔄 자원 관리 6단계

1️⃣ 자원 관리 계획
"어떤 자원이 필요한가?"
- 역할 정의
- 필요 기술 식별
- 조직 구조 설계

2️⃣ 활동 자원 산정
"각 활동에 얼마나 필요한가?"
- WBS 활동별 자원 매핑
- 자원 가용성 확인
- 자원 캘린더 작성

예시:
```
활동: API 개발
필요 자원:
- 백엔드 개발자 2명 × 3주
- 서버 4대 × 3주
- 데이터베이스 라이선스 1개
```

3️⃣ 팀 확보
"어디서 구하는가?"
- 내부: 기존 팀원, 타 부서 협조
- 외부: 채용, 외주, 파견
- 협상 필요

💬 질문: "좋은 인재를 확보하기 어려우면?"
→ 교육, 멘토링, 외부 전문가 컨설팅

4️⃣ 팀 개발
"어떻게 성장시키는가?"
- 팀 빌딩 활동
- 기술 교육
- 멘토링, 코칭
- 협업 문화 조성

💡 Tuckman 팀 개발 5단계:
```
형성(Forming) → 혼란(Storming) → 규범(Norming) 
→ 수행(Performing) → 해산(Adjourning)
```

각 단계 설명:

1. 형성 (Forming): "처음 만났어요"
   - 서로 알아가기
   - 조심스러움
   - 역할 불명확
   → PM 역할: 명확한 목표 제시, 규칙 설정

2. 혼란 (Storming): "갈등 발생!"
   - 의견 충돌
   - 역할 경쟁
   - 불만 표출
   → PM 역할: 갈등 중재, 역할 명확화
   💡 이 단계를 빨리 넘어가야 성과 나옴

3. 규범 (Norming): "이제 적응됐어요"
   - 협력 시작
   - 신뢰 구축
   - 규칙 확립
   → PM 역할: 긍정적 문화 강화

4. 수행 (Performing): "최고 성과!"
   - 자율적 업무
   - 높은 생산성
   - 시너지 효과
   → PM 역할: 지원과 격려

5. 해산 (Adjourning): "안녕..."
   - 프로젝트 종료
   - 아쉬움
   → PM 역할: 성과 인정, 네트워크 유지

5️⃣ 팀 관리
"어떻게 이끄는가?"
- 동기 부여
- 성과 관리
- 갈등 해결
- 커뮤니케이션

6️⃣ 자원 통제
"효율적으로 쓰고 있는가?"
- 활용도 모니터링
- 재배치
- 평준화

💰 자원 제약의 현실
"항상 자원은 부족합니다!"
- 인재 쟁탈전
- 예산 제약
- 시설 공유

→ 효율적 배분과 협상이 중요
```

**교수법**:
- 6단계를 순차적으로 설명
- Tuckman 모델은 실제 팀 사례로
- 화이트보드에 프로세스 흐름도

#### [00:45-01:15] 1.2 RACI 매트릭스 (30분)

**강의 내용**:
```
📊 RACI 매트릭스: 역할과 책임 명확화

RACI란?
- R (Responsible): 실행 책임 (작업 수행자)
- A (Accountable): 최종 책임 (승인자, 1명만!)
- C (Consulted): 사전 협의 (전문가 의견)
- I (Informed): 사후 통보 (정보 수신)

💡 핵심 규칙:
1. 모든 활동에 R이 최소 1명
2. 모든 활동에 A는 정확히 1명만!
3. C와 I는 필요에 따라

예시 1: 웹사이트 개발 프로젝트

활동          │ PM │개발자│디자이너│ QA │고객│CTO
─────────────┼────┼──────┼────────┼────┼────┼────
요구사항 정의 │ A  │  C   │   C    │ I  │ R  │ I
UI/UX 설계   │ A  │  C   │   R    │ I  │ C  │ I
개발         │ A  │  R   │   I    │ I  │ I  │ I
코드 리뷰    │ I  │  R   │   I    │ I  │ I  │ A
테스트       │ A  │  C   │   I    │ R  │ I  │ I
배포         │ A  │  R   │   I    │ C  │ I  │ C

해석:
- 요구사항 정의: 고객이 작성(R), PM이 승인(A)
- 개발: 개발자가 수행(R), PM이 최종 책임(A)
- 코드 리뷰: 개발자가 수행(R), CTO가 승인(A)

💬 질문: "왜 A는 1명만일까요?"
→ 답: 책임 분산 방지, 명확한 의사결정

❌ 흔한 실수:

실수 1: A가 여러 명
```
활동: 배포 결정
A: PM, CTO, 개발 팀장

문제: "누가 최종 결정하나요?"
→ 의사결정 지연, 책임 회피
```

실수 2: R이 없음
```
활동: 보안 테스트
A: PM
C: 보안 팀
I: 개발자

문제: "누가 실제로 하나요?"
→ 작업 누락
```

실수 3: C가 너무 많음
```
활동: UI 변경
C: 디자이너, 개발자, QA, 마케팅, 고객, 경영진

문제: 의견 수렴에 너무 많은 시간
→ 일정 지연
```

✅ RACI 작성 팁:
1. 프로젝트 초기에 작성
2. 팀 전체와 합의
3. 주요 활동 위주 (너무 상세 X)
4. 정기적으로 업데이트
5. 갈등 발생 시 RACI 참조

🎯 실무 적용:
"킥오프 미팅에서 RACI 차트 공유하세요.
'이 작업은 누가 하나요?'라는 질문이 사라집니다!"

예시 2: 갈등 상황 해결

상황:
```
개발자: "이 버그는 제 책임이 아닙니다. 디자인 문제예요!"
디자이너: "아니요, 개발 구현 문제입니다!"
```

해결:
RACI 확인 → 버그 수정 R: 개발자, A: PM
→ "개발자가 수정하되, 디자이너와 협의(C)"

💡 결론:
"RACI는 갈등 예방 도구입니다.
역할이 명확하면 갈등의 80%는 사라집니다."
```

**교수법**:
- RACI 매트릭스 표를 화이트보드에
- 실제 프로젝트 예시 풍부하게
- 실수 사례로 중요성 강조

#### [01:15-01:30] 1.3 RACI 실습 (15분)

**활동 내용**:
```
📝 개별 실습: RACI 매트릭스 작성

시나리오:
"모바일 쇼핑몰 앱 개발 프로젝트"

팀 구성:
- PM (당신)
- 프론트엔드 개발자 2명
- 백엔드 개발자 2명
- 디자이너 1명
- QA 엔지니어 1명
- 마케팅 담당자 1명
- 고객 (의뢰사)

주요 활동:
1. 요구사항 수집
2. 화면 설계 (Wireframe)
3. UI 디자인
4. 프론트엔드 개발
5. 백엔드 API 개발
6. 결제 모듈 통합
7. 테스트
8. 마케팅 자료 준비
9. 앱스토어 배포
10. 사용자 교육

과제:
RACI 매트릭스 완성하기
- 각 활동별 R, A, C, I 배정
- A는 반드시 1명만
- 모든 활동에 R 존재

⏰ 시간: 10분 작성, 5분 발표

💡 강사 순회:
- A가 여러 명인지 체크
- R이 빠진 활동 확인
- 현실적인지 피드백

📢 발표:
- 2명 발표
- 다른 접근 방식 비교
- 강사 피드백

예시 답안:
활동           │PM│FE│BE│디자이너│QA│마케팅│고객
──────────────┼──┼──┼──┼────────┼──┼──────┼────
요구사항 수집  │A │C │C │  C     │I │  C   │ R
화면 설계      │A │C │I │  R     │I │  I   │ C
UI 디자인     │A │C │I │  R     │I │  C   │ C
프론트 개발    │A │R │C │  C     │I │  I   │ I
백엔드 개발    │A │C │R │  I     │I │  I   │ I
결제 통합      │A │C │R │  I     │C │  I   │ C
테스트        │A │C │C │  I     │R │  I   │ I
마케팅 자료    │A │I │I │  C     │I │  R   │ C
앱스토어 배포  │A │R │C │  I     │C │  I   │ I
사용자 교육    │A │I │I │  I     │I │  R   │ C
```

### [01:45-03:00] 섹션 2: 팀 관리 (75분)

#### [01:45-02:15] 2.1 동기 부여 및 갈등 관리 (30분)

**강의 내용**:
```
💪 동기 부여 이론

1️⃣ 매슬로우 욕구 5단계

피라미드 그리기:
```
        ▲ 5. 자아실현 (성장, 잠재력 실현)
       ▲▲ 4. 존중 욕구 (인정, 성취)
      ▲▲▲ 3. 사회적 욕구 (소속, 애정)
     ▲▲▲▲ 2. 안전 욕구 (고용, 건강)
    ▲▲▲▲▲ 1. 생리적 욕구 (식사, 수면)
```

PM 적용:
- 1-2단계: 적절한 급여, 안정적 고용
- 3단계: 팀워크, 소속감
- 4단계: 공개 칭찬, 성과 인정
- 5단계: 도전적 업무, 성장 기회

예시:
```
상황: 주니어 개발자가 의욕 저하
분석: 4단계 (존중) 욕구 미충족
해결: 
✅ 코드 리뷰에서 잘한 점 구체적 칭찬
✅ 팀 회의에서 기여 공개 인정
✅ "당신 덕분에 일정 단축되었습니다"
```

2️⃣ 허츠버그 2요인 이론

위생 요인 (불만족 방지):
❌ 부족하면 불만
✅ 충족해도 만족 증가 X
- 급여, 근무 환경, 복지
- 회사 정책, 관리 방식

동기 요인 (만족 증진):
✅ 있으면 만족 증가
❌ 없어도 불만 증가 X
- 성취감, 인정, 책임
- 성장, 일의 의미

💡 PM의 실수:
"급여만 올리면 동기 부여될 거야"
→ 일시적 효과만 있음
→ 성취감과 인정이 더 중요!

3️⃣ 맥그리거 X-Y 이론

X 이론: "사람은 게으르다"
- 엄격한 통제 필요
- 상벌로 관리
- 미시 관리 (Micromanagement)

Y 이론: "사람은 자기주도적이다"
- 자율성 부여
- 권한 위임
- 신뢰 기반 관리

💬 질문: "어느 이론이 맞나요?"
→ 답: 상황과 사람에 따라 다름
→ 하지만 Y 이론이 현대적, 효과적

🎯 실무 적용:
초보 팀원: X 이론 (명확한 지시)
숙련 팀원: Y 이론 (자율성)

🔥 갈등 관리

갈등의 5가지 원인:
1. 자원 경쟁 (같은 서버를 여러 팀이 쓰고 싶음)
2. 일정 압박 (더 빨리 vs 더 꼼꼼하게)
3. 우선순위 차이 (버그 수정 vs 신기능)
4. 의사소통 문제 (오해, 정보 부족)
5. 성격 차이 (업무 스타일 충돌)

갈등 해결 5가지 기법:

1️⃣ 협력/문제 해결 (Collaborate)
- Win-Win
- 양측 모두 만족
- 시간 필요
- 🏆 가장 효과적!

예시:
```
개발자: "테스트 작성 시간이 없어요"
QA: "테스트 없이는 배포 불가능합니다"

협력 해결:
"테스트 자동화 도구를 도입합시다.
개발자는 단위 테스트만, QA는 통합 테스트를 담당."
→ 양쪽 모두 만족
```

2️⃣ 타협 (Compromise)
- 부분 Win-Win
- 각자 조금씩 양보
- 빠른 해결

예시:
```
마케팅: "기능 20개 필요"
개발: "10개만 가능"

타협: 15개 개발
→ 양쪽 모두 약간 불만이지만 수용
```

3️⃣ 강제/지시 (Force)
- Win-Lose
- PM이 결정
- 긴급 상황

예시:
```
디자이너: "디자인 변경 필요"
개발자: "일정상 불가능"

강제: "PM이 결정: 디자인 변경 없이 진행"
→ 디자이너 불만, 하지만 일정 준수
```

💡 자주 쓰면 팀 사기 저하!
```

### 자주 하는 질문
1. **Q: "갈등이 무서워요. 어떻게 극복하나요?"**
   A: 갈등은 자연스러운 것. 조기 개입과 연습이 답.

2. **Q: "RACI에서 A가 꼭 1명이어야 하나요?"**
   A: 네, 책임 분산 방지. 공동 책임은 무책임.

3. **Q: "팀원이 동기 부여가 안 되면?"**
   A: 1:1 면담으로 진짜 원인 파악. 급여 외 요인 고려.

4. **Q: "고정가와 T&M 중 어느 게 나은가요?"**
   A: 상황에 따라. 범위 명확하면 고정가, 불명확하면 T&M.
