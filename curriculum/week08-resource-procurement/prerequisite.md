> **학습 시간**: 25-30분

## 📚 이 주차를 위한 준비

Week 8에서는 프로젝트 팀 구성, 자원 배분, 조달 프로세스, 갈등 해결을 다룹니다. 
이러한 개념들을 이해하기 위해서는 조직 내 역할과 책임, 그리고 사람 간의 협력과 협상의 기본 원리를 먼저 익혀두는 것이 중요합니다.

## 1️⃣ 자원(Resource)이란?

### 정의
프로젝트를 수행하기 위해 필요한 **모든 것**

### 4가지 유형

#### 인적 자원 (Human Resources)
**사람**

**예시:**
- 프로젝트 팀원
- 개발자, 디자이너
- 외부 전문가

#### 물적 자원 (Physical Resources)
**물건, 장비**

**예시:**
- 컴퓨터, 서버
- 사무실 공간
- 소프트웨어 라이선스

#### 금전적 자원 (Financial Resources)
**돈, 예산**

**예시:**
- 프로젝트 예산
- 예비비
- 운영비

#### 정보 자원 (Information Resources)
**지식, 데이터**

**예시:**
- 과거 프로젝트 문서
- 기술 문서
- 전문 지식

### 실생활 비유
```
🎓 대학 축제 준비:
- 인적: 학생회 임원, 자원봉사자
- 물적: 무대, 음향 장비, 의자
- 금전적: 학생회비 예산 500만원
- 정보: 작년 축제 결과 보고서
```

## 2️⃣ RACI 매트릭스

### 정의
**누가 무엇을 하는지** 명확히 정의하는 표

### 4가지 역할

#### R - Responsible (실행 책임)
**"실제로 일하는 사람"**

**예시:**
- 보고서 작성하는 사람
- 코드 개발하는 사람

#### A - Accountable (최종 책임)
**"결과에 대해 책임지는 사람"** (반드시 1명만!)

**예시:**
- 프로젝트 매니저
- 팀 리더

#### C - Consulted (사전 협의)
**"의견을 물어봐야 하는 사람"** (양방향 소통)

**예시:**
- 기술 전문가
- 법무팀

#### I - Informed (사후 통보)
**"결과만 알려주면 되는 사람"** (일방향 소통)

**예시:**
- 상급자
- 관련 부서

### 실생활 비유
```
🍕 가족 피자 주문:

작업: 피자 주문하기
- R (실행): 막내 (전화로 주문)
- A (최종 책임): 엄마 (메뉴 결정, 결제)
- C (사전 협의): 가족 전체 (토핑 의견)
- I (사후 통보): 할머니 (배달 왔다고 알림)
```

### PM 활용 예시
```
작업: 웹사이트 디자인

작업        │PM│개발자│디자이너│고객
───────────┼──┼─────┼────────┼────
요구사항    │A │  C   │   I    │ R
디자인      │A │  I   │   R    │ C
개발        │A │  R   │   C    │ I
테스트      │A │  R   │   C    │ I
```

## 3️⃣ 팀 개발 5단계 (Tuckman 모델)

### 1단계: 형성 (Forming)
**"서로 알아가는 단계"**

**특징:**
- 어색함, 조심스러움
- 역할 불명확
- PM 의존도 높음

**실생활 비유:**
```
첫 MT (엠티):
- 어색하게 인사
- "이름이 뭐예요?"
- 조용히 앉아있음
```

### 2단계: 혼란 (Storming)
**"갈등이 생기는 단계"**

**특징:**
- 의견 충돌
- 역할 경쟁
- 긴장감 높음

**실생활 비유:**
```
조별 과제 2주차:
- "제가 리더 하겠습니다"
- "그 방법은 비효율적이에요"
- 의견 대립
```

### 3단계: 규범 (Norming)
**"규칙을 만드는 단계"**

**특징:**
- 역할 확정
- 규칙 확립
- 협력 시작

**실생활 비유:**
```
조별 과제 4주차:
- 역할 분담 완료
- 회의 규칙 확립
- 협력 시작
```

### 4단계: 수행 (Performing)
**"높은 성과를 내는 단계"**

**특징:**
- 효율적 협업
- 자율적 문제 해결
- 높은 생산성

**실생활 비유:**
```
조별 과제 마무리:
- 각자 맡은 일 완벽히 수행
- 문제 발생 시 빠른 해결
- 좋은 결과물 완성
```

### 5단계: 해산 (Adjourning)
**"프로젝트 종료"**

**특징:**
- 아쉬움
- 회고
- 해체

**실생활 비유:**
```
조별 과제 제출 후:
- 회식
- "수고했어요"
- 연락처 교환
```

## 4️⃣ 갈등 해결 5가지 기법

### 1. 협력/문제 해결 (Collaborate)
**Win-Win** - 양측 모두 만족

**예시:**
```
갈등: 개발자는 3주 필요, 고객은 2주 요구
해결: 핵심 기능만 2주에 완성, 추가 기능은 다음 버전
결과: 개발자도 만족, 고객도 만족 ✅
```

**실생활 비유:**
```
🍕 피자 토핑 선택:
- A: 페퍼로니 원함
- B: 야채 원함
- 해결: 반반 피자 주문
```

### 2. 타협 (Compromise)
**부분 Win-Win** - 서로 조금씩 양보

**예시:**
```
갈등: 예산 1억 vs 8천만원
해결: 9천만원으로 합의
결과: 둘 다 부분 만족
```

### 3. 강제/지시 (Force)
**Win-Lose** - PM이 결정

**예시:**
```
갈등: 기술 A vs 기술 B
해결: PM이 "기술 A 사용"으로 결정
결과: 한 쪽 만족, 다른 쪽 불만
```

**언제 사용:** 긴급 상황, 안전 문제

### 4. 완화/수용 (Smooth)
**일시적 해결** - 공통점 강조

**예시:**
```
갈등: 디자인 의견 차이
해결: "우선 목표는 프로젝트 성공이니 나중에 다시 논의"
결과: 당장은 넘어감
```

### 5. 회피 (Withdraw)
**해결 안 함** - 미루거나 무시

**예시:**
```
갈등: 사소한 의견 차이
해결: "이건 나중에 다시 얘기해요"
```

**언제 사용:** 중요하지 않은 갈등, 감정 가라앉을 시간 필요

### 효과적인 순서
**협력 > 타협 > 강제 > 완화 > 회피**

## 5️⃣ 조달(Procurement)이란?

### 정의
프로젝트에 필요한 것을 **외부에서 구매하거나 계약**하는 것

### Make vs Buy 결정

#### Make (직접 수행)
**장점:**
- 통제 용이
- 노하우 축적

**단점:**
- 시간 소요
- 자원 필요

#### Buy (외주)
**장점:**
- 시간 절약
- 전문성 활용

**단점:**
- 비용 증가
- 통제 어려움

### 실생활 비유
```
🎂 생일 케이크:

Make (직접 만들기):
- 장점: 원하는 대로, 저렴
- 단점: 시간 5시간, 실패 위험

Buy (케이크 가게):
- 장점: 시간 30분, 전문적
- 단점: 비용 5만원
```

---

## 6️⃣ 계약 유형

### 고정가 계약 (Fixed Price)
**"총액 고정"**

**예시:**
```
웹사이트 개발: 3,000만원 (고정)
→ 작업이 더 들어도 3,000만원만 지급
```

**장점 (구매자):**
- 예산 확실
- 리스크 낮음

**단점 (판매자):**
- 작업 늘면 손해

### 원가 정산 계약 (Cost Reimbursable)
**"실제 비용 + 수수료"**

**예시:**
```
연구 개발: 실제 비용 + 10% 수수료
→ 3,000만원 들면 3,300만원 지급
```

**장점 (판매자):**
- 리스크 낮음
- 실비 보장

**단점 (구매자):**
- 예산 불확실
- 비용 증가 위험

### T&M (Time and Material)
**"시간 × 단가 + 자재비"**

**예시:**
```
컨설팅: 시간당 10만원 × 실제 작업 시간
→ 300시간 작업 시 3,000만원
```

**장점:**
- 유연함
- 범위 불명확할 때 유용

**단점:**
- 예산 관리 어려움

### 실생활 비유
```
🚗 자동차 수리:

고정가: "엔진 수리 100만원"
→ 추가 부품 필요해도 100만원

원가 정산: "실제 부품비 + 공임 20%"
→ 부품 80만원이면 96만원

T&M: "시간당 5만원 × 작업 시간"
→ 10시간 작업 시 50만원
```

## 7️⃣ 협상 기본 원칙

### Win-Win 추구
**양측 모두 이익**

**예시:**
```
가격 협상:
- 판매자: 단가 인하 대신 물량 증가
- 구매자: 물량 증가 대신 단가 인하
→ 둘 다 만족
```

### BATNA 준비
**Best Alternative to Negotiated Agreement**  
= "협상 결렬 시 최선의 대안"

**예시:**
```
외주 업체 협상:
- BATNA: A업체가 안 되면 B업체 선택
→ 협상에서 유리한 위치
```

### 실생활 비유
```
🏠 전월세 계약:
- 집주인 BATNA: 다른 세입자 구하기
- 세입자 BATNA: 다른 집 구하기
→ 둘 다 대안이 있으면 협상 강함
```

## 💬 자주 묻는 질문 (FAQ)

**Q1: RACI에서 A(Accountable)가 왜 1명만 가능한가요?**  
A: 책임 소재를 명확히 하기 위해서입니다. 여러 명이면 "너가 할 줄 알았어" 같은 책임 회피가 발생합니다.

**Q2: 모든 갈등을 협력으로 해결해야 하나요?**  
A: 이상적이지만 현실적으로 어렵습니다. 시간, 중요도, 상황에 따라 적절한 기법을 선택하세요. 긴급하면 강제도 필요합니다.

**Q3: 외주(Buy)가 항상 비싼가요?**  
A: 아닙니다! 내부 인력 비용(급여, 교육, 시간)을 고려하면 외주가 더 저렴한 경우가 많습니다. 총 비용을 비교하세요.

**Q4: 팀이 Storming(혼란) 단계에 너무 오래 머무르면?**  
A: PM이 적극 개입해야 합니다. 명확한 역할 정의, 규칙 수립, 팀 빌딩 활동으로 Norming 단계로 빨리 이동시키세요.

**Q5: 협상에서 BATNA가 없으면 어떻게 하나요?**  
A: 협상 전에 반드시 BATNA를 만드세요. 없으면 협상에서 매우 불리합니다. 최소한 "협상 결렬" 옵션이라도 준비하세요.
