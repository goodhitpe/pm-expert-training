## 📖 Part 1: 자원 관리 (Resource Management)

### 1.1 자원 관리의 중요성

#### 자원 관리 실패의 영향
- **프로젝트 지연**: 자원 부족으로 인한 일정 지연 (65%)
- **품질 저하**: 미숙련 자원 투입으로 품질 문제
- **비용 증가**: 비효율적 자원 활용으로 예산 초과
- **팀 사기 저하**: 과도한 업무 부담, 번아웃

#### 효과적인 자원 관리의 이점
- ✅ 적시 적재적소 자원 배치
- ✅ 팀 생산성 및 사기 향상
- ✅ 일정 및 예산 준수
- ✅ 품질 향상
- ✅ 이해관계자 만족도 증가

### 1.2 자원의 종류

#### 1. 인적 자원 (Human Resources)
- **프로젝트 팀**: PM, 개발자, 디자이너, QA
- **지원 팀**: HR, 재무, 법무
- **외부 자원**: 컨설턴트, 계약직, 프리랜서

#### 2. 물적 자원 (Physical Resources)
- **장비**: 서버, PC, 테스트 장비
- **시설**: 사무실, 회의실, 데이터센터
- **소프트웨어**: 개발 도구, 라이선스

#### 3. 자금 (Financial Resources)
- **예산**: 인건비, 장비비, 운영비
- **예비비**: 리스크 대응 예산

## 📊 2. 자원 관리 프로세스 (PMBOK 기반)

### 2.1 6단계 프로세스

```
1. 자원 관리 계획 수립 (Plan Resource Management)
   ↓
2. 활동 자원 산정 (Estimate Activity Resources)
   ↓
3. 자원 획득 (Acquire Resources)
   ↓
4. 팀 개발 (Develop Team)
   ↓
5. 팀 관리 (Manage Team)
   ↓
6. 자원 통제 (Control Resources)
```

### 2.2 활동 자원 산정 기법

#### 2.2.1 전문가 판단 (Expert Judgment)
**장점**: 경험 기반 빠른 판단  
**단점**: 주관적, 편향 가능

#### 2.2.2 유사 산정 (Analogous Estimating)
**방법**: 과거 프로젝트 데이터 활용  
**예시**: 이전 프로젝트 5명 → 현 프로젝트 6명 추정

#### 2.2.3 상향식 산정 (Bottom-up Estimating)
**방법**: WBS 최하위 활동별 산정 후 합산  
**장점**: 정확도 높음  
**단점**: 시간 소요

#### 2.2.4 매개변수 산정 (Parametric Estimating)
**공식**: 자원 = 생산성 지표 × 작업량

**예시**:
```
작업량: 10,000 라인 코드
생산성: 100 라인/일/명
기간: 30일

필요 인력 = 10,000 / (100 × 30) = 3.33 ≈ 4명
```

## 👥 3. RACI 차트 (Responsibility Assignment Matrix)

### 3.1 RACI 정의

| 역할 | 영문 | 한글 | 설명 |
|-----|------|------|------|
| **R** | Responsible | 실행 책임자 | 실제 작업 수행자 (복수 가능) |
| **A** | Accountable | 최종 책임자 | 승인 및 의사결정 (단 1명) |
| **C** | Consulted | 자문 제공자 | 양방향 의사소통 |
| **I** | Informed | 정보 수신자 | 일방향 정보 전달 |

### 3.2 RACI 작성 원칙

✅ **DO (해야 할 것)**:
- 각 활동마다 A는 반드시 1명
- 모든 활동에 최소 1명의 R
- 명확한 역할 정의
- 이해관계자와 합의

❌ **DON'T (하지 말 것)**:
- A가 2명 이상 (책임 분산)
- R이 없는 활동 (실행자 없음)
- 모든 사람을 C로 지정 (비효율)
- 불필요한 I 과다 지정

### 3.3 실습 예제 1: 웹사이트 개발 프로젝트

| 활동/역할 | PM | 개발 팀장 | 개발자 | 디자이너 | QA | 고객 |
|---------|----|---------:|------:|--------:|---:|-----:|
| **프로젝트 계획 수립** | A/R | C | I | I | I | C |
| **요구사항 수집** | A | R | C | C | C | R |
| **UI/UX 디자인** | A | C | I | R | C | C |
| **데이터베이스 설계** | A | R | R | I | C | I |
| **프론트엔드 개발** | A | R | R | C | I | I |
| **백엔드 개발** | A | R | R | I | I | I |
| **통합 테스트** | A | C | R | I | R | I |
| **사용자 인수 테스트** | A | C | C | I | R | R |
| **배포** | A | R | R | I | C | I |
| **운영 이관** | A | R | R | I | C | R |

**해석**:
- PM은 모든 활동의 A (최종 책임)
- 개발 팀장은 기술 활동의 R
- 고객은 요구사항 및 인수테스트의 R

### 3.4 실습 예제 2: 마케팅 캠페인 프로젝트

| 활동/역할 | 마케팅 디렉터 | 캠페인 매니저 | 콘텐츠 팀 | 디자인 팀 | SNS 팀 | 데이터 분석가 |
|---------|-------------|-------------|---------|----------|--------|------------|
| **캠페인 전략 수립** | A | R | C | C | C | C |
| **콘텐츠 제작** | A | A | R | R | C | I |
| **SNS 포스팅** | I | A | C | I | R | I |
| **광고 집행** | A | R | C | C | R | I |
| **성과 분석** | A | C | I | I | I | R |

## 🚀 4. 팀 개발 (Team Development)

### 4.1 Tuckman의 팀 개발 5단계

```
1단계: 형성기 (Forming)
   ↓
2단계: 혼란기 (Storming)
   ↓
3단계: 규범기 (Norming)
   ↓
4단계: 성과기 (Performing)
   ↓
5단계: 해체기 (Adjourning)
```

#### 1단계: 형성기 (Forming)
**특징**:
- 팀원 간 서로 낯설고 조심스러움
- 역할과 책임이 불명확
- PM에 대한 높은 의존도
- 공손하지만 피상적 관계

**PM 역할**:
- 팀 비전 및 목표 명확히 제시
- 역할 및 책임 정의 (RACI)
- 팀 규범 수립
- 아이스브레이킹 활동

**예시**:
```
"안녕하세요? 저는 백엔드 개발 담당 김철수입니다."
"저희 프로젝트 목표가 뭐죠?"
"회의는 언제 하나요?"
```

#### 2단계: 혼란기 (Storming)
**특징**:
- 의견 충돌 및 갈등 발생
- 권한 및 역할 다툼
- 팀원 간 긴장감 높음
- 생산성 일시적 하락

**PM 역할**:
- 갈등을 건설적으로 해결
- 경청 및 중재
- 팀 규범 상기
- 개별 코칭

**예시**:
```
"제 방법이 더 나은데 왜 안 따르죠?"
"이건 제 역할 아닌데요?"
"회의 시간이 너무 길어요!"
```

#### 3단계: 규범기 (Norming)
**특징**:
- 팀 규범 및 프로세스 정착
- 협력 및 신뢰 형성
- 의견 차이 건설적 해결
- 생산성 증가 시작

**PM 역할**:
- 팀 문화 강화
- 성공 경험 공유
- 협업 도구 활용 지원
- 긍정적 피드백

**예시**:
```
"이 방법으로 가는 게 어떨까요?"
"서로 도와서 잘 해결했네요."
"우리 팀 분위기 좋아졌어요."
```

#### 4단계: 성과기 (Performing)
**특징**:
- 높은 생산성 및 효율성
- 자율적 문제 해결
- 팀워크 및 시너지 극대화
- PM 개입 최소화

**PM 역할**:
- 장애물 제거 (Servant Leadership)
- 권한 위임
- 성과 인정 및 보상
- 도전적 목표 설정

**예시**:
```
"이건 제가 알아서 처리할게요."
"같이 하니까 더 빠르네요."
"다음 스프린트 목표 달성 자신 있어요!"
```

#### 5단계: 해체기 (Adjourning)
**특징**:
- 프로젝트 종료 또는 팀 해산
- 성취감과 아쉬움 공존
- 관계 정리 및 이별

**PM 역할**:
- 성과 축하 및 인정
- 회고 회의 (Retrospective)
- 지식 문서화
- 네트워크 유지 방안

**예시**:
```
"함께해서 좋았습니다."
"다음에 또 같이 일하고 싶어요."
"배운 게 많았습니다."
```

### 4.2 팀 개발 활동

#### 4.2.1 팀 빌딩 (Team Building)
- **아이스브레이킹**: 자기소개, 게임
- **워크샵**: 공동 목표 수립
- **팀 활동**: 회식, 야유회, 봉사활동

#### 4.2.2 교육 및 훈련
- **기술 교육**: 프로그래밍, 도구 사용법
- **소프트 스킬**: 커뮤니케이션, 협상
- **인증 과정**: PMP, 애자일

#### 4.2.3 Co-location (집중 근무)
- **War Room**: 한 공간에서 협업
- **페어 프로그래밍**: 2명이 함께 코딩
- **몹 프로그래밍**: 팀 전체가 함께 작업

## 💪 5. 동기 부여 이론 (Motivation Theories)

### 5.1 Maslow의 욕구 단계 이론

```
        자아실현 욕구
      (Self-Actualization)
           /  \
          /    \
        존중 욕구
       (Esteem)
         /  \
        /    \
     사회적 욕구
    (Belongingness)
       /  \
      /    \
    안전 욕구
    (Safety)
     /  \
    /    \
  생리적 욕구
 (Physiological)
```

#### 5단계 욕구와 PM 대응 방안

| 단계 | 욕구 | PM 대응 방안 |
|-----|------|------------|
| **5단계** | 자아실현 | 도전적 과제, 자율성, 창의적 업무 |
| **4단계** | 존중 | 인정, 승진, 책임, 피드백 |
| **3단계** | 사회적 | 팀워크, 소속감, 커뮤니케이션 |
| **2단계** | 안전 | 고용 안정, 건강, 복지 |
| **1단계** | 생리적 | 적정 임금, 근무 환경 |

#### 실습 예제: 팀원 동기 부여

**시나리오 1**: 신입 개발자 (생리적 + 안전 욕구)
```
문제: 낮은 급여, 고용 불안
대응:
- 적정 급여 협상 (생리적)
- 정규직 전환 계획 제시 (안전)
- 멘토 배정 (사회적)
```

**시나리오 2**: 중견 개발자 (존중 + 자아실현 욕구)
```
문제: 단순 반복 업무, 성장 정체
대응:
- 리드 역할 부여 (존중)
- 신기술 프로젝트 배정 (자아실현)
- 컨퍼런스 발표 기회 (존중)
```

### 5.2 Herzberg의 2요인 이론

#### 위생 요인 (Hygiene Factors)
**정의**: 불만족을 제거하는 요인 (있어도 동기 부여 안 됨)

**요인**:
- 급여 및 복지
- 근무 환경
- 회사 정책
- 상사와의 관계
- 직무 안정성

**PM 대응**:
- 기본적 근무 조건 보장
- 공정한 보상 체계
- 쾌적한 환경 제공

#### 동기 요인 (Motivators)
**정의**: 만족을 증대시키는 요인 (진정한 동기 부여)

**요인**:
- 성취감
- 인정
- 업무 자체
- 책임
- 성장 기회

**PM 대응**:
- 도전적 과제 부여
- 성과 인정 및 칭찬
- 권한 위임
- 교육 기회 제공

#### 비교표

| 위생 요인 | 동기 요인 |
|---------|----------|
| 급여 | 성취감 |
| 복지 | 인정 |
| 정책 | 책임감 |
| 환경 | 성장 |
| 안정성 | 자율성 |

### 5.3 McGregor의 X-Y 이론

#### Theory X (이론 X) - 전통적 관리
**가정**:
- 사람은 본질적으로 일하기 싫어함
- 강제와 통제가 필요
- 책임 회피, 안정 추구

**관리 방식**:
- 엄격한 통제
- 세부 지시
- 감시 및 처벌
- 미세 관리 (Micromanagement)

#### Theory Y (이론 Y) - 인간 중심 관리
**가정**:
- 사람은 일을 즐길 수 있음
- 자율성과 자기 통제 가능
- 책임감 있고 창의적

**관리 방식**:
- 권한 위임
- 목표 중심 관리
- 자율성 부여
- 서번트 리더십

#### PM 적용 가이드

**이론 Y 적용 (권장)**:
```
✅ "이 기능을 어떻게 구현하면 좋을까요?" (참여)
✅ "팀의 의견을 존중합니다." (신뢰)
✅ "실패해도 괜찮아요. 배우는 과정입니다." (실험)
```

**이론 X 적용 (지양)**:
```
❌ "이렇게 하세요. 다른 방법은 안 돼요." (지시)
❌ "왜 진행이 안 되죠? 보고 드렸어야죠." (통제)
❌ "실수하면 안 됩니다." (위협)
```

## 🤝 6. 갈등 해결 기법

### 6.1 갈등의 원인

1. **자원 부족**: 예산, 인력, 장비 부족
2. **우선순위 차이**: 일정 vs 품질
3. **역할 모호성**: RACI 미정의
4. **의사소통 부재**: 정보 공유 부족
5. **성격 차이**: 업무 스타일 불일치

### 6.2 Thomas-Kilmann 갈등 해결 모델

```
           협력성 (Cooperativeness)
               높음
                 ↑
      협력       │      수용
   (Collaborate) │   (Accommodate)
                 │
─────────────────┼─────────────────→
                 │              자기주장성
      회피       │      경쟁       (Assertiveness)
     (Avoid)     │   (Compete)      높음
                 │
                타협
              (Compromise)
```

#### 6.2.1 협력 (Collaborate) ⭐ 최선
**특징**: Win-Win, 문제 해결

**장점**:
- 양측 모두 만족
- 장기적 관계 개선
- 창의적 해결책

**단점**:
- 시간 많이 소요
- 노력 필요

**적용 상황**:
- 중요한 이슈
- 장기 관계
- 충분한 시간

**예시**:
```
갈등: 개발팀은 2주 필요, 영업팀은 1주 요구
협력: 
- 핵심 기능만 1주 내 완성 (영업 만족)
- 추가 기능은 2주 후 배포 (개발 만족)
- 결과: 양측 모두 목표 달성
```

#### 6.2.2 경쟁 (Compete)
**특징**: Win-Lose, 강압적

**장점**:
- 빠른 결정
- 긴급 상황 대응

**단점**:
- 관계 악화
- 반발 초래

**적용 상황**:
- 긴급 결정
- 안전/윤리 이슈
- 원칙적 사안

**예시**:
```
갈등: 보안 취약점 vs 출시 일정
경쟁:
- PM: "보안 문제 해결 전까지 출시 연기"
- 결과: 영업팀 불만이지만 안전 우선
```

#### 6.2.3 수용 (Accommodate)
**특징**: Lose-Win, 양보

**장점**:
- 관계 유지
- 빠른 해결

**단점**:
- 자신의 손해
- 반복 시 불만

**적용 상황**:
- 사소한 이슈
- 관계가 중요
- 자신이 틀렸을 때

**예시**:
```
갈등: 회의 시간 (오전 vs 오후)
수용:
- PM: "오후가 더 좋으시다면 그렇게 하죠."
- 결과: 팀원 만족, PM은 일정 조정
```

#### 6.2.4 회피 (Avoid)
**특징**: Lose-Lose, 무시

**장점**:
- 즉각 대응 불필요
- 시간 벌기

**단점**:
- 문제 악화
- 신뢰 저하

**적용 상황**:
- 사소한 문제
- 감정 고조 시 (진정 필요)
- 정보 부족

**예시**:
```
갈등: 코딩 스타일 논쟁 (열띤 토론)
회피:
- PM: "잠시 휴식하고 30분 후 다시 논의합시다."
- 결과: 감정 진정 후 건설적 대화
```

#### 6.2.5 타협 (Compromise)
**특징**: 중간 지점, 양보

**장점**:
- 공정한 해결
- 빠른 합의

**단점**:
- 양측 다 불만족
- 최선책 아님

**적용 상황**:
- 시간 제약
- 동등한 권한
- 임시 해결책

**예시**:
```
갈등: 개발 2주 vs 영업 1주
타협:
- PM: "10일로 합의하죠."
- 결과: 양측 약간의 양보로 중간점 도출
```

### 6.3 갈등 해결 시나리오 실습

#### 시나리오 1: 자원 경쟁
**상황**: 
- 개발팀: "백엔드 개발자 3명 더 필요해요."
- 디자인팀: "UI 디자이너 2명 더 필요해요."
- 현실: 추가 인원 2명만 가능

**해결책**: 협력 (Collaborate)
```
1. 각 팀의 진짜 필요 파악
   - 개발: 핵심 기능 2명, 부가 기능 1명
   - 디자인: 주요 화면 1명, 아이콘 1명

2. 우선순위 합의
   - 핵심 기능 + 주요 화면 먼저

3. 대안 도출
   - 개발 2명 배정 (핵심 기능)
   - 디자인 외주 활용 (아이콘)
   - 부가 기능은 다음 스프린트

결과: Win-Win
```

#### 시나리오 2: 기술적 의견 충돌
**상황**:
- 시니어 개발자 A: "마이크로서비스 아키텍처"
- 시니어 개발자 B: "모놀리식 아키텍처"
- 양측 모두 자신의 방식 고집

**해결책**: 경쟁 (Compete) + 전문가 판단
```
1. PM이 결정 권한 행사
2. CTO/아키텍트에게 자문
3. 프로젝트 특성 고려
   - 팀 규모: 5명 (소규모)
   - 기간: 3개월 (단기)
   - 복잡도: 낮음

4. 결정: 모놀리식 (단순함이 유리)

결과: 시니어 A 불만이지만 프로젝트 성공이 우선
```

#### 시나리오 3: 일정 vs 품질
**상황**:
- 영업팀: "다음 주 데모 필수!"
- 개발팀: "버그 50개 남았어요. 2주 더 필요"
- 경영진: "고객 계약 걸려있음"

**해결책**: 타협 (Compromise)
```
1. 핵심 기능 우선순위 선정
2. 데모용 버전과 상용 버전 구분
3. 합의안:
   - 데모: 핵심 기능만, 크리티컬 버그 10개 수정
   - 상용: 2주 후, 모든 버그 수정

결과: 영업팀 데모 가능, 개발팀 품질 보장 시간 확보
```

## 📦 Part 2: 조달 관리 (Procurement Management)

### 7.1 조달 관리의 중요성

#### 조달이 필요한 경우
- 내부 자원/역량 부족
- 전문성 필요 (법률, 회계)
- 비용 효율성
- 위험 분산

#### 조달 관리 실패 사례
- **보잉 787**: 70% 아웃소싱 → 품질 문제, 3년 지연
- **영국 NHS IT**: 부실 계약 관리 → 120억 파운드 낭비
- **애플 지도**: TomTom 데이터 품질 문제

### 7.2 조달 관리 프로세스

```
1. 조달 관리 계획 수립 (Plan Procurement Management)
   ↓
2. 조달 수행 (Conduct Procurements)
   ↓
3. 조달 통제 (Control Procurements)
```

### 7.3 계약 유형 (Contract Types)

#### 7.3.1 확정가 계약 (Fixed Price - FP)

##### FFP (Firm Fixed Price) - 확정 고정가
**특징**:
- 가격 고정, 변경 불가
- 구매자 리스크 최소
- 공급자 리스크 최대

**장점** (구매자):
- 예산 확정, 예측 가능
- 관리 부담 낮음

**단점** (구매자):
- 변경 비용 높음
- 공급자 마진에 리스크 포함

**적용**:
- 요구사항 명확
- 표준 제품/서비스
- 짧은 기간

**예시**:
```
프로젝트: 로고 디자인
계약: ₩5,000,000 (확정)
결과: 
- 추가 수정 3회 포함
- 초과 수정 시 추가 비용
```

##### FPIF (Fixed Price Incentive Fee) - 인센티브 고정가
**특징**:
- 기본 가격 + 성과 인센티브
- 비용 절감 시 이익 공유

**공식**:
```
최종 가격 = 목표 비용 + 목표 이익 + (목표 비용 - 실제 비용) × 공유 비율

예시:
- 목표 비용: ₩100M
- 목표 이익: ₩10M
- 공유 비율: 80/20 (구매자/공급자)
- 실제 비용: ₩90M

절감액 = ₩100M - ₩90M = ₩10M
공급자 추가 이익 = ₩10M × 20% = ₩2M
최종 가격 = ₩90M + ₩10M + ₩2M = ₩102M

구매자 절감: ₩110M - ₩102M = ₩8M (80%)
공급자 이익: ₩10M + ₩2M = ₩12M
```

#### 7.3.2 원가 보상 계약 (Cost Reimbursable - CR)

##### CPFF (Cost Plus Fixed Fee) - 원가 가산 고정 수수료
**특징**:
- 실제 원가 + 고정 수수료
- 구매자 리스크 최대
- 공급자 리스크 최소

**장점**:
- 불확실한 작업 수행 가능
- 공급자 동기 부여

**단점**:
- 예산 예측 어려움
- 원가 관리 부담

**적용**:
- 요구사항 불명확
- R&D 프로젝트
- 긴급 작업

**예시**:
```
프로젝트: 신기술 연구개발
계약: 실제 원가 + 15% 수수료

실제 원가: ₩200M
수수료: ₩200M × 15% = ₩30M
최종 금액: ₩230M
```

##### CPIF (Cost Plus Incentive Fee) - 원가 가산 인센티브
**특징**:
- 실제 원가 + 변동 수수료
- 목표 달성 시 인센티브

**적용**:
- 성과 향상 유도
- 중간 정도 불확실성

#### 7.3.3 실비 정산 계약 (Time & Material - T&M)

**특징**:
- 시간당 단가 × 투입 시간
- 재료비 실비 청구
- 유연한 범위 조정

**장점**:
- 빠른 계약 체결
- 범위 변경 용이

**단점**:
- 총 비용 예측 어려움
- 공급자 비효율 위험

**적용**:
- 긴급 작업
- 범위 불명확
- 인력 증강

**예시**:
```
프로젝트: 버그 수정 지원
계약:
- 시니어 개발자: ₩200,000/시간
- 주니어 개발자: ₩100,000/시간
- 최대 상한: ₩10,000,000

실제 투입:
- 시니어 30시간 = ₩6,000,000
- 주니어 20시간 = ₩2,000,000
- 총액: ₩8,000,000
```

### 7.4 계약 유형 비교표

| 구분 | FFP | CPFF | T&M |
|-----|-----|------|-----|
| **구매자 리스크** | 낮음 | 높음 | 중간 |
| **공급자 리스크** | 높음 | 낮음 | 중간 |
| **요구사항** | 명확 | 불명확 | 중간 |
| **관리 부담** | 낮음 | 높음 | 중간 |
| **가격 예측성** | 높음 | 낮음 | 중간 |
| **범위 유연성** | 낮음 | 높음 | 높음 |
| **적용 사례** | 표준 제품 | R&D | 긴급 지원 |

### 7.5 공급업체 선정 기준

#### 선정 기준 (가중치 평가)

| 기준 | 가중치 | 설명 |
|-----|--------|------|
| **가격** | 30% | 총 비용, 가성비 |
| **기술 역량** | 25% | 기술력, 경험 |
| **품질** | 20% | 품질 인증, 실적 |
| **일정** | 15% | 납기 준수 능력 |
| **재무 안정성** | 10% | 재무 건전성 |

#### 실습 예제: 공급업체 평가

**프로젝트**: 클라우드 인프라 구축

| 기준 (가중치) | A사 | B사 | C사 |
|------------|-----|-----|-----|
| **가격 (30%)** | 8점 | 6점 | 9점 |
| **기술 (25%)** | 9점 | 8점 | 7점 |
| **품질 (20%)** | 7점 | 9점 | 8점 |
| **일정 (15%)** | 8점 | 7점 | 8점 |
| **재무 (10%)** | 8점 | 9점 | 6점 |

**가중 점수 계산**:
```
A사 = (8×0.3) + (9×0.25) + (7×0.2) + (8×0.15) + (8×0.1)
    = 2.4 + 2.25 + 1.4 + 1.2 + 0.8
    = 8.05

B사 = (6×0.3) + (8×0.25) + (9×0.2) + (7×0.15) + (9×0.1)
    = 1.8 + 2.0 + 1.8 + 1.05 + 0.9
    = 7.55

C사 = (9×0.3) + (7×0.25) + (8×0.2) + (8×0.15) + (6×0.1)
    = 2.7 + 1.75 + 1.6 + 1.2 + 0.6
    = 7.85
```

**결과**: A사 선정 (8.05점)

**8. 5가지 갈등 해결 기법을 나열하고, 각각 한 문장으로 설명하시오.**

**모범 답안**:
1. **협력 (Collaborate)**: 양측의 니즈를 모두 만족시키는 Win-Win 해결
2. **경쟁 (Compete)**: 자신의 입장을 관철시키는 Win-Lose 해결
3. **수용 (Accommodate)**: 상대방 입장을 받아들이는 Lose-Win 해결
4. **회피 (Avoid)**: 갈등을 무시하거나 연기하는 Lose-Lose 해결
5. **타협 (Compromise)**: 양측이 양보하여 중간점을 찾는 해결

---

**9. FFP와 CPFF 계약의 차이점을 리스크 관점에서 설명하시오.**

**모범 답안**:

**FFP (Firm Fixed Price)**:
- 가격: 고정
- 구매자 리스크: 낮음 (예산 확정)
- 공급자 리스크: 높음 (초과 비용 감수)
- 적용: 요구사항 명확

**CPFF (Cost Plus Fixed Fee)**:
- 가격: 변동 (실제 원가 + 고정 수수료)
- 구매자 리스크: 높음 (예산 예측 어려움)
- 공급자 리스크: 낮음 (원가 보상)
- 적용: 요구사항 불명확

---

**10. Tuckman 모델의 5단계를 순서대로 나열하고, 각 단계에서 PM의 주요 역할을 설명하시오.**

**모범 답안**:

1. **Forming (형성기)**
   - PM 역할: 비전 제시, 역할 정의, 팀 규범 수립

2. **Storming (혼란기)**
   - PM 역할: 갈등 중재, 경청, 개별 코칭

3. **Norming (규범기)**
   - PM 역할: 팀 문화 강화, 협업 지원, 긍정 피드백

4. **Performing (성과기)**
   - PM 역할: 장애물 제거, 권한 위임, 성과 인정

5. **Adjourning (해체기)**
   - PM 역할: 성과 축하, 회고, 지식 문서화
