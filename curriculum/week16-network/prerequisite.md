## 1️⃣ 네트워크(Network)란?

### 쉬운 설명
**네트워크**는 컴퓨터들이 서로 연결되어 데이터를 주고받을 수 있는 시스템입니다.
인터넷도 전 세계 컴퓨터를 연결한 거대한 네트워크입니다.

### 실생활 비유
도로망을 생각해보세요:
- **컴퓨터/스마트폰** = 집이나 건물
- **네트워크 케이블** = 도로
- **데이터** = 택배 상자
- **라우터** = 교차로의 신호등

집에서 택배를 보내면 도로를 따라 목적지에 도착하듯이,
컴퓨터에서 데이터를 보내면 네트워크를 따라 다른 컴퓨터에 도착합니다.

### PM이 알아야 하는 이유
- 모든 온라인 서비스의 기반
- 네트워크 속도가 사용자 경험에 직접 영향
- 네트워크 장애는 서비스 중단으로 직결
- 보안 사고의 대부분이 네트워크를 통해 발생

---

## 2️⃣ 인터넷은 어떻게 작동하나?

### 클라이언트-서버 모델
인터넷의 기본 구조입니다.

**클라이언트**: 요청하는 쪽 (내 컴퓨터, 스마트폰)  
**서버**: 응답하는 쪽 (네이버, 구글의 서버)

### 실생활 비유
- **고객(클라이언트)** 이 식당에 가서
- **"불고기 하나요"** 라고 요청(Request)하면
- **주방(서버)** 에서 요리를 만들어
- **"여기 있습니다"** 하고 응답(Response)

### 웹사이트 접속 과정
```
1. 브라우저에 "naver.com" 입력 (클라이언트 요청)
   ↓
2. DNS 서버가 "naver.com"의 IP 주소 찾기
   ↓
3. 네이버 서버에 연결
   ↓
4. 네이버 서버가 웹페이지 전송 (서버 응답)
   ↓
5. 브라우저가 화면에 표시
```

### PM 관점
- "서버 응답 시간"이 느리면 사용자 이탈 증가
- 클라이언트(앱)과 서버 간 통신 설계 중요
- API 설계 시 요청/응답 구조 이해 필요

## 3️⃣ IP 주소(IP Address)

### 정의
네트워크에 연결된 각 기기의 고유한 주소입니다.

### 실생활 비유
- **IP 주소** = 집 주소
- **도메인 이름** = 건물 이름

"서울시 강남구 테헤란로 123" (IP 주소)를 
"삼성빌딩" (도메인 이름)으로 부르는 것처럼,
"172.217.174.46" (IP 주소)을 
"google.com" (도메인)으로 부릅니다.

### IPv4와 IPv6

**IPv4 (현재 주류)**
- 형식: `192.168.0.1`
- 점으로 구분된 4개 숫자
- 약 43억 개 주소 (부족해지는 중)

**IPv6 (미래 표준)**
- 형식: `2001:0db8:85a3:0000:0000:8a2e:0370:7334`
- 거의 무한대 주소
- 아직 전환 중

### 공인 IP vs 사설 IP

**공인 IP (Public IP)**
- 인터넷에서 접근 가능한 유일한 주소
- ISP(통신사)에서 부여
- 예: 회사 홈페이지 서버

**사설 IP (Private IP)**
- 내부 네트워크에서만 사용
- 예: 회사 내부 PC들
- 보통 `192.168.x.x` 형태

### 실생활 비유
- **공인 IP** = 건물의 도로명 주소 (외부에서 찾아올 수 있음)
- **사설 IP** = 건물 안의 호수 (401호, 402호 등)

### PM이 알아야 하는 이유
- 서버는 공인 IP 필요 (비용 발생)
- 개발/테스트 환경은 사설 IP 사용 가능
- IP 차단/허용으로 보안 정책 수립

## 4️⃣ DNS (Domain Name System)

### 정의
도메인 이름을 IP 주소로 변환해주는 시스템입니다.

### 왜 필요한가?
사람은 "naver.com"을 기억하기 쉽지만,
컴퓨터는 "223.130.195.95" 같은 숫자만 이해합니다.
DNS는 이 둘을 연결하는 전화번호부 역할입니다.

### 실생활 비유
전화번호부:
- **이름** = 도메인 (naver.com)
- **전화번호** = IP 주소 (223.130.195.95)

"홍길동"이라는 이름을 전화번호부에서 찾아
실제 전화번호를 알아내는 것처럼,
"naver.com"을 DNS에서 찾아 IP 주소를 알아냅니다.

### DNS 조회 과정
```
사용자: "google.com"에 접속하고 싶어요
   ↓
DNS 서버: "google.com의 IP는 172.217.174.46입니다"
   ↓
컴퓨터: 172.217.174.46에 연결합니다
```

### PM 관점
- 도메인 구매 및 DNS 설정 필요 (년 1-2만원)
- DNS 변경 시 전파 시간 필요 (24-48시간)
- DNS 장애는 서비스 접속 불가 초래
- CDN 설정 시 DNS 설정 변경 필요

## 5️⃣ 포트(Port)

### 정의
컴퓨터의 특정 서비스에 접근하기 위한 번호입니다.
하나의 IP 주소로 여러 서비스를 구분합니다.

### 실생활 비유
아파트와 비슷합니다:
- **IP 주소** = 아파트 주소 (서울시 강남구 ...)
- **포트 번호** = 동/호수 (101동 503호)

하나의 아파트(IP)에 여러 집(포트)이 있듯이,
하나의 서버(IP)에 여러 서비스(포트)가 있습니다.

### 자주 사용하는 포트

| 포트 | 서비스 | 설명 |
|------|--------|------|
| 80 | HTTP | 일반 웹사이트 |
| 443 | HTTPS | 보안 웹사이트 (자물쇠 아이콘) |
| 22 | SSH | 서버 원격 접속 |
| 3306 | MySQL | 데이터베이스 |
| 25 | SMTP | 이메일 발송 |

### 예시
- `http://example.com:80` → 포트 80 (생략 가능)
- `https://example.com:443` → 포트 443 (생략 가능)
- `example.com:8080` → 포트 8080 (개발 서버)

### PM이 알아야 하는 이유
- 방화벽 설정 시 포트 번호로 차단/허용
- "포트 8080으로 테스트 서버 띄웠습니다" 같은 개발자 멘트 이해
- 보안상 불필요한 포트는 닫아야 함

## 6️⃣ HTTP와 HTTPS

### HTTP (HyperText Transfer Protocol)
웹 브라우저와 웹 서버가 대화하는 규칙입니다.

**특징:**
- 암호화 없음 (누군가 중간에서 볼 수 있음)
- 포트 80 사용
- 주소창에 자물쇠 없음

### HTTPS (HTTP Secure)
HTTP에 보안(암호화)을 추가한 것입니다.

**특징:**
- 데이터 암호화 (중간에서 봐도 해독 불가)
- 포트 443 사용
- 주소창에 자물쇠 🔒 표시
- SSL/TLS 인증서 필요

### 실생활 비유

**HTTP = 엽서**
- 내용이 그대로 노출
- 배달부가 내용을 볼 수 있음
- 빠르지만 안전하지 않음

**HTTPS = 봉인된 편지**
- 내용이 봉투에 감춰짐
- 받는 사람만 열어볼 수 있음
- 조금 느리지만 안전함

### 왜 HTTPS가 중요한가?
- **로그인 정보** (아이디, 비밀번호) 보호
- **개인정보** (주민번호, 카드번호) 보호
- **검색 순위**: 구글은 HTTPS 사이트를 우대
- **신뢰도**: 자물쇠 표시로 사용자 신뢰 증가

### PM 관점
- 모든 서비스는 HTTPS 필수 (개인정보보호법 준수)
- SSL 인증서 비용 발생 (년 수만원~수백만원)
- Let's Encrypt로 무료 인증서 가능
- 인증서 만료 시 서비스 접속 불가 (갱신 관리 필요)

## 7️⃣ 대역폭(Bandwidth)과 지연 시간(Latency)

### 대역폭 (Bandwidth)
단위 시간에 전송할 수 있는 데이터의 양입니다.

**측정 단위**: Mbps (초당 메가비트), Gbps (초당 기가비트)

**실생활 비유**: 도로의 차선 수
- 1차선 도로 = 낮은 대역폭 (10Mbps)
- 10차선 고속도로 = 높은 대역폭 (1Gbps)

### 지연 시간 (Latency)
데이터가 출발지에서 목적지까지 도달하는 시간입니다.

**측정 단위**: ms (밀리초, 1/1000초)

**실생활 비유**: 배송 시간
- 서울에서 부산: 약 4시간 (지연 시간)
- 서울에서 미국: 약 10시간 (지연 시간)

### 대역폭 vs 지연 시간

| 상황 | 대역폭 | 지연 시간 |
|------|--------|----------|
| 영화 다운로드 | 중요 ✅ | 덜 중요 |
| 실시간 화상 회의 | 중요 | 매우 중요 ✅ |
| 온라인 게임 | 보통 | 매우 중요 ✅ |

### 실제 예시
- **서울-부산 서버**: 지연 시간 5-10ms
- **서울-도쿄 서버**: 지연 시간 30-40ms
- **서울-미국 서버**: 지연 시간 150-200ms

### PM이 알아야 하는 이유
- 글로벌 서비스는 여러 지역에 서버 배치 (지연 시간 감소)
- CDN 사용으로 정적 콘텐츠 지연 시간 개선
- 실시간 서비스(게임, 화상회의)는 지연 시간이 핵심
- 대역폭이 크면 비용 증가 (트래픽 요금)

## 8️⃣ 방화벽(Firewall)

### 정의
네트워크 트래픽을 검사하고 차단하는 보안 시스템입니다.

### 실생활 비유
건물의 경비실:
- 출입자를 확인하고
- 허가된 사람만 통과시키고
- 의심스러운 사람은 차단합니다

### 작동 방식
```
외부 접속 시도
   ↓
방화벽 검사
   ├─ 허용된 IP? → 통과 ✅
   ├─ 허용된 포트? → 통과 ✅
   └─ 의심스러운 패턴? → 차단 ❌
```

### 방화벽 규칙 예시
```
규칙 1: 포트 443 (HTTPS) → 모든 IP 허용
규칙 2: 포트 22 (SSH) → 회사 IP만 허용
규칙 3: 나머지 모든 접속 → 차단
```

### PM 관점
- 모든 서버에 방화벽 필수
- 개발 서버는 회사 IP만 접근 가능하도록 설정
- 해킹 시도 대부분은 방화벽에서 차단
- 방화벽 규칙 변경 시 서비스 영향도 검토 필요

---

## 9️⃣ CDN (Content Delivery Network)

### 정의
전 세계에 분산된 서버에 콘텐츠를 복사해두고,
사용자와 가장 가까운 서버에서 제공하는 서비스입니다.

### 왜 필요한가?
한국에 있는 서버를 미국 사용자가 접속하면 느립니다.
CDN은 미국에도 복사본을 두어 빠르게 제공합니다.

### 실생활 비유
**CDN 없이**: 서울 본사 창고에서만 배송
- 부산 고객: 빠름
- 미국 고객: 느림 (국제 배송)

**CDN 사용**: 각 지역마다 물류 창고 설치
- 부산 고객: 부산 창고에서 배송
- 미국 고객: 미국 창고에서 배송
- 모두 빠름! ✅

### 작동 방식
```
미국 사용자가 이미지 요청
   ↓
CDN이 가장 가까운 미국 서버에서 제공
   ↓
빠른 로딩! (한국 서버까지 갈 필요 없음)
```

### 주요 CDN 서비스
- **CloudFlare**: 무료 플랜 있음
- **AWS CloudFront**: AWS와 통합 잘 됨
- **Akamai**: 대기업용, 고가

### PM이 알아야 하는 이유
- 글로벌 서비스에 필수
- 이미지/동영상 같은 정적 파일에 효과적
- CDN 비용은 트래픽 기반 (사용량 증가 시 비용 상승)
- 페이지 로딩 속도 개선 → 사용자 경험 향상

## 🔟 네트워크 보안 기초

### 주요 위협

**1. DDoS 공격 (분산 서비스 거부 공격)**
- 수많은 컴퓨터가 동시에 서버 공격
- 서버가 과부하로 다운됨
- **대응**: CDN, 방화벽, Auto Scaling

**2. 중간자 공격 (Man-in-the-Middle)**
- 통신 중간에서 데이터 가로채기
- **대응**: HTTPS 사용

**3. SQL Injection**
- 악의적인 데이터베이스 명령어 삽입
- **대응**: 입력값 검증, Prepared Statement

### 실생활 비유

**DDoS 공격**
- 수천 명이 동시에 가짜 전화로 콜센터 마비

**중간자 공격**
- 우체국 직원이 편지 몰래 열어보기

**SQL Injection**
- ATM에 특수한 카드를 넣어 해킹

### 보안 체크리스트
- [ ] HTTPS 사용 (SSL/TLS 인증서)
- [ ] 방화벽 설정 (불필요한 포트 차단)
- [ ] 정기적인 보안 업데이트
- [ ] 강력한 비밀번호 정책
- [ ] 접근 권한 관리 (최소 권한 원칙)
- [ ] 정기적인 백업

### PM이 알아야 하는 이유
- 보안 사고는 회사 신뢰도와 직결
- 개인정보보호법 위반 시 과징금 (매출의 3%)
- 보안은 개발 단계부터 고려해야 함 (나중에 추가하면 비용 큼)
- 정기적인 보안 점검 필요

### 자주 묻는 질문 (FAQ)

**Q1: 네트워크를 잘 몰라도 PM을 할 수 있나요?**  
A: 네, 가능합니다! 하지만 기본 개념(IP, DNS, HTTP)만 알아도 개발팀과의 소통이 훨씬 원활해집니다.

**Q2: OSI 7계층을 외워야 하나요?**  
A: 모두 외울 필요는 없습니다. PM은 전체적인 흐름과 주요 프로토콜(HTTP, TCP/IP) 정도만 이해하면 충분합니다.

**Q3: 네트워크 장애가 생기면 어떻게 대응하나요?**  
A: PM은 장애를 직접 해결하지 않습니다. 네트워크 엔지니어가 해결하지만, PM은 사용자 공지, 비즈니스 영향도 파악, 복구 일정 관리를 담당합니다.

**Q4: CDN은 꼭 써야 하나요?**  
A: 국내만 서비스하면 선택사항이지만, 글로벌 서비스나 이미지/동영상이 많다면 필수입니다. 사용자 경험이 크게 개선됩니다.
