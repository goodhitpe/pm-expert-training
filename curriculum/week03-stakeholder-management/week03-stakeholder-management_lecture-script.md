### [00:15-02:00] 섹션 1: 이해관계자 관리 (105분)

#### [00:15-00:45] 1.1 이해관계자 식별 및 분석 (30분)

**강의 내용**:
```
👥 이해관계자(Stakeholder)란?

"프로젝트에 영향을 주거나 받는 모든 개인, 그룹, 조직"

💡 왜 중요한가?
"만족한 이해관계자 → 프로젝트 지원
불만족한 이해관계자 → 프로젝트 방해"

이해관계자 유형:

내부 이해관계자:
- 프로젝트 스폰서 (예산 승인자)
- 프로젝트 팀원
- 경영진 (CEO, CTO, CFO)
- 타 부서 (마케팅, 영업, 재무)

외부 이해관계자:
- 고객 (Client)
- 최종 사용자 (End User)
- 공급업체 (Vendor)
- 파트너사
- 규제 기관

💼 실무 사례: 온라인 쇼핑몰 구축
───────────────────────────────────────
이해관계자 맵:

내부:
✓ CEO (최종 승인자)
✓ CFO (예산 관리)
✓ CTO (기술 검토)
✓ CMO (마케팅 전략)
✓ 개발팀 (구현)
✓ 운영팀 (인수 후 운영)
✓ 고객센터 (사용자 지원)

외부:
✓ 주요 고객사 3개
✓ 결제 대행사 (PG)
✓ 물류 파트너
✓ 외주 개발사
✓ 최종 소비자

총 15개 이해관계자 그룹!
───────────────────────────────────────

📊 이해관계자 분석: 파워-관심 그리드

화이트보드에 크게 그리기:

      높은 관심
         ↑
    B    │    A
    관리 │ 긴밀히
    만족 │  관리
─────────┼─────────→ 높은 파워
    D    │    C
  모니터 │  정보
         │  제공

A (높은 파워, 높은 관심): 긴밀히 관리
- CEO, CTO, 주요 고객
- 전략: 주간 보고, 의사결정 참여
- "가장 중요! 매주 만나기"

B (낮은 파워, 높은 관심): 정보 제공
- 개발팀, 운영팀
- 전략: 일일 스탠드업, 투명한 소통
- "정보를 충분히 제공"

C (높은 파워, 낮은 관심): 관리 만족
- CFO, 규제 기관
- 전략: 월간 요약 보고
- "관심을 갖게 하되 방해하지 않기"

D (낮은 파워, 낮은 관심): 모니터
- 타 부서, 일부 외부 파트너
- 전략: 분기별 간단 보고
- "최소한의 노력"

💡 이해관계자 배치 실습 (15분)

진행 방법:
1. 화이트보드에 그리드 그리기 (5분)
2. 쇼핑몰 사례의 이해관계자를 포스트잇에 적기
3. 적절한 위치에 배치
4. 각 사분면별 전략 논의

예시:
CEO → A (긴밀히 관리)
개발팀 → B (정보 제공)
CFO → C (관리 만족)
```

**교수법**:
- 화이트보드에 그리드 그리며 상호작용
- 포스트잇으로 이해관계자 배치 실습
- 각 사분면의 전략 구체적으로 설명

#### [00:45-01:15] 1.2 이해관계자 참여 전략 (30분)

**강의 내용**:
```
📈 이해관계자 참여 수준

5단계:
비협조 → 저항 → 중립 → 지원 → 주도
(Unaware) (Resistant) (Neutral) (Supportive) (Leading)

💼 실무 사례: 신규 시스템 도입
───────────────────────────────────────
프로젝트: ERP 시스템 교체
현재 레거시 시스템 20년 사용

이해관계자별 참여 수준:

1. CFO (재무팀장)
   현재: 저항 (Resistant)
   "왜 바꾸려고 해? 지금도 잘 되는데!"
   목표: 지원 (Supportive)
   
2. 영업팀장
   현재: 중립 (Neutral)
   "나랑 상관없는데..."
   목표: 지원 (Supportive)
   
3. IT팀장
   현재: 지원 (Supportive)
   "빨리 바꿔야 해요!"
   목표: 주도 (Leading)
───────────────────────────────────────

🎯 참여 전략 수립

1. 저항 → 지원으로 (CFO)

분석:
- 저항 이유: 변화에 대한 두려움, 과거 실패 경험
- 관심사: 비용, 업무 중단 최소화

전략:
✅ 1:1 미팅 (월 2회)
✅ 비용 절감 효과 제시 (ROI 분석)
✅ 단계적 전환 계획 공유
✅ 조기 성과 시연 (Quick Win)
✅ 우려사항 해결 (Risk 대응)

실행:
Week 1: 1:1 미팅, 우려사항 청취
Week 2: ROI 분석 보고서 제출
Week 4: 유사 기업 성공 사례 공유
Week 6: 파일럿 데모 초대
Week 8: 재무팀 의견 반영한 계획 수정

결과: 저항 → 중립 → 지원 (3개월 소요)

2. 중립 → 지원으로 (영업팀장)

분석:
- 중립 이유: 직접적 영향 인식 부족
- 관심사: 영업 효율, 고객 정보

전략:
✅ 혜택 구체화 (영업 보고서 자동화)
✅ 영향도 설명 (고객 이력 통합)
✅ 조기 참여 유도 (요구사항 수집)

3. 지원 → 주도로 (IT팀장)

전략:
✅ 프로젝트 핵심 역할 부여
✅ 의사결정 권한 위임
✅ 교육 및 개발 기회 제공

💡 핵심 원칙

"사람들은 변화를 싫어합니다"
→ WIIFM (What's In It For Me?)
"내게 무슨 이득이 있나?"

"저항을 두려워하지 마세요"
→ 저항은 관심의 표현
→ 무관심이 더 위험

"작은 승리를 만드세요"
→ Quick Win으로 신뢰 구축
→ 점진적으로 참여도 증가
```

**교수법**:
- 5단계 참여 수준 명확히 설명
- 실제 사례로 전략 구체화
- 저항을 극복한 경험 공유

#### [01:15-01:45] 1.3 이해관계자 갈등 해결 (30분)

**강의 내용**:
```
⚔️ 갈등은 필연적입니다

"프로젝트 = 갈등의 연속"
- 우선순위 충돌
- 자원 경쟁
- 의견 차이

🎭 역할극: 갈등 상황 (20분)

시나리오 1: 기능 우선순위 갈등
───────────────────────────────────────
프로젝트: CRM 시스템 개발 (6개월, 3억원)
Month 3, 스프린트 리뷰 회의

등장인물:
- PM (여러분)
- 영업팀장 (높은 파워, 높은 관심)
- 기술팀장 (낮은 파워, 높은 관심)

영업팀장:
"고객이 원하는 건 모바일 앱입니다!
웹은 나중에 해도 되니 모바일 먼저!"

기술팀장:
"지금 아키텍처 변경은 불가능합니다.
모바일은 웹 완성 후 Phase 2입니다!"

(양쪽 다 자기 주장)

PM, 어떻게 하시겠습니까?
───────────────────────────────────────

역할극 진행:
1. 3명씩 그룹 (PM, 영업, 기술)
2. 5분간 상황 연기
3. PM의 해결 방법 제시
4. 전체 공유 및 토론

💡 모범 답안: 5단계 갈등 해결 프로세스

1단계: 경청 (Listen)
PM: "영업팀장님, 왜 모바일이 급한지 말씀해주세요."
영업팀장: "주요 고객사 3곳이 모바일을 요구했습니다.
          이대로면 계약 취소 가능성이 있습니다."

PM: "기술팀장님, 모바일을 왜 나중에 해야 하나요?"
기술팀장: "웹 API를 먼저 완성해야 모바일에서 사용 가능합니다.
          지금 바꾸면 2개월 지연됩니다."

2단계: 공통점 찾기 (Find Common Ground)
PM: "두 분 다 고객 만족이 목표시네요.
    또한 프로젝트 성공을 원하시고요."

3단계: 데이터 기반 분석 (Analyze with Data)
PM: "정확한 영향도를 분석하겠습니다."

분석 결과:
- 모바일 우선: +2개월, +5천만원, 고객 유지
- 웹 우선: 원래 일정, 원래 예산, 고객 이탈 위험

4단계: 대안 제시 (Propose Options)
PM: "3가지 옵션을 제안합니다:

Option A: 웹 먼저 (Original Plan)
- 장점: 일정/예산 준수
- 단점: 고객 이탈 위험

Option B: 모바일 먼저 (영업팀 요청)
- 장점: 고객 유지
- 단점: +2개월, +5천만원

Option C: 하이브리드 (절충안)
- 핵심 API + 모바일 MVP 동시 개발
- 웹은 Phase 1.5로 1개월 후
- +1개월, +2천만원
- 고객에게 조기 데모 가능"

5단계: 의사결정 및 합의 (Decide & Commit)
PM: "저는 Option C를 추천합니다.
    CTO님의 최종 승인을 받아야 합니다."

(CTO 보고 → Option C 승인)

PM: "영업팀장님, 고객에게 4주 후 모바일 데모를 약속하세요.
    기술팀장님, 핵심 API부터 개발해주세요."

───────────────────────────────────────

📚 갈등 해결 5가지 스타일 (Thomas-Kilmann)

1. 경쟁 (Competing): 내 방식대로
   - "제 결정이 맞습니다"
   - 상황: 긴급한 경우, 명확한 근거

2. 협력 (Collaborating): Win-Win
   - "같이 해결책을 찾아봅시다"
   - 상황: 시간 있고, 관계 중요

3. 타협 (Compromising): 중간
   - "각자 양보합시다"
   - 상황: 동등한 파워, 빠른 결정

4. 회피 (Avoiding): 피하기
   - "나중에 얘기합시다"
   - 상황: 사소한 이슈, 감정 식힐 시간

5. 수용 (Accommodating): 양보
   - "알겠습니다, 그렇게 하죠"
   - 상황: 상대가 옳거나, 관계 우선

💡 PM의 선택
"대부분의 경우: 협력 (Collaborating)
긴급 상황: 경쟁 (Competing)
사소한 이슈: 수용 (Accommodating)"
```

**교수법**:
- 역할극으로 생생한 체험
- 실제 갈등 상황 재현
- 5단계 해결 프로세스 시연
- 다양한 스타일 비교 설명

```
- 회의
- 이메일

📊 커뮤니케이션 채널 수
공식: n × (n-1) / 2

팀원 수    채널 수    복잡도
3명        3개        낮음
5명        10개       중간

"팀이 커질수록 의사소통이 기하급수적으로 복잡해집니다!"

📋 커뮤니케이션 계획 수립

5W1H:
- Who: 누구에게?
- What: 무엇을?
- When: 언제?
- Where: 어디서?
- Why: 왜?
- How: 어떻게?

💼 실무 예시: 커뮤니케이션 계획서
───────────────────────────────────────
프로젝트: 모바일 뱅킹 앱 개발

1. 일일 스탠드업
   - 대상: 개발팀 (8명)
   - 내용: 어제, 오늘, 장애물
   - 시간: 매일 9:00, 15분
   - 방법: 대면, 화이트보드 앞
   - 목적: 일일 동기화, 이슈 조기 발견

2. 주간 진행 보고
   - 대상: CTO, 스폰서
   - 내용: 진척률, 이슈, 리스크, 다음 계획
   - 시간: 매주 월요일 14:00, 30분
   - 방법: 회의실, PPT
   - 목적: 투명성, 의사결정

3. 월간 운영위원회
   - 대상: 경영진, 주요 이해관계자
   - 내용: 전체 현황, 주요 결정 사항
   - 시간: 매월 마지막 금요일 10:00, 1시간
   - 방법: 대회의실, 공식 보고
   - 목적: 거버넌스, 승인

4. 분기 리뷰
   - 대상: 전체 이해관계자
   - 내용: 데모, 성과, 교훈
   - 시간: 분기 말, 2시간
   - 방법: 데모 + 발표
   - 목적: 투명성, 피드백

5. 임시 이슈 회의

📨 커뮤니케이션 방법 선택

구두 vs 서면:

구두 (Verbal):
✅ 장점: 빠름, 즉시 피드백, 관계 형성
❌ 단점: 기록 없음, 오해 가능성

상황: 브레인스토밍, 갈등 해결, 동기부여

서면 (Written):
✅ 장점: 기록, 명확성, 법적 증거
❌ 단점: 시간 소요, 감정 전달 어려움

상황: 공식 승인, 계약, 변경 요청

공식 vs 비공식:

공식 (Formal):
- 공식 회의, 보고서, 이메일
- 문서화 필수
- 의사결정, 승인

비공식 (Informal):
- 복도 대화, 메신저, 점심
- 편안함, 솔직함
- 관계 형성, 사전 협의

💡 효과적인 커뮤니케이션 7원칙

1. 명확성 (Clarity)
   - 간결하고 구체적으로
   - 전문 용어 최소화
   - 예시 활용

2. 간결성 (Conciseness)
   - 핵심만 전달
   - "30초 엘리베이터 피치"

3. 구체성 (Concreteness)
   - 모호한 표현 지양
   - 숫자, 데이터 활용
   - "빨리" → "2일 이내"

4. 정확성 (Correctness)
   - 사실 확인
   - 데이터 검증
   - 오타 주의

5. 일관성 (Consistency)
   - 동일한 메시지
   - 표현 통일
   - 브랜드 가이드 준수

6. 완전성 (Completeness)
   - 5W1H 충족
   - 필요한 정보 모두 포함
   - 다음 액션 명시

7. 정중함 (Courtesy)
   - 존중하는 태도
   - 긍정적 표현
   - 감사 표현
```

```
   각 이해관계자는 자신의 입장 표현
   
   발생 이벤트:
   - 영업본부장: "지금도 잘 되는데 왜 바꿔?"
   - IT팀장: "6개월은 너무 짧아요"
   - CEO: "비용 대비 효과가 확실한가?"
   
   PM의 미션:
   - 모든 이해관계자를 최소 중립으로
   - CEO의 지지 확보
   - 구체적 전략 실행
   
③ 디브리핑 (5분)
   - 무엇이 잘 되었나?
   - 무엇이 어려웠나?
   - 다르게 할 점은?

Phase 3: 커뮤니케이션 계획 작성 (15분)

각 그룹:
① 커뮤니케이션 계획서 작성
   이해관계자별:
   - 정보 제공 내용
   - 주기 (일/주/월)
   - 방법 (회의/이메일/보고서)
   - 목적
   
② 주간 진행 보고서 Draft
   - 템플릿 활용
   - 첫 주 가상 상황 작성

Phase 4: 발표 (10분)

각 그룹 2분씩 발표:
① 이해관계자 배치 결과
② 핵심 전략 3가지
③ 킥오프 미팅 인사이트
④ 커뮤니케이션 계획 하이라이트

💡 강사 피드백 포인트:
- 영업본부장(저항)을 어떻게 다루었나?
- CEO(높은 파워)에게 집중했나?
- 커뮤니케이션 계획이 현실적인가?
```

## 💡 강의 운영 팁

### 효과적인 강의를 위한 조언

1. **역할극 적극 활용**
   - 이론보다 체험이 효과적
   - 실제 상황 재현
   - 다양한 관점 경험

2. **실제 사례 공유**
   - 강사의 실패/성공 경험
   - 수강생들의 경험 공유
   - 공감대 형성

3. **상호작용 극대화**
   - "여러분이라면 어떻게 하시겠습니까?"
   - 토론 시간 충분히
   - 다양한 의견 존중

4. **실용적 템플릿 제공**
   - 바로 사용 가능한 양식
   - 회사로 가져가서 활용

5. **감정 코칭**
   - 이해관계자 관리는 감정 관리
   - 좌절감, 분노 다루기
   - 긍정적 마인드셋

### 주의 사항

- ⚠️ 갈등 상황이 불편할 수 있음
  → 안전한 학습 환경 조성
  
- ⚠️ 개인 경험이 부정적일 수 있음
  → 건설적 방향으로 유도
  
- ⚠️ 역할극이 서툴 수 있음
  → 격려와 지지
  
- ⚠️ 시간 관리 중요
  → 토론이 길어질 수 있음
