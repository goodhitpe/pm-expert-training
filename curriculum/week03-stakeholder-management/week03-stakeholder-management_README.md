## 📚 주요 내용

### 1. 이해관계자 관리

#### 1.1 이해관계자란?
- 프로젝트에 영향을 주거나 받는 개인, 그룹, 조직
- 내부/외부 이해관계자 구분

#### 1.2 이해관계자 식별
**주요 이해관계자 유형:**
- 프로젝트 스폰서 (Project Sponsor)
- 프로젝트 팀원
- 고객 (Customer)
- 최종 사용자 (End User)
- 공급업체 (Vendor)
- 경영진
- 규제 기관

#### 1.3 이해관계자 분석

**권력/관심 매트릭스 (Power/Interest Grid)**

![이해관계자 매트릭스](./images/week03-stakeholder-management_images_stakeholder-matrix.svg)

**영향력/관심 매트릭스 활용법:**
- **긴밀히 관리 (High Power, High Interest)**: 주요 의사결정자
- **정보 제공 (Low Power, High Interest)**: 정기적인 커뮤니케이션
- **관리 만족 (High Power, Low Interest)**: 적절한 정보 제공
- **모니터 (Low Power, Low Interest)**: 최소한의 관리

#### 1.4 이해관계자 참여 전략
- **비협조 (Unaware)** → **저항 (Resistant)** → **중립 (Neutral)** → **지원 (Supportive)** → **주도 (Leading)**
- 각 이해관계자의 현재 상태와 목표 상태 정의
- 참여 전략 수립

### 2. 커뮤니케이션 관리

#### 2.1 커뮤니케이션 계획
- **누구에게 (Who)**: 대상 식별
- **무엇을 (What)**: 전달 내용
- **언제 (When)**: 빈도 및 시기
- **어떻게 (How)**: 방법 및 채널

#### 2.2 커뮤니케이션 방법
- **구두 (Verbal)**: 회의, 전화, 발표
- **서면 (Written)**: 이메일, 보고서, 문서
- **비공식 (Informal)**: 대화, 메신저
- **공식 (Formal)**: 공식 회의, 보고서

#### 2.3 효과적인 커뮤니케이션 기법
1. **명확성**: 간결하고 분명한 표현
2. **일관성**: 동일한 메시지 유지
3. **적시성**: 적절한 시기에 전달
4. **피드백**: 양방향 소통
5. **문서화**: 중요 내용 기록

### 3. 회의 운영

#### 3.1 회의 유형
- 킥오프 미팅 (Kick-off Meeting)
- 주간 진행 회의 (Weekly Status Meeting)
- 이슈 해결 회의
- 검토 회의 (Review Meeting)
- 회고 회의 (Retrospective)

#### 3.2 효과적인 회의 진행
**회의 전:**
- 명확한 목적 설정
- 의제 사전 배포
- 참석자 선정
- 자료 준비

**회의 중:**
- 시간 엄수
- 의제 준수
- 액션 아이템 기록
- 결론 도출

**회의 후:**
- 회의록 작성 및 배포
- 액션 아이템 추적
- 후속 조치

#### 3.3 회의록 작성 요령
- 회의 기본 정보 (일시, 장소, 참석자)
- 주요 논의 사항
- 결정 사항
- 액션 아이템 (담당자, 기한)
- 다음 회의 일정

### 4. 보고서 작성

#### 4.1 프로젝트 진행 보고서 (Status Report)
**포함 내용:**
- 프로젝트 개요
- 주요 성과 (Accomplishments)
- 이슈 및 리스크
- 다음 단계 계획
- 지표 (일정, 예산, 품질)

#### 4.2 대시보드 활용
- RAG 상태 표시 (Red/Amber/Green)
- 주요 지표 시각화
- 트렌드 분석

#### 4.3 경영진 보고 (Executive Summary)
- 한 페이지 요약
- 핵심 메시지 강조
- 의사결정 필요 사항 명시
- 시각 자료 활용

## 🎯 실무 적용 예시: 이해관계자 갈등 해결

### 예시 1: 기술팀 vs 영업팀 갈등 (기능 우선순위)

#### 상황

**프로젝트**: CRM 시스템 개발 (6개월, $300K)

**Month 3, 스프린트 리뷰 회의:**

```
영업팀장 (높은 권력, 높은 관심):
"고객이 원하는 건 모바일 앱입니다!
 웹은 나중에 해도 되니 모바일 먼저 만들어주세요."

기술 리더 (중간 권력, 높은 관심):
"아니요, 웹 기반으로 설계했습니다.
 모바일은 React Native로 웹 기반 재사용이 효율적입니다.
 지금 변경하면 3개월 지연됩니다."

영업팀장:
"그럼 고객 잃습니다! 모바일이 우선입니다!"

기술 리더:
"기술적으로 불가능합니다!"

(회의실 긴장감...PM이 개입 필요)
```

#### ❌ 미숙한 PM의 대응

```
PM: "음... 둘 다 맞는 말씀입니다.
     영업팀장님 말씀도 맞고, 기술 리더님 말씀도 맞고...
     좀 더 논의해봅시다."

결과:
- 갈등 해결 안 됨
- 의사결정 지연
- 팀 사기 저하
- PM 리더십 의심
```

#### ✅ 숙련된 PM의 대응

```
PM: "잠깐만요. 먼저 사실 확인하겠습니다."

[이해관계자 분석]
1. 영업팀장 요구의 진짜 이유 파악
PM: "영업팀장님, 구체적으로 어떤 고객이 모바일을 원하시나요?"
영업: "A사입니다. 영업 사원 100명이 외근 중 사용합니다."
PM: "웹 모바일 버전은 어떤가요? 반응형 웹입니다."
영업: "그것도 모바일 아닌가요?"
PM: "네, 맞습니다. 스마트폰 브라우저에서 작동합니다."

→ 발견: 영업팀장이 "네이티브 앱"과 "모바일 웹"을 혼동

2. 기술팀 우려 확인
PM: "현재 아키텍처 설명해주세요."
기술: "웹 프론트(React) + API 백엔드입니다.
      모바일 웹은 동일 코드베이스로 반응형 대응 가능합니다.
      네이티브 앱은 React Native로 재개발 필요합니다."

[데이터 기반 의사결정]
PM: "두 분께 제안드립니다."

옵션 분석:
A. 현재 계획 (반응형 웹)
   - 기간: 6개월 (변경 없음)
   - 비용: $300K (변경 없음)
   - 장점: 빠른 출시, 하나의 코드베이스
   - 단점: 네이티브 앱 대비 UX 약간 떨어짐

B. 네이티브 앱 전환
   - 기간: 9개월 (+3개월)
   - 비용: $450K (+$150K)
   - 장점: 최상의 모바일 UX
   - 단점: 늦은 출시, 높은 비용, 유지보수 복잡

C. 하이브리드 접근 (권장)
   - Phase 1: 반응형 웹 먼저 출시 (6개월, $300K)
   - Phase 2: 네이티브 앱 개발 (런칭 후 3개월, $150K)
   - 장점: 빠른 시장 진입 + 장기적 UX 개선
   - 단점: 단계적 투자 필요

[시연으로 설득]
PM: "영업팀장님, 제 아이폰으로 현재 개발 중인 모바일 웹 보여드릴게요."
(데모 사이트 접속)
영업: "오, 이것도 모바일에서 잘 되네요?"
PM: "네, 앱 설치 없이 브라우저에서 바로 사용 가능합니다."

[합의 도출]
영업팀장: "그럼 Phase 1로 먼저 진행하고,
          고객 피드백 보고 네이티브 앱 검토하면 되겠네요."
기술 리더: "네, 그게 합리적입니다."

PM: "좋습니다. 그럼 Phase 1 6개월 후 고객 피드백 수집하고,
     필요 시 Phase 2 예산 확보하겠습니다.
     결정 사항 문서화해서 공유드리겠습니다."

결과:
✅ 갈등 해결
✅ 데이터 기반 합의
✅ 양측 모두 만족
✅ PM 리더십 인정
```

### 예시 2: 고객 vs PM (끝없는 추가 요구사항)

#### 상황: 범위 크리프 (Scope Creep) 대응

**프로젝트**: 물류 관리 시스템 (4개월, $200K)

```
Week 8 (중반):
고객: "엑셀 내보내기 기능 추가해주세요."
PM: "네, 추가하겠습니다." (구두 합의)

Week 10:
고객: "차트도 필요해요. 대시보드 만들어주세요."
PM: "알겠습니다." (또 구두 합의)

Week 12:
고객: "이메일 알림도 필요합니다."
PM: "음... 일정이..." (당황)

Week 14:
개발팀: "원래 계획에 없던 기능들이 너무 많아요!"
PM: "고객이 원하는데 어떡해요..."

Week 16 (원래 종료일):
- 프로젝트 50%만 완료
- 범위 2배 증가
- 팀 번아웃
- 고객 불만 ("왜 안 끝나요?")
```

#### ✅ 올바른 대응

```
Week 8:
고객: "엑셀 내보내기 기능 추가해주세요."

PM: "이해했습니다. 변경 요청서 작성해드리겠습니다."

[변경 영향도 분석]
PM이 작성한 변경 요청서:

제목: 엑셀 내보내기 기능 추가
요청자: 고객사 김부장
요청일: 2024-02-15

현재 범위:
- 데이터 조회: 화면 표시만

변경 내용:
- 엑셀 다운로드 버튼 추가
- 데이터 엑셀 변환 (xlsx 포맷)
- 100,000행까지 지원

기술 작업:
1. 백엔드: 엑셀 생성 API (Apache POI) - 3일
2. 프론트: 다운로드 버튼 + 진행 표시 - 1일
3. 테스트: 대용량 데이터 테스트 - 1일
총 5일

영향도:
- 일정: +5일 (현재 여유 2일 → 3일 지연)
- 비용: +$5,000
- 리스크: 대용량 데이터 시 성능 이슈 가능

옵션:
A. 승인 - 일정 3일 연장, 비용 $5K 추가
B. 거부 - 현재 범위 유지
C. Phase 2로 이연 - 런칭 후 추가

PM의 제안: 옵션 C (Phase 2)
이유:
- 핵심 기능 아님 (Nice to have)
- 일정 지연 방지
- Phase 2에서 사용자 피드백 반영하여 더 나은 기능 구현

고객에게 설명:
PM: "엑셀 다운로드는 유용한 기능입니다.
     다만 현재 프로젝트에 추가하면 런칭이 3일 지연됩니다.
     
     제안:
     1. 현재: 화면 복사 기능으로 임시 대응
     2. Phase 2 (런칭 후 1개월): 엑셀 다운로드 추가
        + 차트, 이메일 알림 함께 개발
        + 사용자 피드백 반영
     
     이렇게 하면 예정대로 런칭하고,
     Phase 2에서 더 완성도 높은 기능 제공 가능합니다."

고객: "합리적이네요. Phase 2로 하겠습니다."

[변경 통제 프로세스 확립]
PM이 공지:
"앞으로 모든 변경 요청은:
1. 변경 요청서 작성
2. 영향도 분석 (PM)
3. CCB 검토 (주 1회 회의)
4. 승인/거부 결정
5. 문서화 및 공유

즉흥적 변경은 받지 않습니다."

결과:
✅ 범위 크리프 방지
✅ 일정 준수
✅ 고객 신뢰 유지
✅ 팀 보호
```

### 예시 3: 경영진 보고 (기술을 비즈니스로 번역)

#### 상황: 프로젝트 지연 보고

**프로젝트**: 모바일 앱 개발 (3개월 목표, 현재 4개월 소요 예상)

```
❌ 기술 용어 중심 보고:

PM 보고:
"DB 마이그레이션에서 성능 이슈가 발생했습니다.
 인덱스 재구성과 쿼리 최적화가 필요합니다.
 또한 CI/CD 파이프라인 구축이 지연되어
 배포 자동화가 늦어지고 있습니다."

경영진 (CFO):
"...무슨 말인지 모르겠는데, 언제 끝나나요?"

PM:
"음... 한 달 더 필요합니다."

CFO:
"왜 처음에 3개월이라고 했죠? 예산 초과인가요?"

PM:
"기술적으로 예상하지 못한..."

CFO:
"신뢰할 수 없네요."

결과: ❌ 신뢰 상실, 의사결정 지연
```

핵심 이슈: 데이터 이전 복잡도 예상보다 높음

지연으로 인한 영향:

   - 검증 시간 추가 (무결성 확인)
   
   조치:

   - 결과: 배포 15분, 실수 0건
