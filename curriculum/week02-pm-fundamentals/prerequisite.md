## 1️⃣ PMBOK이란?

### 쉬운 설명
**PMBOK (Project Management Body of Knowledge)**은 프로젝트 관리의 모범 사례를 모아놓은 가이드북입니다.

### 실생활 비유
**자동차 운전 교본**을 생각해보세요:
- 교본 = PMBOK (표준 지식)
- 운전 기술 = 프로젝트 관리 기술
- 교통법규 = PM 프로세스와 원칙

교본에 모든 상황이 다 나오진 않지만, 기본을 배우면 다양한 상황에 응용할 수 있습니다.

### PMBOK의 역사
- **1987년**: PMI(Project Management Institute)가 처음 발행
- **현재**: 7판까지 출간 (이 과정은 6판 기준)
- **글로벌 표준**: 전 세계 PM들이 참고하는 표준

### 왜 배워야 하나?
- ✅ **공통 언어**: 전 세계 PM들과 같은 용어 사용
- ✅ **체계적 접근**: 프로젝트 관리의 전체 그림 파악
- ✅ **인증 기반**: PMP(Project Management Professional) 자격증 시험 출제 기준
- ✅ **검증된 방법**: 수십 년간 축적된 모범 사례

## 2️⃣ 프로세스(Process)란?

### 정의
**프로세스**는 특정 결과를 만들기 위한 일련의 활동입니다.

### 실생활 비유
**라면 끓이기 프로세스**:
```
입력(Input): 라면, 물, 냄비, 불
    ↓
활동(Process): 물 끓이기 → 라면 넣기 → 3분 기다리기
    ↓
출력(Output): 완성된 라면
```

프로젝트 관리도 마찬가지로 **입력 → 활동 → 출력** 구조를 가집니다.

### PMBOK의 프로세스 구조
```
입력(Inputs)
   ↓
도구 및 기법(Tools & Techniques)
   ↓
출력(Outputs)
```

### 예시: 프로젝트 헌장 개발 프로세스
- **입력**: 사업 계획서, 계약서, 기업 환경 요인
- **도구**: 전문가 판단, 브레인스토밍, 회의
- **출력**: 프로젝트 헌장

## 3️⃣ 5대 프로세스 그룹

### 정의
프로젝트를 관리하는 5가지 논리적 단계입니다.

### 실생활 비유: 집들이 파티 준비

#### 1. **착수(Initiating)** - "파티 하기로 결정!"
- 왜 파티를 하나? (목적)
- 누구를 초대할까? (이해관계자)
- 대략적인 날짜와 예산은?
- **PM 활동**: 프로젝트 헌장 작성, 이해관계자 식별

#### 2. **기획(Planning)** - "상세 계획 세우기"
- 메뉴는 뭐로 할까?
- 예산을 어떻게 배분할까?
- 누가 무엇을 준비할까?
- 리스크는? (날씨, 재료 부족 등)
- **PM 활동**: 프로젝트 관리 계획서 작성, WBS 작성, 일정/예산 수립

#### 3. **실행(Executing)** - "실제로 준비하기"
- 장 보러 가기
- 요리하기
- 집 청소하기
- 손님 맞이하기
- **PM 활동**: 팀 관리, 작업 수행, 의사소통 관리

#### 4. **감시 및 통제(Monitoring & Controlling)** - "확인하고 조정하기"
- 예산 내에서 진행되나?
- 시간이 부족하면 메뉴 조정
- 손님이 추가되면 음식 더 준비
- **PM 활동**: 진척도 측정, 변경 관리, 이슈 대응

#### 5. **종료(Closing)** - "마무리하기"
- 손님들 배웅
- 설거지하기
- 다음에는 뭘 개선할지 정리
- **PM 활동**: 최종 인도물 전달, 교훈 정리, 문서 보관

### 프로세스 그룹의 특징
- 순차적이지만 **반복적**
- 실행과 감시/통제는 **동시 진행**
- 규모에 따라 강도 조절 가능

## 4️⃣ 10대 지식영역 (Knowledge Areas)

### 정의
프로젝트 관리에 필요한 10가지 전문 분야입니다.

### 실생활 비유: 집 짓기
집을 지을 때 여러 전문가가 필요한 것처럼, PM도 10가지 분야를 관리해야 합니다.

#### 1. **통합 관리(Integration)** - 총괄 건축가
- 모든 것을 하나로 묶음
- 변경사항 조율
- **핵심**: 프로젝트 헌장, 관리 계획서, 변경 통제

#### 2. **범위 관리(Scope)** - 설계도 작성
- 무엇을 만들지 정의
- WBS(작업 분해 구조) 작성
- 범위 변경 통제
- **핵심**: 요구사항 수집, 범위 정의, 범위 검증

#### 3. **일정 관리(Schedule)** - 공사 일정표
- 언제까지 할지 계획
- 크리티컬 패스 파악
- 일정 지연 대응
- **핵심**: 활동 정의, 순서 배열, 기간 산정

#### 4. **원가 관리(Cost)** - 예산 관리
- 얼마나 드는지 산정
- 예산 배분
- 비용 초과 통제
- **핵심**: 원가 산정, 예산 결정, 원가 통제(EVM)

#### 5. **품질 관리(Quality)** - 품질 검사
- 품질 기준 설정
- 품질 보증 활동
- 결함 찾기 및 수정
- **핵심**: 품질 계획, 품질 보증, 품질 통제

#### 6. **자원 관리(Resource)** - 인력 및 장비
- 팀원 구성
- 자원 배분
- 팀 개발 및 관리
- **핵심**: 팀 구성, 자원 확보, 팀 개발

#### 7. **의사소통 관리(Communications)** - 소통 창구
- 누구에게 무엇을 언제 알릴지
- 회의, 보고서 관리
- 정보 공유
- **핵심**: 커뮤니케이션 계획, 정보 배포, 보고

#### 8. **리스크 관리(Risk)** - 위험 관리
- 무엇이 잘못될 수 있나?
- 발생 확률과 영향도 분석
- 대응 계획 수립
- **핵심**: 리스크 식별, 분석, 대응 계획

#### 9. **조달 관리(Procurement)** - 외주 및 구매
- 외부에서 사야 할 것
- 공급업체 선정
- 계약 관리
- **핵심**: 조달 계획, 공급업체 선정, 계약 통제

#### 10. **이해관계자 관리(Stakeholder)** - 관계 관리
- 누가 관련되나?
- 기대사항 관리
- 참여 유도
- **핵심**: 이해관계자 식별, 참여 계획, 참여 관리

## 5️⃣ WBS (Work Breakdown Structure)

### 정의
프로젝트 전체 작업을 작고 관리 가능한 단위로 분해한 계층 구조입니다.

### 실생활 비유
**피자 자르기**를 생각해보세요:
```
피자(프로젝트)
  ├─ 1/4 조각
  ├─ 1/4 조각
  ├─ 1/4 조각
  └─ 1/4 조각
      ├─ 토핑
      ├─ 치즈
      └─ 도우
```

큰 일을 작게 나누면 관리하기 쉽습니다!

### WBS 예시: 웹사이트 구축
```
웹사이트 구축 프로젝트
├─ 1. 기획
│   ├─ 1.1 요구사항 수집
│   ├─ 1.2 기능 정의
│   └─ 1.3 화면 설계
├─ 2. 디자인
│   ├─ 2.1 UI 디자인
│   └─ 2.2 UX 설계
├─ 3. 개발
│   ├─ 3.1 프론트엔드
│   ├─ 3.2 백엔드
│   └─ 3.3 데이터베이스
├─ 4. 테스트
│   ├─ 4.1 단위 테스트
│   ├─ 4.2 통합 테스트
│   └─ 4.3 사용자 테스트
└─ 5. 배포
    ├─ 5.1 서버 설정
    └─ 5.2 운영 배포
```

### WBS 작성 원칙
- **100% 규칙**: 모든 작업을 포함
- **상호 배타성**: 중복 없이 분해
- **적절한 수준**: 너무 크지도 작지도 않게 (8/80 규칙: 8~80시간)
- **산출물 중심**: 활동보다는 결과물로 표현

### PM이 알아야 하는 이유
- 누락된 작업 방지
- 정확한 공수 산정
- 명확한 책임 할당
- 진척도 측정 기준

## 6️⃣ EVM (Earned Value Management)

### 정의
프로젝트의 진행 상황을 **비용과 일정** 관점에서 측정하는 기법입니다.

### 핵심 지표 3가지

#### 1. **PV (Planned Value)** - 계획 가치
- **의미**: "이 시점까지 쓰기로 한 예산"
- **비유**: 여행 예산 100만 원 중 5일차까지 50만 원 쓰기로 계획

#### 2. **EV (Earned Value)** - 획득 가치
- **의미**: "실제로 완료한 작업의 가치"
- **비유**: 5일차인데 계획의 60%만 완료 → 60만 원 가치

#### 3. **AC (Actual Cost)** - 실제 비용
- **의미**: "실제로 쓴 돈"
- **비유**: 5일차까지 실제로 70만 원 사용

### 성과 지표

#### **SV (Schedule Variance)** - 일정 차이
```
SV = EV - PV
```
- **양수**: 일정 앞섬 (좋음)
- **음수**: 일정 지연 (나쁨)
- **예시**: 60만 원 - 50만 원 = +10만 원 (일정 앞섬!)

#### **CV (Cost Variance)** - 비용 차이
```
CV = EV - AC
```
- **양수**: 예산 절감 (좋음)
- **음수**: 예산 초과 (나쁨)
- **예시**: 60만 원 - 70만 원 = -10만 원 (예산 초과!)

### 실생활 예시
**다이어트 목표**: 3개월에 6kg 감량

| 기준 | 계획 | 실제 | 분석 |
|------|------|------|------|
| **PV** | 1개월에 2kg 감량 계획 | - | - |
| **EV** | - | 실제 1.5kg 감량 | 계획의 75% 달성 |
| **AC** | 100시간 운동 계획 | 120시간 운동함 | 노력 20% 초과 |
| **결론** | - | - | 목표 대비 부진하고, 노력도 더 들어감 |

## 7️⃣ 크리티컬 패스(Critical Path)

### 정의
프로젝트에서 **가장 오래 걸리는 경로**로, 하나라도 지연되면 전체 프로젝트가 지연됩니다.

### 실생활 비유
**아침 출근 준비**를 생각해보세요:

```
A. 씻기 (15분) → B. 옷 입기 (5분) → D. 출발 (0분)
                                        ↑
C. 아침 먹기 (10분) ──────────────────┘
```

- **경로 1**: A→B→D = 20분
- **경로 2**: C→D = 10분
- **크리티컬 패스**: 경로 1 (20분) ← 이게 늦어지면 전체가 늦어짐!

C(아침 먹기)는 10분 여유(Float)가 있어서 조금 늦어도 괜찮습니다.

### PM이 알아야 하는 이유
- ⚠️ 크리티컬 패스 작업은 **최우선 관리**
- ⏰ 일정 단축 시 크리티컬 패스에 자원 집중
- 📊 크리티컬 패스는 변할 수 있음 (다른 경로가 더 길어질 수 있음)

### 간단 예시: 웹사이트 구축
```
디자인(10일) → 개발(20일) → 테스트(5일) = 35일 (크리티컬!)
                     ↓
              문서작성(3일) = 3일 (여유 있음)
```

개발이 늦어지면 전체가 늦어지지만, 문서작성은 여유가 있어서 다소 늦어도 OK.

## 💬 자주 묻는 질문 (FAQ)

### Q1: PMBOK를 다 외워야 하나요?
**A**: 아니요! PMBOK는 **참고 자료**입니다. 핵심 개념과 프로세스 흐름을 이해하는 게 중요하고, 필요할 때 찾아보면 됩니다. PMP 시험을 준비하는 게 아니라면 완벽히 외울 필요 없습니다.

### Q2: 5대 프로세스 그룹을 항상 순서대로 해야 하나요?
**A**: 기본적으로 순차적이지만, 실무에서는 **반복**됩니다. 특히 실행과 감시/통제는 동시에 일어나고, 계획도 필요에 따라 재수립합니다. 애자일에서는 이런 반복이 더 자주 일어납니다.

### Q3: 10대 지식영역을 모두 사용해야 하나요?
**A**: 프로젝트 규모와 특성에 따라 다릅니다. 작은 프로젝트는 일부만 사용하고, 큰 프로젝트는 모두 사용합니다. 핵심은 **필요한 것을 선택적으로** 적용하는 것입니다.

### Q4: WBS를 얼마나 상세하게 분해해야 하나요?
**A**: **8/80 규칙**을 권장합니다. 한 작업이 8시간(1일) 미만이면 너무 세분화된 것이고, 80시간(2주) 이상이면 더 분해해야 합니다. 팀 규모와 관리 스타일에 따라 조정하세요.

### Q5: EVM이 어렵게 느껴져요. 꼭 써야 하나요?
**A**: EVM은 강력하지만 모든 프로젝트에 필수는 아닙니다. **큰 프로젝트**나 **공식 보고가 필요한 경우**에 유용합니다. 작은 프로젝트는 간단한 진척률(%)만 추적해도 충분합니다.

### Q6: PMBOK 6판과 7판의 차이는 뭔가요?
**A**: 
- **6판**: 프로세스 중심 (49개 프로세스)
- **7판**: 원칙 중심 (12가지 원칙), 더 유연하고 애자일 친화적

이 과정은 실무에서 여전히 많이 사용되는 **6판 기준**입니다.

### Q7: 크리티컬 패스가 여러 개일 수 있나요?
**A**: 네! 두 경로가 동일하게 가장 길면 둘 다 크리티컬 패스입니다. 이 경우 리스크가 더 높습니다. 하나만 관리해도 되는 게 아니라 **여러 경로를 동시에** 관리해야 하기 때문입니다.
