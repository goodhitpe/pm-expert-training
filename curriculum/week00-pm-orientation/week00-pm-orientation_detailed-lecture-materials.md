## Part 1: 통합 학습의 필요성 (60분)

### 1.1 현대 PM의 역할 변화

#### 전통적 PM (1990년대-2000년대)
```
역할:
- 일정 관리자 (Schedule Manager)
- 예산 통제자 (Budget Controller)
- 문서 작성자 (Documentation Writer)

요구 역량:
- MS Project 활용
- 엑셀 스프레드시트
- 문서 작성 능력
```

#### 현대 PM (2020년대 이후)
```
역할:
- 기술 리더십 (Technical Leadership)
- 전략적 의사결정자 (Strategic Decision Maker)
- 크로스-펑셔널 협업자 (Cross-functional Collaborator)

요구 역량:
- PMBOK 방법론 + 애자일
- 기술 트렌드 이해 (클라우드, AI, DevOps)
- 데이터 기반 의사결정
- 개발자/엔지니어와의 소통 능력
```

### 1.2 실무 사례: 기술 이해도가 프로젝트 성패를 좌우한 사례

#### 사례 1: 전자상거래 플랫폼 구축 프로젝트

**프로젝트 개요:**
- 예산: $500,000
- 기간: 6개월
- 목표: 일 10만 명 동시 접속 가능한 쇼핑몰

**PM A (기술 지식 부족):**
```
Week 1: 킥오프 미팅
PM: "6개월 안에 완성하면 됩니다."
개발팀: "아키텍처 설계부터 해야 합니다."
PM: "빨리 코딩 시작하세요."

Week 8: 중간 점검
PM: "왜 50%밖에 안 됐나요?"
개발팀: "DB 스키마를 3번이나 변경했습니다."
PM: "그게 왜 문제죠?"

Week 12: 런칭 전날
QA: "동시 접속 100명만 되도 서버가 다운됩니다."
PM: "왜 처음부터 말 안 했나요?"
인프라: "아키텍처 설계 단계에서 말씀드렸습니다..."

결과:
- 프로젝트 9개월로 연장 (50% 지연)
- 예산 $200,000 추가 (40% 초과)
- 런칭 후 1주일간 장애 10회
- PM 교체됨
```

**PM B (기술 지식 보유):**
```
Week 1: 아키텍처 설계 회의
PM: "10만 동시 접속이면 마이크로서비스 아키텍처가 필요하겠네요."
개발팀: "맞습니다. 주문, 결제, 상품 서비스를 분리하겠습니다."
PM: "DB는 샤딩 고려하고, 캐시는 Redis 사용하죠."
인프라: "정확히 이해하셨습니다!"

Week 4: 초기 개발 완료
PM: "부하 테스트 결과 어떤가요?"
개발팀: "1만 동시 접속까지 안정적입니다."
PM: "Auto Scaling 설정으로 10만까지 대응 가능하게 해주세요."

Week 6: 성능 최적화
PM: "CDN 설정하고, 이미지는 WebP 포맷으로 변환했나요?"
프론트엔드: "네, 페이지 로딩 속도 70% 개선됐습니다."

Week 24 (6개월): 성공적 런칭
- 목표 기간 내 완료
- 예산 95% 사용 (5% 절감)
- 런칭 후 무장애 운영
- PM 승진, 팀 보너스 지급
```

**차이점 분석:**
| 항목 | PM A | PM B |
|-----|------|------|
| 아키텍처 이해 | ❌ 없음 | ✅ 마이크로서비스, DB 샤딩 이해 |
| 기술 의사결정 참여 | ❌ 개발팀 의존 | ✅ 기술 방향 제시 |
| 리스크 사전 식별 | ❌ 후반부에 발견 | ✅ 초기부터 부하 테스트 |
| 팀 신뢰도 | 낮음 | 높음 |
| 프로젝트 결과 | 실패 | 성공 |

### 1.3 PMBOK와 IT 기술의 시너지

#### 시너지 맵
```
        PMBOK 지식영역              IT 기술 영역           실무 적용
        ─────────────              ──────────            ────────
1. 범위 관리 (Scope)    +  소프트웨어 공학  =  현실적 WBS 작성
   - 요구사항 수집          - SDLC 이해          - 개발 단계별 작업 정의
   - WBS 작성               - 아키텍처           - 기술 부채 고려

2. 일정 관리 (Schedule) +  소프트웨어 공학  =  정확한 일정 산정
   - 활동 정의              - 테스팅 전략        - 단위/통합 테스트 기간
   - 기간 산정              - CI/CD              - 배포 자동화 시간

3. 비용 관리 (Cost)     +  클라우드 컴퓨팅  =  비용 최적화
   - 예산 수립              - AWS 요금 체계      - Reserved Instance
   - 원가 통제              - Auto Scaling       - 트래픽 기반 비용 예측

4. 품질 관리 (Quality)  +  보안 + 테스팅   =  품질 기준 구체화
   - 품질 계획              - OWASP Top 10       - 보안 취약점 0건
   - 품질 보증              - 테스트 커버리지    - 코드 커버리지 80%

5. 리스크 관리 (Risk)   +  네트워크 + 보안  =  기술 리스크 식별
   - 리스크 식별            - 네트워크 장애      - DDoS 대응 계획
   - 리스크 대응            - 데이터 유출        - 암호화, 백업 전략
```

## Part 2: 제약조건 관리 실무 (60분)

### 2.1 프로젝트 제약의 삼각형 (Triple Constraint)

```
         범위 (Scope)
            /\
           /  \
          /    \
         /  품질 \
        /  (Quality)\
       /____________\
    일정            비용
  (Time)         (Cost)

규칙:
- 한 변을 늘리면 다른 변도 영향 받음
- PM의 핵심 역할: 균형 유지
```

### 2.2 범위 관리: 기술 지식으로 범위 크리프 방지

#### 실무 시나리오: 추가 요구사항 대응

**상황:**
고객이 프로젝트 중간에 "소셜 로그인 기능 추가"를 요청

**❌ 기술 지식 없는 PM의 대응:**
```
고객: "소셜 로그인 기능 추가해주세요. 간단하죠?"
PM: "네, 개발팀에 전달하겠습니다."

개발팀: "OAuth 2.0 구현, 소셜 플랫폼 API 연동, 
        기존 인증 시스템 통합, 테스트 필요합니다."
PM: "그게 그렇게 복잡한가요?"

결과:
- 범위 크리프 발생
- 2주 일정 지연
- 추가 비용 발생
- 고객과의 갈등
```

**✅ 기술 지식 있는 PM의 대응:**
```
고객: "소셜 로그인 기능 추가해주세요."
PM: "이해했습니다. 다음 작업이 필요합니다:

1. OAuth 2.0 구현 (3일)
   - Google, Facebook, Kakao 각각 연동
   - 토큰 관리 시스템

2. 기존 시스템 통합 (2일)
   - 사용자 DB 스키마 수정
   - 세션 관리 통합

3. 보안 검토 (1일)
   - CSRF 토큰 검증
   - 리다이렉트 URI 검증

4. 테스트 (2일)
   - 각 플랫폼별 테스트
   - 기존 기능 회귀 테스트

총 소요: 8일, 추가 비용 $10,000

옵션:
A) 현재 프로젝트 범위에 추가 → 일정 2주 연장
B) Phase 2로 이연 → 현재 일정 유지
C) 다른 기능 축소 → 범위 교환

결정해주세요."

고객: "명확한 설명 감사합니다. 옵션 B로 하겠습니다."

결과:
- 명확한 영향도 분석
- 고객의 informed decision
- 범위 크리프 방지
- 신뢰 구축
```

### 2.3 일정 관리: SDLC 이해로 정확한 산정

#### 실무 워크샵: 웹 서비스 일정 산정

**요구사항:**
사용자 관리 기능 (CRUD: Create, Read, Update, Delete)

**❌ 기술 지식 없는 PM의 산정:**
```
작업                    기간
─────────────────────  ────
사용자 관리 기능 개발    5일

총계: 5일
```

**✅ 기술 지식 있는 PM의 산정:**
```
작업                              기간    담당
────────────────────────────────  ────  ──────
1. 요구사항 분석 및 설계            1일   PM + 개발
   - Use Case 정의
   - API 스펙 작성
   - DB 스키마 설계

2. 백엔드 개발                      3일   백엔드
   - User Model 생성
   - CRUD API 구현 (4개 엔드포인트)
   - 입력 검증 및 에러 처리
   - JWT 인증 적용

3. 프론트엔드 개발                  3일   프론트엔드
   - 사용자 목록 화면
   - 등록/수정 폼
   - 삭제 확인 모달
   - API 연동

4. 데이터베이스 작업                1일   DBA
   - 테이블 생성
   - 인덱스 설정
   - 마이그레이션 스크립트

5. 테스트                          2일   QA + 개발
   - 단위 테스트 (백엔드)
   - 통합 테스트 (API)
   - E2E 테스트 (프론트엔드)
   - 성능 테스트

6. 코드 리뷰 및 수정                1일   전체
   - 코드 리뷰
   - 피드백 반영

7. 문서화                          0.5일  개발
   - API 문서
   - 사용자 가이드

총계: 11.5일 → 버퍼 20% 추가 = 14일

리스크:
- DB 스키마 변경 시 +2일
- 복잡한 검증 로직 추가 시 +1일
```

**차이 분석:**
| 항목 | 기술 지식 없음 | 기술 지식 있음 |
|-----|-------------|-------------|
| 산정 기간 | 5일 | 14일 |
| 실제 소요 | 15일 (200% 지연) | 13일 (목표 달성) |
| 정확도 | 33% | 93% |

### 2.4 비용 관리: 클라우드 비용 최적화

#### 실무 사례: SaaS 서비스 운영 비용

**시나리오:**
월간 활성 사용자 10만 명, 일 트래픽 1TB

**❌ 기술 지식 없는 PM의 예산:**
```
항목                금액
─────────────────  ───────
개발 비용           $100,000
서버 비용 (추정)     $10,000/월

연간 총 비용: $220,000
```

**✅ 기술 지식 있는 PM의 예산:**
```
1. 개발 비용: $100,000
   - 프론트엔드: $30,000
   - 백엔드: $40,000
   - 인프라 설정: $10,000
   - QA: $20,000

2. 인프라 비용 (월간):

A. 컴퓨팅 (EC2/ECS)
   - 웹 서버: t3.medium × 3 = $300
   - API 서버: c5.large × 4 = $560
   - 워커: t3.small × 2 = $60
   소계: $920/월

B. 데이터베이스
   - RDS PostgreSQL (db.r5.large)
     → Reserved Instance (1년) = $450/월
     (On-Demand: $670/월, 33% 절감)
   - ElastiCache Redis = $100/월
   소계: $550/월

C. 스토리지
   - S3 (500GB): $11.50
   - S3 Intelligent-Tiering으로 30% 절감
   실제: $8/월

D. 네트워크
   - CloudFront CDN (1TB 트래픽): $85
   - Data Transfer Out: $90
   소계: $175/월

E. 기타 서비스
   - Lambda (이벤트 처리): $20
   - SES (이메일): $10
   - CloudWatch (모니터링): $30
   소계: $60/월

월간 총 인프라 비용: $1,713
연간 인프라 비용: $20,556

3. 운영 비용 (월간):
   - 백업 스토리지: $50
   - 로그 스토리지: $30
   - SSL 인증서: $0 (Let's Encrypt)
   소계: $80/월

총 연간 비용:
- 초기 개발: $100,000
- 인프라: $20,556
- 운영: $960
총계: $121,516

비용 최적화 전략:
1. Reserved Instances 활용 (33% 절감)
2. S3 Intelligent-Tiering (30% 절감)
3. CloudFront 캐시 최적화 (트래픽 비용 40% 절감)
4. Auto Scaling으로 유휴 시간 비용 절감

예상 절감액: 연간 $15,000
```

**투자 대비 효과:**
- 정확한 예산 수립으로 경영진 신뢰 확보
- 비용 최적화로 수익성 15% 개선
- 확장 가능한 아키텍처로 성장 대응

## Part 3: 이해관계자 의사소통 (60분)

### 3.1 기술 팀과의 효과적 의사소통

#### 워크샵: 스프린트 플래닝 미팅 시뮬레이션

**상황:**
2주 스프린트 시작, 팀 역량 40 Story Points

**참석자:**
- PM (당신)
- 백엔드 개발자 2명
- 프론트엔드 개발자 2명
- QA 1명

**스프린트 백로그 (우선순위 순):**
1. 결제 시스템 구현 (21 SP)
2. 상품 검색 개선 (13 SP)
3. 사용자 알림 기능 (8 SP)
4. 관리자 대시보드 (13 SP)
5. 성능 최적화 (5 SP)

**❌ 기술 지식 없는 PM의 진행:**
```
PM: "이번 스프린트에 1, 2, 3번 모두 해주세요."
개발자: "그럼 55 SP인데 역량이 40 SP밖에 안 됩니다."
PM: "야근하면 되지 않나요?"

결과:
- 팀 사기 저하
- 품질 저하
- 번아웃
```

**✅ 기술 지식 있는 PM의 진행:**
```
PM: "먼저 각 항목의 기술적 의존성을 확인하겠습니다.
     결제 시스템은 외부 PG사 API 연동이 있나요?"

백엔드: "네, 이니시스 API 연동이 필요합니다."

PM: "API 문서는 받았나요? 샌드박스 테스트 가능한가요?"

백엔드: "네, 준비되어 있습니다."

PM: "좋습니다. 그럼 결제 시스템이 1순위입니다.
     상품 검색은 Elasticsearch 도입 건이죠?"

백엔드: "네, 맞습니다."

PM: "그럼 인프라 팀에 Elasticsearch 클러스터 준비 요청했나요?"

백엔드: "아직입니다."

PM: "그럼 이번 스프린트에는 어렵겠네요. 
     인프라 준비만 1주 걸립니다.
     이번 스프린트: 결제 시스템 + 알림 기능 (29 SP)
     다음 스프린트: 검색 개선 (ES 준비 완료 후)
     
     알림 기능은 FCM 사용하면 백엔드 부담이 적습니다.
     프론트엔드에서 처리 가능한가요?"

프론트엔드: "네, 가능합니다!"

PM: "그럼 남은 11 SP는 성능 최적화에 쓰겠습니다.
     DB 쿼리 최적화와 이미지 lazy loading 어떤가요?"

개발자들: "완벽합니다!"

결과:
- 현실적인 계획
- 팀 동의
- 높은 완료율
```

### 3.2 경영진 보고: 기술을 비즈니스 언어로 번역

#### 실무 사례: 마이크로서비스 전환 제안

**상황:**
모놀리식 아키텍처의 한계로 마이크로서비스 전환 필요

**❌ 기술 중심 보고:**
```
PM: "모놀리식을 마이크로서비스로 전환하겠습니다."

CEO: "그게 뭔가요?"

PM: "컨테이너화하고 쿠버네티스로 오케스트레이션하는 겁니다."

CEO: "비용은? 기간은? 효과는?"

PM: "개발팀에 물어봐야 할 것 같습니다."

결과: 승인 거부
```

**✅ 비즈니스 중심 보고:**
```
PM 보고서:

제목: "서비스 확장성 확보를 위한 아키텍처 개선 제안"

1. 현재 문제점 (비즈니스 영향)
   
   A. 배포 시간 지연
      - 현재: 한 줄 수정에도 전체 배포 필요 (2시간)
      - 영향: 긴급 버그 수정 지연 → 고객 불만
      - 손실: 배포 중 서비스 중단 → 시간당 $5,000 손실

   B. 장애 전파
      - 현재: 한 기능 장애 시 전체 서비스 다운
      - 사례: 지난달 결제 모듈 장애로 쇼핑몰 전체 4시간 다운
      - 손실: $20,000 + 고객 신뢰 하락

   C. 확장의 한계
      - 현재: 트래픽 증가 시 전체 서버 증설 필요
      - 문제: 비효율적 자원 사용 (CPU 20% 사용률)
      - 낭비: 연간 $50,000 인프라 비용 낭비

2. 제안 솔루션: 마이크로서비스 아키텍처

   핵심 개념 (경영진 버전):
   "레고 블록처럼 서비스를 분리하여 
    각각 독립적으로 개발/배포/확장"

   기대 효과:
   
   A. 배포 속도 개선
      - 현재 2시간 → 개선 후 15분 (88% 단축)
      - 효과: 하루 3회 배포 가능 → 빠른 시장 대응

   B. 장애 격리
      - 한 서비스 장애가 다른 서비스에 영향 없음
      - 효과: 가용성 99.9% → 99.99% (다운타임 1/10)
      - 절감: 연간 장애 손실 $80,000 → $8,000

   C. 비용 최적화
      - 필요한 서비스만 확장
      - 효과: Auto Scaling으로 트래픽 대응
      - 절감: 연간 인프라 비용 $50,000 절감

3. 투자 계획

   초기 투자: $200,000
   - 아키텍처 재설계: $50,000
   - 개발 인력 (6개월): $120,000
   - 인프라 이중화 (전환 기간): $30,000

   연간 운영 비용: 기존과 동일

4. ROI 분석

   연간 편익:
   - 장애 손실 절감: $72,000
   - 인프라 비용 절감: $50,000
   - 개발 생산성 향상: $80,000
   총 편익: $202,000/년

   투자 회수 기간: 12개월
   3년 NPV: $406,000

5. 리스크 및 대응

   리스크 1: 전환 중 서비스 중단
   → 대응: Blue-Green 배포로 무중단 전환

   리스크 2: 팀 학습 곡선
   → 대응: 외부 전문가 멘토링 + 파일럿 프로젝트

   리스크 3: 일정 지연
   → 대응: 단계별 전환 (6개월 → 3단계)

6. 실행 계획

   Phase 1 (Month 1-2): 결제 서비스 분리
   Phase 2 (Month 3-4): 상품/주문 서비스 분리
   Phase 3 (Month 5-6): 사용자/인증 서비스 분리

   각 단계마다 검증 후 다음 단계 진행

CEO: "명확한 분석입니다. 승인합니다."

결과: 승인, 프로젝트 시작
```

**핵심 차이:**
| 항목 | 기술 중심 | 비즈니스 중심 |
|-----|---------|------------|
| 언어 | 기술 용어 | 비즈니스 가치 |
| 초점 | How (어떻게) | Why (왜), What (무엇) |
| 근거 | 기술적 필요성 | ROI, 리스크, 전략 |
| 결과 | 이해 부족 | 이해 + 승인 |

## Part 4: 실습 활동 (30분)

### 활동 1: 제약조건 균형 게임 (15분)

**시나리오:**
모바일 앱 프로젝트 중 고객이 다음 3가지 요청을 동시에 제시

1. 기능 추가: "AI 추천 기능 추가해주세요"
2. 일정 단축: "런칭을 2주 앞당겨주세요"
3. 비용 절감: "예산 10% 줄여주세요"

**과제:**
기술 지식을 활용하여 고객에게 제시할 3가지 옵션을 만드세요.

**모범 답안 예시:**
```
옵션 A: AI 추천 기능 우선
- 범위: AI 기능 추가
- 일정: 기존 일정 유지 (2주 연장 불가)
- 비용: $20,000 추가
- 대신: 다른 기능 2개 축소
- 기술: Cloud ML API 활용으로 개발 시간 단축

옵션 B: 일정 우선
- 범위: AI 기능 → 간단한 추천 알고리즘으로 대체
- 일정: 2주 단축 가능
- 비용: 기존 유지
- 방법: MVP 접근, Phase 2로 AI 이연

옵션 C: 비용 우선
- 범위: 핵심 기능만 (AI 제외)
- 일정: 기존 유지
- 비용: 10% 절감
- 방법: 오픈소스 활용, 클라우드 비용 최적화
```

### 활동 2: 의사소통 롤플레이 (15분)

**시나리오:**
개발자가 "기술 부채 해소를 위한 리팩토링 2주 필요"라고 요청

**역할:**
- A조: PM 역할 (기술 이해)
- B조: 경영진 역할 (비즈니스 중심)

**과제:**
PM이 경영진에게 리팩토링의 필요성을 설득하는 대화 만들기

**모범 답안 예시:**
```
PM: "현재 코드베이스에 기술 부채가 누적되어 있습니다."

경영진: "기술 부채요?"

PM: "쉽게 말씀드리면, 빨리 만들다 보니 코드가 복잡해져서
     새로운 기능 추가할 때 시간이 2배로 걸립니다."

경영진: "구체적으로 얼마나 손해인가요?"

PM: "현재 신규 기능 하나 개발에 4주 걸립니다.
     리팩토링 후에는 2주로 단축 가능합니다.
     
     계산:
     - 리팩토링 투자: 2주 ($10,000)
     - 분기당 신규 기능 5개
     - 절감: 5개 × 2주 = 10주 ($50,000)
     - 분기 ROI: 400%"

경영진: "버그는 안 생기나요?"

PM: "리팩토링 중 자동화 테스트를 100% 작성합니다.
     오히려 버그가 줄어듭니다.
     현재 버그 수정에 주당 5시간 소요
     → 리팩토링 후 1시간으로 감소"

경영진: "언제 시작하면 되나요?"

PM: "다음 스프린트부터 단계적으로 진행하겠습니다."
```

## 📊 오리엔테이션 요약

### 핵심 메시지

1. **기술 지식은 선택이 아닌 필수**
   - 현대 PM의 80%가 기술 배경 또는 기술 교육 이수
   - 기술 이해 없이는 정확한 의사결정 불가능

2. **PMBOK + IT 기술 = 시너지**
   - PMBOK: 관리 프레임워크 제공
   - IT 기술: 실무 적용 능력 제공
   - 통합: 탁월한 PM

3. **제약조건 관리의 핵심은 기술 이해**
   - 범위: 기술 실현 가능성 판단
   - 일정: SDLC 기반 정확한 산정
   - 비용: 인프라/라이선스 비용 예측
   - 품질: 기술 품질 기준 설정

4. **의사소통은 양방향**
   - 개발팀: 기술 용어로 소통
   - 경영진: 비즈니스 가치로 번역
