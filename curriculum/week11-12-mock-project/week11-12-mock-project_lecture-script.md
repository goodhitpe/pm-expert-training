## 🎯 모의 프로젝트 구조

### 프로젝트 특성
```
📊 프로젝트 시나리오 (3가지 중 선택)

Option 1: 전자상거래 플랫폼 구축
- 목표: B2C 온라인 쇼핑몰 개발
- 기간: 4개월 (가상)
- 예산: $200,000
- 팀: 8명 (PM, 개발자 4, 디자이너 1, QA 2)
- 핵심 기능: 상품 관리, 장바구니, 결제, 리뷰

Option 2: 모바일 헬스케어 앱 개발
- 목표: 운동/식단 관리 모바일 앱
- 기간: 3개월 (가상)
- 예산: $150,000
- 팀: 6명 (PM, 개발자 3, 디자이너 1, QA 1)
- 핵심 기능: 운동 기록, 식단 분석, AI 추천

Option 3: 사내 협업 시스템 구축
- 목표: 기업용 협업 플랫폼
- 기간: 5개월 (가상)
- 예산: $250,000
- 팀: 10명 (PM, 개발자 5, 디자이너 2, QA 2)
- 핵심 기능: 메신저, 문서 공유, 일정 관리

💡 선택 기준
- 팀 관심사
- 실무 연관성
- 기술 스택 친숙도
```

### 팀 구성
```
👥 팀 편성 (3-5명/팀)

역할 분담:
1. 프로젝트 매니저 (PM) - 1명
   - 전체 프로젝트 총괄
   - 계획 수립 및 진척 관리
   - 이해관계자 커뮤니케이션

2. 기술 리드 (Tech Lead) - 1명
   - 기술 의사결정
   - 아키텍처 설계
   - 개발자 지원

3. 개발자 (Developers) - 1-2명
   - 기능 개발
   - 코드 리뷰
   - 단위 테스트

4. QA 엔지니어 - 1명
   - 테스트 계획 수립
   - 품질 검증
   - 버그 관리

💡 강사의 역할
- 프로젝트 스폰서 (승인 권한)
- 코치/멘토 (조언 제공)
- 이해관계자 (요구사항 제시)
- 평가자 (최종 평가)

"강사는 답을 주지 않고, 질문을 통해 스스로 찾도록 돕습니다"
```

### [01:45-03:15] 프로젝트 계획 수립 (90분)

**활동 내용**:
```
1️⃣ 범위 정의 및 WBS 작성 (45분)

📋 범위 명세서 작성:
- 프로젝트 범위
- 인도물 목록
- 인수 기준
- 제외 사항 (Out of Scope)

🌳 WBS (Work Breakdown Structure) 작성:
- 3단계까지 분해
- 작업 패키지 50-100개 수준
- 포스트잇으로 화이트보드에 붙이며 작성
- 디지털화 (Excel, MindMap)

예시 WBS:
1. 프로젝트 관리
   1.1 착수
   1.2 계획
   1.3 감시/통제
2. 요구사항 분석
   2.1 사용자 요구사항
   2.2 시스템 요구사항
3. 설계
   3.1 UI/UX 설계
   3.2 시스템 설계
4. 개발
   4.1 백엔드 개발
   4.2 프론트엔드 개발
5. 테스트
   5.1 단위 테스트
   5.2 통합 테스트
6. 배포
   6.1 배포 준비
   6.2 운영 이관

💡 코칭 포인트:
"이 작업을 더 작게 나눌 수 있나요?"
"이 작업의 완료 기준은 무엇인가요?"

2️⃣ 일정 계획 수립 (45분)

📅 활동 순서:
1. 활동 순서 배열
   - 선후행 관계 정의
   - 의존성 파악
   - 네트워크 다이어그램

2. 활동 기간 산정
   - 각 작업 패키지의 소요 시간 추정
   - 3점 추정법 활용 (낙관/가능/비관)
   - 버퍼 시간 고려

3. 간트 차트 작성
   - MS Project 또는 Excel
   - 주요 마일스톤 표시
   - 임계 경로 식별

📊 마일스톤 예시:
- M1: 요구사항 확정 (Week 2)
- M2: 설계 완료 (Week 4)
- M3: 개발 완료 (Week 10)
- M4: 테스트 완료 (Week 13)
- M5: 최종 배포 (Week 16)

💻 도구 활용:
- Jira Timeline
- MS Project
- Excel 간트 차트
- Miro (온라인 협업)

💡 코칭 포인트:
"임계 경로가 무엇인가요? 왜 중요한가요?"
"이 일정이 현실적인가요?"
```

**교수법**:
- 팀별 작업, 강사 순회 코칭

**활동 내용**:
```
1️⃣ 리스크 관리 계획 (45분)

📋 리스크 등록부 작성:
1. 리스크 식별 (브레인스토밍)

3. 리스크 대응 전략
   - 회피 (Avoid)
   - 완화 (Mitigate)
   - 전가 (Transfer)
   - 수용 (Accept)

4. 리스크 책임자 배정
   - 각 리스크마다 담당자
   - 모니터링 계획

예시 리스크:
| ID | 리스크 | 가능성 | 영향 | 점수 | 대응 전략 | 책임자 |
|----|--------|--------|------|------|-----------|--------|
| R1 | 주요 개발자 이탈 | 3 | 5 | 15 | 완화: 백업 인력 준비 | PM |
| R2 | API 통합 지연 | 4 | 4 | 16 | 완화: 조기 PoC | Tech Lead |
| R3 | 보안 취약점 발견 | 3 | 5 | 15 | 완화: 보안 테스트 조기 시작 | QA |

💡 코칭 포인트:
"이 리스크가 실제로 발생하면 어떻게 될까요?"
"대응 전략의 비용은 얼마나 될까요?"

2️⃣ 예산 계획 (30분)

💰 예산 구조:
1. 인건비 (60-70%)
   - PM, 개발자, 디자이너, QA 인건비

2. 인프라 비용 (10-15%)

   - 데이터베이스
   - CDN, 스토리지

3. 소프트웨어 라이선스 (5-10%)
   - 개발 도구
   - 디자인 툴
   - 프로젝트 관리 도구

4. 기타 비용 (10-15%)
   - 교육/훈련
   - 출장
   - 예비비 (10%)

📊 예산 산출:
- 각 항목별 상세 산출

- 누적 예산 곡선 (S-Curve)

💡 코칭 포인트:
"예산이 부족하면 어디서 줄일 수 있나요?"
"예비비가 왜 필요한가요?"
```

   - 실제 프로젝트처럼 긴장감 조성

   - 막히면 힌트 제공
   - 완전히 틀린 방향이면 즉시 수정
   - 하지만 스스로 해결하도록 유도

5. **지속적 피드백**
   - 즉각적이고 구체적인 피드백
   - 긍정과 개선 균형
   - 팀별 맞춤 조언
