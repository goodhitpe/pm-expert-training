## 1. WBS (Work Breakdown Structure) / 작업 분해 구조

### 용어
**WBS (Work Breakdown Structure) / 작업 분해 구조**

### 설명
프로젝트 범위를 관리 가능한 작은 구성 요소로 계층적으로 분해한 구조입니다. WBS는 프로젝트의 전체 작업을 체계적으로 정리하여 계획, 실행, 통제의 기초를 제공하는 프로젝트 관리의 핵심 도구입니다.

### 세부 요소 및 설명

#### 1.1 WBS의 목적

**주요 목적**
1. **범위의 시각화**: 프로젝트 전체 범위를 한눈에 파악
2. **작업 추정의 기초**: 각 작업의 시간, 비용, 자원 추정
3. **책임 배분**: 작업 패키지별 담당자 할당
4. **진척도 측정**: 완료된 작업 추적 기준
5. **커뮤니케이션 도구**: 이해관계자 간 공통 이해 형성

**WBS의 중요성**
- 프로젝트 실패의 30%가 불명확한 범위 정의에서 비롯
- WBS 없는 프로젝트는 작업 누락, 중복, 혼란 발생
- 일정, 예산, 자원 계획의 기초

#### 1.2 WBS 구성 요소

**계층 구조**
```
Level 0: 프로젝트 (Project)
  ├─ Level 1: 주요 인도물 또는 단계 (Major Deliverables/Phases)
  │   ├─ Level 2: 하위 인도물 또는 활동 (Sub-deliverables/Activities)
  │   │   ├─ Level 3: 세부 활동 (Detailed Activities)
  │   │   │   └─ Level 4: 작업 패키지 (Work Packages)
```

**예시: 모바일 앱 개발 프로젝트**
```
모바일 앱 개발 프로젝트
├─ 1. 프로젝트 관리
│   ├─ 1.1 프로젝트 계획
│   ├─ 1.2 진행 관리
│   └─ 1.3 프로젝트 종료
├─ 2. 요구사항 분석
│   ├─ 2.1 사용자 조사
│   ├─ 2.2 요구사항 수집
│   └─ 2.3 요구사항 문서화
├─ 3. 설계
│   ├─ 3.1 UI/UX 설계
│   ├─ 3.2 시스템 아키텍처 설계
│   └─ 3.3 데이터베이스 설계
├─ 4. 개발
│   ├─ 4.1 프론트엔드 개발
│   ├─ 4.2 백엔드 개발
│   └─ 4.3 API 통합
├─ 5. 테스트
│   ├─ 5.1 단위 테스트
│   ├─ 5.2 통합 테스트
│   └─ 5.3 사용자 수용 테스트
└─ 6. 배포
    ├─ 6.1 앱스토어 등록
    ├─ 6.2 사용자 교육
    └─ 6.3 출시 및 모니터링
```

#### 1.3 WBS 작성 원칙

**1. 100% 규칙 (100% Rule)**
- 상위 항목은 하위 항목의 100%를 포함해야 함
- 모든 프로젝트 작업이 WBS에 포함되어야 함
- 누락된 작업이 없어야 함

**2. 상호 배타성 (Mutual Exclusivity)**
- 동일 레벨의 항목들은 서로 중복되지 않아야 함
- 각 작업은 한 곳에만 위치
- 명확한 경계 설정

**3. 결과 지향 (Outcome-Oriented)**
- 작업(동사)이 아닌 결과물(명사) 중심
- 나쁜 예: "데이터베이스 개발하기"
- 좋은 예: "데이터베이스 스키마"

**4. 적절한 분해 수준**
- **8-80 시간 규칙**: 작업 패키지는 8-80시간 소요
- **보고 주기 규칙**: 보고 주기(예: 1주) 내 완료 가능
- 너무 상세하면 관리 부담, 너무 거시적이면 통제 어려움

**5. 검증 가능성**
- 각 작업의 완료 여부를 명확히 판단 가능
- 측정 가능한 인도물 정의

#### 1.4 WBS 작성 방법

**하향식 접근 (Top-Down Approach)**
1. 프로젝트 전체부터 시작
2. 주요 인도물 또는 단계로 분해
3. 각 항목을 더 작은 구성 요소로 분해
4. 작업 패키지 수준까지 반복

**장점**: 체계적, 전체 구조 명확
**단점**: 세부 작업 누락 가능

**상향식 접근 (Bottom-Up Approach)**
1. 모든 세부 작업 나열 (브레인스토밍)
2. 유사한 작업 그룹화
3. 그룹을 더 큰 카테고리로 묶음
4. 계층 구조 형성

**장점**: 작업 누락 최소화
**단점**: 시간 소요, 중복 가능

**하이브리드 접근**
- 상위 레벨은 하향식
- 하위 레벨은 상향식
- 실무에서 가장 많이 사용

#### 1.5 WBS 표현 방법

**1. 계층도 (Hierarchical/Tree Structure)**
```
프로젝트
  ├─ 1.0 설계
  │   ├─ 1.1 UI 설계
  │   └─ 1.2 DB 설계
  └─ 2.0 개발
      ├─ 2.1 프론트엔드
      └─ 2.2 백엔드
```
- 가장 직관적
- 전체 구조 한눈에 파악

**2. 들여쓰기 목록 (Outline/Indented List)**
```
1. 설계
  1.1 UI 설계
  1.2 DB 설계
2. 개발
  2.1 프론트엔드
  2.2 백엔드
```
- 간단하고 작성 용이
- 문서 작성에 적합

**3. 표 형식 (Tabular)**
| WBS ID | 작업명 | 레벨 | 담당자 |
|--------|--------|------|--------|
| 1.0 | 설계 | 1 | 팀A |
| 1.1 | UI 설계 | 2 | 디자이너 |

- 추가 정보 포함 용이
- 스프레드시트 활용

#### 1.6 WBS 사전 (WBS Dictionary)

각 작업 패키지의 상세 정보를 기술한 문서:

**포함 내용**
- **WBS ID**: 고유 식별자 (예: 1.2.3)
- **작업명**: 작업 패키지 이름
- **작업 설명**: 수행할 작업의 상세 내용
- **담당자**: 책임자 및 팀
- **선행 작업**: 이 작업 전에 완료되어야 할 작업
- **소요 기간**: 예상 작업 기간
- **인도물**: 작업 완료 시 생성되는 산출물
- **승인 기준**: 작업 완료를 판단하는 기준
- **필요 자원**: 인력, 장비, 자재 등

**예시**
```
WBS ID: 3.2
작업명: 데이터베이스 스키마 설계
설명: 앱의 데이터 구조를 정의하고 ER 다이어그램 작성
담당자: DB 설계자 (김철수)
선행 작업: 2.3 요구사항 문서화
소요 기간: 5일
인도물: ER 다이어그램, 테이블 정의서
승인 기준: 아키텍트 검토 및 승인
필요 자원: DB 설계 도구, 요구사항 문서
```

#### 1.7 WBS와 다른 도구의 관계

**WBS → 일정 (Schedule)**
- WBS의 작업 패키지가 일정의 활동(Activity)이 됨
- 활동 순서, 기간 추정, 간트 차트 작성

**WBS → 예산 (Budget)**
- 작업 패키지별 비용 추정
- 상향식 예산 산정

**WBS → 자원 배분**
- 작업 패키지별 필요 자원 식별
- 자원 할당 및 조달 계획

**WBS → 리스크 식별**
- 각 작업 패키지의 잠재적 리스크 파악

### 추가 설명

**WBS 작성 시 흔한 실수**

1. **동사 사용 (작업 중심)**
   - 잘못: "데이터베이스 개발하기"
   - 올바름: "데이터베이스"

2. **불균형한 분해**
   - 일부는 너무 상세, 일부는 너무 거시적
   - 동일 레벨은 유사한 상세도 유지

3. **프로젝트 관리 작업 누락**
   - WBS에 PM 활동도 포함 필요
   - 예: 회의, 보고서 작성, 리스크 관리

4. **100% 규칙 위반**
   - 일부 작업 누락
   - 상위 항목이 하위 항목의 합보다 큼

5. **중복 작업**
   - 동일한 작업이 여러 곳에 존재

**WBS 검증 방법**
- 팀 리뷰: 모든 팀원이 검토
- 전문가 판단: 경험 있는 PM 검토
- 체크리스트: 100% 규칙, 상호 배타성 등 확인
- 과거 프로젝트 비교: 유사 프로젝트의 WBS 참조

## 2. 작업 패키지 (Work Package)

### 용어
**작업 패키지 (Work Package)**

### 설명
WBS의 최하위 레벨에 위치한 관리 가능한 최소 단위의 작업입니다. 작업 패키지는 명확하게 정의되고, 측정 가능하며, 담당자가 할당된 실질적인 작업 단위입니다.

### 세부 요소 및 설명

#### 2.1 작업 패키지의 특징

**필수 특성**
1. **명확성 (Clarity)**: 무엇을 해야 하는지 명확히 정의
2. **측정 가능성 (Measurability)**: 완료 여부를 객관적으로 판단 가능
3. **책임 배정 (Accountability)**: 명확한 담당자 존재
4. **독립성 (Independence)**: 다른 작업 패키지와 독립적으로 수행 가능
5. **완결성 (Completeness)**: 명확한 시작과 종료 지점

**적절한 크기**
- **8-80 시간 규칙**: 일반적으로 8시간(1일)에서 80시간(2주) 소요
- **보고 주기**: 진행 보고 주기 내에 완료 가능한 크기
- **프로젝트 규모에 따라 조정**: 대규모 프로젝트는 더 클 수 있음

#### 2.2 작업 패키지의 구성 요소

**기본 정보**
- **WBS ID**: 고유 식별 번호 (예: 3.2.1)
- **작업명**: 간결하고 명확한 이름
- **설명**: 작업 내용의 상세 설명

**일정 정보**
- **시작일**: 예상 시작 날짜
- **종료일**: 예상 완료 날짜
- **소요 기간**: 순수 작업 시간
- **선행 작업**: 이 작업 전에 완료되어야 할 작업들
- **후속 작업**: 이 작업 후에 시작할 작업들

**자원 정보**
- **담당자**: 작업을 책임지는 사람 (1명)
- **참여자**: 작업에 참여하는 팀원들
- **필요 자원**: 장비, 도구, 자재 등
- **예상 공수**: 인원 x 시간 (예: 2인 x 5일 = 10인일)

**비용 정보**
- **예상 비용**: 작업 수행에 필요한 총 비용
- **비용 항목**: 인건비, 자재비, 외주비 등

**품질 정보**
- **인도물**: 작업 완료 시 생성되는 산출물
- **승인 기준**: 작업 완료를 판단하는 명확한 기준
- **검증 방법**: 품질 확인 방법

#### 2.3 작업 패키지 정의 프로세스

**1단계: 작업 식별**
- WBS 분해를 통해 작업 패키지 도출
- 모든 프로젝트 작업이 작업 패키지로 표현

**2단계: 작업 상세화**
- WBS 사전 작성
- 각 작업 패키지의 상세 정보 기술

**3단계: 담당자 할당**
- 적절한 역량을 가진 담당자 배정
- RACI 차트 작성

**4단계: 기간 및 비용 추정**
- 전문가 판단, 유사 추정, 상향식 추정 등
- 예비비 고려

**5단계: 검증 및 승인**
- 담당자와 협의
- 프로젝트 팀 검토
- PM 승인

#### 2.4 작업 패키지 vs 활동 (Activity)

| 구분 | 작업 패키지 | 활동 |
|------|------------|------|
| 정의 | WBS의 최하위 작업 | 일정 계획의 작업 단위 |
| 출처 | WBS | 작업 패키지를 더 분해 |
| 목적 | 범위 정의 | 일정 수립 |
| 크기 | 더 큼 (수일~수주) | 더 작음 (수시간~수일) |
| 관계 | 1개 작업 패키지 → 1개 이상의 활동 |

**변환 예시**
```
작업 패키지: 3.2 데이터베이스 스키마 설계
  ↓ 분해
활동들:
- 3.2.1 엔터티 정의
- 3.2.2 관계 정의
- 3.2.3 ER 다이어그램 작성
- 3.2.4 정규화 수행
- 3.2.5 테이블 정의서 작성
```

#### 2.5 작업 패키지 관리

**진척도 추적**
- 시작됨 / 진행 중 / 완료됨
- 완료율 (0%, 25%, 50%, 75%, 100%)
- Earned Value (EV) 계산 기준

**이슈 관리**
- 작업 패키지별 이슈 추적
- 지연 원인 분석
- 대응 조치

**변경 관리**
- 작업 패키지 범위 변경
- 담당자 변경
- 기간/비용 변경

### 추가 설명

**작업 패키지 정의의 중요성**

명확한 작업 패키지 정의는:
1. **정확한 추정**: 세부적일수록 추정 정확도 향상
2. **명확한 책임**: 담당자가 무엇을 해야 하는지 확실히 앎
3. **효과적인 추적**: 진척도를 객관적으로 측정 가능
4. **리스크 감소**: 작업 누락이나 오해 최소화

**작업 패키지 크기 결정 요인**

1. **프로젝트 규모**: 대규모일수록 작업 패키지도 클 수 있음
2. **리스크 수준**: 고위험 작업은 더 작게 분해
3. **복잡도**: 복잡할수록 작게 분해
4. **통제 필요성**: 긴밀한 통제가 필요하면 작게
5. **보고 빈도**: 보고 주기 내 완료 가능한 크기

**실무 Tip**

- 작업 패키지명은 명사형으로 (결과물 중심)
- 각 작업 패키지는 1명의 책임자
- 여러 명이 참여해도 책임자는 1명
- 완료 기준을 명확히 정의
- 담당자와 충분히 협의하여 정의

## 3. 범위 크리프 (Scope Creep)

### 용어
**범위 크리프 (Scope Creep) / 범위 확대**

### 설명
공식적인 변경 통제 절차 없이 프로젝트 범위가 통제되지 않은 상태로 지속적으로 확대되는 현상입니다. 범위 크리프는 프로젝트 일정 지연, 예산 초과, 팀 소진의 주요 원인 중 하나입니다.

### 세부 요소 및 설명

#### 3.1 범위 크리프의 특징

**주요 특성**
1. **점진적 발생**: 한 번에 큰 변경이 아닌 작은 변경들이 누적
2. **비공식적**: 공식 변경 요청 없이 발생
3. **감지 어려움**: 초기에는 영향이 작아 눈치채기 어려움
4. **누적 효과**: 시간이 지나면서 큰 영향으로 누적

**범위 크리프의 원인**

1. **불명확한 초기 범위**
   - 모호한 요구사항
   - 불완전한 범위 정의
   - 가정과 제약조건 미정의

2. **약한 변경 통제**
   - 변경 관리 프로세스 부재
   - 변경 승인 절차 미준수
   - "작은 변경"이라는 이유로 승인 생략

3. **이해관계자 압력**
   - "간단한 추가"라는 요청
   - "이미 하는 김에" 추가 요구
   - 정치적 압력으로 인한 범위 추가

4. **커뮤니케이션 부족**
   - 범위에 대한 이해 차이
   - 기대사항 불일치
   - PM의 "No" 말하기 어려움

5. **기술적 문제**
   - 예상치 못한 기술적 복잡도
   - 통합 문제
   - 기술 변화

#### 3.2 범위 크리프의 영향

**직접적 영향**
- **일정 지연**: 추가 작업으로 완료일 연기
- **예산 초과**: 예상치 못한 비용 발생
- **품질 저하**: 시간 압박으로 품질 희생

**간접적 영향**
- **팀 사기 저하**: "끝나지 않는 프로젝트" 인식
- **번아웃**: 지속적인 야근과 스트레스
- **이해관계자 불만**: 목표일 못 맞춤
- **프로젝트 실패**: 심각한 경우 프로젝트 중단

**측정 가능한 통계**
- 범위 크리프가 있는 프로젝트의 70%가 일정 지연
- 평균적으로 초기 범위 대비 30-50% 증가
- 프로젝트 실패의 45%가 범위 크리프와 연관

#### 3.3 범위 크리프 vs 정당한 변경

**범위 크리프 (피해야 함)**
- 공식 프로세스 없이 추가
- 영향 분석 없음
- 문서화되지 않음
- "작은 것"이라는 이유로 승인 생략

**정당한 범위 변경 (허용)**
- 공식 변경 요청서 제출
- 영향 분석 (일정, 비용, 품질)
- 변경 통제 위원회 (CCB) 검토
- 승인 후 기준선 업데이트
- 모든 이해관계자에게 통보

#### 3.4 범위 크리프 방지 전략

**1. 명확한 초기 범위 정의**
- 상세한 범위 기술서 작성
- WBS 작성
- 제외 범위(Out-of-Scope) 명시
- 승인 기준 명확화

**2. 강력한 변경 통제 프로세스**
```
변경 요청 접수
  ↓
영향 분석 (일정, 비용, 품질, 리스크)
  ↓
CCB 검토 및 평가
  ↓
승인/거부 결정
  ↓
(승인 시) 계획 업데이트 및 통보
  ↓
실행 및 추적
```

**3. 효과적인 커뮤니케이션**
- 범위에 대한 공통 이해 확립
- 정기적인 범위 검증
- 이해관계자 기대 관리
- "No" 말하는 법 익히기

**4. 문서화 및 승인**
- 모든 변경 요청 문서화
- 승인 서명 받기
- 변경 로그 유지
- 추적 가능성 확보

**5. 정기적인 범위 검증**
- 마일스톤마다 범위 검증
- 인도물 공식 승인
- 완료 기준 확인

#### 3.5 범위 크리프 대응 방법

**발견 시 즉시 조치**
1. **중지**: 추가 작업 즉시 중단
2. **평가**: 영향 분석 수행
3. **협의**: 이해관계자와 논의
4. **결정**: 승인/거부/연기
5. **문서화**: 결정 사항 기록

**"No" 말하는 기술**
- 사실 기반: "이 추가는 일정을 2주 지연시킵니다"
- 대안 제시: "다음 단계/버전에서 고려하겠습니다"
- 의사결정권자 에스컬레이션: "스폰서의 승인이 필요합니다"
- 우선순위 조정: "이것을 추가하려면 무엇을 제외할까요?"

**트레이드오프 제시**
```
요청: 신규 기능 A 추가

옵션 1: 일정 연장
- 기능 A 추가
- 완료일: +2주 연기

옵션 2: 예산 증액
- 기능 A 추가
- 추가 인력 투입
- 추가 비용: +2000만원

옵션 3: 범위 조정
- 기능 A 추가
- 기능 B 제외
- 일정/예산 유지

옵션 4: 연기
- 기능 A는 Phase 2로 연기
- 현재 프로젝트 영향 없음
```

### 추가 설명

**범위 크리프의 심리학**

**골드 플레이팅 (Gold Plating)**
- 팀이 요구되지 않은 추가 기능을 자발적으로 추가
- "더 좋게 만들자"는 선의의 의도
- 하지만 범위 크리프와 동일한 부정적 결과

**완벽주의**
- "조금만 더"의 함정
- 완료 기준에 도달했음에도 계속 수정
- 타임박싱과 완료 기준 명확화로 대응

**변화에 대한 저항 vs 유연성**
- 너무 경직: 필요한 변경도 거부 → 프로젝트 실패
- 너무 유연: 모든 요청 수용 → 범위 크리프
- 균형: 공식 프로세스로 필요한 변경은 수용

**실무 사례**

**성공 사례: 변경 통제로 범위 크리프 방지**
- 프로젝트: 전사 ERP 시스템 구축
- 상황: 중간에 50여 건의 변경 요청
- 대응: 모든 요청을 CCB에서 평가
- 결과: 15건만 승인, 나머지는 Phase 2로 연기
- 성과: 목표일 준수, 예산 내 완료

**실패 사례: 범위 크리프로 인한 프로젝트 실패**
- 프로젝트: 신제품 개발
- 상황: "작은 개선" 요청들을 지속적으로 수용
- 결과: 초기 6개월 → 18개월 소요, 예산 3배 초과
- 교훈: 초기 범위 준수와 변경 통제의 중요성

## 4. MoSCoW 기법

### 용어
**MoSCoW 기법 / MoSCoW 우선순위화**

### 설명
요구사항의 우선순위를 4가지 카테고리(Must, Should, Could, Won't)로 분류하는 우선순위화 기법입니다. 애자일 및 전통적 프로젝트 관리에서 널리 사용되며, 제한된 시간과 자원 내에서 최대 가치를 제공하는 데 도움을 줍니다.

### 세부 요소 및 설명

#### 4.1 MoSCoW 카테고리

**M - Must have (필수)**
- **정의**: 프로젝트 성공을 위해 반드시 필요한 요구사항
- **특징**: 
  - 없으면 프로젝트 실패로 간주
  - 법적/규제적 필수 사항
  - 비즈니스 목표 달성에 필수
- **질문**: "이것 없이 출시할 수 있나요?" → No
- **예시**:
  - 로그인 기능 (보안 필수)
  - 결제 기능 (전자상거래)
  - 개인정보 암호화 (법적 요구)

**S - Should have (중요)**
- **정의**: 매우 중요하지만 필수는 아닌 요구사항
- **특징**:
  - 없어도 프로젝트는 출시 가능
  - 하지만 큰 불편이나 가치 손실
  - 가능한 포함하고 싶음
- **질문**: "이것 없이 출시할 수 있나요?" → Yes, but difficult
- **예시**:
  - 비밀번호 찾기 기능
  - 이메일 알림
  - 상세 통계 리포트

**C - Could have (희망)**
- **정의**: 있으면 좋지만 없어도 큰 영향 없는 요구사항
- **특징**:
  - Nice-to-have
  - 자원이 남으면 구현
  - 쉽게 제외 가능
- **질문**: "이것이 제외되면?" → 별 영향 없음
- **예시**:
  - 다크 모드
  - 애니메이션 효과
  - 추가 정렬 옵션

**W - Won't have (제외)**
- **정의**: 이번 출시에서는 포함하지 않는 요구사항
- **특징**:
  - 명시적으로 제외
  - 미래 버전에서 고려 가능
  - 범위 명확화
- **질문**: "언제 할 것인가?" → 이번에는 안 함
- **예시**:
  - 소셜 미디어 통합 (Phase 2)
  - AI 추천 기능 (미래 고려)
  - 다국어 지원 (우선순위 낮음)

#### 4.2 MoSCoW 적용 프로세스

**1단계: 요구사항 수집**
- 모든 이해관계자로부터 요구사항 수집
- 브레인스토밍, 인터뷰, 워크숍 등

**2단계: 초기 분류**
- 각 요구사항을 M/S/C/W로 분류
- 팀 워크숍으로 진행

**3단계: 비율 조정**
- 권장 비율 확인
- Must: 60% 이하
- Should: 20%
- Could: 20%
- 비율 초과 시 재검토

**4단계: 검증**
- 이해관계자와 협의
- 비즈니스 가치 확인
- 스폰서 승인

**5단계: 문서화**
- 요구사항 명세서에 기록
- 우선순위와 이유 명시

#### 4.3 MoSCoW 사용 시나리오

**시나리오 1: 고정 기한 프로젝트**
- 상황: 출시일이 고정됨 (예: 행사 날짜)
- 사용: Must를 먼저 완료, 시간 남으면 Should/Could
- 결과: 핵심 기능 보장, 나머지는 상황에 따라

**시나리오 2: 고정 예산 프로젝트**
- 상황: 예산 증액 불가
- 사용: 예산 내에서 최대한 많은 가치 제공
- 결과: Must 보장, Should 일부, Could 제외 가능

**시나리오 3: 애자일 스프린트**
- 상황: 2주 스프린트 계획
- 사용: Must와 Should를 스프린트에 포함
- 결과: 스프린트 목표 달성, 점진적 가치 제공

**시나리오 4: 범위 크리프 방지**
- 상황: 추가 요청 발생
- 사용: 새 요청을 MoSCoW로 분류
- 결과: Could나 Won't로 분류하여 연기

#### 4.4 MoSCoW vs 다른 우선순위화 기법

**MoSCoW vs High/Medium/Low**
| 구분 | MoSCoW | High/Med/Low |
|------|--------|-------------|
| 카테고리 수 | 4개 | 3개 |
| 명확성 | 더 명확 | 다소 모호 |
| 적용 | 범위 관리 중심 | 일반적 우선순위 |
| 장점 | 범위 협상에 유리 | 간단함 |

**MoSCoW vs Kano 모델**
- Kano: 고객 만족도 기반 (필수/성능/매력)
- MoSCoW: 프로젝트 성공 기반
- 보완적 사용 가능

**MoSCoW vs RICE (Reach/Impact/Confidence/Effort)**
- RICE: 정량적 점수 계산
- MoSCoW: 정성적 카테고리 분류
- RICE가 더 객관적, MoSCoW가 더 간단

#### 4.5 MoSCoW 적용 시 주의사항

**흔한 실수**

1. **너무 많은 Must**
   - 문제: 모든 것이 Must → 우선순위화 실패
   - 해결: "정말 없으면 출시 불가능한가?" 재검토
   - 가이드: Must는 전체의 60% 이하

2. **모호한 기준**
   - 문제: Must와 Should 경계 불명확
   - 해결: 명확한 정의와 예시 제공
   - 도구: 의사결정 트리 사용

3. **Won't의 부재**
   - 문제: 제외 항목 명시하지 않음
   - 해결: 명시적으로 Won't 정의
   - 효과: 범위 크리프 방지

4. **이해관계자 간 불일치**
   - 문제: 각자 다른 우선순위 주장
   - 해결: 퍼실리테이션, 투표, 스폰서 결정
   - 방법: 워크숍으로 합의 도출

**성공 요인**
- 명확한 정의와 예시
- 이해관계자 참여
- 정기적인 재검토
- 스폰서의 최종 결정권

### 추가 설명

**MoSCoW의 유래**
- 1994년 Dai Clegg가 개발
- DSDM (Dynamic Systems Development Method)의 일부
- 현재는 애자일, 전통적 PM 모두에서 사용

**MoSCoW의 장점**

1. **간단하고 이해하기 쉬움**: 4개 카테고리만 기억
2. **빠른 의사결정**: 복잡한 계산 불필요
3. **효과적인 커뮤니케이션**: 비기술자도 이해
4. **범위 협상에 유리**: "Could를 제외하면 일정 단축"
5. **범위 크리프 방지**: 새 요청을 Could/Won't로 분류

**MoSCoW의 한계**

1. **주관적**: 정량적 기준 부족
2. **상대적 우선순위 불명확**: Must 내에서의 순서 불명확
3. **변화에 취약**: 상황 변화 시 재분류 필요

**실무 Tip**
- Must를 다시 P1, P2, P3로 세분화
- 정기적으로 재검토 (특히 애자일)
- 비즈니스 가치와 기술적 의존성 함께 고려

## 5. RTM (Requirements Traceability Matrix) / 요구사항 추적성 매트릭스

### 용어
**RTM (Requirements Traceability Matrix) / 요구사항 추적성 매트릭스**

### 설명
요구사항을 프로젝트 생명주기 전반에 걸쳐 추적하는 문서입니다. 각 요구사항이 설계, 개발, 테스트, 인도물과 어떻게 연결되는지 보여주며, 모든 요구사항이 충족되었는지 확인하는 도구입니다.

### 세부 요소 및 설명

#### 5.1 RTM의 목적

**주요 목적**
1. **완전성 보장**: 모든 요구사항이 구현되고 테스트되었는지 확인
2. **변경 영향 분석**: 요구사항 변경 시 영향 범위 파악
3. **범위 검증**: 불필요한 기능(Gold Plating) 방지
4. **컴플라이언스**: 규제 준수 증명
5. **진척도 추적**: 요구사항별 완료 상태 모니터링

**RTM이 필요한 프로젝트**
- 규제가 엄격한 산업 (의료, 금융, 항공)
- 대규모/복잡한 프로젝트
- 외주 프로젝트 (계약 이행 증명)
- 안전이 중요한 시스템
- 고객 승인이 필요한 프로젝트

#### 5.2 RTM의 구조

**기본 구성 요소**

1. **요구사항 정보**
   - 요구사항 ID
   - 요구사항 설명
   - 우선순위 (MoSCoW 등)
   - 출처 (누가 요청했는가)
   - 상태 (승인됨/구현 중/완료)

2. **전방 추적 (Forward Traceability)**
   요구사항 → 설계 → 개발 → 테스트 → 인도물
   - 설계 문서 ID
   - 코드 모듈 ID
   - 테스트 케이스 ID
   - 인도물 ID

3. **후방 추적 (Backward Traceability)**
   인도물 → 테스트 → 개발 → 설계 → 요구사항
   - 각 산출물이 어떤 요구사항에서 비롯되었는지 확인

**RTM 예시**

| Req ID | 요구사항 | 우선순위 | 출처 | 설계 ID | 코드 ID | 테스트 ID | 인도물 | 상태 |
|--------|---------|---------|------|---------|---------|----------|--------|------|
| REQ-001 | 사용자 로그인 | Must | 사용자팀 | DES-101 | MOD-201 | TC-001, TC-002 | User Module | 완료 |
| REQ-002 | 비밀번호 찾기 | Should | 사용자팀 | DES-102 | MOD-202 | TC-003 | User Module | 완료 |
| REQ-003 | 고객 검색 | Must | 영업팀 | DES-201 | MOD-301 | TC-010, TC-011 | Search Module | 진행중 |
| REQ-004 | 엑셀 내보내기 | Could | 영업팀 | DES-203 | - | - | - | 미시작 |

#### 5.3 RTM 작성 프로세스

**1단계: 요구사항 수집 및 정리**
- 모든 요구사항 식별
- 고유 ID 부여
- 우선순위 설정

**2단계: RTM 초기화**
- 템플릿 생성
- 요구사항 목록 입력
- 버전 관리 설정

**3단계: 전방 추적 연결**
- 설계 단계: 설계 문서 ID 연결
- 개발 단계: 코드 모듈 ID 연결
- 테스트 단계: 테스트 케이스 ID 연결
- 배포 단계: 인도물 ID 연결

**4단계: 지속적 업데이트**
- 진행 상황에 따라 상태 업데이트
- 새로운 요구사항 추가
- 변경사항 반영

**5단계: 검증 및 보고**
- 커버리지 분석 (100% 달성?)
- 누락 요구사항 식별
- 이해관계자에게 보고

#### 5.4 RTM의 유형

**1. 전방 추적 (Forward Traceability)**
```
요구사항 → 설계 → 구현 → 테스트
```
- **목적**: 모든 요구사항이 구현되고 테스트되었는지 확인
- **질문**: "이 요구사항은 어떻게 구현되었나?"

**2. 후방 추적 (Backward Traceability)**
```
테스트 → 구현 → 설계 → 요구사항
```
- **목적**: 불필요한 기능이 없는지 확인
- **질문**: "이 기능은 어떤 요구사항에서 나왔나?"

**3. 양방향 추적 (Bidirectional Traceability)**
- 전방 + 후방 추적
- 가장 완전한 형태
- 변경 영향 분석에 유용

#### 5.5 RTM 활용 시나리오

**시나리오 1: 요구사항 변경**
```
REQ-005 변경 요청
  ↓ RTM 확인
영향 받는 항목:
- 설계: DES-301
- 코드: MOD-401, MOD-402
- 테스트: TC-020, TC-021, TC-022
  ↓ 영향 분석
예상 공수: 5일, 비용: 250만원
```

**시나리오 2: 테스트 커버리지 확인**
```
총 요구사항: 50개
테스트 케이스 연결: 45개
미연결: 5개 (REQ-008, REQ-015, ...)
  ↓ 조치
누락된 테스트 케이스 작성
```

**시나리오 3: 프로젝트 종료 시 검증**
```
체크리스트:
☑ 모든 Must 요구사항 구현 완료
☑ 모든 Should 요구사항 구현 완료
☐ Could 요구사항 중 3개 미구현 (합의됨)
☑ 모든 구현 항목 테스트 완료
  ↓ 결과
프로젝트 범위 100% 달성 (Could 제외)
```

#### 5.6 RTM 관리 Best Practices

**Do's (해야 할 것)**
1. **일찍 시작**: 프로젝트 초기부터 작성
2. **지속적 업데이트**: 실시간 반영
3. **도구 활용**: 수동 관리는 오류 발생
4. **정기 검토**: 주간 또는 마일스톤마다 확인
5. **자동화**: 가능한 도구로 자동 연결

**Don'ts (피해야 할 것)**
1. **미뤄두기**: 나중에 작성하면 매우 어려움
2. **수동 관리**: Excel만으로는 한계
3. **방치**: 한번 만들고 업데이트 안 함
4. **과도한 상세**: 너무 세부적이면 관리 부담

**성공 요인**
- 경영진 지원: RTM 작성 시간 인정
- 팀 교육: 중요성과 사용법 교육
- 적절한 도구: 프로젝트 규모에 맞는 도구 선택
- 프로세스 통합: 일상 작업 프로세스에 포함

### 추가 설명

**RTM의 가치**

**정량적 가치**
- 결함 발견 비용 75% 감소 (조기 발견)
- 테스트 커버리지 100% 보장
- 재작업 50% 감소

**정성적 가치**
- 고객 신뢰 향상
- 프로젝트 투명성 증대
- 규제 준수 증명

**RTM이 구한 실제 사례**

**사례 1: 의료기기 소프트웨어**
- 상황: FDA 승인 필요
- 문제: 모든 요구사항의 테스트 증명 요구
- 해결: RTM으로 100% 커버리지 증명
- 결과: FDA 승인 획득, 6개월 일정 단축

**사례 2: 금융 시스템 구축**
- 상황: 감사 기관의 컴플라이언스 검사
- 문제: 보안 요구사항 이행 증명 필요
- 해결: RTM으로 모든 보안 요구사항 추적
- 결과: 감사 통과, 벌금 회피

**RTM의 한계**

1. **관리 부담**: 지속적 업데이트 필요
2. **초기 투자**: 설정에 시간 소요
3. **도구 의존성**: 적절한 도구 없으면 어려움
4. **형식적 위험**: 형식만 갖추고 실효성 없을 수 있음

**대안 및 보완**
- 작은 프로젝트: 간소화된 RTM (핵심만)
- 애자일: User Story Mapping + 자동화 도구
- 대규모: 전용 요구사항 관리 도구 (DOORS, Jama 등)

## 6. CCB (Change Control Board) / 변경 통제 위원회

### 용어
**CCB (Change Control Board) / 변경 통제 위원회**

### 설명
프로젝트의 변경 요청을 검토하고 승인 또는 거부하는 공식적인 의사결정 그룹입니다. CCB는 범위, 일정, 예산에 영향을 미치는 변경을 통제하여 프로젝트의 무분별한 확대(범위 크리프)를 방지합니다.

### 세부 요소 및 설명

#### 6.1 CCB의 구성

**일반적인 CCB 구성원**
1. **의장 (Chair)**: 프로젝트 스폰서 또는 PM
2. **핵심 구성원**:
   - 프로젝트 매니저
   - 기술 리더/아키텍트
   - 비즈니스 대표
   - 품질 담당자
   - 고객 대표 (해당 시)
3. **선택 구성원** (변경 유형에 따라):
   - 재무 담당자 (예산 관련)
   - 법무 담당자 (계약 관련)
   - 보안 담당자 (보안 관련)

**CCB 구성 원칙**
- **다양성**: 다양한 관점 반영
- **권한**: 의사결정 권한 보유
- **가용성**: 정기 회의 참석 가능
- **책임성**: 결정에 대한 책임 수용
- **소수 정예**: 5-7명 권장 (효율성)

#### 6.2 CCB의 역할과 책임

**주요 역할**
1. **변경 요청 검토**
   - 변경의 타당성 평가
   - 비즈니스 가치 판단
   - 기술적 실현 가능성 검토

2. **영향 분석 검토**
   - 일정 영향
   - 비용 영향
   - 품질 및 리스크 영향
   - 다른 작업/프로젝트 영향

3. **의사결정**
   - 승인 (Approve)
   - 거부 (Reject)
   - 연기 (Defer) - 추가 정보 필요
   - 조건부 승인

4. **우선순위 결정**
   - 여러 변경 요청 간 우선순위
   - 자원 배분 결정

5. **기준선 관리**
   - 승인된 변경에 대한 기준선 업데이트 승인

**책임사항**
- 프로젝트 목표와의 정렬성 보장
- 이해관계자 이익 균형
- 결정의 문서화
- 투명한 의사결정 프로세스 유지

#### 6.3 CCB 운영 프로세스

**1. 변경 요청 접수**
```
변경 제안자
  ↓ 변경 요청서 제출
PM (또는 변경 관리자)
  ↓ 1차 검토 및 분류
긴급성/영향도 평가
```

**2. 영향 분석**
```
PM 및 관련 전문가
  ↓ 분석 수행
- 일정 영향
- 비용 영향
- 품질 영향
- 리스크 평가
- 대안 제시
  ↓
분석 보고서 작성
```

**3. CCB 회의**
```
정기 회의 (주간/격주) 또는 긴급 회의
  ↓
변경 요청 발표 (PM 또는 제안자)
  ↓
질의응답 및 논의
  ↓
의사결정 (투표 또는 합의)
```

**4. 결정 통보 및 실행**
```
CCB 결정
  ↓
결정 사항 문서화
  ↓
관련자에게 통보
  ↓
(승인 시) 계획 업데이트 및 실행
```

#### 6.4 CCB 회의 운영

**정기 CCB 회의**
- **빈도**: 주간 또는 격주 (프로젝트 규모에 따라)
- **시간**: 1-2시간
- **의제**:
  1. 지난 회의 결정사항 검토
  2. 신규 변경 요청 검토
  3. 보류 중인 변경 재검토
  4. 승인된 변경 진행 상황 확인

**긴급 CCB 회의**
- **시기**: 긴급하고 중대한 변경 발생 시
- **소집**: 24-48시간 이내
- **참석**: 핵심 구성원만 (최소 정족수)

**의사결정 방법**
1. **만장일치**: 모든 구성원 동의 (이상적이나 시간 소요)
2. **다수결**: 과반수 이상 동의
3. **가중 투표**: 역할에 따라 투표권 차등
4. **의장 결정**: 합의 불가 시 의장이 최종 결정

#### 6.5 CCB 의사결정 기준

**승인 고려 사항**
1. **비즈니스 가치**: 변경이 비즈니스 목표에 기여하는가?
2. **긴급성**: 이번 릴리스에 포함되어야 하는가?
3. **실현 가능성**: 기술적으로 가능한가?
4. **영향 허용 범위**: 일정/예산 영향이 수용 가능한가?
5. **리스크**: 잠재적 리스크가 관리 가능한가?

**거부 사유**
- 프로젝트 목표와 무관
- 영향이 너무 큼 (일정/예산 초과)
- 기술적으로 불가능
- 리스크가 너무 높음
- 다른 우선순위가 더 높음
- 범위 크리프

**연기 사유**
- 정보가 불충분함
- 추가 분석 필요
- 다른 변경과의 조율 필요
- 시기상조 (Phase 2 고려)

#### 6.6 변경 요청서 (Change Request) 구조

**필수 항목**
- CR ID
- 제출일 및 제출자
- 변경 제목 및 설명
- 변경 이유/배경
- 현재 상태 vs 제안 상태
- 영향 분석 (일정/비용/품질/리스크)
- 대안
- 권고사항

**CCB 검토 결과**
- 결정 (승인/거부/연기)
- 결정 이유
- 조건 (조건부 승인 시)
- 승인자 및 날짜

### 추가 설명

**CCB의 중요성**

**통계**
- CCB가 없는 프로젝트의 65%가 범위 크리프 경험
- CCB가 있는 프로젝트는 일정 준수율 40% 향상
- 명확한 변경 관리 프로세스로 프로젝트 성공률 2배 증가

**CCB 운영의 실제 사례**

**성공 사례: IT 시스템 구축**
- 프로젝트: 전사 ERP 시스템
- CCB 구성: 스폰서(의장), PM, IT리더, 비즈니스 대표, 재무팀
- 활동:
  - 12개월 동안 80건의 변경 요청 접수
  - 주간 CCB 회의 (매주 금요일)
  - 25건 승인, 40건 Phase 2 연기, 15건 거부
- 결과:
  - 목표일 준수 (12개월)
  - 예산 5% 이내 초과 (허용 범위)
  - 이해관계자 만족도 높음

**실패 사례: CCB 형식화**
- 프로젝트: 제품 개발
- 문제: CCB는 있었으나 형식적으로 운영
  - 거의 모든 변경 승인 (95%)
  - 영향 분석 미흡
  - 정치적 압력에 의한 결정
- 결과:
  - 범위 50% 증가
  - 일정 6개월 지연
  - 예산 2배 초과

**CCB 운영 Tip**

1. **명확한 권한**: CCB의 결정은 최종 결정
2. **정기적 운영**: 회의가 너무 드물면 병목
3. **신속한 결정**: 변경 요청 후 1주 이내 결정
4. **투명성**: 결정 이유 명확히 공유
5. **일관성**: 유사한 사례는 유사하게 결정

### 별첨

#### CCB 헌장 (CCB Charter) 템플릿

```markdown
# 변경 통제 위원회 (CCB) 헌장

**프로젝트**: 고객 관리 시스템 구축
**작성일**: 2024-03-01
**승인자**: 프로젝트 스폰서

## 1. CCB 목적
프로젝트의 모든 변경 요청을 검토하고 승인/거부하여, 프로젝트 목표 달성과 범위/일정/예산 통제를 보장한다.

## 2. CCB 구성
| 역할 | 이름 | 소속 | 책임 |
|------|------|------|------|
| 의장 | 김부장 | 영업본부 | 최종 의사결정 |
| PM | 홍길동 | PMO | 변경 관리 프로세스 운영 |
| 기술 리더 | 이철수 | IT팀 | 기술적 실현 가능성 평가 |
| 비즈니스 대표 | 박영희 | 영업팀 | 비즈니스 가치 평가 |
| 품질 담당 | 최민수 | QA팀 | 품질 영향 평가 |

**정족수**: 최소 3명 (의장 포함 필수)

## 3. CCB 권한
- 모든 변경 요청에 대한 승인/거부/연기 권한
- 변경 우선순위 결정 권한
- 기준선 업데이트 승인 권한
- 예비비 사용 승인 (범위 내)

## 4. CCB 프로세스

### 4.1 변경 요청 제출
- 변경 요청서 (CR) 양식 사용
- PM에게 제출
- PM이 1차 검토 및 분류

### 4.2 영향 분석
- PM 및 관련 전문가가 수행
- 일정/비용/품질/리스크 분석
- 대안 제시

### 4.3 CCB 회의
- **정기 회의**: 매주 금요일 14:00-16:00
- **긴급 회의**: 필요 시 48시간 이내 소집
- **의제**: 신규 CR, 보류 CR, 후속 조치

### 4.4 의사결정
- 가능한 합의 도출
- 합의 불가 시 다수결 (과반수)
- 동률 시 의장이 결정

### 4.5 결정 통보
- 결정 후 24시간 이내 통보
- 제안자, PM, 관련 팀원에게 이메일 발송

## 5. 변경 분류 및 처리

### Level 1: 소규모 (Minor)
- 영향: 일정 ≤ 2일, 비용 ≤ 100만원
- 승인: PM 단독 승인 가능
- 보고: 다음 CCB 회의에서 보고

### Level 2: 중규모 (Medium)
- 영향: 일정 ≤ 1주, 비용 ≤ 500만원
- 승인: 정기 CCB 회의에서 검토

### Level 3: 대규모 (Major)
- 영향: 일정 > 1주 또는 비용 > 500만원
- 승인: 긴급 CCB 회의 + 스폰서 승인 필수

## 6. 의사결정 기준
변경 요청 평가 시 다음 기준 고려:
1. 프로젝트 목표와의 정렬성
2. 비즈니스 가치
3. 긴급성
4. 기술적 실현 가능성
5. 일정/비용 영향
6. 리스크

## 7. 회의록 및 문서화
- 모든 CCB 회의록은 24시간 이내 작성
- 변경 로그 유지
- 승인된 변경은 기준선 업데이트

## 8. 에스컬레이션
- CCB 내 합의 불가 시: 스폰서에게 에스컬레이션
- 스폰서의 결정이 최종

## 승인
- **작성**: PM 홍길동
- **검토**: CCB 구성원
- **승인**: 스폰서 김부장
- **승인일**: 2024-03-01
```
