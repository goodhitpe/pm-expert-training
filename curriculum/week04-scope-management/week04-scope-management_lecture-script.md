### [00:15-02:00] 섹션 1: 범위 관리 기초 (105분)

#### [00:15-00:45] 1.1 범위 관리란? (30분)

**강의 내용**:
```
📐 범위(Scope)란?

"프로젝트가 무엇을 만들 것인가"

2가지 범위:
1. 제품 범위 (Product Scope)
   - 최종 산출물의 특성과 기능
   - "무엇을 만들 것인가?"
   
2. 프로젝트 범위 (Project Scope)
   - 산출물을 만들기 위한 작업
   - "어떻게 만들 것인가?"

💼 예시: 모바일 앱 개발
───────────────────────────────────────
제품 범위:
✓ 로그인 기능
✓ 상품 검색
✓ 장바구니
✓ 결제
✓ 주문 내역
✓ 푸시 알림

프로젝트 범위:
✓ 요구사항 분석
✓ UI/UX 설계
✓ 데이터베이스 설계
✓ 백엔드 API 개발
✓ 프론트엔드 개발
✓ 테스트
✓ 배포
───────────────────────────────────────

📊 범위 관리 6가지 프로세스

1. 범위 관리 계획 수립
   "어떻게 범위를 관리할 것인가?"
   
2. 요구사항 수집
   "고객이 무엇을 원하는가?"
   
3. 범위 정의
   "정확히 무엇을 만들 것인가?"
   
4. WBS 작성
   "작업을 어떻게 나눌 것인가?"
   
5. 범위 검증
   "만든 것이 맞는가?"
   
6. 범위 통제
   "변경을 어떻게 관리할 것인가?"

💡 왜 범위 관리가 어려운가?

1. 고객이 원하는 것을 모름
   "만들고 나서 '이게 아닌데...'

2. 요구사항이 계속 변함
   "처음에는 이거였는데..."

3. 애매한 표현
   "사용자 친화적으로" → 정의는?
   "빠르게" → 얼마나 빠르게?

4. 숨은 요구사항
   "당연히 될 줄 알았어요"

⚠️ 범위 관리 실패의 결과

사례: 쇼핑몰 구축 프로젝트
───────────────────────────────────────
초기 범위: 기본 쇼핑몰 (3개월, 2억원)

Month 1:
고객: "SNS 로그인도 되겠죠?"
PM: "네..." (+1주)

Month 2:
고객: "쿠폰 기능은 필수예요"
PM: "추가하겠습니다" (+2주)

Month 3:
고객: "모바일 앱도 만들어주세요"
PM: "..." (+2개월, +1억원)

결과:
- 일정: 3개월 → 5개월 (67% 지연)
- 예산: 2억 → 3억 (50% 초과)
- 팀 사기: 최악
───────────────────────────────────────

💡 해결책: 명확한 범위 정의 + 변경 통제
```

**교수법**:
- 제품 범위와 프로젝트 범위 구분 명확히
- 실패 사례로 경각심 부여
- 실무에서 겪는 어려움 공감

#### [00:45-01:30] 1.2 요구사항 수집 (45분)

**강의 내용**:
```
📝 요구사항 수집 기법

"고객이 무엇을 원하는지 알아내기"

8가지 기법:

1️⃣ 인터뷰 (Interview)
   - 1:1 또는 소그룹
   - 깊이 있는 정보
   - 시간 소요 많음
   
   질문 유형:
   - 개방형: "어떤 기능이 필요하신가요?"
   - 폐쇄형: "결제 기능이 필요하신가요?"
   
   💡 팁:
   ✅ 5W1H 질문
   ✅ "왜?"를 3번 이상
   ❌ 유도 질문 금지

2️⃣ 설문조사 (Survey)
   - 많은 사용자로부터
   - 정량적 데이터
   - 표준화된 질문
   
   💼 예시:
   "새로운 기능 중 가장 원하는 것은?"
   - AI 추천: 45%
   - 음성 검색: 30%
   - AR 쇼핑: 15%
   - 기타: 10%

3️⃣ 워크샵 (Workshop)
   - 여러 이해관계자 함께
   - 2-4시간 집중 세션
   - 합의 도출
   
   진행 방법:
   ① 목표 설명 (10분)
   ② 브레인스토밍 (30분)
   ③ 그룹핑 (20분)
   ④ 우선순위 (20분)
   ⑤ 합의 (20분)

4️⃣ 포커스 그룹 (Focus Group)
   - 대표 사용자 6-10명
   - 퍼실리테이터 주도
   - 심층 논의
   
   💡 주의: 목소리 큰 사람 주의

5️⃣ 관찰 (Observation)
   - 현장 방문
   - 실제 사용 관찰
   - 숨은 니즈 발견
   
   💼 예시: 고객센터 관찰
   "하루 종일 고객센터를 관찰했더니
   FAQ 검색 기능이 절실했음"

6️⃣ 프로토타입 (Prototype)
   - 초기 버전 제작
   - 피드백 수집
   - 반복 개선
   
   유형:
   - 와이어프레임 (간단)
   - 목업 (중간)
   - MVP (실제)

7️⃣ 벤치마킹 (Benchmarking)
   - 경쟁사 분석
   - 업계 표준
   - 베스트 프랙티스
   
   💼 예시:
   "쿠팡, 네이버, 11번가의
   결제 프로세스 분석"

8️⃣ 문서 분석 (Document Analysis)
   - 기존 문서 검토
   - 레거시 시스템 분석
   - 규정/법규

💼 실습: 요구사항 수집 역할극 (15분)

시나리오:
"카페 주문 앱 개발"

역할:
- PM (인터뷰어)
- 카페 사장님 (고객)

PM의 미션:
15분 인터뷰로 요구사항 10개 이상 수집

인터뷰 질문 예시:
1. "어떤 문제를 해결하고 싶으신가요?"
2. "현재 주문 프로세스는 어떻게 되나요?"
3. "가장 불편한 점은 무엇인가요?"
4. "앱에 꼭 있어야 하는 기능은?"
5. "경쟁 카페의 앱을 사용해보신 적 있나요?"

진행:
① 2명씩 짝 (5분 인터뷰)
② 역할 바꾸기 (5분 인터뷰)
③ 수집한 요구사항 공유 (5분)

📋 요구사항 분류

기능적 요구사항 (Functional):
"시스템이 무엇을 해야 하는가?"
- 사용자 로그인
- 상품 검색
- 결제 처리
- 주문 취소

비기능적 요구사항 (Non-functional):
"시스템이 어떻게 동작해야 하는가?"
- 성능: 응답 시간 2초 이내
- 보안: SSL 암호화, 개인정보 보호
- 가용성: 99.9% 업타임
- 사용성: 3번 클릭 이내 구매
- 확장성: 동시 접속 10만명

기술적 요구사항:
- iOS 15 이상
- Android 10 이상
- AWS 클라우드
- PostgreSQL 데이터베이스

규제/법적 요구사항:
- 전자금융거래법 준수
- 개인정보보호법
- 전자상거래법

💡 요구사항 작성 원칙: SMART

S (Specific): 구체적
   ❌ "빠른 검색"
   ✅ "검색 결과 2초 이내 표시"

M (Measurable): 측정 가능
   ❌ "사용자 친화적"
   ✅ "80% 사용자가 3번 클릭 이내 구매"

A (Achievable): 달성 가능
   ❌ "100% 완벽한 보안"
   ✅ "업계 표준 보안 프로토콜 적용"

R (Relevant): 관련성
   ❌ "게임 기능" (쇼핑몰에...)
   ✅ "리뷰 작성 기능"

T (Time-bound): 기한
   ❌ "언젠가 추가"
   ✅ "Phase 1에 포함"
```

**교수법**:
- 8가지 기법을 구체적 예시로 설명
- 역할극으로 실제 인터뷰 체험
- SMART 원칙으로 좋은/나쁜 요구사항 비교

#### [01:30-02:00] 1.3 요구사항 우선순위화 (30분)

**강의 내용**:
```
🎯 요구사항 우선순위화

"모든 것을 다 할 수는 없다!"

문제:
- 요구사항 100개 수집
- 예산: 2억원
- 기간: 3개월
→ 모두 불가능!

해결책: 우선순위 결정

📊 MoSCoW 기법

M (Must have): 필수
- 이것 없으면 프로젝트 실패
- 법적 요구사항
- 핵심 기능
- 예: 로그인, 결제

S (Should have): 중요
- 없으면 불편하지만 가능
- 다음 단계에서 추가 가능
- 예: 위시리스트, 쿠폰

C (Could have): 선택
- 있으면 좋음
- 예산/시간 여유 시
- 예: SNS 공유, 상품 비교

W (Won't have): 제외
- 이번에는 안 함
- 향후 버전에서 고려
- 예: AR 쇼핑, 음성 주문

💼 실무 예시: 쇼핑몰 앱
───────────────────────────────────────
수집된 요구사항: 80개

MoSCoW 분류:

Must Have (20개):
✓ 사용자 회원가입/로그인
✓ 상품 검색 및 상세 보기
✓ 장바구니
✓ 결제 (신용카드, 계좌이체)
✓ 주문 내역
✓ 배송 조회
✓ 고객 센터
...

Should Have (25개):
○ 위시리스트
○ 상품 리뷰
○ 쿠폰/할인
○ SNS 로그인
○ 푸시 알림
...

Could Have (20개):
△ 상품 비교
△ 최근 본 상품
△ 친구 추천
△ 포인트 적립
...

Won't Have (15개):
✗ AR 가상 착용
✗ AI 스타일리스트
✗ 음성 주문
✗ 라이브 커머스
...

결정:
Phase 1 (3개월): Must Have만
Phase 2 (2개월): Should Have
Phase 3 (향후): Could Have 검토
───────────────────────────────────────

📈 Kano 모델

고객 만족도를 기준으로 분류:

1. 기본 요구사항 (Must-be)
   - 없으면 불만
   - 있어도 만족도 증가 미미
   - 예: 검색 기능

2. 성능 요구사항 (Performance)
   - 더 좋을수록 만족도 증가
   - 예: 검색 속도, 상품 수

3. 매력 요구사항 (Attractive)
   - 없어도 불만 없음
   - 있으면 크게 만족
   - 예: 개인화 추천, 무료 배송

전략:
1. 기본 요구사항 100% 충족
2. 성능 요구사항 적절히
3. 매력 요구사항 1-2개 선택

💡 실습: MoSCoW 분류 (10분)

시나리오:
"헬스장 예약 앱"

다음 요구사항을 MoSCoW로 분류:
1. 회원 등록
2. 수업 예약
3. 결제
4. 출석 체크
5. 운동 일지
6. 식단 추천
7. PT 강사와 채팅
8. 운동 영상
9. 커뮤니티
10. 웨어러블 연동

개인 작업 (5분) → 그룹 토론 (5분)

📋 요구사항 추적성 매트릭스 (RTM)

"요구사항 → 구현 → 테스트 연결"

예시:
───────────────────────────────────────
ID  요구사항      우선순위  설계   개발   테스트  상태
R01 로그인        Must     D01   C01   TC01   완료
R02 상품검색      Must     D02   C02   TC02   진행중
R03 위시리스트    Should   D03   -     -      대기
R04 AR쇼핑       Won't    -     -     -      제외
───────────────────────────────────────

💡 이점:
✅ 모든 요구사항이 구현되었는지 확인
✅ 변경 영향도 분석
✅ 테스트 커버리지 확인
✅ 감사 추적 (Audit Trail)
```

**교수법**:
- MoSCoW 기법 명확히 설명
- 실제 프로젝트 분류 예시
- Kano 모델로 고객 관점 이해
- RTM의 중요성 강조

### [02:15-04:30] 섹션 2: WBS 작성 (135분)

#### [02:15-02:45] 2.1 WBS 개념 및 원칙 (30분)

**강의 내용**:
```
🏗️ WBS (Work Breakdown Structure)

"작업 분해 구조"

정의:
"프로젝트의 모든 작업을 계층적으로 분해한 것"

💡 왜 WBS인가?

1. 코끼리를 어떻게 먹나?
   → 한 입씩!
   
2. 큰 프로젝트를 어떻게 관리하나?
   → 작은 작업으로 나눠서!

WBS의 이점:
✅ 전체 작업을 한눈에
✅ 누락 방지
✅ 일정/비용 산정 기초
✅ 책임 배분
✅ 진척도 측정

📐 WBS 작성 원칙

1️⃣ 100% 규칙
"상위 항목 = 하위 항목의 100% 합"

예시:
웹사이트 (100%)

└── 테스트

"너무 깊지도, 얕지도 않게"

Level 2: 단계 또는 인도물

너무 크면: 통제 불가

"무엇을(What), 어떻게(How)가 아님"

❌ "개발하기", "테스트하기"
✅ "로그인 모듈", "테스트 케이스"
```

   - WBS 작성은 직접 해봐야 이해
   - 포스트잇으로 물리적 작업

   - Scope Creep 경험 공유
   - 실패 사례로 교훈
   - 성공 전략 제시

   - 변경 요청서
   - UAT 체크리스트

   - WBS 실습이 길어질 수 있음
   - 타이머 사용
   - 단계별 시간 공지
