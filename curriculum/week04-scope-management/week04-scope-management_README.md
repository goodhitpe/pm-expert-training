### 2. 요구사항 관리

#### 2.1 요구사항 문서화
**요구사항 유형:**
- 기능적 요구사항 (Functional Requirements)
- 비기능적 요구사항 (Non-functional Requirements)
- 기술적 요구사항 (Technical Requirements)
- 품질 요구사항 (Quality Requirements)
- 규제/법적 요구사항 (Regulatory Requirements)

#### 2.2 요구사항 우선순위화
**MoSCoW 기법:**
- **M**ust have: 필수 요구사항
- **S**hould have: 중요하지만 필수는 아님
- **C**ould have: 가능하면 포함
- **W**on't have: 이번에는 미포함

#### 2.3 요구사항 추적성 매트릭스 (RTM)
- 요구사항 ↔ 설계
- 요구사항 ↔ 개발
- 요구사항 ↔ 테스트
- 요구사항 ↔ 인도물

### 3. WBS (Work Breakdown Structure)

#### 3.1 WBS의 중요성
- 프로젝트 범위의 시각화
- 작업 추정의 기초
- 책임 배분의 근거
- 진척도 측정 기준

#### 3.2 WBS 작성 원칙
- **100% 규칙**: 상위 항목은 하위 항목의 100%를 포함
- **상호 배타성**: 하위 항목 간 중복 없음
- **적절한 수준**: 3-4 레벨 권장
- **작업 패키지**: 최하위 레벨 (8-80 시간 규칙)

#### 3.3 WBS 작성 방법

**하향식 접근 (Top-Down):**
```
프로젝트
├── 단계 1
│   ├── 활동 1.1
│   │   ├── 작업 1.1.1
│   │   └── 작업 1.1.2
│   └── 활동 1.2
├── 단계 2
│   ├── 활동 2.1
│   └── 활동 2.2
└── 단계 3
```

**상향식 접근 (Bottom-Up):**
- 모든 작업 나열
- 유사한 작업 그룹화
- 계층 구조 형성

#### 3.4 WBS 템플릿 활용
- 프로젝트 관리 작업
- 설계 작업
- 개발 작업
- 테스트 작업
- 배포 작업

#### 3.5 WBS 사전 (WBS Dictionary)
각 작업 패키지에 대한 상세 설명:
- WBS ID
- 작업 설명
- 담당자
- 소요 기간
- 선행 작업
- 산출물
- 승인 기준

## 🎯 실무 적용 예시: 데이터베이스 설계와 범위 관리

### 예시 프로젝트: 전자상거래 플랫폼 구축

#### 1. 요구사항 수집 시 데이터베이스 관점 포함

**시나리오:**
고객이 "사용자 관리 기능" 요구

**❌ DB 지식 없이 수집한 요구사항:**
```
FR-001: 사용자는 가입할 수 있다
FR-002: 사용자는 로그인할 수 있다
FR-003: 사용자는 정보를 수정할 수 있다
```
문제: 모호하고 불완전, 개발 단계에서 재작업 발생

**✅ DB 지식 활용한 요구사항:**
```
FR-001: 사용자 가입
  - 입력 항목: 이메일, 비밀번호, 이름, 휴대폰
  - 이메일 중복 검증 필수
  - 비밀번호 암호화 저장 (bcrypt, 10 rounds)
  - DB 테이블: users
  - 유니크 제약: email

FR-002: 사용자 로그인
  - 로그인 이력 저장 (보안 추적)
  - DB 테이블: login_history (user_id, ip_address, timestamp)
  - 실패 5회 시 계정 잠금 (30분)
  - 세션 저장: Redis (TTL 30분)

FR-003: 사용자 정보 수정
  - 수정 가능 항목: 이름, 휴대폰, 프로필 이미지
  - 이메일 변경 시 재인증 필요
  - 수정 이력 저장: user_audit_log
  - 트랜잭션 처리 (ACID 보장)

NFR-001: 성능 요구사항
  - 로그인 응답 시간 < 500ms
  - DB 인덱스: users(email), login_history(user_id, timestamp)
  - 동시 접속 10,000명 지원

NFR-002: 데이터 무결성
  - 외래키 제약 설정
  - 참조 무결성 보장
  - 트랜잭션 격리 수준: READ_COMMITTED
```

결과: 구체적이고 실현 가능한 요구사항, 개발자와 즉시 작업 가능

#### 2. WBS 작성 시 데이터베이스 작업 포함

**❌ DB 작업 누락된 WBS:**
```
1.0 사용자 관리 기능
  1.1 화면 디자인
  1.2 프론트엔드 개발
  1.3 백엔드 API 개발
  1.4 테스트
```
문제: DB 설계/마이그레이션 시간 미반영 → 일정 지연

**✅ DB 작업 포함한 WBS:**
```
1.0 사용자 관리 기능 (20일)
  
  1.1 요구사항 분석 및 데이터 모델링 (3일)
      1.1.1 비즈니스 요구사항 분석
      1.1.2 ERD (Entity Relationship Diagram) 작성
      1.1.3 정규화 (3NF까지)
      1.1.4 인덱스 전략 수립
  
  1.2 데이터베이스 설계 및 구축 (4일)
      1.2.1 테이블 스키마 설계
            - users (id, email, password, name, phone, created_at)
            - login_history (id, user_id, ip, timestamp)
            - user_audit_log (id, user_id, field, old_value, new_value)
      1.2.2 인덱스 생성
            - users: UNIQUE(email), INDEX(phone)
            - login_history: INDEX(user_id, timestamp)
      1.2.3 제약조건 설정
            - 외래키, NOT NULL, CHECK 제약
      1.2.4 마이그레이션 스크립트 작성
            - Flyway/Liquibase 설정
  
  1.3 백엔드 API 개발 (5일)
      1.3.1 User Model 및 Repository
      1.3.2 Registration API (POST /api/users)
      1.3.3 Login API (POST /api/auth/login)
      1.3.4 Profile Update API (PUT /api/users/:id)
      1.3.5 비즈니스 로직 (이메일 중복 검증, 암호화)
  
  1.4 프론트엔드 개발 (4일)
      1.4.1 회원가입 화면
      1.4.2 로그인 화면
      1.4.3 프로필 수정 화면
      1.4.4 API 연동
  
  1.5 테스트 (3일)
      1.5.1 DB 단위 테스트
            - 트랜잭션 롤백 테스트
            - 제약조건 검증
            - 인덱스 성능 테스트
      1.5.2 API 통합 테스트
      1.5.3 부하 테스트 (10,000 동시 사용자)
  
  1.6 데이터 마이그레이션 (1일)
      1.6.1 기존 데이터 변환 스크립트
      1.6.2 검증 및 롤백 계획
```

#### 3. 범위 변경 시 DB 영향도 분석

**시나리오:**
고객 요청: "사용자 등급 시스템 추가 (일반/VIP/VVIP)"

**영향도 분석 (DB 관점):**

```
1. 범위 영향:
   
   A. 데이터베이스 스키마 변경
      변경 전:
      users (id, email, password, name, phone, created_at)
      
      변경 후:
      users (id, email, password, name, phone, tier, created_at)
      + user_tier_history (id, user_id, tier, changed_at, reason)
      
      작업: ALTER TABLE + 새 테이블 생성 (1일)

   B. 기존 데이터 마이그레이션
      - 기존 사용자 100만 명 → 기본 등급 '일반' 설정
      - 마이그레이션 시간: 30분 (다운타임)
      - 작업: 데이터 백업 + 마이그레이션 스크립트 (0.5일)

   C. 연관 기능 수정
      - 등급별 할인율 적용 (주문 테이블 영향)
      - 등급별 포인트 적립률 (포인트 테이블 영향)
      - 영향 범위: 주문/결제 모듈 (5개 API)

2. 일정 영향:
   - DB 설계 및 마이그레이션: 1.5일
   - 백엔드 API 수정: 3일
   - 프론트엔드 수정: 2일
   - 테스트 (기존 기능 회귀 포함): 2일
   총 8.5일 → 버퍼 포함 10일 (2주)

3. 비용 영향:
   - 개발 비용: $8,000 (2주 × 2명)
   - DB 마이그레이션 다운타임 손실: $2,000 (30분 × $4,000/시간)
   - 총 $10,000

4. 리스크:
   - 리스크 1: 마이그레이션 실패 시 서비스 중단
     심각도: 높음
     대응: Blue-Green 배포 + 롤백 계획 준비
   
   - 리스크 2: 외래키 제약 위반
     심각도: 중간
     대응: 마이그레이션 전 데이터 무결성 검증

5. CCB 제안:
   옵션 A: 즉시 적용 (2주 일정 추가, $10,000)
   옵션 B: Phase 2로 이연 (현재 일정 유지)
   옵션 C: 간소화 버전 (등급만 추가, 기능 없음, 3일)
   
   권장: 옵션 C → 옵션 A (단계적 접근)
```

#### 4. 실무 체크리스트: 범위 정의 시 DB 고려사항

```
✅ 요구사항 단계:
□ ERD(Entity Relationship Diagram) 작성했는가?
□ 테이블 간 관계 (1:1, 1:N, N:M) 명확한가?
□ 데이터 타입과 제약조건 정의했는가?
□ 정규화 필요 여부 검토했는가?
□ 인덱스 전략 수립했는가?

✅ WBS 작성 단계:
□ DB 설계 작업 항목 포함했는가? (3-5일)
□ 마이그레이션 스크립트 작성 시간 배정했는가?
□ DB 성능 테스트 시간 포함했는가?
□ 백업 및 복구 계획 수립 시간 있는가?

✅ 범위 검증 단계:
□ DB 스키마가 요구사항 모두 반영하는가?
□ 데이터 무결성 제약이 설정되었는가?
□ 성능 요구사항 (응답 시간) 충족하는가?
□ 확장성 (Scale-out) 고려했는가?

✅ 변경 관리 단계:
□ 스키마 변경 시 마이그레이션 계획 있는가?
□ 기존 데이터 보존 전략 수립했는가?
□ 다운타임 최소화 방안 마련했는가?
□ 롤백 시나리오 준비했는가?
```

### 실전 사례 연구

**프로젝트:** 대형 쇼핑몰 DB 재설계

**초기 범위 (모호함):**
"사용자, 상품, 주문 관리"

**문제 발생:**
- 개발 중반에 "장바구니는 어디에?" 발견
- 리뷰 테이블 설계 누락
- 결제 내역과 주문 내역 분리 필요성 뒤늦게 인지
→ 결과: 3주 지연, 예산 30% 초과

**개선된 범위 (DB 중심 상세화):**
```
1.0 데이터베이스 설계 (완전한 ERD)
  
  1.1 사용자 도메인
      - users (기본 정보)
      - user_addresses (배송지)
      - user_payment_methods (결제 수단)
  
  1.2 상품 도메인
      - products (상품 마스터)
      - product_categories (카테고리)
      - product_images (이미지)
      - product_inventory (재고)
  
  1.3 주문 도메인
      - orders (주문 헤더)
      - order_items (주문 상세)
      - order_status_history (상태 변경 이력)
  
  1.4 결제 도메인
      - payments (결제 정보)
      - payment_transactions (PG사 거래)
  
  1.5 기타
      - carts (장바구니)
      - wishlists (찜 목록)
      - reviews (리뷰)
      - coupons (쿠폰)

  총 18개 테이블, 관계 명확화
```

**결과:**
- 명확한 범위로 재작업 0건
- 일정 준수 (16주 완료)
- 확장 용이 (신규 기능 추가 시 테이블 구조 변경 불필요)

## 📝 핵심 용어
- **WBS (Work Breakdown Structure)**: 작업 분해 구조
- **작업 패키지 (Work Package)**: WBS의 최하위 레벨 작업
- **범위 크리프 (Scope Creep)**: 통제되지 않은 범위 확대
- **MoSCoW**: 요구사항 우선순위화 기법
- **RTM (Requirements Traceability Matrix)**: 요구사항 추적성 매트릭스
- **CCB (Change Control Board)**: 변경 통제 위원회
