## 1️⃣ 범위(Scope)란?

### 정의
**범위**는 "프로젝트에서 무엇을 할 것인가?"와 "무엇을 하지 않을 것인가?"를 명확히 정의하는 것입니다.

### 실생활 비유
```
집들이 파티 준비:
✅ 범위 안 (할 일):
  - 손님 20명 초대
  - 음식 준비 (피자 3판, 치킨 2마리, 음료)
  - 거실 청소
  - 음악 플레이리스트 준비

❌ 범위 밖 (하지 않을 일):
  - 손님 50명 초대 (예산 초과)
  - 집 전체 대청소 (시간 부족)
  - 라이브 밴드 초청 (비용 과다)
```

### 왜 중요한가?
범위를 명확히 하지 않으면:
- 일이 끝없이 늘어남 (Scope Creep)
- 예산 초과
- 일정 지연
- 팀 혼란

---

## 2️⃣ 제품 범위 vs 프로젝트 범위

### 제품 범위 (Product Scope)
**"무엇을 만드는가?"** - 최종 산출물의 특징과 기능

**예시 (모바일 쇼핑 앱):**
- 상품 검색 기능
- 장바구니 기능
- 결제 기능
- 주문 내역 조회

### 프로젝트 범위 (Project Scope)
**"어떻게 만드는가?"** - 제품을 만들기 위해 수행할 작업

**예시 (모바일 쇼핑 앱):**
- 요구사항 분석
- UI/UX 디자인
- 앱 개발 (iOS, Android)
- 서버 구축
- 테스트
- 배포

### 쉬운 비유
```
결혼식:
- 제품 범위: 완성된 결혼식 (예식, 피로연, 사진)
- 프로젝트 범위: 결혼식 준비 작업 
  (예식장 예약, 드레스 맞춤, 청첩장 발송, 리허설 등)
```

## 3️⃣ 요구사항(Requirements)이란?

### 정의
**요구사항**은 프로젝트가 만족시켜야 할 조건이나 기능

### 요구사항의 종류

#### 기능적 요구사항
**"이런 기능이 있어야 한다"**

**예시 (은행 앱):**
- 로그인 기능
- 계좌 조회 기능
- 이체 기능

#### 비기능적 요구사항
**"이런 품질 수준이어야 한다"**

**예시 (은행 앱):**
- 보안: 데이터 암호화
- 성능: 2초 이내 응답
- 가용성: 99.9% 서비스 유지

### 실생활 비유
```
이사할 집 찾기:
- 기능적 요구사항: 방 3개, 화장실 2개, 주차 1대
- 비기능적 요구사항: 학교 도보 10분 이내, 조용한 환경, 채광 좋을 것
```

---

## 4️⃣ MoSCoW 우선순위

### 정의
요구사항의 중요도를 4단계로 분류하는 기법

### 4가지 분류

#### M - Must have (필수)
**"반드시 있어야 한다"** - 이것 없으면 프로젝트 의미 없음

**예시 (배달 앱):**
- 음식 주문 기능
- 결제 기능

#### S - Should have (중요)
**"있어야 한다"** - 중요하지만, 없어도 일단 출시 가능

**예시:**
- 리뷰 작성 기능
- 단골 가게 즐겨찾기

#### C - Could have (가능하면)
**"있으면 좋다"** - 시간과 예산이 허락하면 포함

**예시:**
- 쿠폰 선물하기
- SNS 공유 기능

#### W - Won't have (제외)
**"이번에는 하지 않는다"** - 나중 버전에서 고려

**예시:**
- AI 추천 시스템
- 음성 주문 기능

### 쉬운 비유
```
여름 휴가 짐 싸기:
- M: 여권, 항공권, 숙박 바우처
- S: 여벌 옷, 세면도구
- C: 카메라, 선글라스
- W: 노트북, 운동화
```

---

## 5️⃣ WBS (Work Breakdown Structure)란?

### 정의
**작업 분해 구조** - 프로젝트를 작고 관리 가능한 단위로 쪼개는 것

### 왜 필요한가?
- 큰 프로젝트를 이해하기 쉽게
- 누가 무엇을 할지 명확히
- 예산과 시간 산정 가능
- 진척도 측정 가능

### 실생활 비유
```
🎂 생일 케이크 만들기

레벨 1: 생일 케이크 만들기

레벨 2:
├── 재료 준비
├── 반죽 만들기
├── 굽기
├── 데코레이션
└── 포장

레벨 3 (재료 준비 세부):
├── 재료 준비
    ├── 밀가루 500g 계량
    ├── 달걀 5개 준비
    ├── 설탕 200g 계량
    └── 버터 녹이기
```

### PM 관점
```
웹사이트 리뉴얼 프로젝트

레벨 1: 웹사이트 리뉴얼

레벨 2:
├── 기획
├── 디자인
├── 개발
├── 테스트
└── 오픈

레벨 3 (개발 세부):
├── 개발
    ├── 프론트엔드 개발
    ├── 백엔드 개발
    └── 데이터베이스 구축
```

## 6️⃣ WBS의 100% 규칙

### 정의
**상위 항목 = 하위 항목들의 합**

### 원칙
- 하위 항목을 모두 합치면 상위 항목이 100% 완료
- 누락되거나 중복되는 작업이 없어야 함

### 실생활 비유
```
❌ 잘못된 예 (회사 워크샵):
워크샵 (100%)
├── 장소 예약 (30%)
└── 식사 준비 (40%)
→ 합계 70%만 됨! 교통, 프로그램 등 누락 ❌

✅ 올바른 예:
워크샵 (100%)
├── 장소 예약 (25%)
├── 교통 준비 (25%)
├── 식사 준비 (25%)
└── 프로그램 기획 (25%)
→ 합계 100% ✅
```

---

## 7️⃣ 범위 크리프 (Scope Creep)

### 정의
**통제되지 않은 범위 확대** - 계획에 없던 일이 자꾸 추가되는 현상

### 왜 발생하나?
- 고객이 자꾸 "이것도 해주세요" 요청
- 팀원이 "이것도 하면 좋을 것 같아요" 제안
- 초기 범위 정의가 불명확

### 실생활 비유
```
베란다 페인트칠 프로젝트:

계획: 베란다 벽만 페인트칠
↓
고객: "창틀도 해주세요"
↓
고객: "천장도 해주세요"
↓
고객: "거실도 같이 해주세요"
↓
결과: 예산 2배, 기간 3배 초과 😱
```

### 예방 방법
1. **명확한 범위 문서화**
2. **변경 요청 프로세스 수립**
3. **추가 작업 = 추가 비용/시간 명시**
4. **"No"라고 말할 수 있어야 함**

## 💬 자주 묻는 질문 (FAQ)

**Q1: 범위를 너무 넓게 잡으면 어떻게 되나요?**  
A: 프로젝트가 끝나지 않습니다. 시간과 예산이 부족해집니다. 처음부터 명확하고 현실적인 범위를 정하는 것이 중요합니다.

**Q2: 고객이 범위 변경을 요청하면 무조건 거절해야 하나요?**  
A: 아닙니다! 정식 변경 요청 프로세스를 통해 영향도(시간, 비용)를 분석하고, 승인 받으면 진행합니다. "Yes"와 "No" 사이의 협상이 필요합니다.

**Q3: WBS를 몇 단계까지 분해해야 하나요?**  
A: 보통 3-4단계가 적절합니다. 최하위 단계(작업 패키지)는 8-80시간 규칙을 따릅니다. 너무 세분화하면 관리가 어렵고, 너무 크면 통제가 안 됩니다.

**Q4: 요구사항이 계속 바뀌는데 어떻게 하나요?**  
A: 초기에 요구사항을 충분히 수집하고 검증하는 것이 중요합니다. 하지만 변경은 불가피하므로, 변경 관리 프로세스를 수립하세요.

**Q5: MoSCoW에서 모든 것을 Must have로 하면 안 되나요?**  
A: 안 됩니다! Must have가 너무 많으면 우선순위의 의미가 없습니다. 정말 필수인 것만 Must have로 지정하세요. (전체의 60% 이하 권장)
