# Week 4: 범위 관리

## 📝 학습 목표
- 프로젝트 범위 정의 및 관리 방법 학습
- 요구사항 수집 기법 습득
- WBS(작업 분해 구조) 작성 및 활용

## 📚 주요 내용

### 1. 범위 관리 프로세스

#### 1.1 범위 관리 계획 수립
- 범위 정의 방법 결정
- 범위 검증 방법 결정
- 범위 통제 방법 결정

#### 1.2 요구사항 수집
**수집 기법:**
- 인터뷰 (Interview)
- 설문조사 (Survey)
- 포커스 그룹 (Focus Group)
- 워크샵 (Workshop)
- 관찰 (Observation)
- 프로토타입 (Prototype)
- 벤치마킹 (Benchmarking)

#### 1.3 범위 정의
- 제품 범위 (Product Scope): 최종 산출물의 특성
- 프로젝트 범위 (Project Scope): 산출물을 만들기 위한 작업

#### 1.4 WBS 작성
- 프로젝트 작업의 계층적 분해
- 관리 가능한 작업 패키지로 구성

#### 1.5 범위 검증
- 인도물 승인 프로세스
- 고객/스폰서의 공식 승인

#### 1.6 범위 통제
- 범위 변경 관리
- 범위 크리프 (Scope Creep) 방지

### 2. 요구사항 관리

#### 2.1 요구사항 문서화
**요구사항 유형:**
- 기능적 요구사항 (Functional Requirements)
- 비기능적 요구사항 (Non-functional Requirements)
- 기술적 요구사항 (Technical Requirements)
- 품질 요구사항 (Quality Requirements)
- 규제/법적 요구사항 (Regulatory Requirements)

#### 2.2 요구사항 우선순위화
**MoSCoW 기법:**
- **M**ust have: 필수 요구사항
- **S**hould have: 중요하지만 필수는 아님
- **C**ould have: 가능하면 포함
- **W**on't have: 이번에는 미포함

#### 2.3 요구사항 추적성 매트릭스 (RTM)
- 요구사항 ↔ 설계
- 요구사항 ↔ 개발
- 요구사항 ↔ 테스트
- 요구사항 ↔ 인도물

### 3. WBS (Work Breakdown Structure)

#### 3.1 WBS의 중요성
- 프로젝트 범위의 시각화
- 작업 추정의 기초
- 책임 배분의 근거
- 진척도 측정 기준

#### 3.2 WBS 작성 원칙
- **100% 규칙**: 상위 항목은 하위 항목의 100%를 포함
- **상호 배타성**: 하위 항목 간 중복 없음
- **적절한 수준**: 3-4 레벨 권장
- **작업 패키지**: 최하위 레벨 (8-80 시간 규칙)

#### 3.3 WBS 작성 방법

**하향식 접근 (Top-Down):**
```
프로젝트
├── 단계 1
│   ├── 활동 1.1
│   │   ├── 작업 1.1.1
│   │   └── 작업 1.1.2
│   └── 활동 1.2
├── 단계 2
│   ├── 활동 2.1
│   └── 활동 2.2
└── 단계 3
```

**상향식 접근 (Bottom-Up):**
- 모든 작업 나열
- 유사한 작업 그룹화
- 계층 구조 형성

#### 3.4 WBS 템플릿 활용
- 프로젝트 관리 작업
- 설계 작업
- 개발 작업
- 테스트 작업
- 배포 작업

#### 3.5 WBS 사전 (WBS Dictionary)
각 작업 패키지에 대한 상세 설명:
- WBS ID
- 작업 설명
- 담당자
- 소요 기간
- 선행 작업
- 산출물
- 승인 기준

### 4. 범위 변경 관리

#### 4.1 변경 요청 프로세스
1. 변경 요청 접수
2. 영향 분석 (일정, 예산, 품질)
3. 변경 통제 위원회 (CCB) 검토
4. 승인/거부 결정
5. 승인 시 계획 업데이트
6. 이해관계자에게 통보

#### 4.2 범위 크리프 방지
- 명확한 범위 정의
- 변경 관리 프로세스 수립
- 정기적인 범위 검증
- 이해관계자와의 지속적인 소통

## 📝 핵심 용어
- **WBS (Work Breakdown Structure)**: 작업 분해 구조
- **작업 패키지 (Work Package)**: WBS의 최하위 레벨 작업
- **범위 크리프 (Scope Creep)**: 통제되지 않은 범위 확대
- **MoSCoW**: 요구사항 우선순위화 기법
- **RTM (Requirements Traceability Matrix)**: 요구사항 추적성 매트릭스
- **CCB (Change Control Board)**: 변경 통제 위원회
