### [00:15-01:45] 섹션 1: 프로젝트 매니저란? (90분)

#### [00:15-00:45] 1.1 PM의 정의와 역할 (30분)

**강의 내용**:
```
📖 프로젝트 매니저의 정의
"프로젝트 매니저는 프로젝트의 목표를 달성하기 위해 
팀과 자원을 관리하는 전문가입니다."

핵심 역할 5가지 설명:
1. 프로젝트 계획 수립 및 실행
   - "마치 건축가가 설계도를 그리듯이..."
   
2. 팀 조율 및 의사소통 관리
   - "오케스트라 지휘자처럼 팀을 조율합니다"
   
3. 일정, 예산, 범위 관리
   - "이 3가지를 Triple Constraint라고 부릅니다"
   - 화이트보드에 삼각형 그리기
   
4. 리스크 식별 및 대응
   - "문제가 발생하기 전에 미리 예측하고 준비합니다"
   
5. 이해관계자 관리
   - "프로젝트에 관련된 모든 사람들의 기대를 관리합니다"

💬 질문: "이 중에서 가장 중요한 역할은 무엇일까요?"
```

**교수법**:
- 슬라이드로 역할 시각화
- 각 역할에 대한 간단한 실무 사례 공유
- 학습자들과 상호작용

```
📱 스타트업 PM 사례 (모바일 앱 개발)

  ✅ 다재다능
  ✅ 기술 깊이 필요
  ❌ 프로세스 부족
  ❌ 리소스 제약

🏢 대기업 PM 사례 (금융 시스템 개발)
- 대기업의 체계적 프로세스
- "대기업에서는 프로세스가 잘 정립되어 있지만 의사결정이 느립니다"
- 주요 특징:
  ✅ 명확한 프로세스
  ✅ 충분한 리소스
  ✅ 체계적 위험 관리
  ❌ 느린 의사결정

💡 그룹 토론 (10분)
"여러분은 스타트업 PM과 대기업 PM 중 어느 쪽이 더 맞을 것 같나요?"
- 소그룹(3-4명)으로 나눠 토론
```

#### [01:15-01:45] 1.3 PM의 기술 이해 필요성 (30분)

**강의 내용**:
```
🚨 실전 시나리오: 시스템 다운 상황

개발자: "원인 파악 중입니다..."

PM: "CloudWatch 보니 DB 커넥션 풀이 고갈됐네요."
개발자: "맞습니다. 느린 쿼리 때문인 것 같습니다."
PM: "긴급 조치:
1. Auto Scaling으로 서버 2대 추가
2. 느린 쿼리 임시 비활성화
3. DB 커넥션 풀 사이즈 증가"

💬 질문: "두 PM의 차이는 무엇인가요?"
→ 기술 이해 → 빠른 의사결정 → 팀 신뢰 확보

🎯 결론
"IT PM은 코딩을 잘할 필요는 없지만, 
기술을 이해하고 개발자와 소통할 수 있어야 합니다."
```

**교수법**:
- 역할극 또는 시나리오 설명

### [02:00-03:15] 섹션 2: 프로젝트 vs 운영 업무 (75분)

#### [02:00-02:30] 2.1 프로젝트의 정의와 특성 (30분)

**강의 내용**:
```
📖 프로젝트란?
"고유한 제품, 서비스, 결과를 만들기 위한 한시적 노력"

3가지 핵심 특성:
1. 한시성 (Temporary)
   - 명확한 시작과 종료
   - "영원한 프로젝트는 없습니다"
   
2. 고유성 (Unique)
   - 각 프로젝트는 독특한 결과물
   - "같은 프로젝트는 두 번 없습니다"
   
3. 점진적 구체화 (Progressive Elaboration)
   - 프로젝트 진행에 따라 세부사항 구체화
   - "처음부터 모든 것을 알 수는 없습니다"

📊 비교표 활용
| 구분 | 프로젝트 | 운영 업무 |
|------|----------|-----------|
| 기간 | 한시적 | 지속적 |
| 목표 | 고유한 결과물 | 서비스 유지 |
| 팀 | 임시 조직 | 영구 조직 |
| 예산 | 프로젝트별 | 부서별 |
```

**교수법**:
- 정의 명확히 설명
- 비교표로 시각화
- 실제 사례 제시

**강의 내용**:
```
🛒 사례 A: 온라인 쇼핑몰 구축
- 기간: 6개월
- 팀: 임시 조직
- 예산: 5억 원
→ 판정: ✅ 프로젝트

🛠️ 사례 B: 쇼핑몰 운영 및 유지보수
- 기간: 지속적
- 팀: 영구 조직
- 활동: 일일 모니터링, 버그 수정
→ 판정: ❌ 운영 업무

🔄 사례 C: 쇼핑몰 v2.0 업그레이드
- 기간: 3개월
- 목표: AI 추천 기능 추가
- 팀: 임시 TF
→ 판정: ✅ 프로젝트

💡 핵심 통찰
"운영 중에도 프로젝트가 발생할 수 있고,
프로젝트 완료 후 운영으로 전환됩니다."
```

**교수법**:
- 3가지 사례 순차 분석

- 실제 업무에서의 전환 과정 설명

### [03:15-04:45] 섹션 3: 프로젝트 생명주기 (90분)

#### [03:15-03:45] 3.1 생명주기 개요 (30분)

**강의 내용**:
```
🔄 프로젝트 생명주기 5단계
착수 → 계획 → 실행 → 감시/통제 → 종료

화이트보드에 흐름도 그리기:
┌────────┐   ┌────────┐   ┌────────┐   ┌──────────┐   ┌────────┐
│ 착수   │→ │ 계획   │→ │ 실행   │→ │ 감시/통제 │→ │ 종료   │
└────────┘   └────────┘   └────────┘   └──────────┘   └────────┘

각 단계별 핵심 활동:

1️⃣ 착수 (Initiating)
- 프로젝트 헌장 작성
- 이해관계자 식별
- 고수준 요구사항 정의
→ "프로젝트를 공식적으로 시작합니다"

2️⃣ 계획 (Planning)
- 상세 범위 정의
- 일정 및 예산 수립
- 리스크 관리 계획
→ "어떻게 할 것인가를 계획합니다"

3️⃣ 실행 (Executing)
- 계획된 활동 수행
- 팀 관리 및 조율
- 품질 보증
→ "계획을 실행합니다"

4️⃣ 감시/통제 (Monitoring & Controlling)
- 진척도 추적
- 변경 관리
- 이슈 및 리스크 관리
→ "계획대로 가고 있는지 확인합니다"

5️⃣ 종료 (Closing)
- 최종 인도물 전달
- 문서화 및 교훈 정리
- 팀 해산
→ "프로젝트를 마무리합니다"

💡 중요 포인트
"실행과 감시/통제는 동시에 진행됩니다!"
```

**교수법**:
- 화이트보드에 흐름도 그리며 설명
- 각 단계의 핵심 활동 명확히 전달
- 실제 프로젝트 진행 과정 비유

```
- 주요 이해관계자: 개발팀, 인프라팀, 보안팀, 파트너사

3️⃣ 실행 단계 (10주)

PM의 일일 활동:
09:00 - 스탠드업 미팅
10:00 - 이슈 해결
14:00 - 진척도 업데이트 (Jira)
15:00 - 이해관계자 커뮤니케이션

4️⃣ 감시/통제 (진행 중 지속)

변경 요청: "네이버페이 추가"
- 영향도: +1주, +$5,000
- 결정: 승인 (예산 여유 활용)

5️⃣ 종료 단계 (1주)
- 최종 인도물:
  ✅ API 서비스 (Production 배포)
  ✅ API 문서
  ✅ 개발자 가이드
  ✅ 운영 매뉴얼

- 교훈 정리:
  ✅ 잘한 점: 보안팀 조기 참여
  ❌ 개선 필요: API 스펙 변경 많음

💬 질문: "각 단계에서 PM이 가장 신경 써야 할 것은?"
```

**교수법**:
- 실제 프로젝트처럼 시간 순서로 설명

- PM의 실제 활동 구체적으로 묘사

### [04:45-05:15] 섹션 4: 실습 - 프로젝트 계획 시뮬레이션 (30분)

```
🎯 그룹 실습: 간단한 프로젝트 기획

4. 예산 (얼마가 필요한가?)
5. 주요 이해관계자
6. 예상되는 리스크 3가지

활동 진행:
1. 4-5명씩 그룹 구성 (2분)

- 기간이 현실적인가?
- 이해관계자를 충분히 고려했는가?
- 리스크가 실질적인가?
```

- 그룹 활동으로 협업 경험

**강의 내용**:

## 💡 강의 운영 팁

### 효과적인 강의를 위한 조언

1. **상호작용 극대화**
   - 일방적 강의보다는 질문과 토론 유도
   - 학습자 경험 공유 시간 확보

2. **실무 연결**
   - 모든 개념을 실무 사례와 연결
   - 학습자들의 업무 맥락에서 설명

3. **시각 자료 활용**
   - 화이트보드 적극 활용
   - 비교표, 흐름도 등 시각화

4. **휴식 시간 준수**
   - 집중력 유지를 위해 휴식 필수
   - 75-90분마다 15분 휴식

5. **유연한 시간 조정**
   - 학습자 수준에 따라 속도 조절
   - 중요한 부분에 더 많은 시간 투자

### 주의 사항

- ⚠️ 기술 용어는 쉽게 풀어서 설명
- ⚠️ IT 비전공자 눈높이 고려
- ⚠️ 실습 시간 충분히 확보
- ⚠️ 과도한 이론 지양, 실무 중심
