#### [00:15-00:45] 1.1 활동 정의 및 기간 산정 (30분)

**강의 내용**:
```
📐 일정 관리 6단계

WBS → 활동 정의 → 순서 배열 → 기간 산정 → 일정 개발 → 통제

1단계: 활동 정의
"WBS의 작업 패키지를 구체적 활동으로"

💼 예시:
WBS: "로그인 기능 개발" (40시간)

활동으로 분해:
1.1 로그인 화면 UI (8h)
1.2 인증 로직 구현 (12h)
1.3 세션 관리 (8h)
1.4 에러 처리 (4h)
1.5 단위 테스트 (8h)
────────────────────
합계: 40시간 ✓

💡 마일스톤 (Milestone)
"중요한 시점, 기간 0"

예시:
✓ 프로젝트 착수
✓ 요구사항 승인
✓ 설계 완료
✓ 개발 완료
✓ UAT 통과
✓ 프로젝트 종료

마일스톤의 특징:
- 기간 없음 (0일)
- 중요한 결정 또는 인도물
- 진척도 측정 기준
- 이해관계자 소통 포인트

📊 활동 기간 산정 기법

1️⃣ 전문가 판단
"경험 있는 사람에게 물어보기"

장점: 빠름, 전문성
단점: 주관적, 낙관적 편향

2️⃣ 유사 산정 (Analogous Estimating)
"과거 유사 프로젝트 참고"

예시:
과거 프로젝트 A: 쇼핑몰 개발 3개월
현재 프로젝트 B: 쇼핑몰 개발
→ 약 3개월 소요 예상

장점: 빠름, 적은 정보로 가능
단점: 부정확, 큰 프로젝트만 가능

3️⃣ 모수 산정 (Parametric Estimating)
"단위당 생산성 × 수량"

예시:
개발자 생산성: 100 LOC/일
총 코드: 10,000 LOC
→ 10,000 / 100 = 100일

또는:
화면 1개: 3일
총 화면: 20개
→ 20 × 3 = 60일

장점: 정확도 높음
단점: 데이터 필요

4️⃣ 3점 산정 (Three-Point Estimating) ← 가장 권장!

"낙관적, 보통, 비관적 고려"

PERT 공식:
기대값 = (O + 4M + P) / 6

O (Optimistic): 낙관적 (모든 게 완벽)
M (Most Likely): 가장 가능성 높음
P (Pessimistic): 비관적 (모든 게 최악)

💼 예시: API 개발
───────────────────────────────────────
낙관적 (O): 3일
- 모든 게 명확
- 라이브러리 바로 적용
- 버그 없음

가능성 높음 (M): 5일
- 일반적 상황
- 약간의 이슈

비관적 (P): 9일
- 문서 불명확
- 라이브러리 호환 문제
- 예상치 못한 버그

기대값 = (3 + 4×5 + 9) / 6
       = (3 + 20 + 9) / 6
       = 32 / 6
       = 5.33일

→ 6일로 계획!
───────────────────────────────────────

표준편차 = (P - O) / 6
        = (9 - 3) / 6
        = 1일

의미: 68% 확률로 4.33~6.33일 소요

💡 버퍼 (Buffer)
"안전 여유"

유형:
1. 활동 버퍼: 각 활동에 추가
2. 프로젝트 버퍼: 전체 일정에 추가
3. 피딩 버퍼: 주공정 전에 추가

권장: 전체 일정의 10-20%

예시:
순수 작업: 100일
버퍼 15%: 15일
────────────────
총 일정: 115일

🎯 실습: 기간 산정 (10분)

시나리오: "회원 관리 기능"

활동 목록:
1. 회원가입 화면
2. 이메일 인증
3. 로그인/로그아웃
4. 프로필 관리
5. 비밀번호 변경

각 활동의 3점 산정:
활동 1: O=?, M=?, P=?, 기대값=?
...

개인 작업 (5분) → 그룹 토론 (5분)
```

**교수법**:
- 4가지 산정 기법 비교
- 3점 산정 공식 계산 시연

#### [00:45-01:30] 1.2 활동 순서 배열 및 네트워크 다이어그램 (45분)

**강의 내용**:
```
🔗 활동 순서 배열 (Sequence Activities)

"어떤 순서로 할 것인가?"

의존 관계 4가지 유형:

1️⃣ FS (Finish-to-Start) ← 가장 일반적 (90%)
"A가 끝나야 B 시작"

예시:
설계 완료 → 개발 시작
개발 완료 → 테스트 시작

[설계] ──FS──> [개발] ──FS──> [테스트]

2️⃣ SS (Start-to-Start)
"A가 시작하면 B도 시작 가능"

예시:
테스트 케이스 작성과 개발 동시 시작

[개발] ──SS──> [테스트케이스 작성]

3️⃣ FF (Finish-to-Finish)
"A가 끝나면 B도 종료"

예시:
문서 작성과 검토가 함께 종료

[문서작성] ──FF──> [문서검토]

4️⃣ SF (Start-to-Finish) ← 거의 안 씀
"A가 시작하면 B 종료"

예시:
야간 작업 시작 → 주간 작업 종료

💡 리드와 래그 (Lead & Lag)

리드 (Lead): 앞당기기 (음수)
"선행 활동이 끝나기 전에 시작"

예시:
[개발] ──FS-2일──> [테스트]
"개발이 끝나기 2일 전에 테스트 시작"

래그 (Lag): 대기 시간 (양수)
"선행 활동 후 기다림"

예시:
[콘크리트 타설] ──FS+7일──> [철근 작업]
"콘크리트 양생 7일 대기"

IT 예시:
[배포] ──FS+3일──> [모니터링]
"안정화 3일 후 본격 모니터링"

📊 네트워크 다이어그램 (Network Diagram)

"활동 간 관계를 시각화"

2가지 표기법:

1. PDM (Precedence Diagramming Method) ← 권장
"노드가 활동"

    ┌─────┐    ┌─────┐    ┌─────┐
    │ A   │───>│ B   │───>│ C   │
    │ 5일 │    │ 3일 │    │ 2일 │
    └─────┘    └─────┘    └─────┘

2. ADM (Arrow Diagramming Method)
"화살표가 활동"
(거의 안 씀)

💼 실습: 네트워크 다이어그램 (25분)

시나리오: "간단한 웹사이트 개발"

활동 목록:
────────────────────────────────────────
ID  활동              기간  선행활동
────────────────────────────────────────
A   요구사항 분석     3일   -
B   UI 설계          4일   A
C   백엔드 설계       3일   A
D   프론트엔드 개발    8일   B
E   백엔드 개발       6일   C
F   통합              2일   D, E
G   테스트           3일   F
H   배포             1일   G
────────────────────────────────────────

실습 진행:

Step 1: 네트워크 다이어그램 그리기 (10분)
- A4 용지에 그리기
- 박스로 활동 표시
- 화살표로 의존 관계
- 기간 표시

Step 2: 경로 찾기 (5분)
"시작부터 끝까지 모든 경로"

경로 1: A → B → D → F → G → H
경로 2: A → C → E → F → G → H

각 경로의 총 기간 계산

Step 3: 주공정 찾기 (5분)
"가장 긴 경로가 주공정"

경로 1: 3 + 4 + 8 + 2 + 3 + 1 = 21일
경로 2: 3 + 3 + 6 + 2 + 3 + 1 = 18일

주공정 (Critical Path): 경로 1 (21일)
→ 프로젝트 최소 기간 = 21일

Step 4: 여유 시간 계산 (5분)
"각 활동의 여유"

총 여유 (Total Float):
"지연해도 프로젝트에 영향 없는 시간"

경로 1 활동: 여유 0일 (Critical!)
경로 2 활동 C: 21 - 18 = 3일 여유
활동 E: 3일 여유

자유 여유 (Free Float):
"다음 활동에 영향 없는 시간"

💡 인사이트

주공정 (Critical Path):
✓ 가장 긴 경로
✓ 여유 시간 0
✓ 한 활동이라도 지연되면 전체 지연!
✓ 집중 관리 필요

비주공정 (Non-Critical):
✓ 여유 시간 있음
✓ 약간 지연해도 OK
✓ 자원 조정 가능

🚨 주의 사항

1. 주공정은 변할 수 있음
   예: 비주공정이 지연되면...

2. 여러 주공정 존재 가능
   예: 두 경로가 같은 길이

3. 여유 시간 = 방치 아님
   예: 버퍼로 활용
```

**교수법**:
- 4가지 의존 관계를 명확한 예시로
- 네트워크 다이어그램 그리기 실습
- 주공정 개념을 단계적으로 설명

#### [01:30-02:00] 1.3 크리티컬 패스 분석 (30분)

**강의 내용**:
```
🎯 크리티컬 패스 메소드 (CPM)

"프로젝트 기간을 결정하는 핵심 경로"

📊 CPM 계산 4단계

예시로 설명:
────────────────────────────────────────
활동  기간  선행
A     3     -
B     5     A
C     4     A
D     2     B
E     3     C
F     4     D, E
────────────────────────────────────────

Step 1: 순방향 계산 (Forward Pass)
"빠른 시작(ES), 빠른 종료(EF)"

ES (Early Start): 가장 빨리 시작 가능한 시간
EF (Early Finish): 가장 빨리 끝날 수 있는 시간
EF = ES + 기간

        ES  EF
활동 A: 0   3   (0 + 3)
활동 B: 3   8   (3 + 5)
활동 C: 3   7   (3 + 4)
활동 D: 8   10  (8 + 2)
활동 E: 7   10  (7 + 3)
활동 F: 10  14  (10 + 4) ← 프로젝트 종료!

Step 2: 역방향 계산 (Backward Pass)
"늦은 종료(LF), 늦은 시작(LS)"

LF (Late Finish): 늦어도 끝내야 하는 시간
LS (Late Start): 늦어도 시작해야 하는 시간
LS = LF - 기간

        LS  LF
활동 F: 10  14  (14 - 4)
활동 E: 7   10  (10 - 3)
활동 D: 8   10  (10 - 2)
활동 C: 6   10  (10 - 4) ← 주의! LF는 10
활동 B: 3   8   (8 - 5)
활동 A: 0   3   (3 - 3)

Step 3: 여유 시간 (Float/Slack) 계산
"총 여유 = LS - ES (또는 LF - EF)"

활동  ES  EF  LS  LF  총여유
A     0   3   0   3   0      ← Critical!
B     3   8   3   8   0      ← Critical!
C     3   7   6   10  3      ← 3일 여유
D     8   10  8   10  0      ← Critical!
E     7   10  7   10  0      ← Critical!
F     10  14  10  14  0      ← Critical!

Step 4: 크리티컬 패스 식별
"여유 시간 0인 활동들"

Critical Path: A → B → D → F
또는: A → E → F

→ 2개의 주공정!

💡 주공정 활용

1. 일정 단축
   "주공정 활동만 단축해야 효과"
   
   비주공정 C를 3일로 단축?
   → 효과 없음! (여유 3일이 2일로)
   
   주공정 B를 4일로 단축?
   → 프로젝트 1일 단축! (14일 → 13일)

2. 리스크 관리
   "주공정 활동에 리스크 집중"
   
   활동 B (주공정): 리스크 높음 → 대책 필수
   활동 C (비주공정): 리스크 있어도 여유 3일

3. 자원 배분
   "주공정에 최고 자원"
   
   최고 개발자 → 활동 B, D 배정
   주니어 개발자 → 활동 C 배정 (여유 있음)

4. 진척도 관리
   "주공정 지연 = 프로젝트 지연"
   
   매일 체크: 주공정 활동 진척
   주간 회의: 주공정 상태 보고

💼 실무 사례: 지연 대응
───────────────────────────────────────
프로젝트: 모바일 앱 개발
주공정: 설계 → 개발 → 테스트
총 기간: 60일

Week 5:
"개발 활동이 5일 지연!"

분석:
- 개발은 주공정 활동
- 5일 지연 = 프로젝트 5일 지연!

대응 옵션:
1. 일정 연장 (60일 → 65일)
   → 고객에게 협상

2. 범위 축소
   → 우선순위 낮은 기능 제거

3. Fast Tracking
   → 테스트를 개발 중에 시작 (리스크↑)

4. Crashing
   → 개발자 추가 투입 (비용↑)

5. 비주공정에서 자원 차용
   → 디자이너를 개발 지원

결정: 옵션 3 + 5
→ 3일 회복 (62일로 마감)
───────────────────────────────────────

#### [04:30-05:00] 3.1 일정 압축 기법 (30분)

**강의 내용**:
```
⚡ 일정 압축 (Schedule Compression)

"일정을 단축해야 할 때"

상황:
- 경영진: "2주 앞당겨주세요"
- 고객: "다음 달까지 가능한가요?"
- 시장: "경쟁사가 먼저 출시했어요"

💡 주의: 일정 단축 = Trade-off
- 비용 증가 또는
- 범위 축소 또는
- 리스크 증가

2가지 압축 기법:

1️⃣ Fast Tracking (빠른 추적)

"병렬 실행"

정의:
원래 순차적인 활동을 병렬로

예시:
Before (순차):
[설계 10일] → [개발 20일] = 30일

After (병렬):
[설계 10일]
    └─> [개발 20일 (Day 6부터)] = 26일
        (4일 단축!)

💼 실무 예시:
───────────────────────────────────────
프로젝트: 전자상거래 사이트

순차 (원래):
설계 완료 → 개발 시작 → 완료 후 테스트
= 8 + 20 + 10 = 38일

Fast Tracking:
설계 50% 완료 시 개발 시작
개발 50% 완료 시 테스트 시작
= 8 + 14 + 6 = 28일 (10일 단축!)
───────────────────────────────────────

✅ 장점:
- 비용 증가 없음
- 빠른 적용

❌ 단점/리스크:
- 재작업 가능성 ↑
  (설계 변경 시 개발 수정)
- 의사소통 복잡도 ↑
- 품질 리스크 ↑

💡 적용 팁:
- 리스크 낮은 활동만
- 명확한 인터페이스
- 긴밀한 협업

2️⃣ Crashing (자원 투입)

"자원 추가 투입"

정의:
비용을 들여서 시간 단축

예시:
Before:
개발자 2명 → 20일

After:
개발자 4명 → 12일 (8일 단축)
단, 비용 2배

💼 실무 예시:
───────────────────────────────────────
활동: 백엔드 API 개발
원래: 개발자 2명 × 20일 = 40 man-days
비용: 40 × 50만원 = 2,000만원

Crashing:
개발자 4명 × 12일 = 48 man-days
비용: 48 × 50만원 = 2,400만원
추가 비용: 400만원

단축: 8일
비용/일: 50만원 (효율 좋음!)
───────────────────────────────────────

✅ 장점:
- 확실한 단축
- 통제 가능

❌ 단점:
- 비용 증가
- 효율 저하 (Brooks' Law)
  "9명의 임산부가 1개월에 아기를 낳을 수 없다"
- 커뮤니케이션 오버헤드

💡 적용 팁:
- 주공정 활동만 Crash
- 비용/일 효율 계산
- 추가 인원 온보딩 시간 고려

📊 Crashing 의사결정

1단계: 주공정 식별
"비주공정을 Crash해도 소용없음!"

2단계: 각 활동의 Crash 비용 계산

예시:
────────────────────────────────────────
활동  일반  Crash  단축  추가비용  비용/일
────────────────────────────────────────
A     10일  7일    3일   300만     100만
B     15일  12일   3일   600만     200만
C     8일   6일    2일   150만     75만
D     12일  10일   2일   500만     250만
────────────────────────────────────────

3단계: 효율 순서로 Crash

목표: 5일 단축

1순위: C (75만/일) → 2일 단축, 150만원
2순위: A (100만/일) → 3일 단축, 300만원
────────────────────
합계: 5일 단축, 450만원

🎯 실습: 압축 시뮬레이션 (10분)

시나리오:

목표: 10일 단축

A (10일) → B (15일) → C (12일) → D (13일)
= 50일

────────────────────────────────────────
활동  단축가능  비용/일
────────────────────────────────────────
A     3일      100만원
B     4일      150만원
C     3일      80만원
D     5일      200만원
────────────────────────────────────────

📊 일정 성과 지표

1. SV (Schedule Variance)
   = EV - PV
   (획득 가치 - 계획 가치)
   
2. SPI (Schedule Performance Index)
   = EV / PV
   
   해석:
   SPI > 1: 앞서감 ✓
   SPI = 1: 계획대로 =
   SPI < 1: 지연 ⚠️

예시:
PV (계획): 5,000만원 (50%)
EV (실제): 4,000만원 (40%)

SV = 4,000 - 5,000 = -1,000만원 (지연)
SPI = 4,000 / 5,000 = 0.8 (20% 지연)

💡 일정 통제 모범 사례

1. 조기 경보 시스템
   - 주간 SPI 모니터링
   - 0.95 이하 시 경보
   - 즉시 대응

2. 예측
   - ETC (Estimate to Complete)
   - EAC (Estimate at Complete)
   - TCPI (To-Complete Performance Index)
   
   EAC = 원래 기간 / SPI
   예: 60일 / 0.8 = 75일 예상

3. 투명한 보고
   - 문제 숨기지 않기
   - 빨리 알리기
   - 해결책 제시

4. 정기적 업데이트

   - 주간: 전체 일정
   - 월간: 전체 재계획

5. 변경 관리
   - 일정 변경도 승인 필요
   - 영향도 분석
   - 문서화
