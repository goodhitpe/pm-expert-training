### 2. 활동 순서 배열

#### 2.1 의존 관계 유형
- **FS (Finish-to-Start)**: A가 끝나야 B 시작 (가장 일반적)
- **SS (Start-to-Start)**: A가 시작하면 B 시작
- **FF (Finish-to-Finish)**: A가 끝나면 B 종료
- **SF (Start-to-Finish)**: A가 시작하면 B 종료 (드물게 사용)

#### 2.2 네트워크 다이어그램
**PDM (Precedence Diagramming Method):**
```
[활동 A] --FS--> [활동 B] --FS--> [활동 C]
              |                      |
              +----SS----> [활동 D] -+
```

#### 2.3 리드와 래그
- **리드 (Lead)**: 선행 활동이 끝나기 전에 후속 활동 시작 (예: -2일)
- **래그 (Lag)**: 선행 활동이 끝난 후 대기 시간 (예: +3일)

### 3. 활동 기간 산정

#### 3.1 산정 기법

**3점 산정 (PERT):**
- 낙관적 (Optimistic): O
- 가능성 높음 (Most Likely): M
- 비관적 (Pessimistic): P

**공식:**
- 기대값 (Expected) = (O + 4M + P) / 6
- 표준편차 (Standard Deviation) = (P - O) / 6

**예시:**
- O = 3일, M = 5일, P = 9일
- 기대값 = (3 + 20 + 9) / 6 = 5.33일
- 표준편차 = (9 - 3) / 6 = 1일

### 5. 크리티컬 패스 분석 (CPM)

#### 5.1 크리티컬 패스란?
- 프로젝트의 가장 긴 경로
- 여유 시간(Float)이 0인 경로
- 이 경로의 지연은 전체 프로젝트 지연으로 이어짐

#### 5.2 순방향 계산 (Forward Pass)
- 조기 시작일 (Early Start, ES)
- 조기 종료일 (Early Finish, EF)
- EF = ES + 기간 - 1

#### 5.3 역방향 계산 (Backward Pass)
- 늦은 종료일 (Late Finish, LF)
- 늦은 시작일 (Late Start, LS)
- LS = LF - 기간 + 1

#### 5.4 여유 시간 (Float/Slack)
- **총 여유 시간 (Total Float)**: TF = LS - ES 또는 LF - EF
- **자유 여유 시간 (Free Float)**: 후속 활동에 영향 없이 지연 가능한 시간

#### 5.5 크리티컬 패스 활용
- 우선순위 활동 식별
- 자원 집중 배분
- 리스크 관리 집중
- 일정 압축 대상 선정

### 6. 일정 압축 기법

#### 6.1 크래싱 (Crashing)
- 자원 추가 투입으로 기간 단축
- 비용 증가 수반
- 크리티컬 패스 활동에 적용

**예시:**
- 개발자 2명 → 4명 (기간 10일 → 6일)

#### 6.2 패스트 트래킹 (Fast Tracking)
- 순차 활동을 병렬 수행
- 재작업 리스크 증가
- 의존 관계 완화

**예시:**
- 설계 완료 전 개발 시작

**네트워크 다이어그램:**
```
[요구사항 분석] --FS--> [설계] --FS--> [개발 환경 구축]
                          |              |
                          |              +--FS--> [백엔드 개발]
                          |              |            |
                          |              +--FS--> [프론트엔드 개발]
                          |                           |
```

```
경로 3: 요구사항(2) → DB설계(3) → 통합테스트(1) → 시스템테스트(1)

크리티컬 패스: 경로 1 (16주)
크리티컬 활동: 요구사항 → 설계 → 환경구축 → 백엔드개발 → 통합 → 시스템 → 성능 → 배포
```

#### 3. 3점 산정 (PERT) 적용 예시

**활동: Payment Integration (결제 연동)**

```
시나리오별 분석:

낙관적 (O): 3일
- PG사 API 문서가 완벽
- 샘플 코드 제공
- 한 번에 성공

가능성 높음 (M): 5일
- 일반적인 개발 과정
- 몇 번의 디버깅 필요
- 테스트 케이스 작성

비관적 (P): 10일
- API 문서 불명확, PG사 지원 필요
- 예외 처리 복잡
- 테스트 환경 구축 어려움
- 인증 이슈 발생

PERT 계산:
기대값 = (3 + 4×5 + 10) / 6 = 33 / 6 = 5.5일
표준편차 = (10 - 3) / 6 = 1.17일

68% 확률로 4.33일 ~ 6.67일 사이에 완료
95% 확률로 3.16일 ~ 7.84일 사이에 완료

일정 계획: 6일 배정 (버퍼 포함)
```

#### 4. 일정 압축 기법 적용

**옵션 분석:**

**A. 크래싱 (Crashing) - 자원 추가**
```
크리티컬 활동 분석:

1. 백엔드 개발 (5주 → 3주)
   - 현재: 2명
   - 추가: 2명 (총 4명)
   - 단축: 2주
   - 추가 비용: $20,000 (2주 × 2명 × $5,000)
   
   기술적 고려사항:
   ✅ 가능: 마이크로서비스 구조로 병렬 개발 가능
   ⚠️ 주의: 코드 충돌, 커뮤니케이션 오버헤드 증가

2. 통합 테스트 (1주 → 0.5주)
   - 현재: QA 1명
   - 추가: QA 1명 (총 2명)
   - 단축: 0.5주
   - 추가 비용: $2,500
   
   ✅ 가능: 테스트 케이스 분할 실행 가능

3. 시스템 테스트 (1주 → 0.5주)
   - 자동화 테스트 도구 도입 (Selenium Grid)
   - 단축: 0.5주
   - 추가 비용: $5,000 (도구 + 설정)

크래싱 총 효과: 3주 단축
총 추가 비용: $27,500
```

**B. 패스트 트래킹 (Fast Tracking) - 병렬 수행**
```
1. 설계 중 개발 시작 (리스크 높음)
   Before: 설계(3주) --FS--> 개발
   After:  설계(3주) --SS(+2주)--> 개발
   단축: 2주
   
   ⚠️ 리스크:
   - 설계 변경 시 재작업 발생 가능성 30%
   - 예상 재작업 비용: $10,000
   
   완화 전략:
   - 모듈별 설계 완료 시 개발 시작
   - 핵심 API부터 우선 개발

2. 통합 테스트 중 시스템 테스트 시작
   Before: 통합(1주) --FS--> 시스템(1주)
   After:  통합(1주) --SS(+0.5주)--> 시스템(1주)
   단축: 0.5주
   
   ✅ 리스크 낮음: 일부 기능은 통합 완료 시 즉시 시스템 테스트 가능

패스트 트래킹 총 효과: 2.5주 단축
추가 비용: $0 (재작업 리스크만)
```

**C. 범위 축소 (Scope Reduction)**
```
1. Phase 1과 Phase 2로 분리
   Phase 1 (16주): 핵심 기능만 (MVP)
   - User Management
   - Product Catalog
   - Basic Order Processing
   
   Phase 2 (4주, 런칭 후): 부가 기능
   - Advanced Search (Elasticsearch)
   - Recommendation Engine
   - Admin Analytics Dashboard
   
   단축: 4주
   추가 비용: $0
   
   ✅ 권장: 시장 진입 빠르고 리스크 낮음
```

**최종 제안:**
```
통합 전략: 크래싱 + 범위 축소

1. 백엔드 개발 크래싱 (2주 단축, $20,000)
2. Phase 2 기능 이연 (4주 단축, $0)
3. 총 단축: 6주 (20주 → 14주)
4. 목표 초과 달성: 16주 목표 대비 2주 여유
5. 총 비용: $20,000
6. 리스크: 낮음

경영진 보고:
"목표 16주보다 2주 빠른 14주에 런칭 가능합니다.
 추가 비용 $20,000으로 Phase 1 기능을 먼저 출시하고,
 Phase 2 기능은 시장 피드백 반영 후 4주 내 추가하겠습니다."
```

#### 5. 기술 의존성을 고려한 일정 최적화

**병렬화 가능 영역:**
```
✅ 가능:
- 프론트엔드 + 백엔드 (API 스펙 확정 후)
- 여러 마이크로서비스 동시 개발
- 단위 테스트 + 개발 (TDD)
- UI 디자인 + 백엔드 개발

❌ 불가능 (순차 필수):
- API 설계 전에 API 개발 불가
- DB 마이그레이션 전에 데이터 사용 불가
- 코드 작성 전에 테스트 불가 (TDD 제외)
- 인프라 구축 전에 배포 불가
```

**실전 최적화 사례:**
```
Before (순차 개발): 20주
[설계] → [백엔드] → [프론트엔드] → [통합]

After (병렬 개발): 16주
[설계]
   ├→ [백엔드 User Service]     ──┐
   ├→ [백엔드 Product Service]  ──┼→ [통합]
   ├→ [백엔드 Order Service]    ──┘
   └→ [프론트엔드] (API Mock 사용) ─┘

조건:
1. API 스펙 먼저 확정
2. Mock API 서버 제공 (Postman/Swagger)
3. Git 브랜치 전략 (Feature Branch)
4. 일일 통합 (Continuous Integration)
```
