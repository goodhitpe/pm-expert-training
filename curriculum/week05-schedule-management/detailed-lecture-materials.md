## 1. 일정 관리 (Schedule Management)

### 용어
**일정 관리 (Schedule Management)**

### 설명
프로젝트 활동을 정의하고, 순서를 배열하며, 기간을 산정하여 프로젝트를 적시에 완료하기 위한 정책, 절차, 문서화를 포함하는 프로세스입니다.

### 세부 요소 및 설명

#### 1.1 일정 관리의 6가지 프로세스
- **활동 정의**: WBS 작업 패키지를 구체적 활동으로 분해
- **활동 순서 배열**: 활동 간 논리적 관계 설정
- **활동 기간 산정**: 각 활동에 필요한 시간 추정
- **일정 개발**: 활동, 기간, 자원을 통합하여 일정 수립
- **일정 통제**: 프로젝트 일정 상태를 감시하고 변경 관리

#### 1.2 일정 관리의 중요성
- 프로젝트 성공의 핵심 요소 (지연은 비용 증가로 직결)
- 이해관계자 기대 관리
- 자원 효율적 활용
- 리스크 조기 식별 및 대응

#### 1.3 일정 관리 계획서 (Schedule Management Plan)
프로젝트 일정을 개발, 관리, 실행, 통제하는 방법을 정의하는 문서:
- 일정 모델 개발 방법론
- 정확도 수준 (±10%, ±5%)
- 측정 단위 (시간, 일, 주)
- 조직 절차 연계
- 일정 베이스라인 설정 방법
- 성과 측정 방법
- 보고 형식 및 빈도

### 추가 설명

일정 관리의 성공 요인:
1. **현실적인 추정**: 과거 데이터 활용, 전문가 의견 수렴
2. **명확한 의존성**: 활동 간 관계 정확히 파악
3. **적절한 버퍼**: 불확실성 고려한 예비 시간
4. **지속적 모니터링**: 주간/격주 진척 점검
5. **능동적 조정**: 문제 발생 시 신속한 대응

일정 관리 실패의 주요 원인:
- 비현실적인 낙관적 추정 (45%)
- 범위 변경 미반영 (30%)
- 의존성 파악 미흡 (15%)
- 리스크 고려 부족 (10%)

### 별첨

#### 일정 관리 베스트 프랙티스
1. **상향식 추정**: 작은 단위로 나누어 정확도 향상
2. **버퍼 관리**: 크리티컬 체인 기법 활용
3. **마일스톤 관리**: 주요 완료 시점 명확히 설정
4. **시각화**: 간트 차트로 일정 공유
5. **정기 리뷰**: 주간 일정 회의 진행

#### 실제 프로젝트 사례
- **성공 사례**: 삼성전자 갤럭시 출시 프로젝트 - 철저한 마일스톤 관리로 출시일 준수
- **실패 사례**: 공공 IT 시스템 구축 - 의존성 미파악으로 6개월 지연

## 2. 간트 차트 (Gantt Chart)

### 용어
**간트 차트 (Gantt Chart)**

### 설명
프로젝트 일정을 막대 그래프 형태로 시각화한 도구로, 각 활동의 시작일, 종료일, 기간, 의존 관계를 한눈에 파악할 수 있습니다.

### 세부 요소 및 설명

#### 2.1 간트 차트의 구성 요소
- **작업 목록 (세로축)**: WBS 기반 활동 나열
- **시간축 (가로축)**: 날짜, 주, 월 단위
- **작업 막대**: 활동 기간을 막대로 표시
- **의존 관계 화살표**: 활동 간 연결 표시
- **마일스톤**: 주요 완료 시점 (다이아몬드 모양)
- **진척도**: 막대 내부 색상으로 완료율 표시
- **오늘 날짜 선**: 현재 진행 상황 파악

#### 2.2 간트 차트 작성 단계
1. **WBS에서 활동 추출**: 작업 패키지를 활동으로 분해
2. **활동 나열**: 계층 구조대로 정렬
3. **기간 산정**: 각 활동의 소요 시간 결정
4. **의존 관계 설정**: FS, SS, FF, SF 관계 정의
5. **자원 배분**: 담당자 및 자원 할당
6. **일정 최적화**: 크리티컬 패스 분석 및 조정
7. **마일스톤 표시**: 주요 검토 시점 명시
8. **베이스라인 설정**: 초기 계획 저장

#### 2.3 간트 차트 활용
- **계획 단계**: 일정 수립 및 검토
- **실행 단계**: 진척도 추적 및 보고
- **커뮤니케이션**: 이해관계자와 일정 공유
- **의사결정**: 일정 지연 시 대응 방안 수립

#### 2.4 간트 차트의 장단점
**장점:**
- 직관적이고 이해하기 쉬움
- 전체 일정을 한눈에 파악
- 진척도 시각화 용이
- 이해관계자 커뮤니케이션에 효과적

**단점:**
- 복잡한 프로젝트는 가독성 저하
- 의존 관계 표현에 한계
- 정적 스냅샷 (동적 변경 추적 어려움)
- 크리티컬 패스 식별 어려움

### 추가 설명

간트 차트는 1910년대 Henry Gantt가 개발한 이후 100년 이상 사용되는 검증된 도구입니다. 현대에는 소프트웨어 도구를 통해 더욱 강력한 기능을 제공합니다.

**간트 차트 작성 도구:**
- **Microsoft Project**: 전문가용, 복잡한 프로젝트
- **Excel**: 간단한 프로젝트, 수동 작성
- **Smartsheet**: 협업 중심, 클라우드 기반
- **Monday.com**: 직관적 UI, 소규모 팀
- **Asana**: 태스크 관리 + 타임라인

**효과적인 간트 차트 작성 팁:**
1. 너무 상세하지 않게: 적정 수준의 세분화
2. 색상 활용: 단계별, 팀별 구분
3. 정기 업데이트: 주간 단위 진척도 반영
4. 버전 관리: 변경 이력 추적
5. 인쇄 고려: A3 크기로 출력 가능하게

## 3. 크리티컬 패스 (Critical Path)

### 용어
**크리티컬 패스 (Critical Path, CP)**

### 설명
프로젝트 네트워크 다이어그램에서 가장 긴 경로로, 여유 시간(Float)이 0인 활동들의 연속입니다. 이 경로의 지연은 전체 프로젝트 완료일에 직접 영향을 미칩니다.

### 세부 요소 및 설명

#### 3.1 크리티컬 패스 방법론 (CPM, Critical Path Method)
1957년 DuPont와 Remington Rand에서 개발한 일정 분석 기법:
- 네트워크 다이어그램 작성
- 순방향 계산으로 조기 일정 산출
- 역방향 계산으로 늦은 일정 산출
- 여유 시간 계산으로 크리티컬 패스 식별

#### 3.2 주요 개념 및 용어
- **ES (Early Start)**: 조기 시작일 - 활동을 가장 빨리 시작할 수 있는 날짜
- **EF (Early Finish)**: 조기 종료일 - 활동을 가장 빨리 끝낼 수 있는 날짜
- **LS (Late Start)**: 늦은 시작일 - 프로젝트 지연 없이 시작 가능한 최대 날짜
- **LF (Late Finish)**: 늦은 종료일 - 프로젝트 지연 없이 종료 가능한 최대 날짜
- **TF (Total Float)**: 총 여유 시간 = LS - ES 또는 LF - EF
- **FF (Free Float)**: 자유 여유 시간 - 후속 활동에 영향 없이 지연 가능한 시간

#### 3.3 순방향 계산 (Forward Pass)
프로젝트 시작부터 종료까지 계산하여 조기 일정 산출:
1. 시작 활동의 ES = 0 (또는 프로젝트 시작일)
2. EF = ES + 기간
3. 후속 활동의 ES = 선행 활동의 EF 중 최대값
4. 프로젝트 완료일 = 최종 활동의 EF

#### 3.4 역방향 계산 (Backward Pass)
프로젝트 종료부터 시작까지 계산하여 늦은 일정 산출:
1. 최종 활동의 LF = EF (프로젝트 완료일)
2. LS = LF - 기간
3. 선행 활동의 LF = 후속 활동의 LS 중 최소값
4. 시작 활동까지 계속 계산

#### 3.5 크리티컬 패스 식별
- TF = 0인 활동들을 연결
- 시작부터 종료까지 끊김 없이 연결된 경로
- 여러 개의 크리티컬 패스가 존재할 수 있음

#### 3.6 크리티컬 패스 활용
- **우선순위 결정**: CP 활동에 관리 집중
- **자원 배분**: 중요 자원을 CP 활동에 우선 배정
- **리스크 관리**: CP 활동의 리스크 중점 모니터링
- **일정 단축**: CP 활동 대상 크래싱/패스트 트래킹
- **진척 관리**: CP 활동의 지연 방지

### 추가 설명

**크리티컬 패스의 특성:**
1. 프로젝트의 최단 완료 기간 결정
2. 여유 시간이 없어 지연 시 전체 일정 영향
3. 프로젝트 진행 중 변경될 수 있음
4. 일정 압축 시 주 대상이 됨

**크리티컬 패스 관리 전략:**
1. **일일 모니터링**: CP 활동의 진척 매일 확인
2. **자원 우선 배정**: 최고 역량 자원 투입
3. **리스크 버퍼**: CP에 시간 버퍼 추가 고려
4. **병렬화 검토**: 가능한 활동 병렬 수행
5. **조기 경보**: 지연 징후 즉시 보고

**실무 활용 사례:**
- 건설 프로젝트: 기초 작업 → 골조 → 외벽 → 내부 마감
- IT 프로젝트: 요구사항 → 설계 → 개발 → 테스트 → 배포
- 제품 출시: 설계 → 시제품 → 양산 준비 → 마케팅 → 출시

### 별첨

#### 크리티컬 패스 계산 예제
```
활동  기간  선행  ES  EF  LS  LF  TF  CP
A     3     -     0   3   0   3   0   ●
B     5     A     3   8   3   8   0   ●
C     4     A     3   7   4   8   1
D     6     B     8   14  8   14  0   ●
E     3     C     7   10  11  14  4
F     2     D,E   14  16  14  16  0   ●

크리티컬 패스: A → B → D → F (총 16일)
```

## 4. 활동 기간 산정 (Activity Duration Estimating)

### 용어
**활동 기간 산정 (Activity Duration Estimating)**

### 설명
각 활동을 완료하는 데 필요한 작업 기간을 추정하는 프로세스로, 자원 가용성, 역량, 생산성을 고려하여 현실적인 기간을 결정합니다.

### 세부 요소 및 설명

#### 4.1 기간 산정 기법

**4.1.1 유사 산정 (Analogous Estimating)**
- **정의**: 과거 유사 프로젝트 데이터를 기반으로 추정
- **장점**: 빠르고 비용 효율적, 정보 부족 시 유용
- **단점**: 정확도 낮음 (±25%), 프로젝트 유사성에 의존
- **활용**: 프로젝트 초기, 개략 산정

**예시:**
- 과거 모바일 앱 개발: 3개월
- 현재 유사 앱 개발 추정: 3개월 ± 25% = 2.25~3.75개월

**4.1.2 모수 산정 (Parametric Estimating)**
- **정의**: 통계적 관계와 변수를 활용한 산정
- **장점**: 정확도 높음 (±10%), 객관적
- **단점**: 신뢰할 만한 데이터 필요
- **활용**: 반복적 작업, 데이터 축적된 영역

**예시:**
- 코딩 생산성: 100줄/일
- 필요 코드: 5,000줄
- 산정 기간: 5,000 / 100 = 50일

**4.1.3 3점 산정 (Three-Point Estimating, PERT)**
- **정의**: 낙관, 가능성 높음, 비관 시나리오 고려
- **장점**: 불확실성 반영, 리스크 고려
- **단점**: 추정에 시간 소요
- **활용**: 불확실성 높은 활동

**공식:**
- **PERT (Beta 분포)**: tE = (tO + 4tM + tP) / 6
- **삼각 분포**: tE = (tO + tM + tP) / 3
- **표준편차**: σ = (tP - tO) / 6

**예시:**
- 낙관적 (tO): 3일
- 가능성 높음 (tM): 5일
- 비관적 (tP): 9일
- 기대값: (3 + 20 + 9) / 6 = 5.33일
- 표준편차: (9 - 3) / 6 = 1일

**4.1.4 상향식 산정 (Bottom-Up Estimating)**
- **정의**: 세부 활동별로 산정 후 합산
- **장점**: 가장 정확 (±5%), 상세함
- **단점**: 시간 많이 소요, 비용 높음
- **활용**: 프로젝트 실행 단계, 정확성 중요 시

#### 4.2 기간 산정 시 고려사항
- **자원 가용성**: 풀타임 vs 파트타임
- **역량 수준**: 초급 vs 전문가 (생산성 10배 차이)
- **학습 곡선**: 초기 느림 → 점진적 향상
- **의존성**: 선행 작업 완료 대기
- **제약사항**: 마감일, 예산, 리소스 제한
- **리스크**: 기술적 난이도, 외부 의존성

#### 4.3 버퍼 관리
- **프로젝트 버퍼**: 크리티컬 패스 끝에 배치 (전체 기간의 50%)
- **피딩 버퍼**: 비 크리티컬 패스가 CP와 만나는 지점
- **리소스 버퍼**: 핵심 자원 필요 전 알림

### 추가 설명

**산정 정확도 향상 방법:**
1. **역사적 데이터 활용**: 과거 프로젝트 기록 분석
2. **전문가 의견**: 경험 많은 팀원 참여
3. **델파이 기법**: 전문가 그룹의 합의 도출
4. **점진적 정교화**: 프로젝트 진행에 따라 재산정
5. **학습 곡선 적용**: 반복 작업의 효율 향상 반영

**일반적인 산정 오류:**
- 낙관적 편향 (Planning Fallacy): 실제보다 짧게 추정
- 불확실성 무시: 리스크 고려 부족
- 자원 다중 할당: 100% 생산성 가정
- 간접 시간 누락: 회의, 휴가, 병가 미고려

### 별첨

#### 산정 정확도 비교
| 기법 | 정확도 | 소요 시간 | 적용 단계 |
|------|--------|-----------|-----------|
| 유사 산정 | ±25% | 낮음 | 착수 |
| 모수 산정 | ±10% | 중간 | 계획 초기 |
| 3점 산정 | ±10~15% | 중간 | 계획 중기 |
| 상향식 산정 | ±5% | 높음 | 계획 후기 |

#### 참고 자료
- PMBOK Guide Chapter 6.5
- "Software Estimation: Demystifying the Black Art" - Steve McConnell
- "Agile Estimating and Planning" - Mike Cohn

## 5. 일정 압축 (Schedule Compression)

### 용어
**일정 압축 (Schedule Compression)**

### 설명
범위 변경 없이 프로젝트 일정을 단축하는 기법으로, 주로 크래싱(Crashing)과 패스트 트래킹(Fast Tracking) 두 가지 방법이 사용됩니다.

### 세부 요소 및 설명

#### 5.1 크래싱 (Crashing)
**정의**: 추가 자원 투입으로 활동 기간 단축
**특징**: 비용 증가, 범위 불변, 위험도 낮음

**크래싱 프로세스:**
1. 크리티컬 패스 식별
2. CP 활동 중 크래싱 가능 활동 선정
3. 각 활동의 크래싱 비용 계산
4. 비용 대비 효과 분석
5. 가장 경제적인 활동부터 크래싱
6. 새로운 CP 확인 (변경 가능)
7. 목표 달성까지 반복

**크래싱 가능 자원:**
- 인력 추가 (개발자, 작업자)
- 교대 근무 (2교대 → 3교대)
- 초과 근무
- 전문 장비/도구 추가
- 외주/아웃소싱

**크래싱 예시:**
```
활동 A: 10일, 비용 1,000만원
크래싱 옵션:
- 8일로 단축: 추가 비용 200만원 (100만원/일)
- 6일로 단축: 추가 비용 600만원 (150만원/일)

선택: 비용 효율 고려하여 8일로 단축
```

**크래싱 시 주의사항:**
- 브룩스의 법칙: "지연된 프로젝트에 인력 추가는 더 지연시킴"
- 한계 효율: 일정 수준 이상 단축 불가
- 품질 저하 위험: 과도한 압박
- 팀 사기 저하: 지속적 초과 근무

#### 5.2 패스트 트래킹 (Fast Tracking)
**정의**: 순차적 활동을 병렬로 수행
**특징**: 비용 증가 최소, 재작업 리스크 증가

**패스트 트래킹 프로세스:**
1. 크리티컬 패스 분석
2. 순차 활동 중 병렬화 가능 식별
3. 병렬화 시 리스크 평가
4. 의존성 완화 방안 수립
5. 병렬 실행 계획 수립
6. 긴밀한 조정 및 모니터링

**패스트 트래킹 예시:**
```
기존: 설계 완료 → 개발 시작 (순차)
변경: 설계 80% 완료 시점에 개발 착수 (병렬)

효과: 전체 기간 20% 단축
리스크: 설계 변경 시 재개발 가능성
```

**병렬화 가능한 사례:**
- 설계와 구매 병행
- 개발과 테스트 환경 구축 병행
- 제품 개발과 마케팅 준비 병행
- 건설 기초 작업과 자재 발주 병행

**패스트 트래킹 시 주의사항:**
- 재작업 비용 발생 가능
- 긴밀한 의사소통 필요
- 변경 관리 프로세스 강화
- 리스크 모니터링 집중

#### 5.3 크래싱 vs 패스트 트래킹 비교
| 구분 | 크래싱 | 패스트 트래킹 |
|------|--------|---------------|
| 방법 | 자원 추가 | 병렬 수행 |
| 비용 | 증가 | 최소 증가 |
| 리스크 | 낮음 | 높음 (재작업) |
| 적용 | 자원 추가 가능 시 | 의존성 완화 가능 시 |
| 효과 | 확실함 | 불확실함 |

#### 5.4 일정 압축 전략
1. **최소 비용 단축**: CP에서 단위당 비용 최소 활동부터
2. **리스크 균형**: 크래싱과 패스트 트래킹 혼합
3. **단계적 적용**: 목표 달성까지 점진적 압축
4. **대안 검토**: 범위 축소, 품질 기준 조정도 고려

### 추가 설명

**일정 압축이 필요한 상황:**
- 시장 기회 포착 (경쟁사보다 빠른 출시)
- 계약상 패널티 회피
- 자원 가용성 제약 (핵심 인력 이탈 예정)
- 외부 의존성 (전시회, 행사 일정)

**일정 압축의 한계:**
- 물리적 한계: 임신 기간은 9개월 (인력 추가 불가)
- 품질 저하: 과도한 압축은 결함 증가
- 비용 급증: 한계 비용 체증
- 팀 번아웃: 지속 불가능

### 별첨

#### 일정 압축 의사결정 트리
```
일정 단축 필요?
├─ Yes → 크리티컬 패스 식별
│        ├─ 자원 추가 가능? → Yes → 크래싱 적용
│        │                    No ↓
│        └─ 병렬화 가능? → Yes → 패스트 트래킹
│                          No → 범위/품질 조정 협의
└─ No → 정상 일정 진행
```

#### 참고 자료
- PMBOK Guide: Schedule Compression
- "The Mythical Man-Month" - Frederick Brooks
- "Critical Chain" - Eliyahu M. Goldratt

## 6. 일정 베이스라인 (Schedule Baseline)

### 용어
**일정 베이스라인 (Schedule Baseline)**

### 설명
승인된 프로젝트 일정의 기준선으로, 실제 성과를 측정하고 비교하는 기준이 됩니다. 프로젝트 계획서의 일부로 공식 변경 통제 절차를 통해서만 변경됩니다.

### 세부 요소 및 설명

#### 6.1 베이스라인의 구성 요소
- **시작 마일스톤**: 프로젝트 착수일
- **주요 마일스톤**: 단계별 완료 시점
- **활동 일정**: 각 활동의 시작일/종료일
- **종료 마일스톤**: 프로젝트 완료일
- **버퍼**: 일정 예비 시간

#### 6.2 베이스라인 설정 시기
- 프로젝트 계획 완료 후
- 이해관계자 승인 획득 후
- 프로젝트 실행 개시 전
- 주요 변경 승인 후 (재설정)

#### 6.3 베이스라인 활용
- **성과 측정**: 계획 대비 실제 진척 비교
- **변경 관리**: 일정 변경 영향 분석
- **의사소통**: 이해관계자 보고
- **교훈 수집**: 프로젝트 종료 후 분석

#### 6.4 베이스라인 vs 현재 일정
| 구분 | 베이스라인 | 현재 일정 |
|------|-----------|----------|
| 정의 | 승인된 계획 | 최신 상태 |
| 변경 | 공식 절차 필요 | 수시 업데이트 |
| 목적 | 성과 측정 기준 | 진행 관리 |

### 추가 설명

베이스라인 관리는 프로젝트 통제의 핵심입니다. 베이스라인 없이는 프로젝트가 계획 대비 얼마나 진행되었는지, 지연되었는지 판단할 수 없습니다.

**베이스라인 재설정 상황:**
- 중대한 범위 변경 승인
- 프로젝트 목표 변경
- 주요 가정 변경
- 계획 대비 ±20% 이상 편차

### 별첨

#### 참고 자료
- PMBOK Guide: Performance Measurement Baseline
