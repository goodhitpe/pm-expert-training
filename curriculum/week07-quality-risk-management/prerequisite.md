## 1️⃣ 품질(Quality)이란?

### 정의
**품질**은 **요구사항을 충족하는 정도**입니다.

### 착각하기 쉬운 개념
❌ "비싼 것 = 품질 좋은 것" (잘못된 이해)  
✅ "요구사항 만족 = 품질 좋은 것" (올바른 이해)

### 실생활 비유
```
🚗 자동차 구매:

시나리오 1: 출퇴근용 차량 필요
- 요구사항: 연비 좋고, 저렴하고, 주차 쉬운 차
- 선택: 경차
- 품질: 높음 ✅ (요구사항 충족)

시나리오 2: 출퇴근용 차량 필요
- 요구사항: 연비 좋고, 저렴하고, 주차 쉬운 차
- 선택: 고급 대형 세단
- 품질: 낮음 ❌ (비싸고, 주차 어려움 = 요구사항 불충족)
```

### PM 관점
```
프로젝트 요구사항: "2주 내에 간단한 홍보 전단지 디자인"

결과 A: 2주 만에 완성, 깔끔한 디자인
→ 높은 품질 ✅

결과 B: 4주 걸림, 화려한 디자인
→ 낮은 품질 ❌ (일정 요구사항 미충족)
```

## 2️⃣ 등급(Grade) vs 품질(Quality)

### 등급이란?
제품이나 서비스의 **기능 수준이나 사양**

### 품질이란?
**결함이 없는 정도**

### 중요한 차이점
- **낮은 등급 + 높은 품질**: 가능 ✅
- **높은 등급 + 낮은 품질**: 문제 있음 ❌

### 실생활 비유
```
🏨 호텔 예약:

모텔 (낮은 등급):
- 시설: 기본적
- 서비스: 간단
- 결함: 없음 (깨끗, 조용, 예약대로)
→ 낮은 등급, 높은 품질 ✅

5성급 호텔 (높은 등급):
- 시설: 최고급
- 서비스: 프리미엄
- 결함: 있음 (더러운 침대, 시끄러움, 예약 오류)
→ 높은 등급, 낮은 품질 ❌
```

### PM 관점
```
웹사이트 개발:

기본 웹사이트 (낮은 등급):
- 기능: 게시판, 로그인만
- 디자인: 단순
- 버그: 없음
→ 낮은 등급, 높은 품질 ✅ (요구사항 충족)

고급 웹사이트 (높은 등급):
- 기능: AI 추천, 챗봇 등 다양
- 디자인: 화려
- 버그: 많음
→ 높은 등급, 낮은 품질 ❌ (결함 많음)
```

## 3️⃣ 품질 보증(QA) vs 품질 통제(QC)

### 품질 보증 (Quality Assurance, QA)
**"일을 제대로 하고 있는가?"** - 프로세스 중심

**예시:**
- 개발 표준 준수 확인
- 코드 리뷰 절차 진행
- 테스트 계획대로 진행

**실생활 비유:**
```
🍜 음식점:
- 위생 수칙 준수 (손 씻기, 장갑 착용)
- 레시피대로 조리
- 조리 온도 체크
→ 과정을 제대로 했는가?
```

### 품질 통제 (Quality Control, QC)
**"결과물이 괜찮은가?"** - 산출물 중심

**예시:**
- 완성된 코드 테스트
- 버그 발견 및 수정
- 최종 검수

**실생활 비유:**
```
🍜 음식점:
- 완성된 음식 맛보기
- 온도 측정
- 비주얼 확인
→ 결과물이 괜찮은가?
```

### 쉬운 구분
```
자동차 제조:
- QA: 조립 과정 매뉴얼 준수, 작업자 교육
- QC: 완성차 테스트 주행, 외관 검사

결혼식 준비:
- QA: 체크리스트대로 준비, 리허설 진행
- QC: 리허설 후 문제점 발견 및 수정
```

## 4️⃣ 리스크(Risk)란?

### 정의
**발생할 수도 있는 불확실한 사건**

### 2가지 유형

#### 부정적 리스크 (위협, Threat)
**나쁜 일이 일어날 가능성**

**예시:**
- 핵심 직원 퇴사
- 예산 삭감
- 외주 업체 납기 지연

#### 긍정적 리스크 (기회, Opportunity)
**좋은 일이 일어날 가능성**

**예시:**
- 경쟁사 프로젝트 실패로 시장 기회 확대
- 신기술로 개발 기간 단축
- 예산 추가 확보

### 실생활 비유
```
🌤️ 주말 야외 결혼식:
- 부정적 리스크: 비가 올 수 있음 ⚠️
- 긍정적 리스크: 날씨가 예상보다 더 좋을 수 있음 ✨
```

---

## 5️⃣ 리스크의 2가지 요소

### 발생 확률 (Probability)
**얼마나 일어날 가능성이 있는가?**

**예시:**
- 높음: 50-80%
- 중간: 20-50%
- 낮음: 5-20%

### 영향도 (Impact)
**발생하면 얼마나 큰 영향을 미치는가?**

**예시:**
- 높음: 프로젝트 실패 수준
- 중간: 일정/예산 10-20% 영향
- 낮음: 사소한 불편

### 실생활 비유
```
🚗 출퇴근 시나리오:

리스크 1: 교통 체증
- 확률: 높음 (70%)
- 영향: 낮음 (10-15분 지연)
→ 높은 확률 × 낮은 영향 = 중간 리스크

리스크 2: 자동차 고장
- 확률: 낮음 (5%)
- 영향: 높음 (출근 불가)
→ 낮은 확률 × 높은 영향 = 중간 리스크

리스크 3: 사고
- 확률: 매우 낮음 (1%)
- 영향: 매우 높음 (입원)
→ 매우 낮은 확률 × 매우 높은 영향 = 중간 리스크
```

---

## 6️⃣ 리스크 매트릭스

### 정의
확률과 영향도를 조합하여 **리스크 우선순위** 결정

### 매트릭스 예시
```
영향도
높음  │  중    높    매우높음
중간  │  낮    중    높음
낮음  │ 매우낮 낮    중간
      └────────────────
       낮음  중간  높음  
            확률
```

### 실생활 비유
```
집들이 파티 리스크:

매우 높음 (즉시 대응):
- 가스 누출 (낮은 확률 × 높은 영향)

높음 (우선 대응):
- 음식 부족 (중간 확률 × 중간 영향)

중간 (모니터링):
- 손님 늦게 도착 (높은 확률 × 낮은 영향)

낮음 (무시 가능):
- 음악 플레이리스트 실수 (낮은 확률 × 낮은 영향)
```

## 7️⃣ 리스크 대응 전략

### 부정적 리스크 (위협) 대응

#### 1. 회피 (Avoid)
**리스크 원인을 제거**

**예시:**
```
리스크: 신기술 사용으로 실패 가능성
대응: 검증된 기술 사용 (리스크 제거)
```

#### 2. 전가 (Transfer)
**제3자에게 리스크 이전**

**예시:**
```
리스크: 서버 장애
대응: 클라우드 서비스 사용 (업체가 책임)
```

#### 3. 완화 (Mitigate)
**확률이나 영향도 감소**

**예시:**
```
리스크: 핵심 직원 퇴사
대응: 백업 인력 교육, 문서화 강화
```

#### 4. 수용 (Accept)
**리스크를 인정하고 대응 계획 준비**

**예시:**
```
리스크: 환율 변동
대응: 예비비 확보, 발생 시 대응
```

### 실생활 비유
```
🌧️ 소풍 날 비 올 리스크:

회피: 실내 장소로 변경 (비 리스크 제거)
전가: 여행사에 맡김 (여행사가 책임)
완화: 우산 준비, 텐트 설치 (영향 최소화)
수용: 그냥 가기, 비 오면 일찍 철수 (계획 B 준비)
```

## 💬 자주 묻는 질문 (FAQ)

**Q1: 모든 리스크에 대응해야 하나요?**  
A: 아닙니다! 우선순위가 낮은 리스크(낮은 확률 × 낮은 영향)는 모니터링만 하거나 무시할 수 있습니다. 자원을 효율적으로 사용하세요.

**Q2: QA와 QC 중 무엇이 더 중요한가요?**  
A: 둘 다 중요하지만, QA(프로세스)가 더 근본적입니다. 과정을 제대로 하면 결과물도 좋아집니다. "예방이 치료보다 낫다"는 원칙입니다.

**Q3: 리스크를 100% 제거할 수 있나요?**  
A: 불가능합니다. 프로젝트는 본질적으로 불확실성을 가지고 있습니다. 목표는 제거가 아닌 "관리"입니다.

**Q4: 품질을 높이면 비용이 많이 드나요?**  
A: 단기적으로는 그럴 수 있지만, 장기적으로는 비용 절감입니다. 재작업, 고객 불만, A/S 비용이 줄어듭니다. "품질 비용"은 투자입니다.

**Q5: 긍정적 리스크(기회)도 관리해야 하나요?**  
A: 네! 기회를 놓치지 않도록 적극적으로 관리하세요. 예를 들어, 경쟁사 실패 시 빠르게 시장 진입할 준비를 해두세요.
