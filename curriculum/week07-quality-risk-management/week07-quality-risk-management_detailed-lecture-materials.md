## 📖 Part 1: 품질 관리 (Quality Management)

### 1.1 품질 관리의 중요성

#### 품질 실패의 영향
- **비용 증가**: 재작업으로 인한 추가 비용 (프로젝트 예산의 10-20%)
- **일정 지연**: 품질 문제 해결로 인한 지연
- **고객 불만**: 신뢰 상실 및 계약 해지
- **명성 손상**: 브랜드 이미지 하락

#### 품질 관리 vs 품질 보증 vs 품질 통제

| 구분 | 품질 관리(QM) | 품질 보증(QA) | 품질 통제(QC) |
|-----|--------------|--------------|--------------|
| **정의** | 전체 품질 체계 | 프로세스 준수 | 산출물 검증 |
| **초점** | 정책/표준 수립 | 프로세스 개선 | 결함 발견 |
| **시점** | 프로젝트 전체 | 작업 수행 중 | 산출물 완성 후 |
| **활동** | 품질 계획 수립 | 프로세스 감사 | 테스트/검사 |
| **책임** | PM | QA 팀 | QC 팀 |

### 1.2 품질 비용 (Cost of Quality, CoQ)

#### CoQ 4가지 범주

```
품질 비용 (CoQ)
├─ 적합 비용 (Cost of Conformance)
│  ├─ 예방 비용 (Prevention Costs)
│  │  - 품질 계획 수립
│  │  - 교육 및 훈련
│  │  - 프로세스 개선
│  │  - 품질 도구 도입
│  └─ 평가 비용 (Appraisal Costs)
│     - 검사 및 테스트
│     - 감사 및 리뷰
│     - 계측 장비 유지
│
└─ 부적합 비용 (Cost of Non-Conformance)
   ├─ 내부 실패 비용 (Internal Failure Costs)
   │  - 재작업
   │  - 폐기
   │  - 지연
   └─ 외부 실패 비용 (External Failure Costs)
      - 보증 수리
      - 제품 회수
      - 소송 비용
      - 명성 손상
```

#### CoQ 계산 예제

**시나리오**: 소프트웨어 프로젝트
- 프로젝트 예산: ₩500,000,000
- 예방 비용: ₩20,000,000 (코드 리뷰, 교육)
- 평가 비용: ₩30,000,000 (테스트)
- 내부 실패: ₩25,000,000 (버그 수정)
- 외부 실패: ₩15,000,000 (고객 패치)

```
적합 비용 = ₩20M + ₩30M = ₩50M (10%)
부적합 비용 = ₩25M + ₩15M = ₩40M (8%)
총 품질 비용 = ₩90M (18%)
```

**분석**:
- ✅ 예방 비용 증가 → 외부 실패 비용 감소
- ⚠️ 외부 실패 비용 비중 높음 → 테스트 강화 필요
- 🎯 목표: 부적합 비용을 5% 이하로 감축

## 🔧 2. 7가지 기본 품질 도구 (7 Basic Quality Tools)

### 2.1 특성요인도 (Fishbone Diagram / Ishikawa Diagram)

#### 정의
문제의 원인을 체계적으로 분석하기 위한 도구

#### 6M 분류법
```
                      문제/결과
                          │
        ┌────────┬────────┼────────┬────────┐
        │        │        │        │        │
      Man     Machine  Material Method  Measurement Environment
    (인력)    (장비)    (재료)   (방법)   (측정)     (환경)
```

#### 실습 예제: 소프트웨어 버그 증가 문제

```
                  소프트웨어 버그 증가
                          │
    ┌─────────┬─────────┼─────────┬─────────┐
    │         │         │         │         │
  인력       장비      재료      방법      측정
    │         │         │         │         │
경험 부족   개발 환경   라이브러리  코드리뷰   테스트 커버리지
높은 이직률  버전 불일치  구버전 사용  생략      미측정
교육 부족   테스트 도구  문서 부족   급한 일정  자동화 부족
```

**활용법**:
1. 문제 정의 (우측에 기재)
2. 주요 범주 식별 (6M)
3. 각 범주별 원인 브레인스토밍
4. 원인의 원인 파악 (Why-Why 분석)
5. 근본 원인 선정

### 2.2 파레토 차트 (Pareto Chart)

#### 정의
80/20 법칙을 활용한 주요 문제 식별 도구

#### 실습 예제: 고객 불만 분석

| 불만 유형 | 건수 | 비율 | 누적% |
|---------|------|------|-------|
| 응답 지연 | 120 | 40% | 40% |
| 기능 오류 | 90 | 30% | 70% |
| UI 불편 | 45 | 15% | 85% |
| 문서 부족 | 30 | 10% | 95% |
| 기타 | 15 | 5% | 100% |
| **합계** | 300 | 100% | - |

```
건수  │
140  │ ██████
120  │ ██████           100%
100  │ ██████ ████      90%
 80  │ ██████ ████      80%
 60  │ ██████ ████ ██   70%
 40  │ ██████ ████ ██   60%
 20  │ ██████ ████ ██ █ 50%
  0  └─────────────────────
      응답   기능  UI  문서 기타
```

**해석**:
- ✅ 상위 2개 항목(응답 지연, 기능 오류)이 70% 차지
- 🎯 우선순위: 응답 지연 개선에 집중
- 💡 투자 효과: 2개 문제 해결로 70% 불만 감소

### 2.3 관리도 (Control Chart)

#### 정의
프로세스가 통계적 관리 상태인지 확인하는 도구

#### 관리 한계선
```
UCL (Upper Control Limit)  ──────────  +3σ
                                       +2σ
중심선 (Center Line)        ──────────  평균
                                       -2σ
LCL (Lower Control Limit)  ──────────  -3σ
```

#### 실습 예제: 개발 생산성 모니터링

**데이터**: 주간 완성 스토리 포인트

| 주차 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|-----|---|---|---|---|---|---|---|---|
| SP | 25 | 28 | 26 | 30 | 24 | 27 | 29 | 26 |

```
평균 = 26.9
표준편차 = 2.1
UCL = 26.9 + (3 × 2.1) = 33.2
LCL = 26.9 - (3 × 2.1) = 20.6

SP   │
 35  │ UCL ──────────────────
 30  │     •   •       •
 25  │ •       •   • •     •
 20  │ LCL ──────────────────
  0  └───────────────────────
      1   2   3   4   5   6   7   8
```

**해석**:
- ✅ 모든 점이 관리 한계 내 → 안정적 프로세스
- 🎯 평균 26.9 SP/주 유지 가능
- ⚠️ 특이 패턴 없음 (7개 연속 상승 등)

### 2.4 산점도 (Scatter Diagram)

```

  •                  •            •   •
    •              •                • •
      •          •                •   •
        •      •                    •
          •  •                    •   •
```

|-----------|----|----|----|----|----|----|-----|

- 🎯 권장: 최소 80% 커버리지 유지

## ⚠️ Part 2: 리스크 관리 (Risk Management)

### 3.1 리스크 관리의 중요성

#### 리스크 관리 실패 사례
- **삼풍백화점 붕괴** (1995): 리스크 무시로 502명 사망
- **세월호 침몰** (2014): 안전 리스크 미대응
- **후쿠시마 원전 사고** (2011): 자연재해 리스크 과소평가
- **보잉 737 MAX** (2019): 기술 리스크 은폐

#### 리스크 vs 이슈

| 구분 | 리스크 (Risk) | 이슈 (Issue) |
|-----|-------------|------------|
| **정의** | 미래 불확실성 | 현재 문제 |
| **발생 여부** | 발생할 수도 있음 | 이미 발생함 |
| **대응** | 사전 예방 | 즉각 해결 |
| **관리** | 리스크 관리 | 이슈 관리 |
| **예시** | "서버 장애 가능성" | "서버 다운됨" |

### 3.2 리스크 관리 프로세스

```
1. 리스크 관리 계획 수립 (Plan Risk Management)
   ↓
2. 리스크 식별 (Identify Risks)
   ↓
3. 리스크 정성적 분석 (Qualitative Analysis)
   ↓
4. 리스크 정량적 분석 (Quantitative Analysis)
   ↓
5. 리스크 대응 계획 수립 (Plan Risk Responses)
   ↓
6. 리스크 대응 실행 (Implement Risk Responses)
   ↓
7. 리스크 모니터링 (Monitor Risks)
```

### 3.3 리스크 식별 기법

#### 3.3.1 브레인스토밍
- **장점**: 많은 아이디어 도출
- **단점**: 품질 편차, 시간 소요
- **활용**: 프로젝트 초기 단계

#### 3.3.2 델파이 기법
- **장점**: 전문가 의견 수렴, 편향 감소
- **단점**: 시간 많이 소요
- **활용**: 중요한 의사결정

#### 3.3.3 SWOT 분석
```
내부 요인:
- Strengths (강점): 활용 방안
- Weaknesses (약점): 개선 필요

외부 요인:
- Opportunities (기회): 기회 리스크
- Threats (위협): 위협 리스크
```

#### 3.3.4 체크리스트
- 과거 프로젝트 리스크 목록
- 산업별 공통 리스크
- 조직 리스크 데이터베이스

### 3.4 리스크 매트릭스 (Risk Matrix)

#### 확률-영향 매트릭스

```
영향 │  극상  │   5   │  10  │  15  │  20  │  25  │
(I)  │  상    │   4   │   8  │  12  │  16  │  20  │
     │  중    │   3   │   6  │   9  │  12  │  15  │
     │  하    │   2   │   4  │   6  │   8  │  10  │
     │  극하  │   1   │   2  │   3  │   4  │   5  │
     └────────┴───────┴──────┴──────┴──────┴──────┘
                극하    하     중     상    극상
                      확률 (P)
```

**리스크 점수 = 확률 × 영향**

#### 리스크 등급 분류

| 점수 | 등급 | 색상 | 대응 우선순위 |
|-----|------|------|-------------|
| 15-25 | 극상 | 🔴 빨강 | 즉시 대응 |
| 8-12 | 상 | 🟠 주황 | 우선 대응 |
| 4-6 | 중 | 🟡 노랑 | 모니터링 |
| 1-3 | 하 | 🟢 초록 | 수용 가능 |

#### 실습 예제: 웹 서비스 런칭 프로젝트

| ID | 리스크 | 확률 | 영향 | 점수 | 등급 |
|----|-------|------|------|------|------|
| R1 | 서버 과부하 | 4 | 5 | 20 | 🔴 극상 |
| R2 | 보안 취약점 | 3 | 5 | 15 | 🔴 극상 |
| R3 | 핵심 인력 이탈 | 2 | 4 | 8 | 🟠 상 |
| R4 | 타사 API 장애 | 3 | 3 | 9 | 🟠 상 |
| R5 | 브라우저 호환성 | 3 | 2 | 6 | 🟡 중 |
| R6 | 문서 미비 | 4 | 1 | 4 | 🟡 중 |
| R7 | 디자인 변경 | 2 | 1 | 2 | 🟢 하 |

## 🎯 4. 리스크 대응 전략

### 4.1 위협(Threat) 대응 전략

#### 4.1.1 회피 (Avoid)
**정의**: 리스크를 제거하거나 위협을 차단

**예시**:
- 불확실한 기술 대신 검증된 기술 사용
- 문제 있는 협력사 계약 취소
- 리스크 높은 기능 제외

**시나리오**: 신기술 도입 리스크
```
리스크: 신기술(블록체인) 미숙으로 프로젝트 지연
대응: 검증된 기존 기술(데이터베이스)로 회피
결과: 리스크 완전 제거, 안정적 진행
```

#### 4.1.2 전가 (Transfer)
**정의**: 리스크를 제3자에게 이전

**예시**:
- 보험 가입
- 아웃소싱
- 고정가 계약 (비용 리스크 전가)
- 성과 보증 계약

**시나리오**: 서버 장애 리스크
```
리스크: 자체 서버 운영 시 장애 발생 가능
대응: AWS 같은 클라우드 서비스 사용 (SLA 99.99%)
결과: 인프라 리스크를 클라우드 업체에 전가
비용: 월 ₩5,000,000 (보험료 개념)
```

#### 4.1.3 완화 (Mitigate)
**정의**: 리스크 발생 확률 또는 영향을 감소

**예시**:
- 백업 시스템 구축
- 프로토타입 개발
- 여분 인력 확보
- 정기 점검 강화

**시나리오**: 핵심 인력 이탈 리스크
```
리스크: 핵심 개발자 퇴사 시 프로젝트 중단
대응 전:
- 확률: 3 (중)
- 영향: 5 (극상)
- 점수: 15 (극상)

완화 조치:
1. 지식 문서화 (위키 작성)
2. 페어 프로그래밍 (지식 공유)
3. 백업 인력 교육
4. 보상 체계 개선

대응 후:
- 확률: 2 (하) - 퇴사 확률 감소
- 영향: 3 (중) - 퇴사해도 영향 감소
- 점수: 6 (중)
```

#### 4.1.4 수용 (Accept)
**정의**: 리스크를 인정하고 대응하지 않음

**적용 조건**:
- 리스크 점수가 낮음 (1-3)
- 대응 비용이 리스크 영향보다 큼
- 대응 방법이 없음

**2가지 유형**:
1. **능동적 수용**: 예비비/예비 시간 확보
2. **수동적 수용**: 아무 조치 없음

**시나리오**: 사무실 정전 리스크
```
리스크: 정전으로 인한 작업 중단 (연 1-2회, 1시간)
분석:
- 확률: 1 (극하)
- 영향: 2 (하)
- 점수: 2 (하)
대응: 수용 (UPS 구매보다 그냥 기다리는 게 경제적)
```

### 4.2 기회(Opportunity) 대응 전략

#### 4.2.1 활용 (Exploit)
**정의**: 기회를 확실하게 실현

**예시**:
- 우수 인재 확보
- 신기술 조기 도입
- 시장 선점

#### 4.2.2 향상 (Enhance)
**정의**: 기회 발생 확률 또는 영향 증대

**예시**:
- 마케팅 강화
- 품질 개선
- 고객 지원 확대

#### 4.2.3 공유 (Share)
**정의**: 기회를 제3자와 공유

**예시**:
- 조인트 벤처
- 파트너십
- 라이선싱

#### 4.2.4 수용 (Accept)
**정의**: 기회를 인지하되 적극적 조치 없음

## 📊 5. 리스크 정량적 분석

### 5.1 EMV (Expected Monetary Value) 분석

#### 공식
```
EMV = 확률 × 영향(금액)
```

#### 실습 예제 1: 단일 리스크

**리스크**: 서버 장애
- 발생 확률: 20%
- 발생 시 손실: ₩50,000,000

```
EMV = 0.2 × (-₩50,000,000) = -₩10,000,000
```

**해석**: 서버 장애 리스크의 기대 금전적 가치는 -₩10M

#### 실습 예제 2: 대응 방안 비교

**옵션 A: 예방 조치 (백업 서버 구축)**
- 비용: ₩15,000,000
- 장애 확률 감소: 20% → 2%
- EMV = 0.02 × (-₩50,000,000) - ₩15,000,000 = -₩16,000,000

**옵션 B: 아무 조치 없음**
- 비용: ₩0
- 장애 확률: 20%
- EMV = 0.2 × (-₩50,000,000) = -₩10,000,000

**결정**: 옵션 B 선택 (EMV가 덜 부정적)

### 5.2 의사결정 트리 (Decision Tree)

#### 구성 요소
```
□ 의사결정 노드 (Decision Node)
○ 확률 노드 (Chance Node)
─ 가지 (Branch)
```

#### 실습 예제: 신제품 출시 의사결정

**시나리오**:
- 제품 개발 비용: ₩100M
- 시장 조사 비용: ₩20M

```
                           성공 (60%)
              제품 출시 ─○─────────── 수익 ₩300M
             ↗          └─ 실패 (40%) 수익 ₩0
□ 의사결정                
             ↘          ┌─ 긍정 (70%)
              시장조사 ─○─────────── 제품 출시 → 성공 (80%) ₩300M
                        │            └─ 실패 (20%) ₩0
                        └─ 부정 (30%) 출시 중단 ₩0
```

**계산**:

**옵션 1: 즉시 출시**
```
EMV = 0.6 × ₩300M + 0.4 × ₩0 - ₩100M
    = ₩180M - ₩100M
    = ₩80M
```

**옵션 2: 시장조사 후 결정**
```
긍정적 결과 (70%):
  EMV = 0.8 × ₩300M + 0.2 × ₩0 - ₩100M
      = ₩240M - ₩100M
      = ₩140M

부정적 결과 (30%):
  EMV = ₩0 (출시 중단)

전체 EMV = 0.7 × ₩140M + 0.3 × ₩0 - ₩20M
         = ₩98M - ₩20M
         = ₩78M
```

**결정**: 옵션 1 (즉시 출시) 선택
- EMV ₩80M > ₩78M
- 시장조사 비용 대비 가치 낮음

**1. 품질 비용(CoQ) 중 "예방 비용"에 해당하는 것은?**
- A) 제품 회수 비용
- B) 재작업 비용
- C) 교육 및 훈련 비용
- D) 검사 비용

**정답**: C
**해설**: 예방 비용은 품질 문제를 사전에 방지하기 위한 비용으로, 교육, 훈련, 프로세스 개선 등이 포함됩니다.

---

**2. 파레토 차트의 기본 원리는?**
- A) 정규 분포
- B) 80/20 법칙
- C) 중심 극한 정리
- D) 확률 분포

**정답**: B
**해설**: 파레토 차트는 80/20 법칙을 기반으로, 소수의 원인이 대부분의 결과를 만든다는 원리를 활용합니다.

**3. 리스크 매트릭스에서 확률 4, 영향 5인 리스크의 점수는?**
- A) 9
- B) 15
- C) 20
- D) 25

**정답**: C
**해설**: 리스크 점수 = 확률 × 영향 = 4 × 5 = 20

---

**4. 다음 중 위협(Threat) 대응 전략이 아닌 것은?**
- A) 회피 (Avoid)
- B) 전가 (Transfer)
- C) 활용 (Exploit)
- D) 완화 (Mitigate)

**정답**: C
**해설**: 활용(Exploit)은 기회(Opportunity) 대응 전략입니다. 위협 대응 전략은 회피, 전가, 완화, 수용입니다.

- A) ₩10M의 이익 예상
- B) ₩10M의 손실 예상
- C) ₩10M의 비용 발생
- D) ₩10M의 투자 필요

**정답**: B
**해설**: EMV가 음수(-)이면 손실을 의미합니다. -₩10M은 평균적으로 ₩10M의 손실이 예상됨을 뜻합니다.

1. Man (인력): 사람의 기술, 경험, 교육 수준
2. Machine (장비): 사용하는 도구, 장비, 시스템

6. Environment (환경): 작업 환경, 외부 요인, 조건

**7. 품질 비용(CoQ)의 4가지 범주를 설명하고, 각각의 예시를 2가지씩 제시하시오.**

**적합 비용 (Cost of Conformance)**:

**부적합 비용 (Cost of Non-Conformance)**:

   - 예시: 재작업, 폐기 처분
4. 외부 실패 비용 (External Failure)

**모범 답안**:

| 구분 | 리스크 | 이슈 |
|-----|-------|------|
| 시점 | 미래 (발생 가능성) | 현재 (이미 발생) |
| 대응 | 사전 예방 | 사후 해결 |
| 확률 | 0-100% | 100% (확정) |
| 관리 | 리스크 관리 계획 | 이슈 해결 프로세스 |

**9. 완화(Mitigate) 전략과 전가(Transfer) 전략의 차이를 설명하고, 각각의 예시를 제시하시오.**

**모범 답안**:

**완화 (Mitigate)**:
- 정의: 리스크 확률 또는 영향을 감소시킴
- 예시: 백업 시스템 구축, 프로토타입 개발
- 결과: 리스크 점수 감소 (예: 20 → 6)

**전가 (Transfer)**:
- 정의: 리스크를 제3자에게 이전
- 예시: 보험 가입, 아웃소싱
- 결과: 리스크 소유권 이전 (비용 발생)

**핵심 차이**: 완화는 리스크를 줄이고, 전가는 리스크를 넘김

---

**10. 의사결정 트리를 활용하여 EMV를 계산하는 과정을 단계별로 설명하시오.**

**모범 답안**:

**단계 1**: 의사결정 대안 식별
- 예: 옵션 A, 옵션 B

**단계 2**: 각 대안의 가능한 결과 나열
- 예: 성공/실패

**단계 3**: 각 결과의 확률 설정
- 예: 성공 60%, 실패 40%

**단계 4**: 각 결과의 금전적 가치 계산
- 예: 성공 시 ₩300M, 실패 시 ₩0

**단계 5**: EMV 계산
- EMV = (확률1 × 가치1) + (확률2 × 가치2) - 비용
- 예: (0.6 × ₩300M) + (0.4 × ₩0) - ₩100M = ₩80M

**단계 6**: 각 대안의 EMV 비교 및 최적 대안 선택
