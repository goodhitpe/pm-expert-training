## 1️⃣ 서버(Server)란?

### 쉬운 설명
**서버**는 다른 컴퓨터(클라이언트)에게 서비스를 제공하는 컴퓨터입니다.
우리가 웹사이트를 볼 때, 그 내용을 보내주는 컴퓨터가 서버입니다.

### 실생활 비유
식당을 생각해보세요:
- **손님** = 클라이언트 (내 컴퓨터, 스마트폰)
- **웨이터** = 서버
- **주방** = 데이터베이스

손님이 "불고기를 주세요"라고 요청하면, 웨이터가 주방에서 요리를 가져와 제공합니다.
마찬가지로 브라우저가 "웹페이지를 주세요"라고 요청하면, 서버가 데이터를 가져와 전송합니다.

### PM이 알아야 하는 이유
- 서버는 24시간 켜져 있어야 하는 컴퓨터 (전기료, 관리 비용)
- 서버 성능이 서비스 속도와 동시 접속자 수에 영향
- 서버 장애 = 서비스 중단 (백업 서버 필요)
- 사용자가 많아지면 서버 추가 필요 (확장성 계획)

## 2️⃣ 온프레미스(On-Premise) vs 클라우드(Cloud)

### 온프레미스 (전통적 방식)
회사에서 직접 서버를 구매해서 사무실에 두고 관리하는 방식입니다.

**장점:**
- ✅ 완전한 통제 가능
- ✅ 보안에 민감한 데이터 직접 관리

**단점:**
- ❌ 초기 비용 높음 (서버 구매, 공간, 전기)
- ❌ 유지보수 필요 (고장 나면 직접 수리)
- ❌ 확장 어려움 (서버 추가 시 구매부터 설치까지 시간 소요)

### 클라우드 (현대적 방식)
인터넷으로 서버를 빌려 쓰는 방식입니다. AWS, Azure, Google Cloud 등이 제공합니다.

**장점:**
- ✅ 초기 비용 낮음 (사용한 만큼만 지불)
- ✅ 유지보수 불필요 (제공자가 관리)
- ✅ 쉬운 확장 (클릭 몇 번으로 서버 추가)
- ✅ 전 세계 어디서든 빠른 서비스 가능

**단점:**
- ❌ 장기 사용 시 비용 높을 수 있음
- ❌ 인터넷 의존

### 실생활 비유

**온프레미스 = 자가용**
- 내 차를 사서 주차장에 보관
- 초기 구매 비용 높음
- 보험, 수리, 주차비 등 관리 필요

**클라우드 = 카셰어링/택시**
- 필요할 때만 빌려서 사용
- 초기 비용 없음
- 유지보수 걱정 없음

### PM 관점
- 스타트업은 클라우드로 시작 (초기 비용 절감)
- 대기업은 하이브리드 (중요 데이터는 온프레미스, 나머지는 클라우드)
- 클라우드 비용 관리 중요 (사용하지 않는 서버 정리)

## 3️⃣ IaaS vs PaaS vs SaaS

클라우드 서비스는 제공 범위에 따라 3가지로 나뉩니다.

### IaaS (Infrastructure as a Service) - 인프라 제공

**제공 항목**: 가상 서버, 스토리지, 네트워크  
**사용자가 관리**: 운영체제, 애플리케이션, 데이터  
**예시**: AWS EC2, Azure VM, Google Compute Engine

**실생활 비유**: 빈 아파트 임대
- 건물과 기본 설비만 제공
- 가구, 인테리어는 직접 해야 함

**언제 사용?**
- 완전한 통제가 필요한 경우
- 특별한 개발 환경이 필요한 경우

### PaaS (Platform as a Service) - 플랫폼 제공

**제공 항목**: 운영체제, 개발 도구, 데이터베이스 환경  
**사용자가 관리**: 애플리케이션, 데이터만  
**예시**: AWS Elastic Beanstalk, Google App Engine, Heroku

**실생활 비유**: 가구 포함 아파트
- 기본 가구와 가전제품이 갖춰진 상태
- 짐만 풀면 바로 살 수 있음

**언제 사용?**
- 빠르게 개발하고 배포하고 싶을 때
- 인프라 관리에 신경 쓰고 싶지 않을 때

### SaaS (Software as a Service) - 소프트웨어 제공

**제공 항목**: 완성된 소프트웨어  
**사용자가 관리**: 설정과 데이터만  
**예시**: Gmail, Google Docs, Slack, Salesforce, Notion

**실생활 비유**: 호텔
- 모든 게 준비되어 있음
- 그냥 사용하기만 하면 됨

**언제 사용?**
- 이미 만들어진 서비스를 바로 쓰고 싶을 때
- 개발 없이 업무용 도구가 필요할 때

### 비교표

| 구분 | 관리 범위 | 유연성 | 예시 |
|------|----------|--------|------|
| **IaaS** | 많음 | 높음 | AWS EC2 |
| **PaaS** | 중간 | 중간 | Heroku |
| **SaaS** | 적음 | 낮음 | Gmail |

### PM이 알아야 하는 이유
- 프로젝트 특성에 따라 적절한 서비스 모델 선택
- IaaS는 개발 시간 길지만 유연함
- PaaS는 빠른 출시 가능
- SaaS는 즉시 사용, 커스터마이징 제한

## 4️⃣ 주요 클라우드 제공자

### AWS (Amazon Web Services)
- **시장 점유율**: 1위 (약 32%)
- **특징**: 가장 많은 서비스, 가장 많은 사용자
- **강점**: 서비스 다양성, 레퍼런스 풍부
- **대표 고객**: Netflix, Airbnb, 삼성전자

### Microsoft Azure
- **시장 점유율**: 2위 (약 23%)
- **특징**: Windows, Office와 통합 용이
- **강점**: 엔터프라이즈, 하이브리드 클라우드
- **대표 고객**: BMW, 삼성SDS, LG

### Google Cloud Platform (GCP)
- **시장 점유율**: 3위 (약 10%)
- **특징**: 구글의 인프라 활용
- **강점**: 빅데이터 분석, 인공지능/머신러닝
- **대표 고객**: Spotify, Twitter, Snapchat

### 실생활 비유
- **AWS** = 이마트 (가장 크고, 모든 게 있음)
- **Azure** = 현대백화점 (기업 고객 특화)
- **GCP** = 코스트코 (데이터 분석용품 강세)

### PM 관점
- AWS가 가장 레퍼런스 많아 채용 쉬움
- Azure는 기존에 Microsoft 제품 쓰면 유리
- GCP는 데이터 분석/AI 프로젝트에 강점
- 멀티 클라우드 전략도 가능 (여러 제공자 동시 사용)

## 5️⃣ 가상 머신(Virtual Machine)

### 정의
하나의 물리적 서버를 여러 개의 가상 서버로 나누는 기술입니다.

### 실생활 비유
큰 사무실을 칸막이로 나누는 것과 같습니다:
- **물리 서버** = 전체 사무실 건물
- **가상 머신** = 칸막이로 나눈 각각의 공간
- 각 공간은 독립적으로 사용 가능

### 장점
- 하나의 서버로 여러 서비스 운영 가능
- 비용 절감
- 필요할 때 쉽게 생성/삭제

### 예시
AWS EC2 인스턴스를 생성하면:
- 몇 분 만에 가상 서버 하나 생김
- Linux나 Windows 설치된 상태로 제공
- 필요 없으면 종료하면 비용 청구 중단

### PM이 알아야 하는 이유
- "EC2 인스턴스 2개 추가 필요"라는 개발자 요청의 의미 이해
- 가상 머신 = 서버 = 비용 발생
- 테스트 서버는 사용 안 할 때 종료해서 비용 절감

---

## 6️⃣ 스토리지(Storage)

### 정의
클라우드에서 파일을 저장하는 공간입니다.

### 종류

**객체 스토리지 (Object Storage)**
- **예시**: AWS S3, Azure Blob Storage
- **용도**: 이미지, 동영상, 백업 파일
- **특징**: 용량 무제한, 저렴, 빠른 접근

**블록 스토리지 (Block Storage)**
- **예시**: AWS EBS, Azure Disk
- **용도**: 데이터베이스, 서버 디스크
- **특징**: 빠른 속도, 서버에 직접 연결

### 실생활 비유
- **객체 스토리지** = 창고 (대용량 물건 보관)
- **블록 스토리지** = 책상 서랍 (자주 쓰는 것 보관)

### PM 관점
- 사용자 업로드 이미지/동영상 → S3 같은 객체 스토리지
- 스토리지 비용은 사용량에 비례 (콘텐츠 많으면 비용 상승)
- 데이터 삭제 정책 필요 (오래된 파일 자동 삭제)

---

## 7️⃣ 확장성(Scalability)

### 정의
사용자가 늘어나도 서비스가 원활히 동작하도록 자원을 늘리는 것입니다.

### 수직 확장 (Scale-Up)
서버 한 대의 성능을 올리는 것
- CPU 2코어 → 4코어
- 메모리 4GB → 8GB

**실생활 비유**: 작은 식당을 큰 식당으로 이전

**장점**: 간단함  
**단점**: 한계가 있음, 비용 급격히 상승

### 수평 확장 (Scale-Out)
서버 대수를 늘리는 것
- 서버 1대 → 3대

**실생활 비유**: 식당 지점 늘리기

**장점**: 무한 확장 가능  
**단점**: 관리 복잡, 부하 분산 필요

### Auto Scaling (자동 확장)
트래픽에 따라 자동으로 서버를 늘리거나 줄이는 기능

**시나리오**:
- 평소: 서버 2대로 운영
- 점심시간: 사용자 급증 → 자동으로 5대로 증가
- 새벽: 사용자 감소 → 자동으로 1대로 감소

### PM이 알아야 하는 이유
- 출시 초기에는 작은 서버로 시작 → 비용 절감
- 사용자 급증에 대비한 확장 계획 필요
- Auto Scaling 설정하면 새벽에 비용 절감
- 이벤트/마케팅 전에 서버 확장 계획

## 8️⃣ 로드 밸런서(Load Balancer)

### 정의
여러 서버로 트래픽을 골고루 분산시키는 장비입니다.

### 실생활 비유
대형마트의 계산대 안내 직원:
- 손님(요청)이 오면
- 가장 한가한 계산대(서버)로 안내
- 모든 계산대의 대기 줄이 비슷해짐

### 작동 방식
```
사용자들
  ↓
로드 밸런서
  ├─→ 서버1 (30% 트래픽)
  ├─→ 서버2 (40% 트래픽)
  └─→ 서버3 (30% 트래픽)
```

### 장점
- 서버 한 대가 고장 나도 다른 서버로 전환 (고가용성)
- 트래픽 분산으로 성능 향상
- 점진적 배포 가능 (일부 서버만 업데이트)

### PM이 알아야 하는 이유
- 서비스 안정성의 핵심 요소
- "로드 밸런서 추가 필요"는 트래픽 증가 대응책
- 로드 밸런서도 비용 발생 (서비스 안정성 vs 비용)

## 9️⃣ 리전(Region)과 가용 영역(Availability Zone)

### 리전 (Region)
클라우드 제공자가 서버를 두는 지리적 위치

**예시**:
- 서울 리전
- 도쿄 리전
- 미국 버지니아 리전

### 가용 영역 (Availability Zone, AZ)
하나의 리전 안에 있는 독립적인 데이터센터

**예시**: 서울 리전 안에
- AZ-A (강남 데이터센터)
- AZ-B (판교 데이터센터)
- AZ-C (가산 데이터센터)

### 실생활 비유
- **리전** = 서울시
- **가용 영역** = 강남구, 서초구, 송파구

### 왜 중요한가?
- 가까운 리전 선택 = 빠른 속도
- 여러 가용 영역 사용 = 한 곳 장애 시 다른 곳에서 서비스 계속

### 고가용성 구성 예시
```
서울 리전
  ├─ AZ-A: 서버1, 데이터베이스1
  ├─ AZ-B: 서버2, 데이터베이스2
  └─ AZ-C: 백업
```
AZ-A가 화재로 중단되어도, AZ-B에서 서비스 계속 가능!

### PM 관점
- 한국 사용자 대상 → 서울 리전 선택
- 글로벌 서비스 → 여러 리전에 배포
- 중요한 서비스는 다중 AZ 구성 (비용은 높지만 안정적)

## 🔟 클라우드 비용 관리

### 과금 방식

**종량제 (Pay-as-you-go)**
- 사용한 만큼만 지불
- 시간 단위 (서버를 1시간 켜면 1시간 비용)
- 스토리지는 용량 단위

**예약 인스턴스**
- 1년 또는 3년 장기 약정
- 최대 70% 할인
- 장기 프로젝트에 적합

### 비용 최적화 팁

**1. 사용하지 않는 리소스 제거**
- 테스트 서버는 퇴근 시 종료
- 오래된 백업 삭제
- 사용하지 않는 서버 정리

**2. 적절한 인스턴스 크기**
- 과도하게 큰 서버 선택 금물
- 모니터링 후 적정 크기 선택

**3. Auto Scaling 활용**
- 새벽 시간에는 서버 최소화
- 피크 시간에만 증가

### 실생활 비유
- 에어컨을 24시간 켜두면 전기세 폭탄
- 필요할 때만 켜고, 외출 시 끄기
- 클라우드도 마찬가지!

### PM이 알아야 하는 이유
- 클라우드 비용은 "변동 비용" (관리 안 하면 급증)
- 월별 비용 리포트 확인 필요
- 예산 알림 설정 (예: 100만원 초과 시 알림)
- 개발팀과 비용 최적화 정기 회의

### 자주 묻는 질문 (FAQ)

**Q1: 클라우드는 안전한가요?**  
A: 네! 대형 클라우드 제공자는 전문 보안팀이 24/7 관리합니다. 오히려 소규모 회사의 온프레미스보다 안전한 경우가 많습니다.

**Q2: 클라우드 비용이 비싸다던데요?**  
A: 잘 관리하면 온프레미스보다 저렴합니다. 초기 투자 없이 시작할 수 있고, 사용한 만큼만 내면 됩니다. 다만 방치하면 비용이 급증할 수 있어 관리가 중요합니다.

**Q3: AWS, Azure, GCP 중 뭘 선택해야 하나요?**  
A: 대부분의 경우 AWS를 선택합니다 (레퍼런스와 인력 풀이 가장 많음). Microsoft 제품을 많이 쓴다면 Azure, 데이터 분석 중심이면 GCP도 좋습니다.

**Q4: 클라우드를 몰라도 PM을 할 수 있나요?**  
A: 할 수는 있지만, 현대의 대부분 서비스가 클라우드 기반이라 알아두면 훨씬 유리합니다. 기본 개념만 알아도 개발팀과 소통이 원활해집니다.
